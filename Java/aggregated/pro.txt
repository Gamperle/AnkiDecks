# Java Pro Questions
# Format: Front[TAB]Back

When should you use `String.intern()` and what are the caveats?	`String.intern()` returns the pool's canonical copy of the string (adds it if not present). Benefits: memory savings when many strings share the same value (e.g., repeated keys or identifiers). Caveats: `intern()` is relatively expensive (hashing, synchronization) ‚Äî only intern strings that will have many duplicates with long lifetimes. `new String("abc")` creates a new object outside the pool; string literals are automatically interned. In Java 8+, the String Pool resides in the heap and is GC'd like normal objects (unlike PermGen in Java 7-). For large-scale deduplication, consider `String.intern()` or, from Java 8, `StringTable` GC options like `-XX:+UseStringDeduplication` (G1GC only).
What is the difference between static binding and dynamic binding in Java?	Static binding (early binding) ‚Äî method to call is determined at compile time. Applies to: `private`, `static`, and `final` methods (not overridable). Also applies to overloaded methods (resolved by declared parameter types). Dynamic binding (late binding) ‚Äî method determined at runtime via vtable lookup. Applies to every instance method that can be overridden. The JVM dispatches on the actual runtime type, not the declared reference type ‚Äî this is what makes polymorphism work. Performance: dynamic dispatch adds a vtable lookup, but HotSpot inlines monomorphic/bimorphic call sites aggressively.
What is a covariant return type and what are bridge methods?	A covariant return type means an overriding method may return a subtype of the declared return type. Example: `Object.clone()` returns `Object`; a subclass can override with `protected MyClass clone()`. The compiler generates a synthetic **bridge method** with the original return type that calls the narrowed-return-type version ‚Äî this preserves binary compatibility and supports generics erasure. Bridge methods are also generated for generic type parameters: `Comparable<String>.compareTo(Object)` bridges to `compareTo(String)`. Visible via `javap -verbose` as `ACC_BRIDGE ACC_SYNTHETIC`.
What is the difference between method overloading and overriding?	Overloading: same method name, different parameter types/count in the same class (or subclass). Resolved at **compile time** by the declared static type of the arguments. Overriding: same signature in a subclass, providing specialized behaviour. Resolved at **runtime** by the actual object type (dynamic dispatch). Key pitfalls: (1) Overloading is NOT polymorphic ‚Äî calling `method(Object)` vs `method(String)` depends on the declared type of the variable. (2) Adding an overload can silently break callers if auto-promotion selects a different overload. (3) `@Override` annotation should always be used to confirm true override intent.
What are covariance, contravariance, and invariance in Java generics vs arrays?	Arrays are **covariant** ‚Äî `String[]` is-a `Object[]`. This is unsound: you can write `Object[] arr = new String[1]; arr[0] = 42;` which compiles but throws `ArrayStoreException` at runtime. Generics are **invariant** ‚Äî `List<String>` is NOT a `List<Object>`. This is type-safe but requires wildcards for flexibility. Use-site variance: `List<? extends T>` (covariant/producer) ‚Äî read-only; `List<? super T>` (contravariant/consumer) ‚Äî write-only. PECS: Producer Extends, Consumer Super. Declaration-site variance does not exist in Java (cf. Kotlin `out`/`in`).
What is a static initializer block and what is `ExceptionInInitializerError`?	A static initializer block `static { ... }` runs once when the class is first loaded by the JVM, before any constructor or static method. Execution order: static fields and blocks in textual order. Use for complex static initialization that can't fit in a field initializer (e.g., populating an immutable map). If the static initializer throws any exception, the JVM wraps it in `ExceptionInInitializerError`. Subsequent attempts to use the class throw `NoClassDefFoundError` (the class is flagged as failed). This is a common source of confusing startup failures ‚Äî look for `ExceptionInInitializerError` in the root cause chain.
What is the difference between `ReentrantLock` and `synchronized`?	Both provide mutual exclusion and are reentrant (same thread can re-acquire). `ReentrantLock` extras: `tryLock()` (non-blocking attempt), `tryLock(timeout, unit)` (timed), `lockInterruptibly()` (can be interrupted while waiting), fairness constructor `new ReentrantLock(true)` (longest-waiting thread wins), multiple `Condition` variables via `lock.newCondition()`. `synchronized` is simpler, automatically released on exit/exception. `ReentrantLock` must be unlocked in a `finally` block. Use `synchronized` for simple cases; `ReentrantLock` when you need `tryLock`, fairness, or multiple conditions.
What is `StampedLock` and what is optimistic reading?	`StampedLock` (Java 8+) offers three modes via a stamp (long token): (1) Write lock ‚Äî exclusive: `long stamp = lock.writeLock()`. (2) Read lock ‚Äî shared: `long stamp = lock.readLock()`. (3) Optimistic read ‚Äî non-blocking, no actual lock: `long stamp = lock.tryOptimisticRead()` then validate with `lock.validate(stamp)` afterward; if invalidated, fall back to a real read lock. Higher throughput than `ReentrantReadWriteLock` for read-heavy workloads. Caution: NOT reentrant; does not support `Condition` variables.
What is `CyclicBarrier` and how does it differ from `CountDownLatch`?	`CyclicBarrier` makes a fixed number of threads all wait for each other at a barrier point before proceeding together. It is **reusable** ‚Äî resets after each cycle. An optional `Runnable` barrier action runs when the barrier trips. Ideal for iterative parallel algorithms (e.g., simulation phases). `CountDownLatch`: one or more threads wait for N other threads to complete a task. Count is one-way to zero. **Not reusable**. Threads calling `countDown()` do not wait themselves.
What is `Phaser` and when is it more flexible than `CyclicBarrier`?	`Phaser` (Java 7+) is the most flexible barrier: (1) Dynamic parties ‚Äî register/deregister with `register()` / `arriveAndDeregister()` at any time. (2) Multi-phase ‚Äî automatically advances to next phase when all parties arrive; reusable like `CyclicBarrier`. (3) Tiered phasers ‚Äî parent-child hierarchy reduces contention at large party counts. (4) Override `onAdvance(phase, parties)` for per-phase logic or early termination. Supports both blocking (`arriveAndAwaitAdvance()`) and non-blocking (`arrive()`) participation.
What are `BlockingQueue` implementations and when to use each?	`ArrayBlockingQueue(capacity)` ‚Äî bounded, array-backed, FIFO, optional fairness. `LinkedBlockingQueue(optional capacity)` ‚Äî optionally bounded, linked nodes, higher throughput than `ArrayBlockingQueue`. `SynchronousQueue` ‚Äî zero capacity: each `put()` must wait for a matching `take()` (rendezvous); used in cached thread pools. `PriorityBlockingQueue` ‚Äî unbounded, priority-ordered. `DelayQueue` ‚Äî elements become available only after a per-element delay. `LinkedTransferQueue` ‚Äî unbounded, supports `transfer()` (blocks until a consumer takes the element). Use in producer-consumer patterns as the handoff mechanism.
What are the `CompletableFuture` additions in Java 9+?	`orTimeout(duration, unit)` ‚Äî completes exceptionally with `TimeoutException` if not done in time. `completeOnTimeout(value, duration, unit)` ‚Äî completes with a default value on timeout instead of an exception. `completeAsync(Supplier, executor)` ‚Äî asynchronously completes using a supplier. `failedFuture(Throwable)` ‚Äî creates an already-failed future. `newIncompleteFuture()` ‚Äî creates a new incomplete future of the same type (useful for subclassing). Combined with the Java 8 combinators (`thenApply`, `thenCompose`, `allOf`, etc.), these cover most async timeout and fallback patterns without external libraries.
What is the Fork/Join framework and when should you use it?	`ForkJoinPool` (Java 7+) is designed for divide-and-conquer parallel algorithms. Tasks extend `RecursiveTask<V>` (returns a value) or `RecursiveAction` (no return). Tasks `fork()` subtasks asynchronously and `join()` their results. Uses **work-stealing**: idle threads steal tasks from the tails of other threads' deques, maximizing CPU use. `ForkJoinPool.commonPool()` is used by parallel streams and `CompletableFuture` by default. Use when: tasks naturally split into independent subtasks, work is CPU-bound, and subtasks are roughly equal in cost. Avoid for I/O-bound work (use virtual threads instead).
What is `LockSupport` and how does it relate to `AQS`?	`LockSupport.park()` suspends the current thread (like `Object.wait()` but without requiring a lock). `LockSupport.unpark(thread)` permits the given thread to resume. Key differences from `synchronized`/`Object.wait()`: (1) No monitor needed ‚Äî can park/unpark without holding any lock. (2) Unpark can be called before park ‚Äî the permit is stored, so the subsequent `park()` returns immediately (no spurious block). (3) Non-reentrant permit ‚Äî only one stored permit. `AbstractQueuedSynchronizer` (AQS) uses `LockSupport` internally as the primitive to block/unblock threads in its wait queue. `ReentrantLock`, `Semaphore`, `CountDownLatch`, and `CyclicBarrier` are all built on AQS.
What is `submit()` vs `execute()` in `ExecutorService`?	`execute(Runnable)` is defined on `Executor`; it runs the task but discards any result or exception ‚Äî if the task throws, the exception is swallowed by the uncaught exception handler of that thread. `submit(Runnable)` and `submit(Callable<V>)` return a `Future<V>`; exceptions are captured inside the `Future` and thrown when you call `future.get()`. Always prefer `submit()` over `execute()` so you can detect and handle exceptions. Common pitfall: calling `executor.submit(task)` and ignoring the returned `Future` ‚Äî exceptions are silently dropped.
What is `scheduleAtFixedRate` vs `scheduleWithFixedDelay`?	Both are on `ScheduledExecutorService`. `scheduleAtFixedRate(command, initialDelay, period, unit)` ‚Äî fires every `period` regardless of task duration. If the task takes longer than `period`, the next execution starts immediately after it finishes (no overlap, but rate may slow). `scheduleWithFixedDelay(command, initialDelay, delay, unit)` ‚Äî waits `delay` after each task completes before scheduling the next. Use `AtFixedRate` for heartbeats / metrics collection where wall-clock frequency matters. Use `WithFixedDelay` when you need a guaranteed rest between executions (e.g., polling with cooldown).
What is `invokeAll()` vs `invokeAny()` in `ExecutorService`?	`invokeAll(Collection<Callable<T>>)` submits all tasks and blocks until ALL complete (or timeout). Returns a `List<Future<T>>` ‚Äî some may have exceptions. `invokeAny(Collection<Callable<T>>)` submits all tasks and returns the result of the FIRST one to succeed; cancels remaining tasks. Use `invokeAll` for parallel batch processing where you need every result. Use `invokeAny` for redundant requests where the first response wins (e.g., querying multiple servers, speculative execution). Both accept an optional timeout overload.
What is a `Condition` and how do you use `await()`, `signal()`, `signalAll()`?	A `Condition` is the `java.util.concurrent.locks` equivalent of `Object.wait/notify`, but tied to a `Lock` rather than a monitor. Obtain one from a `Lock`: `Condition notFull = lock.newCondition()`. A single `ReentrantLock` can have multiple Conditions (e.g., `notFull` and `notEmpty` for a bounded buffer), whereas `Object` only has one wait-set per monitor. `await()` ‚Äî atomically releases the lock and suspends the thread until signalled or interrupted; re-acquires the lock before returning. `signal()` ‚Äî wakes one waiting thread on this Condition. `signalAll()` ‚Äî wakes all waiting threads. Always call `await()`/`signal()` while holding the associated lock; always re-check the condition in a `while` loop to guard against spurious wake-ups.
What are the `Executors` factory types and when should you use each?	`Executors.newFixedThreadPool(n)` ‚Äî `n` threads, unbounded `LinkedBlockingQueue`; good for CPU-bound tasks with known parallelism. `newCachedThreadPool()` ‚Äî creates new threads on demand, reuses idle ones; unbounded ‚Äî can exhaust resources under load; good for many short-lived tasks. `newSingleThreadExecutor()` ‚Äî single thread, tasks serialised; useful for ordered background processing. `newScheduledThreadPool(n)` ‚Äî supports `schedule()`, `scheduleAtFixedRate()`, `scheduleWithFixedDelay()`. `newWorkStealingPool()` (Java 8+) ‚Äî backed by `ForkJoinPool` with parallelism = available processors; work-stealing; good for parallel divide-and-conquer. `newVirtualThreadPerTaskExecutor()` (Java 21+) ‚Äî creates one virtual thread per task; ideal for high-concurrency I/O-bound workloads. Prefer explicit `ThreadPoolExecutor` construction for production code to control queue size and rejection policy.	Reactive programming models data flows as asynchronous streams of events. Producers push data; operators transform/filter; consumers process asynchronously. **Back-pressure** is the mechanism by which a slow consumer signals a fast producer to slow down ‚Äî preventing unbounded buffering and OOM. In Java: `java.util.concurrent.Flow` (Reactive Streams spec): `Subscriber.onSubscribe()` receives a `Subscription`; the subscriber calls `subscription.request(n)` to signal it can handle `n` more items. The publisher must not emit more than requested. Libraries: Project Reactor (`Flux`, `Mono`), RxJava (`Observable`, `Flowable` ‚Äî Flowable has back-pressure, Observable does not).
What is CAS (Compare-And-Swap) and what is the ABA problem?	CAS is a hardware-level atomic instruction: given a memory address, an expected value, and a new value ‚Äî it atomically updates the memory only if it currently contains the expected value, returning success/failure. The foundation of lock-free programming. All `java.util.concurrent.atomic` classes use CAS via `VarHandle`. ABA problem: CAS can incorrectly succeed if a value changes A‚ÜíB‚ÜíA; solved with `AtomicStampedReference`.
What is the difference between `thenApply()` and `thenCompose()` in `CompletableFuture`?	`thenApply(Function<T,U>)` transforms the result synchronously ‚Äî analogous to `map()`. If the function itself returns a `CompletableFuture<U>`, you get nested `CompletableFuture<CompletableFuture<U>>`. `thenCompose(Function<T, CompletableFuture<U>>)` chains a dependent async computation ‚Äî analogous to `flatMap()`. Flattens to `CompletableFuture<U>`. Use `thenCompose` when the next step is itself async (e.g., another HTTP call), to avoid nesting.
What is double-checked locking (DCL) and when is `volatile` essential?	DCL is a lazy initialization pattern that avoids synchronization overhead after initialization: check the field without a lock; if null, lock and check again, then initialize. Without `volatile`, the JVM may reorder the write to the object's fields with the assignment of the reference ‚Äî another thread sees a non-null but partially constructed object. `volatile` on the field inserts a happens-before edge ensuring all writes to the object are visible before the reference is published. Prefer the initialization-on-demand holder (Bill Pugh) pattern or enum singleton as simpler alternatives.
What is type erasure in Java and what are its runtime implications?	At compile time, generics provide type-checking. At runtime, all generic type parameters are erased and replaced with their bounds (usually `Object`). `List<String>` becomes `List` at bytecode level; `List<Integer>` also becomes `List`. Implications: (1) Cannot use generics in `instanceof` checks: `x instanceof List<String>` is illegal (use `instanceof List<?>`). (2) Cannot create generic arrays: `new T[]` is illegal. (3) Cannot catch a generic type: `catch (MyException<T> e)` is illegal. (4) Static fields cannot be of type T. (5) Overloads only differing by type parameter won't compile (same erasure). Type information is retained in class metadata (via `getGenericSuperclass()`, reflection) but not in object instances themselves.
What are bridge methods and why does the compiler generate them?	When a class implements or extends a generic type, the compiler generates synthetic "bridge methods" to maintain binary compatibility with the erased base type. Example: `class IntBox implements Comparable<Integer>`. The interface requires `compareTo(Object)` (erased) but the class provides `compareTo(Integer)`. The compiler generates a bridge `compareTo(Object e) { return compareTo((Integer) e); }` that casts and delegates. This allows code compiled against the raw interface to call the typed implementation. Bridge methods are an implementation detail; you'll see them in `getDeclaredMethods()` with `isBridge() == true`.
What is PECS and when should you use wildcards?	PECS = Producer Extends, Consumer Super. It guides which wildcard to use when designing APIs: (1) If a collection is a **Producer** (you read values out of it) ‚Üí use `? extends T`. The collection produces T values; you can safely read T references. Example: `void copy(List<? extends Number> src, ...)`. (2) If a collection is a **Consumer** (you write values into it) ‚Üí use `? super T`. The collection can accept T values and any supertype's consumers work. Example: `void addNumbers(List<? super Integer> dest)`. (3) If you both read and write ‚Üí use exact type `T` (no wildcard). Rule: you **cannot add** to a `? extends T` list (except null), and you can only read as `Object` from a `? super T` list.
What is a recursive (F-bounded) type bound and where is it used?	A recursive type bound constrains a type parameter using itself: `<T extends Comparable<T>>`. This says "T must be comparable to itself." The Java `Comparable` interface uses this: `interface Comparable<T>` combined with `T extends Comparable<T>` ensures type-safe comparison. Used in: (1) Builder patterns: `class Builder<B extends Builder<B>>` allows `this` references to return the concrete subtype. (2) Enum: `enum` inhericitly uses `Enum<E extends Enum<E>>`. (3) Fluent APIs where each method returns `this` typed to the concrete class. Common in APIs like Google Guava, Lombok `@SuperBuilder`.
What is heap pollution and what causes it?	Heap pollution occurs when a variable of a parameterized type refers to an object that is not of that type ‚Äî violating the type safety guarantee. It typically happens via: (1) Raw types: `List rawList = new ArrayList<>(); rawList.add("hello"); List<Integer> nums = rawList; Integer x = nums.get(0); // ClassCastException`. (2) Unchecked casts: `(List<Integer>) someObject`. (3) Varargs with generics: `void foo(List<T>... lists)` ‚Äî the varargs creates an array of the generic type, which is inherently unsafe. The `@SafeVarargs` annotation suppresses the unchecked warning when you've manually verified safety. At runtime, heap pollution ‚Üí `ClassCastException` at an unexpected location.
How can you recover generic type information at runtime using reflection?	You can access retained generic type info via `java.lang.reflect.Type` and its subtypes. Methods: (1) `getClass().getGenericSuperclass()` returns a `ParameterizedType` if the superclass is generic: `((ParameterizedType) getClass().getGenericSuperclass()).getActualTypeArguments()[0]` gives the type argument. Pattern used by Gson's `TypeToken<List<String>> {}` (anonymous subclass) and Jackson's `TypeReference`. (2) `Method.getGenericParameterTypes()` and `getGenericReturnType()` ‚Äî return type arguments declared in source. (3) `Field.getGenericType()`. Note: This works because class file metadata retains type signatures even after erasure. It does NOT work on instances ‚Äî only on class structure.
What are the differences between `List<?>`, `List<Object>`, and raw `List`?	They are all different: (1) `List<?>` (unbounded wildcard) ‚Äî a list of some unknown type. You can read as `Object`, but you cannot add anything except null (unsafe). Useful for code that doesn't care about the type: `printAll(List<?> list)`. (2) `List<Object>` ‚Äî a list that holds any `Object`. You can add `String`, `Integer`, etc. But `List<String>` is NOT a `List<Object>` (generics are invariant). A method `void foo(List<Object> l)` won't accept a `List<String>`. (3) Raw `List` ‚Äî no type parameter at all. Bypasses all generic checks; heap pollution risk; generates unchecked warnings. Only for interop with pre-Java 5 APIs. Summary: prefer `<?>` for read-only unknowns, `<T>` with bounds for flexibility, avoid raw types.
What is a `FileChannel` and when would you use it over `Files`?	`FileChannel` is an NIO channel for reading, writing, mapping, and locking files. Advantages over `Files`: (1) Memory-mapped files ‚Äî `map(MapMode, position, size)` maps a file region directly into memory for very fast bulk reads/writes. (2) File locking ‚Äî `lock()` / `tryLock()` for inter-process coordination. (3) Positioned reads/writes ‚Äî read/write at specific positions without seeking. (4) Efficient transfers between channels ‚Äî `transferTo()` / `transferFrom()`. Open with `FileChannel.open(path, StandardOpenOption.READ)`. Essential for high-performance or low-level file work.
What is the difference between blocking and non-blocking I/O in Java NIO?	Traditional I/O (java.io) is **blocking** ‚Äî `read()` and `write()` block the calling thread until data is available or written. NIO (java.nio) supports **non-blocking** mode: `channel.configureBlocking(false)`. A non-blocking `read()` returns 0 immediately if no data is ready, without blocking. A `Selector` multiplexes many non-blocking channels onto a single thread: register channels with `channel.register(selector, SelectionKey.OP_READ)`; call `selector.select()` to block until at least one channel is ready; iterate `selector.selectedKeys()` to handle ready channels. This enables high-concurrency servers with few threads. Java 21+ virtual threads largely replace NIO selector-based servers by making blocking I/O cheap.
What are memory-mapped files and when should you use them?	A memory-mapped file maps a file (or region) directly into the process address space via `FileChannel.map(MapMode.READ_ONLY, 0, size)`, returning a `MappedByteBuffer`. Reads/writes to the buffer become reads/writes to the file's pages in the OS page cache. Benefits: (1) Zero-copy ‚Äî no explicit `read()`/`write()` calls; OS demand-pages file content. (2) Shared memory between processes ‚Äî multiple processes map the same file. (3) Very fast for large sequential or random-access reads. Pitfalls: (1) The mapping occupies virtual address space (32-bit process limit). (2) `MappedByteBuffer` cannot be unmapped explicitly in Java ‚Äî reference must go out of scope + GC. (3) File changes from another process are immediately visible. Use for large read-only datasets, log scanning, inter-process communication.
What is the difference between `Serializable` and `Externalizable`?	`Serializable` is a marker interface ‚Äî Java handles all serialization automatically using reflection to read/write fields. Add `serialVersionUID` for version control; use `transient` to exclude fields. `Externalizable` extends `Serializable` and requires implementing `writeExternal(ObjectOutput)` and `readExternal(ObjectInput)` ‚Äî you control exactly what is serialized/deserialized. Externalizable is faster (no reflection) and more compact, but more code. It also requires a public no-arg constructor (called during deserialization before `readExternal`). Modern alternatives (Jackson, Protobuf) are preferred over both for new code ‚Äî Java serialization has security (deserialization gadget chains) and fragility concerns.
What is Java reflection and what can you do with it?	Reflection (`java.lang.reflect`) lets you inspect and manipulate classes, fields, methods, and constructors at runtime ‚Äî even private ones (with `setAccessible(true)`). Key classes: `Class<?>`, `Method`, `Field`, `Constructor`. Capabilities: instantiate classes by name (`Class.forName("...")`), invoke methods dynamically (`Method.invoke(target, args)`), read/write field values, inspect annotations, discover parameter types. Used heavily by Spring (DI, AOP), Hibernate (ORM mapping), JUnit (test discovery), Jackson (JSON serialization). Java 9+ module system restricts reflective access to non-exported packages ‚Äî `--add-opens` or `opens` in `module-info.java` may be needed.
What is the performance cost of reflection and how do you mitigate it?	Reflection is 10‚Äì100√ó slower than direct calls due to: (1) Security checks on every `invoke()` call (reduced after `setAccessible(true)`). (2) No JIT inlining ‚Äî reflective call sites are megamorphic. (3) Boxing/unboxing of primitive arguments and return values. (4) Dynamic method lookup overhead. Mitigations: (1) Cache `Method`, `Field`, `Constructor` objects ‚Äî lookup is expensive, re-invocation is cheaper. (2) Use `MethodHandle` instead ‚Äî JIT-friendly, can reach near-direct-call speed after warmup. (3) Use code generation at compile time (Lombok, MapStruct, annotation processors) to avoid reflection entirely at runtime. (4) Use ByteBuddy to generate adapter classes at startup rather than reflecting on every call.
How do you use `Class<?>` to inspect a class at runtime?	`Class<?>` is the entry point for reflection. Obtain it via: `MyClass.class` (literal), `obj.getClass()` (instance), `Class.forName("com.example.MyClass")` (by name). Key methods: `getDeclaredFields()` ‚Äî all fields declared in this class (incl. private). `getFields()` ‚Äî public fields incl. inherited. `getDeclaredMethods()` ‚Äî all methods declared here. `getMethods()` ‚Äî public methods incl. inherited. `getDeclaredConstructors()`, `getAnnotation(Class)`, `isAssignableFrom(Class)`, `getGenericSuperclass()`. Note: `getDeclared*()` vs `get*()` ‚Äî declared includes private but not inherited; get* includes inherited but only public. Call `setAccessible(true)` on a `Field` or `Method` to bypass access controls.
What are Java annotations and how are they used?	Annotations are metadata markers attached to code elements (classes, methods, fields, parameters, packages). They don't affect runtime behavior directly but are processed by: (1) Compiler: `@Override`, `@SuppressWarnings`, `@FunctionalInterface`, `@Deprecated`. (2) Annotation processors (APT) at compile time: Lombok, MapStruct, Dagger generate code. (3) Frameworks at runtime via reflection: Spring (`@Component`, `@Autowired`), JUnit (`@Test`), Jackson (`@JsonProperty`). Custom annotation syntax: `@interface MyAnnotation { String value() default ""; int priority() default 0; }`. Can be `@Repeatable` for multiple uses on the same element.
What are the five meta-annotations and what does each control?	Meta-annotations annotate annotation type declarations: (1) `@Retention(RetentionPolicy)` ‚Äî how long the annotation is retained: `SOURCE` (discarded after compilation, e.g., `@Override`), `CLASS` (in .class file but not at runtime, the default), `RUNTIME` (accessible via reflection at runtime, required for Spring/JUnit annotations). (2) `@Target(ElementType...)` ‚Äî which elements may be annotated: `TYPE`, `METHOD`, `FIELD`, `PARAMETER`, `CONSTRUCTOR`, `LOCAL_VARIABLE`, `ANNOTATION_TYPE`, `PACKAGE`, `TYPE_USE`. (3) `@Documented` ‚Äî include annotation in Javadoc. (4) `@Inherited` ‚Äî annotation on a superclass is inherited by subclasses (class-level only). (5) `@Repeatable(Container.class)` ‚Äî the annotation can be applied multiple times to the same element; requires a container annotation type.
What is compile-time annotation processing and how does it differ from runtime reflection?	Annotation processing runs custom `AbstractProcessor` subclasses during `javac` compilation. Processors use `javax.lang.model` API to inspect annotated source elements and generate new `.java` files or resources. Multiple rounds: generated files can trigger further processing. The processor is run by the compiler ‚Äî no JVM overhead at runtime. Examples: Lombok (generates getters/setters/builders at compile time ‚Äî no reflection at runtime), MapStruct (generates mapper implementations), Dagger (generates DI graph), AutoValue (generates value classes). Runtime reflection reads annotations at runtime via `element.getAnnotation(MyAnnotation.class)` ‚Äî slower and requires `@Retention(RUNTIME)`. Annotation processing produces zero-overhead abstractions; runtime reflection has per-call cost.
What is `Method.invoke()` and what are its pitfalls?	`Method.invoke(Object target, Object... args)` calls the method reflectively. Steps: get `Method` via `Class.getDeclaredMethod("name", paramTypes)`, call `setAccessible(true)` to bypass visibility, then `invoke(target, args)`. Pitfalls: (1) Checked exceptions are wrapped in `InvocationTargetException` ‚Äî unwrap with `getCause()`. (2) Wrong argument types throw `IllegalArgumentException`. (3) Private method requires `setAccessible(true)` ‚Äî may fail with `InaccessibleObjectException` in a module. (4) Null target for instance method throws `NullPointerException`. (5) Performance ‚Äî call overhead on every invocation. For repeated calls, cache the `Method` object or convert to a `MethodHandle` for JIT-friendly invocation.
What is escape analysis and how does the JVM use it?	Escape analysis is a JIT optimization that determines whether an object's reference can escape its creation method or thread. If an object does not escape: (1) Stack allocation ‚Äî the object can be allocated on the stack and collected when the frame exits, reducing GC pressure. (2) Scalar replacement ‚Äî the object is broken into its primitive fields and allocated in CPU registers. (3) Lock elision ‚Äî `synchronized` on a provably thread-local object is eliminated entirely. Enabled by default: `-XX:+DoEscapeAnalysis`. Disable with `-XX:-DoEscapeAnalysis` to diagnose allocation issues.
What is false sharing and how do you avoid it with `@Contended`?	False sharing occurs when two threads write to different variables that happen to reside in the same CPU cache line (typically 64 bytes). Although the variables are logically independent, the CPU must invalidate and reload the entire cache line on every write by the other CPU, causing a significant performance hit in high-throughput concurrent code. Detection: use Java Flight Recorder or JMH profiling; look for high L1/L2 cache miss rates. Solution: pad the variable to fill a full cache line. Java 8+: annotate the field or class with `@jdk.internal.vm.annotation.Contended` (requires `-XX:-RestrictContended`). Alternatively pad manually with 7 unused `long` fields around the hot field.
What is JVMTI and how are Java agents used for instrumentation?	JVMTI (JVM Tool Interface) is the native API for inspecting and controlling the JVM ‚Äî used by profilers, debuggers, and agents. A Java agent is a JAR with a `premain(String args, Instrumentation inst)` method. Attach at startup with `-javaagent:agent.jar`. A dynamic agent uses `VirtualMachine.attach(pid)` to attach to a running JVM and invokes `agentmain()`. Agents register a `ClassFileTransformer` that intercepts every class as it is loaded; the transformer can modify bytecode before the class is defined. Uses: APM (New Relic, Datadog, Elastic APM), code coverage (JaCoCo), test mocking, AOP weaving. Tools built on agents: ByteBuddy, ASM.
What is AOT Class Loading & Linking (Java 24, JEP 483)?	JEP 483 adds a class-data cache that persists the result of class loading and linking across JVM runs. Training run: `java -XX:AOTCache=app.aot -cp ... MyApp` ‚Äî warms the JVM, records loaded and linked class data into the cache file. Production run: `java -XX:AOTCache=app.aot -cp ... MyApp` ‚Äî restores pre-loaded and pre-linked class data from the cache, skipping parsing and verification for cached classes. Startup time improvement: 10‚Äì40% reduction for large applications. Difference from CDS (Class Data Sharing): CDS only caches JDK bootstrap class data; JEP 483 extends this to application classes. Complements JEP 490 (AOT Method Profiling) which persists JIT profile data.
What are `StackWalker` API (Java 9+) and `ProcessHandle` API (Java 9+)?	`StackWalker` replaces `Thread.getStackTrace()` for efficient, lazy stack frame traversal. `StackWalker.getInstance(Option.RETAIN_CLASS_REFERENCE).walk(stream -> stream.limit(5).collect(toList()))` ‚Äî only walks as many frames as requested; much faster than `getStackTrace()` which allocates all frames eagerly. `ProcessHandle` represents an operating system process. `ProcessHandle.current()` ‚Äî current JVM process. `.pid()`, `.info()` (command, arguments, start time, CPU time). `.children()` / `.descendants()` ‚Äî stream of child/descendent processes. `.onExit()` ‚Äî returns a `CompletableFuture<ProcessHandle>` that completes when the process exits. Used to manage subprocesses started with `ProcessBuilder`.
What is the difference between `WeakReference`, `SoftReference`, and `PhantomReference`?	All three allow an object to be GC'd while still holding a reference. Strengths: `SoftReference` ‚Äî cleared before OOM; good for memory-sensitive caches. `WeakReference` ‚Äî cleared at the next GC cycle when no strong/soft refs remain; good for canonical maps (`WeakHashMap`), listener registries. `PhantomReference` ‚Äî never returns the referent (`get()` always returns null); enqueued into a `ReferenceQueue` *after* the object is finalized but *before* its memory is freed; used for post-mortem cleanup (replacing `finalize()`). All can be used with a `ReferenceQueue` to track when objects are collected.
What causes `StackOverflowError` and how can you prevent or diagnose it?	`StackOverflowError` is thrown when the JVM stack runs out of space ‚Äî typically caused by infinite or very deep recursion (each method call pushes a new stack frame). Common causes: missing base case in recursion, mutual recursion, extremely deep call chains. Diagnosis: the error includes a partial stack trace. Prevention: (1) Add or fix the base case. (2) Convert recursion to iteration using an explicit `Deque` stack. (3) Use trampolining for tail-recursive algorithms. (4) Increase stack size with `-Xss` (e.g., `-Xss4m`) ‚Äî temporary fix only. Virtual threads (Java 21+) have growable stacks and are immune to `StackOverflowError` for typical use.
What are the variants of `OutOfMemoryError`?	`OutOfMemoryError` has distinct messages identifying the memory space that exhausted: (1) `Java heap space` ‚Äî heap full; tune `-Xmx` or fix leaks. (2) `GC overhead limit exceeded` ‚Äî GC spending >98% of time reclaiming <2% of heap; app effectively stalled. (3) `Metaspace` (or PermGen pre-Java 8) ‚Äî too many classes loaded; common with classloader leaks or code generation libraries. (4) `Unable to create native thread` ‚Äî OS process limit on threads; reduce thread count or increase OS limits. (5) `Direct buffer memory` ‚Äî `ByteBuffer.allocateDirect()` exceeded `-XX:MaxDirectMemorySize`. (6) `Native memory` ‚Äî off-heap memory exhausted. Use heap dumps and `jmap -histo:live` to diagnose.
How do you trigger and analyse a heap dump?	Trigger: `jmap -dump:format=b,file=heap.hprof <pid>`, or add JVM flag `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/` for automatic dumps on OOM. Analysis tools: **Eclipse MAT (Memory Analyser Tool)** ‚Äî best-in-class. Use the **Leak Suspects Report** to identify the likely root. Key concepts: **retained heap** ‚Äî total heap freed if the object were collected (includes its entire reachability subgraph). **Dominator tree** ‚Äî shows which objects hold the most retained heap. **GC roots** ‚Äî static fields, thread locals, JNI globals; objects reachable from roots cannot be collected. Shallow heap vs retained heap: a 16-byte object holding a 100MB array has tiny shallow heap but huge retained heap.
What is JMX and how do you use it for JVM monitoring?	JMX (Java Management Extensions) provides a standardized API for monitoring and managing JVM resources. JVM exposes built-in **MBeans** (Managed Beans) via `ManagementFactory`: `MemoryMXBean` ‚Äî heap/non-heap usage. `ThreadMXBean` ‚Äî thread counts, deadlock detection (`findDeadlockedThreads()`). `GarbageCollectorMXBean` ‚Äî GC counts and cumulative times. `RuntimeMXBean` ‚Äî uptime, JVM input arguments. Connect with `jconsole`, `jvisualvm`, or programmatically: `ManagementFactory.getMemoryMXBean().getHeapMemoryUsage()`. Enable remote JMX: `-Dcom.sun.management.jmxremote.port=9010 -Dcom.sun.management.jmxremote.authenticate=false`. Prometheus exporters (e.g., JMX Exporter agent) scrape MBeans and expose metrics for Grafana dashboards.
What is deoptimization in the JVM?	Deoptimization discards a compiled (JIT-optimized) method and falls back to interpreted mode when a speculative assumption is invalidated. Example: HotSpot optimistically inlines a virtual call assuming only one subtype exists (monomorphic call site); if a second subtype is loaded later, the inlined code is no longer valid and is deoptimized. Other triggers: class loading (`instanceof` assumptions), uncommon traps (rarely-executed branches that were never compiled), OSR (on-stack replacement) reversals. A high deoptimization rate (`-XX:+PrintCompilation` output showing `made not entrant`) indicates polymorphic code or unstable type hierarchies. Fix: reduce polymorphism, avoid reflective class loading after warm-up.
What are the three phases of class loading in the JVM?	Class loading has three phases: **(1) Loading** ‚Äî the ClassLoader reads the `.class` file bytes from the source (filesystem, JAR, network) and creates a `Class<?>` object in the method area. **(2) Linking** ‚Äî three sub-steps: **Verification** checks bytecode correctness and safety (type checks, branch targets); **Preparation** allocates memory for static fields and sets them to default values (zero, null, false); **Resolution** replaces symbolic references (class names, method names) in the constant pool with direct references (pointers) ‚Äî may be eagerly or lazily resolved. **(3) Initialization** ‚Äî runs the class's `<clinit>` static initializer block and initialises static fields to their declared values. Initialization is triggered lazily on first active use (object creation, static method call, static field read/write).
What is Java Flight Recorder (JFR) and how do you use it?	JFR (open-sourced in Java 11, JEP 328) is a low-overhead, always-on profiling framework built into the JVM. Overhead is typically < 1%. It records predefined and custom events (GC, JIT, thread activity, I/O, exceptions, allocations). Start programmatically: `jcmd <pid> JFR.start duration=60s filename=app.jfr` or via `-XX:StartFlightRecording=duration=60s,filename=app.jfr` JVM flag. Stop: `jcmd <pid> JFR.stop` / `JFR.dump`. Analyse recordings in JDK Mission Control (JMC). Events appear in categories: GC (pause times, reasons), JIT (compilations, deoptimisations), threads (locks, sleep, blocking), heap (TLAB sizes, allocation rates). Custom events: extend `jdk.jfr.Event` with `@Label`, `@Description` annotations. Ideal for production profiling when `jstack`/`jmap` are too invasive.
What is the `RandomGenerator` interface (Java 17+)?	`java.util.random.RandomGenerator` (Java 17+, JEP 356) is a common interface over all random number generators. Allows code to accept any RNG implementation without knowing the concrete type. Algorithm families available: **LXM family** (`L64X128MixRandom`, `L64X256MixRandom`, etc.) ‚Äî excellent statistical quality, very fast, not secure. **Xoshiro/Xoroshiro** (`Xoshiro256PlusPlus`, `Xoroshiro128PlusPlus`) ‚Äî fast, small state. `SplittableRandom` ‚Äî supports `split()` for independent streams in parallel code. Obtain by name: `RandomGenerator rng = RandomGeneratorFactory.of("L64X128MixRandom").create()`. `SecureRandom` also implements `RandomGenerator`. `RandomGeneratorFactory.all()` returns a stream of all available factories.
What is `HexFormat` (Java 17+)?	`java.util.HexFormat` (Java 17+, JEP 378) provides utility methods for formatting and parsing hex strings. Create: `HexFormat hex = HexFormat.of()` (lowercase, no separator); `HexFormat.ofDelimiter(":")` (colon-separated bytes). Formatting: `hex.formatHex(byte[])` ‚Äî byte array to hex string; `hex.toHexDigits(int)` ‚Äî integer to 8-digit hex; `hex.toHexDigits(byte)` ‚Äî byte to 2-digit hex. Parsing: `hex.parseHex(String)` ‚Äî hex string to `byte[]`; throws `IllegalArgumentException` on invalid input. Replaces manual `String.format("%02x", b)` loops, `DatatypeConverter.printHexBinary()`, and similar boilerplate. Useful for logging binary data, cryptographic keys, and protocol debugging.
What is Generational ZGC (Java 21+, JEP 439)?	ZGC (Z Garbage Collector) was non-generational until Java 21: it treated the entire heap as one space, which is safe but means short-lived objects are swept at the same cost as long-lived ones. Generational ZGC (JEP 439, production-ready Java 21) adds a young/old generation split, exploiting the **generational hypothesis** (most objects die young). Young generation is collected much more frequently than old generation. Benefits: higher throughput (less work per collection), lower allocation stall frequency, similar pause times (sub-millisecond). Enable: `-XX:+UseZGC -XX:+ZGenerational` (Java 21), or just `-XX:+UseZGC` in Java 23+ where generational is the default. Compare: with non-generational ZGC every minor allocation storm triggers a full-heap scan.
How do you analyse GC logs and tune G1GC pause times?	Enable GC logging: `-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m`. Key metrics: **pause time** (`GC(x) Pause Young/Old`), **allocation rate** (promotion rate + Eden size flush frequency), **GC frequency** (how often Minor/Full GC fires). GC viewers: GCEasy.io, GCViewer, JDK Mission Control. G1 tuning: `-XX:MaxGCPauseMillis=200` (default) ‚Äî G1 tries to stay under this target; lowering it forces smaller regions and more frequent collections. `-XX:G1HeapRegionSize=n[m]` ‚Äî fixed region size (1‚Äì32 MB, must be power of 2); larger regions reduce fragmentation at the cost of reclamation granularity. `-XX:G1NewSizePercent` / `G1MaxNewSizePercent` ‚Äî tune young gen size range. Symptom: long Mixed GC pauses ‚Üí increase `-XX:G1MixedGCCountTarget`; humongous object allocations ‚Üí choose region size > half the big object size.
What is Java object layout and how do you inspect it with JOL?	Every Java object on the heap has a **header** (12‚Äì16 bytes on 64-bit JVM): **mark word** (8 bytes) ‚Äî stores GC age, identity hash code, lock state (biased/thin/fat), monitor pointer. **Klass pointer** (4 bytes with compressed oops, 8 bytes without) ‚Äî points to the class metadata. After the header come the instance fields (aligned/padded to 8-byte boundaries). **Object alignment**: each object starts on an 8-byte boundary; padding bytes fill the space after fields to the next alignment boundary. JOL (Java Object Layout) tool: `ClassLayout.parseClass(MyClass.class).toPrintable()` ‚Äî prints field offsets, sizes, padding. Why it matters: understanding layout helps minimise memory usage and avoid false sharing. Compact records and `@Contended` both rely on layout knowledge.
What is AOT Method Profiling (Java 24+, JEP 490)?	JEP 490 (Java 24) complements AOT Class Loading (JEP 483) by persisting JIT **profile data** across JVM runs. Training run: the JVM collects profiling data (which methods are hot, which branches are taken, which call sites are monomorphic/bimorphic) and dumps it to an AOT cache file alongside the class-data cache. Production run: the JIT reads the persisted profile at startup, allowing C2 to apply aggressive optimisations (inlining decisions, speculative optimisations) earlier ‚Äî without waiting for the warm-up period. Result: applications reach peak performance faster, reducing the latency spike during the first minutes of operation that is characteristic of JIT-compiled Java. Combined with JEP 483, startup time and time-to-peak-performance can both be improved significantly.	OSR allows the JVM to replace an interpreter-executing method with its compiled version *while the method is still on the stack* (e.g., mid-loop). Without OSR, a hot loop would run interpreted until the method returns and is re-entered. OSR inserts a transition point at the loop back-edge: if the loop body becomes hot, the interpreter hands off to the compiled version. Visible in `-XX:+PrintCompilation` as `%` suffix on the method. OSR compilations are slightly less optimized than normal compilations because the live local variables constrain register allocation. Long-running loop-heavy code is the primary beneficiary.
What is a custom ClassLoader used for?	Custom ClassLoaders (extending `ClassLoader`, overriding `findClass()`) enable: (1) Non-standard sources ‚Äî loading classes from a network, database, or encrypted JARs. (2) Hot reloading ‚Äî discard old ClassLoader instance to "reload" classes at runtime (Spring DevTools, JRebel). (3) Class isolation ‚Äî plugin systems (OSGi, Java EE containers) give each plugin its own ClassLoader namespace to prevent version conflicts. (4) Bytecode transformation ‚Äî decrypt or instrument classes before defining them. Note: classes from different ClassLoaders are incompatible even with the same binary name.
What is method inlining in the JIT compiler?	Method inlining replaces a method call site with the method body directly, eliminating call overhead (stack frame setup/teardown, argument passing) and enabling further optimizations (constant folding, escape analysis, loop unrolling). Triggers: method is hot and small enough (‚â§35 bytecodes for hot methods). `final`, `private`, and `static` methods are easier to inline (no virtual dispatch). Too-large methods won't inline ‚Äî keep hot-path methods small. HotSpot inlines up to 9 levels deep. Use `-XX:+PrintInlining` to observe inlining decisions.
What is HotSpot's tiered compilation?	HotSpot uses five tiers: Level 0 (interpreted), Level 1 (C1 compiled, no profiling), Level 2 (C1, limited profiling), Level 3 (C1, full profiling ‚Äî method/branch/type data collected), Level 4 (C2 fully optimized ‚Äî uses Level 3 profiles for inlining, devirtualization, loop transforms). Methods advance through tiers based on invocation and back-edge counts. Hot methods reach Level 4; less hot stay at Level 1‚Äì3. Default since Java 8. Controlled with `-XX:+TieredCompilation`.
What is the difference between G1GC (`-XX:+UseG1GC`) and ZGC (`-XX:+UseZGC`)?	G1GC (default since Java 9): region-based, concurrent marking + evacuation, targets predictable pause times (default 200ms via `-XX:MaxGCPauseMillis`). Good balance of throughput and latency for most server workloads. ZGC (production-ready Java 15+): ultra-low latency, sub-millisecond pauses regardless of heap size, concurrent relocation, scalable to multi-TB heaps. Generational ZGC is the default in Java 23+. Slightly lower throughput than G1. Use ZGC for latency-sensitive apps (trading, real-time); G1 for general-purpose applications.
What is the Prototype pattern and what are its pitfalls in Java?	Prototype creates new objects by copying (cloning) an existing prototype rather than using `new`. In Java: implement `Cloneable` and override `clone()`. Pitfalls: (1) `Object.clone()` performs a SHALLOW copy by default ‚Äî fields pointing to mutable objects still share the same reference with the original. (2) `Cloneable` is a marker interface ‚Äî not calling `super.clone()` breaks the contract. (3) `clone()` bypasses constructors, which can break class invariants. Safer alternatives: copy constructors (`new Person(other)`) or copy factory methods, which give full control over what is copied.
What is the Proxy pattern and what are the three types in Java?	Proxy provides a surrogate or placeholder to control access to another object. Three main types in Java: (1) Static Proxy ‚Äî hand-written class implementing the same interface, delegating to the real object. (2) JDK Dynamic Proxy ‚Äî `java.lang.reflect.Proxy.newProxyInstance(classLoader, interfaces, InvocationHandler)` ‚Äî works only for interfaces; handler intercepts all calls. (3) CGLIB Proxy ‚Äî generates a subclass at runtime (used by Spring for concrete classes without interfaces). Uses: lazy initialization, access control, logging, transaction management (Spring AOP), caching.
What is the Flyweight pattern?	Flyweight uses shared objects to efficiently support a large number of fine-grained objects, where many share identical intrinsic state. Intrinsic state is shared and immutable; extrinsic state is passed in per-use. JDK examples: `Integer.valueOf(n)` caches `Integer` objects for ‚àí128 to 127 ‚Äî if you call `valueOf(100)` twice you get the SAME object; `Boolean.valueOf()` returns one of two cached instances; `String.intern()` stores only one copy of each distinct string literal in the pool. Common mistake: using `new Integer(n)` (deprecated) bypasses the cache.
What is the Chain of Responsibility pattern?	Chain of Responsibility passes a request along a chain of handlers, where each handler decides to process the request or pass it to the next handler. JDK/Java examples: `ClassLoader` delegation model ‚Äî child loader asks parent first; `javax.servlet.FilterChain` ‚Äî each filter calls `chain.doFilter()` to pass to the next; Spring Security's security filter chain; `java.util.logging` Logger parent delegation. The chain can be terminated at any point by not calling the next handler.
What is the Command pattern?	Command encapsulates a request as an object, allowing it to be parameterized, queued, logged, or undone. JDK examples: `Runnable` ‚Äî encapsulates a unit of work; `Callable` ‚Äî encapsulates a computation with a return value; `java.awt.event.ActionListener` ‚Äî encapsulates a UI action. Benefits: decouples invoker from executor; enables undo/redo (by storing inverse commands); enables macro commands (composite of commands); supports queuing and scheduling.
What is the Visitor pattern and where does it appear in the JDK?	Visitor lets you add operations to an object structure (like a composite tree) without modifying its classes. Define a `Visitor` interface with a `visit(ConcreteElement)` method per element type; each element implements `accept(Visitor)` which calls `visitor.visit(this)`. JDK examples: `java.nio.file.FileVisitor` ‚Äî `Files.walkFileTree(path, visitor)` calls `visitFile`, `preVisitDirectory`, `postVisitDirectory`, etc.; `javax.lang.model.element.ElementVisitor` used in annotation processors. New in Java 24: `ClassFile` API uses visitors for inspecting bytecode elements.
Where can you see the Facade pattern in the JDK?	`java.net.URL` hides the complexity of socket creation, protocol handling, and connection management behind a simple `openStream()` API. `javax.xml.parsers.DocumentBuilderFactory` provides a simple facade over complex XML parsing machinery. `SLF4J` facades multiple logging backends (Logback, Log4j2, java.util.logging) behind a single API. `javax.faces.context.FacesContext` in JSF. In Spring: `JdbcTemplate` facades raw JDBC (exception translation, resource management). Use Facade when a subsystem has many moving parts ‚Äî expose only what clients need.
What is the Bridge pattern and how does it differ from Adapter?	Bridge decouples an abstraction from its implementation so that the two can vary independently. You create two separate inheritance hierarchies ‚Äî one for the abstraction and one for the implementation ‚Äî and the abstraction holds a reference (the bridge) to an implementation. Example: `Shape` (abstraction) with `DrawingAPI` (implementation) ‚Äî you can add new shapes and new drawing APIs independently. JDK example: JDBC ‚Äî `DriverManager` + `Connection` bridge allows different database drivers without changing the client code. Key difference from Adapter: Adapter reconciles incompatible EXISTING interfaces; Bridge is designed up-front to allow abstraction and implementation to evolve independently.
What is the Composite pattern?	Composite lets clients treat individual objects (leaves) and compositions of objects (composites) uniformly through a common interface. Forms a tree structure where both leaves and composites implement the same `Component` interface. JDK examples: `java.awt.Component` (leaf) and `java.awt.Container` (composite ‚Äî holds other `Component`s); `java.nio.file.Path` represents files or directories uniformly; `javax.swing.JComponent` hierarchy. Use when you have a part-whole hierarchy and want clients to ignore the difference between leaf and composite operations. Common methods: `add(Component)`, `remove(Component)`, `getChildren()`, `operation()`.
What is the Interpreter pattern?	Interpreter defines a grammar for a language and provides an interpreter to deal with that grammar. Each grammar rule becomes a class; an abstract syntax tree (AST) is evaluated by calling `interpret()` on the tree. JDK examples: `java.util.regex.Pattern` ‚Äî compiles a regex grammar into a matcher; `javax.el.ExpressionLanguage` ‚Äî evaluates EL expressions in JSP/JSF; Spring Expression Language (SpEL). Limitations: the class explosion for complex grammars makes it unwieldy ‚Äî most real parsers use generated code (ANTLR) or hand-rolled recursive descent instead.
What is the Mediator pattern?	Mediator defines an object that encapsulates how a set of objects interact, reducing direct dependencies between them (lowering coupling). Objects communicate through the mediator rather than directly. JDK/Java examples: `java.util.concurrent.ExecutorService` mediates between task submitters and worker threads; Spring's `ApplicationEventPublisher` ‚Äî components publish events and subscribe without knowing each other; ATC (Air Traffic Control) ‚Äî classic textbook example. In UI: `JFrame` as mediator between Swing components. Benefits: simplifies maintenance ‚Äî interaction rules are in one place. Risk: mediator itself becomes a "god object" if overloaded.
What is the Memento pattern?	Memento captures and externalises an object's internal state without violating encapsulation, so the object can be restored to that state later (undo/redo). Roles: Originator (object whose state is saved), Memento (snapshot), Caretaker (stores mementos without inspecting them). Java implementation options: serialisation to store/restore complete object state; copy constructors; records (immutable snapshot). JDK examples: `Serializable` objects used as snapshots; `javax.swing.undo.UndoManager` + `UndoableEdit`. Pitfall: deep object graphs require deep copying ‚Äî shallow memento breaks on mutable nested objects.
What is the Abstract Factory pattern?	Abstract Factory provides an interface for creating **families of related objects** without specifying their concrete classes. Clients use the factory interface; the concrete factory determines which product family is instantiated. Classic example: UI toolkit factory ‚Äî `GUIFactory` with `createButton()` and `createCheckbox()`; `WindowsFactory` and `MacFactory` implement it and return platform-specific widgets. Difference from Factory Method: Factory Method creates one product; Abstract Factory creates a family of related products (Button + Checkbox + Dialog all consistent for one platform). JDK examples: `DocumentBuilderFactory.newInstance()` ‚Äî returns a concrete XML parser implementation; `java.sql.Connection` + `Statement` + `ResultSet` are a family created by `DriverManager`-obtained connections.	State allows an object to alter its behaviour when its internal state changes ‚Äî the object appears to change its class. Extract each state into a separate class implementing a `State` interface; the context delegates behaviour to its current state object. JDK/Java examples: `java.lang.Thread` ‚Äî lifecycle states (NEW, RUNNABLE, BLOCKED, WAITING, TERMINATED) each with distinct behaviour; TCP connection states (LISTEN, ESTABLISHED, CLOSE_WAIT); `javax.faces.lifecycle.Lifecycle` phases. Difference from Strategy: Strategy is typically set once and focuses on interchangeable algorithms; State transitions happen automatically in response to events and the state itself can trigger transitions.
What is JPMS and what problems does it solve?	JPMS (Java Platform Module System), introduced in Java 9 via Project Jigsaw, addresses: (1) **Classpath hell** ‚Äî unclear which JARs provide which types; conflicting versions silently shadow each other. (2) **Encapsulation gaps** ‚Äî all public classes on the classpath are accessible to all other code, including internal JDK APIs. (3) **Slow startup / large runtime** ‚Äî the entire JDK was monolithic. JPMS introduces named modules: each has a `module-info.java` declaring what it requires and what it exports. Only exported packages are accessible to dependent modules, even if classes are `public`. Enables `jlink` to create minimal JDK runtimes.
What are the key directives in `module-info.java`?	A `module-info.java` lives at the source root and uses these directives: `requires <module>` ‚Äî declares a dependency. `requires transitive <module>` ‚Äî transitive: dependents of this module also see it (e.g., a library that exposes API types from another module). `exports <package>` ‚Äî makes the package accessible to all modules. `exports <package> to <module>` ‚Äî qualified export to a specific module only. `opens <package>` ‚Äî allows full reflective access at runtime (needed for frameworks like Spring, Jackson). `opens <package> to <module>` ‚Äî qualified open. `uses <service>` ‚Äî declares this module consumes a service. `provides <service> with <impl>` ‚Äî registers an implementation for `ServiceLoader`.
What is the difference between named, automatic, and unnamed modules?	(1) **Named module** ‚Äî has a `module-info.class`; participates fully in the module system with explicit `requires`/`exports`. (2) **Automatic module** ‚Äî a regular JAR on the module path (not classpath) without `module-info.class`. Gets a module name from its JAR filename or `Automatic-Module-Name` manifest entry. It exports all its packages and reads all other modules ‚Äî a migration aid. (3) **Unnamed module** ‚Äî code on the classpath. Reads all named and automatic modules, but named modules cannot `requires` the unnamed module. Allows old classpath code to call module code but not vice versa. Migration path: classpath ‚Üí automatic module ‚Üí named module.
How does `--add-opens` work and why is it sometimes needed?	`--add-opens <module>/<package>=<target-module>` opens a package for deep reflective access at runtime, bypassing strong encapsulation. Used when frameworks (Spring, Hibernate, Jackson, test runners) need to reflectively access JDK internals that are no longer `opens` by default. Example: `--add-opens java.base/java.lang=ALL-UNNAMED`. Add to JVM args in `maven-surefire-plugin`, `java` command, or `JAVA_TOOL_OPTIONS`. Also `--add-exports` for compile-time access to non-exported packages. Both are escape hatches; prefer migrating dependencies to module-aware versions. Java 17+ enforces strong encapsulation by default (JEP 396/403).
What is the Flow API (Java 9+) and what does it provide?	`java.util.concurrent.Flow` is the standard Java API for reactive streams (Reactive Streams specification). It defines four nested interfaces: `Flow.Publisher<T>` ‚Äî produces items; `subscribe(Subscriber)` method. `Flow.Subscriber<T>` ‚Äî consumes items; `onSubscribe`, `onNext`, `onError`, `onComplete`. `Flow.Subscription` ‚Äî the link between Publisher and Subscriber; `request(n)` for back-pressure, `cancel()` to stop. `Flow.Processor<T,R>` ‚Äî both Publisher and Subscriber (pipeline stage). Core concept: **back-pressure** ‚Äî the subscriber controls the rate by calling `subscription.request(n)`. Java provides only `SubmissionPublisher<T>` as a concrete publisher. Project Reactor (Spring) / RxJava bridge to Flow via `FlowAdapters`.
How does `ServiceLoader` work and how does the module system integrate with it?	`ServiceLoader<S>` dynamically discovers implementations of a service interface at runtime. Classic usage: place a file `META-INF/services/<fully.qualified.Interface>` in a JAR containing the implementation class names (one per line). Module system usage: in `module-info.java`, the consumer declares `uses com.example.Plugin;` and the provider declares `provides com.example.Plugin with com.example.MyPlugin;`. Load: `ServiceLoader<Plugin> loader = ServiceLoader.load(Plugin.class); for (Plugin p : loader) { ... }` or `loader.findFirst()`. Benefits: loose coupling, replaceable implementations without recompilation. Used by the JDK itself (e.g., `java.nio.file.spi.FileSystemProvider`, JDBC drivers, `javax.xml` parsers).
What is a multi-release JAR and how does it work?	A multi-release JAR (JEP 238, Java 9+) allows a single JAR to contain version-specific class files for multiple Java versions. Structure: `META-INF/versions/11/com/example/Foo.class` overrides the root-level `com/example/Foo.class` when running on Java 11+. The `META-INF/MANIFEST.MF` must contain `Multi-Release: true`. Build support: Maven `maven-compiler-plugin` `<release>` config per version, Gradle `multiReleaseJar`. Use case: libraries that want to take advantage of newer API (e.g., `StackWalker`, `VarHandle`) while still supporting older JVMs. The root-level classes are the fallback for any JVM that doesn't understand multi-release (Java 8 and earlier).
What is the Class-File API and why was it introduced?	The Class-File API (JEP 484, finalized in Java 24) provides a standard JDK API for reading, writing, and transforming Java `.class` files. Before it, the ecosystem relied on third-party libraries like ASM, Javassist, and BCEL. Problems with that approach: (1) Each JDK release could change the class-file format, requiring ASM updates before frameworks could work on new JDKs. (2) The JDK itself internally used ASM (which caused versioning conflicts). (3) No standard, stable API. The Class-File API is maintained in sync with JVM spec changes, eliminating the "ASM lag" problem. It is already used internally by the JDK (`javac`, proxy generation, `MethodHandles` internals).
How do you read and inspect a class file with the Class-File API?	Use `ClassFile.of()` to get the entry point, then `parse()` to read bytes: `ClassFile cf = ClassFile.of(); ClassModel model = cf.parse(Path.of("MyClass.class"));`. Navigate the model: `model.thisClass().asInternalName()` ‚Üí class name. `model.methods()` ‚Üí list of `MethodModel`. `model.fields()` ‚Üí list of `FieldModel`. Per method: `method.methodName().stringValue()`, `method.flags()`, and `method.code().ifPresent(code -> code.forEach(element -> { if (element instanceof Instruction i) { ... } }))`. The model is a rich tree of `ClassElement`, `MethodElement`, `CodeElement` subtypes ‚Äî visitor-like iteration without requiring explicit visitors.
How do you generate a new class from scratch with the Class-File API?	`ClassFile.of().build(ClassDesc.of("com.example.Generated"), classBuilder -> { classBuilder .withFlags(ClassFile.ACC_PUBLIC) .withSuperclass(ClassDesc.of("java.lang.Object")) .withMethod("greet", MethodTypeDesc.of(CD_String), ClassFile.ACC_PUBLIC, methodBuilder -> { methodBuilder.withCode(codeBuilder -> { codeBuilder .ldc("Hello from generated class") .areturn(); }); }); });`. Returns a `byte[]` which can be loaded with a `ClassLoader`. `ClassDesc` encodes class descriptors type-safely; `MethodTypeDesc` encodes method signatures; `CodeBuilder` exposes typed instruction-emitting methods (`invokevirtual`, `aload`, `ireturn`, etc.).
How does `ClassFile.transformClass()` work for bytecode transformation?	`transformClass` takes a `ClassModel` and a `ClassTransform` and produces a new `byte[]` with the transformation applied. `ClassTransform` is a functional interface: `(ClassBuilder builder, ClassElement element) -> { /* inspect element, emit to builder */ }`. To selectively modify methods: `ClassTransform transform = ClassTransform.transformingMethods( method -> method.methodName().stringValue().startsWith("log"), (methodBuilder, methodElement) -> { // intercept or modify method elements methodBuilder.accept(methodElement); });`. Chain transforms: `transform1.andThen(transform2)`. This is the pattern used for AOP-like instrumentation, profiling agents, coverage tools, and framework code generation ‚Äî without a separate ASM dependency.
How does the Class-File API compare to ASM?	Both read/write class files at the bytecode level. Differences: (1) **Stability** ‚Äî Class-File API is a JDK standard, versioned with the JVM spec; ASM requires explicit updates per JDK version. (2) **API style** ‚Äî Class-File uses immutable models + functional builders; ASM uses a mutable visitor pattern (harder to compose). (3) **Type safety** ‚Äî Class-File uses `ClassDesc`, `MethodTypeDesc`, `DirectMethodHandleDesc` to avoid string-based errors; ASM uses raw internal name strings (`Ljava/lang/String;`). (4) **Performance** ‚Äî comparable; both zero-copy parse where possible. (5) **Ecosystem** ‚Äî ASM has a massive existing ecosystem and is mature; Class-File API is new (Java 24) but will grow. For new projects targeting Java 24+, Class-File API is the forward-looking choice; for maximum compatibility, ASM remains practical.
What happens during a TLS handshake?	(1) ClientHello ‚Äî client sends supported TLS versions, cipher suites, a random nonce. (2) ServerHello ‚Äî server selects TLS version, cipher suite, sends its certificate, its random nonce. (3) Certificate validation ‚Äî client verifies the server certificate's chain of trust to a trusted root CA, signature, expiry, and hostname. (4) Key Exchange ‚Äî e.g., ECDHE: both sides compute the same shared premaster secret without transmitting it. (5) Session keys derived ‚Äî both sides derive symmetric encryption and MAC keys from the shared secret + nonces. (6) Finished ‚Äî both sides send a HMAC of the handshake transcript to confirm integrity. All subsequent traffic is encrypted with the session keys.
How do you configure SSL/TLS in Java?	Use `SSLContext`: `SSLContext ctx = SSLContext.getInstance("TLS"); ctx.init(keyManagers, trustManagers, new SecureRandom());`. `KeyManager` holds the server's (or client's) certificate and private key ‚Äî loaded from a `KeyStore`. `TrustManager` determines which server certificates to trust ‚Äî loaded from a `TrustStore` (contains trusted CA certificates). Apply to `HttpsURLConnection`: `conn.setSSLSocketFactory(ctx.getSocketFactory())`. For `HttpClient` (Java 11+): `HttpClient.newBuilder().sslContext(ctx).build()`. For Spring Boot: configure via `server.ssl.*` properties.
What is the difference between a Keystore and a Truststore in Java?	A Keystore holds private keys and associated certificate chains ‚Äî identifying YOU to others. Used when YOUR server needs to present a certificate (server auth) or when using mutual TLS (client cert auth). A Truststore holds trusted CA certificates ‚Äî determining which certificates YOU trust. Used to validate the certificates presented by peers. In Java: both use `KeyStore` class; distinguished by role. JVM's default truststore is `$JAVA_HOME/lib/security/cacerts` (contains common root CAs). Formats: `JKS` (Java-specific, legacy), `PKCS12` (standard, recommended). Managed with `keytool`.
What is a certificate chain and how is it validated?	A certificate chain is a sequence: your server cert ‚Üí intermediate CA cert(s) ‚Üí root CA cert. Each cert is signed by the private key of the cert above it. Validation: (1) Check each certificate's signature using the public key of the issuing CA. (2) Verify each cert is not expired (`notBefore`/`notAfter`). (3) Check certificate revocation (CRL or OCSP). (4) Verify the leaf cert's hostname matches the `Subject Alternative Name` (SAN) / Common Name. (5) Verify the root CA is in the truststore. If any step fails, the TLS connection is rejected.
How do you create digital signatures in Java?	`java.security.Signature`. Signing (with private key): `Signature sig = Signature.getInstance("SHA256withRSA"); sig.initSign(privateKey); sig.update(dataBytes); byte[] signature = sig.sign();`. Verification (with public key): `sig.initVerify(publicKey); sig.update(dataBytes); boolean valid = sig.verify(signature);`. Common algorithms: `SHA256withRSA`, `SHA256withECDSA` (faster, smaller signatures). Digital signatures provide authenticity (only the private-key holder could sign) and non-repudiation.
What is the Java Cryptography Architecture (JCA) and its provider model?	JCA is the framework governing cryptographic operations in Java (`java.security`). It is algorithm-agnostic: applications request services by name (`MessageDigest.getInstance("SHA-256")`) and JCA dispatches to the appropriate provider. Providers (e.g., SunEC, SunJCE, BouncyCastle) register algorithm implementations. Add a provider: `Security.addProvider(new BouncyCastleProvider())` or list it in `java.security` config file. This enables swapping implementations for compliance (e.g., FIPS-certified providers) without changing application code.
What is ECDH key agreement and how is it used in Java?	ECDH (Elliptic Curve Diffie-Hellman) allows two parties to establish a shared secret over an insecure channel without transmitting the secret itself. Both parties generate ephemeral key pairs; they exchange public keys; each computes the same shared secret using the other's public key and their own private key. In Java: `KeyAgreement ka = KeyAgreement.getInstance("ECDH"); ka.init(myPrivateKey); ka.doPhase(theirPublicKey, true); byte[] sharedSecret = ka.generateSecret();`. Derive a symmetric key from the shared secret using HKDF or SHA-256. TLS uses a similar approach (ECDHE ‚Äî ephemeral ECDH) for forward secrecy.
How do you derive a key from a password using PBKDF2 in Java?	`SecretKeyFactory skf = SecretKeyFactory.getInstance("PBKDF2WithHmacSHA256"); PBEKeySpec spec = new PBEKeySpec(password.toCharArray(), salt, 310_000, 256); byte[] keyBytes = skf.generateSecret(spec).getEncoded(); SecretKey key = new SecretKeySpec(keyBytes, "AES");`. Key parameters: salt ‚Äî random 16+ bytes per credential (generated with `SecureRandom`); iterations ‚Äî high enough that hashing takes ~100ms (tune to your hardware; OWASP recommends 600,000 for PBKDF2-HMAC-SHA256); output length ‚Äî 256 bits for AES-256. NEVER hardcode salt; store it alongside the derived key.
What does AES-GCM authentication tag protect against?	AES-GCM (Galois/Counter Mode) is authenticated encryption: it produces a ciphertext AND an authentication tag. The tag is a MAC (Message Authentication Code) covering both the ciphertext and optional Additional Authenticated Data (AAD ‚Äî like a header or context). If an attacker modifies even one bit of the ciphertext, IV, or AAD, tag verification fails and `javax.crypto.AEADBadTagException` is thrown ‚Äî the data is rejected. This prevents chosen-ciphertext attacks, padding oracle attacks, and undetected tampering. Critical: never reuse an IV with the same key ‚Äî IV reuse with GCM is catastrophic and breaks both confidentiality and authenticity.
What are the best practices for secrets management in Java applications?	(1) Never hardcode secrets (API keys, DB passwords, crypto keys) in source code or config files checked into version control. (2) Use environment variables or a secrets management service: HashiCorp Vault (`vault-java-driver`), AWS Secrets Manager, Azure Key Vault, GCP Secret Manager. (3) In Spring Boot: `@Value("${DB_PASSWORD}")` reads from environment; Spring Cloud Vault integrates Vault natively. (4) Rotate secrets regularly; automate rotation where possible. (5) Audit secret access. (6) In Kubernetes: use Secrets objects mounted as environment variables or files (not stored in env vars in pod specs in plaintext YAML). Seal with Sealed Secrets or external-secrets-operator.
What is the Circuit Breaker pattern?	Circuit Breaker prevents cascading failures by monitoring calls to a remote service. States: CLOSED (normal operation ‚Äî calls pass through; failures counted). OPEN (failure threshold exceeded ‚Äî calls immediately fail without reaching the remote service, allowing it time to recover). HALF-OPEN (after a timeout, limited calls are allowed through to test recovery). If successful, returns to CLOSED; if failing, returns to OPEN. Java implementations: Resilience4j `@CircuitBreaker`, Spring Cloud Circuit Breaker. Configurable: failure rate threshold, slow call threshold, wait duration in OPEN state.
What is the Retry pattern with exponential backoff and jitter?	Retry automatically re-executes a failed operation, useful for transient failures (network blips, temporary service unavailability). Naive retry (fixed interval, unlimited) can cause thundering herd problems. Best practice: exponential backoff ‚Äî double the wait time after each failure: 1s, 2s, 4s, 8s... Plus jitter (random variation) ‚Äî avoids all clients retrying at the same moment after a shared outage. Limit maximum retries (e.g., 3) and maximum delay (e.g., 30s). Resilience4j: `@Retry(name="myService")` with `waitDuration`, `maxAttempts`, `enableExponentialBackoff`, and `randomizedWaitFactor`.
What is the Bulkhead pattern?	Bulkhead isolates components so a failure in one doesn't cascade to others, named after watertight compartments in ships. In Java: assign separate, bounded thread pools to different downstream services ‚Äî if one service is slow and fills its pool, other services are unaffected. Resilience4j supports two bulkhead implementations: `ThreadPoolBulkhead` (separate thread pool per service) and `SemaphoreBulkhead` (max concurrent calls). Key config: `maxConcurrentCalls`, `maxWaitDuration`. Essential for services that call multiple downstream dependencies.
What is the Outbox Pattern?	The Outbox Pattern reliably publishes events to a message broker as part of a database transaction, solving the dual-write problem (atomically updating DB AND publishing an event is impossible without distributed transactions). Implementation: write the event to an `outbox` table in the SAME database transaction as the data change. A separate "relay" service reads the outbox and publishes events to the broker, removing them after confirmation. This guarantees at-least-once delivery. CDC (Change Data Capture) tools like Debezium can act as the relay by reading the database's WAL.
What is CQRS (Command Query Responsibility Segregation)?	CQRS separates the model used to write data (Commands that change state) from the model used to read data (Queries that return data). Benefits: read models can be optimized independently (denormalized projections, caches), write models can focus on enforcing business rules, scalability (scale reads and writes independently). In practice: Commands go to a service that updates the domain model and emits events; Queries read from a separate, optimized read model updated by event handlers. Often combined with Event Sourcing. Overhead: eventual consistency between write and read models; increased complexity ‚Äî only adopt when the benefits justify it.
What is a Domain Event?	A Domain Event is a record of something significant that happened in the domain, expressed in domain language: `OrderPlaced`, `PaymentFailed`, `UserRegistered`. They are immutable value objects (use `record`). Benefits: decoupled components react to events without tight coupling; natural fit for audit logging; enables Event Sourcing. In Spring: publish with `ApplicationEventPublisher.publishEvent(event)`; handle with `@EventListener` (sync) or `@TransactionalEventListener(AFTER_COMMIT)` for reliably publishing after the transaction commits.
What is JMH?	Java Microbenchmark Harness ‚Äî the official OpenJDK tool for writing correct, reproducible Java microbenchmarks. It handles JIT warm-up, dead-code elimination, and statistical analysis that naive `System.nanoTime()` benchmarks get wrong.
Why is writing a Java microbenchmark without JMH unreliable?	Without JMH: the JIT may eliminate "dead" computations with no observable side effects; warm-up effects bias early measurements; GC pauses skew results; OS scheduling adds noise. JMH controls all of these.
How do you add JMH to a Maven project?	Add `org.openjdk.jmh:jmh-core` and `org.openjdk.jmh:jmh-generator-annprocess` as dependencies (or use the `jmh-maven-plugin`). JMH uses annotation processing to generate benchmark infrastructure.
How do you run JMH benchmarks?	Build a fat JAR with `mvn package` (or use the jmh plugin), then `java -jar target/benchmarks.jar`. Alternatively call `new Runner(options).run()` from a main method.
What does `@Benchmark` do?	Marks a method as a benchmark. JMH calls this method in a tight loop, measures throughput or latency, handles warm-up, and reports statistics.
What does `@BenchmarkMode` control?	The measurement mode: `Mode.Throughput` (ops/time), `Mode.AverageTime` (time/op), `Mode.SampleTime` (distribution of latencies), `Mode.SingleShotTime` (one cold invocation), `Mode.All`.
What does `@OutputTimeUnit` do?	Sets the time unit for reported results: `TimeUnit.MILLISECONDS`, `MICROSECONDS`, `NANOSECONDS`. Example: `@OutputTimeUnit(TimeUnit.NANOSECONDS)`.
What does `@Warmup` configure?	`@Warmup(iterations = 5, time = 1, timeUnit = TimeUnit.SECONDS)` ‚Äî runs the benchmark for warm-up without recording results. Allows the JIT to fully optimise the hot path before measurement begins.
What does `@Measurement` configure?	The actual measurement phase: `@Measurement(iterations = 10, time = 1, timeUnit = TimeUnit.SECONDS)`.
What does `@Fork` do?	Runs the benchmark in a separate JVM process: `@Fork(value = 3)` runs 3 forks. Each fork starts with a fresh JVM, eliminating contamination from previous benchmarks' JIT profiles. `@Fork(0)` disables forking (useful for debugging in IDE).
What does `@State` do?	Marks a class as a state container for benchmarks. `@State(Scope.Benchmark)` ‚Äî one instance shared across all threads. `@State(Scope.Thread)` ‚Äî one instance per thread. `@State(Scope.Group)` ‚Äî one per thread group.
What are `@Setup` and `@TearDown`?	`@Setup(Level.Trial)` / `@TearDown(Level.Trial)` ‚Äî run once per benchmark trial (all forks). `Level.Iteration` ‚Äî per iteration. `Level.Invocation` ‚Äî per benchmark method call (use sparingly; introduces overhead).
What is dead code elimination (DCE) and how does JMH address it?	The JIT can detect that a computation's result is unused and eliminate it. JMH addresses this by consuming return values via `Blackhole.consume(result)` or by having benchmarks return the result directly (JMH sinks it automatically).
What is `Blackhole`?	A JMH helper that prevents the JIT from eliminating computations: `public void benchmark(Blackhole bh) { bh.consume(expensiveOp()); }`. The `Blackhole.consume()` call has defined side-effects that block DCE without significant overhead.
What is constant folding and how do you avoid it in benchmarks?	The JIT can compute a result at compile/JIT time if all inputs are constants. Use `@State`-held fields rather than literals, so values appear variable to the JIT.
What does `@Param` do?	Injects parameterised values into `@State` fields: `@Param({"10", "100", "1000"}) public int size;`. JMH runs the full benchmark for each parameter combination, enabling scaling analysis.
What does a JMH output line look like?	`Benchmark          Mode  Cnt     Score     Error  Units`\n`MyBench.test       avgt   10   123.456 ¬±  2.345  ns/op`. Score is the mean, Error is the 99.9% confidence interval.
What does a high `Error` value indicate in JMH results?	High variance ‚Äî the benchmark is unstable. Causes: insufficient warm-up, GC pauses, external interference, non-deterministic code paths. Increase iterations/forks or use `@Fork(jvmArgs="-Xmx4g")`.
What is `Mode.SampleTime` useful for?	Captures the latency distribution (p50, p90, p99, p99.9), not just the average. Important for latency-sensitive code where outliers matter more than mean performance.
What class/method granularity is appropriate for JMH?	Micro: individual operations (serialisation, parsing, a single method). Not appropriate for end-to-end integration benchmarks ‚Äî use tools like Gatling or k6 for those.
Should you use `@Fork(0)` in production benchmarks?	No ‚Äî only for debugging. Without forking, benchmark isolation is broken; previous benchmarks' JIT profiles pollute the current one, giving unreliable results.
What JVM flags are useful for benchmarking?	`-XX:+PrintCompilation` (see JIT activity), `-XX:+LogCompilation` (detailed JIT log), `-prof gc` (GC profiling via JMH profiler), `-prof stack` (stack sampling). Add via `@Fork(jvmArgsPrepend=...)`.
What is Compare-And-Swap (CAS) and how does it underpin lock-free algorithms?	CAS is a CPU-level atomic instruction: `compareAndSwap(address, expectedValue, newValue)` ‚Äî atomically: if the current value at address equals expected, replace it with new and return true; otherwise return false without modifying memory. This is the foundation of all Java lock-free code. CAS eliminates mutex locks: instead of "lock ‚Üí read ‚Üí modify ‚Üí write ‚Üí unlock", lock-free code does "read ‚Üí compute new value ‚Üí CAS(old, new); retry on failure." The key property: exactly one thread wins a CAS; others retry without blocking. All `java.util.concurrent.atomic` classes use CAS internally (`AtomicInteger.compareAndSet(expect, update)`). On x86 this is the `LOCK CMPXCHG` instruction.
What is the ABA problem and how does `AtomicStampedReference` solve it?	ABA: thread reads value A, another thread changes A‚ÜíB‚ÜíA. The first thread's CAS succeeds (sees A as expected) but doesn't detect that an intermediate change occurred ‚Äî the data structure may be in an inconsistent state (e.g., node was removed and re-added to a linked list). `AtomicStampedReference<V>` pairs the value with a monotonically incrementing integer stamp (version number). The CAS checks both value AND stamp: `ref.compareAndSet(expectedRef, newRef, expectedStamp, expectedStamp + 1)`. Since the stamp is always incremented, A‚ÜíB‚ÜíA results in stamps 0‚Üí1‚Üí2; the original thread checks stamp=0 which no longer matches stamp=2 ‚Üí CAS fails correctly. `AtomicMarkableReference` is similar but uses a boolean mark instead of a counter.
When should you use `LongAdder`/`LongAccumulator` over `AtomicLong`?	`AtomicLong` is a single memory location updated with CAS. Under high contention, many threads compete for the same CAS, causing repeated retries (spinning) ‚Äî throughput degrades significantly. `LongAdder` maintains an array of cells (per-thread stripes) + a base cell. Increment: each thread updates its own cell with low contention. `sum()`: adds all cells ‚Äî not atomic, so see a recent-but-not-instant value. Use when you need: (1) High-throughput parallel increments (counters, accumulators). (2) The final sum, not a per-operation precise value. Use `AtomicLong` when you need `get()` to reflect every prior update instantly (e.g., sequence number generation, flag, compare-and-act). `LongAccumulator` generalizes `LongAdder` to arbitrary accumulation functions: `new LongAccumulator(Long::max, Long.MIN_VALUE)`.
What are `VarHandle` memory ordering modes and when does each apply?	`VarHandle` (Java 9+) gives controlled access to fields/array elements with explicit memory ordering. Modes from weakest to strongest: **Plain** ‚Äî no ordering guarantees; JIT may reorder freely. Use only for provably single-threaded access. **Opaque** ‚Äî reads/writes are atomic (no tearing) and not reordered with other opaque accesses on the same variable. **Acquire (read) / Release (write)** ‚Äî acquire read sees all writes before the corresponding release write on another thread. Classic "publish a new object" pattern: write fields, then release-set flag; reader acquire-reads flag, then safely reads fields. **Volatile** ‚Äî full sequential consistency (total order); all threads see the same order of volatile ops. Costs increase with strength. Use the weakest mode that meets your correctness requirement.
How do you implement a correct non-blocking stack using CAS?	A non-blocking (Treiber) stack stores a `AtomicReference<Node<T>> top`. Push: `Node<T> newNode = new Node<>(item); do { newNode.next = top.get(); } while (!top.compareAndSet(newNode.next, newNode));`. Pop: `Node<T> node; Node<T> next; do { node = top.get(); if (node == null) return empty; next = node.next; } while (!top.compareAndSet(node, next)); return node.item;`. Progress guarantee: **lock-free** ‚Äî at least one thread makes progress per finite number of steps (no thread can be blocked by another). Not wait-free (individual threads may spin). ABA risk: if nodes are recycled and re-pushed, the same pointer may reappear. Mitigate with `AtomicStampedReference` or a memory allocator that never reuses pointers.
What distinguishes lock-free, wait-free, and obstruction-free algorithms?	These are formally defined progress guarantees for non-blocking algorithms: (1) **Obstruction-Free** ‚Äî a thread makes progress if it runs alone (no contention). Weakest; rarely used. (2) **Lock-Free** ‚Äî at least one thread in the system makes progress in finite steps regardless of what other threads do. Failed CAS retries don't count as progress for the retrying thread but another thread succeeds. Most `java.util.concurrent` algorithms (e.g., `AtomicInteger`, `ConcurrentLinkedQueue`) are lock-free. (3) **Wait-Free** ‚Äî every thread makes progress in a bounded number of steps regardless of contention. Strongest; very difficult to implement; few practical examples. Lock-free algorithms can cause starvation of individual threads; wait-free eliminates that. For most real-world Java code, lock-free is the practical target.
What is annotation processing and how does `AbstractProcessor` work?	Annotation processing runs at compile time via the Java Compiler Plugin (APT). Subclass `javax.annotation.processing.AbstractProcessor` and annotate it with `@SupportedAnnotationTypes("com.example.MyAnnotation")` and `@SupportedSourceVersion(SourceVersion.RELEASE_17)`. Override `process(Set<? extends TypeElement> annotations, RoundEnvironment roundEnv)`. Use `roundEnv.getElementsAnnotatedWith(MyAnnotation.class)` to get annotated elements (`TypeElement`, `MethodElement`, etc.). Generate source files via `processingEnv.getFiler().createSourceFile(className).openWriter()`. Register the processor in `META-INF/services/javax.annotation.processing.Processor` or via `@AutoService`. Examples: Lombok (code generation), Dagger (DI graph), MapStruct (mapper generation), Immutables.
How do JDK dynamic proxies work and what are their limitations?	JDK dynamic proxies created via `Proxy.newProxyInstance(classLoader, interfaces, handler)` generate a class at runtime that implements given interfaces. All method calls are dispatched to the `InvocationHandler.invoke(Object proxy, Method method, Object[] args)` method. Uses: AOP, logging, mocking (Mockito), JPA lazy loading. Limitations: (1) Can only proxy **interfaces** ‚Äî not concrete classes. (2) Method must be declared in an interface to be intercepted. (3) Calls from within the target object to itself (`this.method()`) bypass the proxy. When a class (not interface) must be proxied, CGLIB or ByteBuddy generates a subclass proxy at runtime by subclassing with bytecode generation.
What is the difference between JDK proxy and CGLIB/ByteBuddy proxies?	JDK proxy: generates a class implementing specified interfaces; only intercepts interface methods; uses `InvocationHandler`; part of the JDK (`java.lang.reflect.Proxy`), no extra dependency. CGLIB (Code Generation Library): subclasses the target class; intercepts all non-final, non-private methods; requires CGLIB JAR (bundled in Spring). Used by Spring AOP when no interface exists. ByteBuddy: modern, powerful, type-safe bytecode library; used by Mockito 2+, Hibernate 5+. All have the same internal-call limitation ‚Äî `this.method()` bypasses the proxy since the original instance doesn't know about the proxy wrapper. Spring recommends prefer-interface design and `@Transactional` on public methods for this reason.
What are GraalVM Native Image constraints and how do you handle reflection?	GraalVM Native Image performs AOT (ahead-of-time) compilation to a native binary. It uses closed-world analysis ‚Äî it must know all classes, methods, and resources at build time. Dynamic features that break by default: reflection, JDK proxies, serialization, resources (`Class.forName()`, `getDeclaredMethods()`, etc.). Fix with **reachability metadata** JSON files: `reflect-config.json` (list classes/methods/fields used reflectively), `proxy-config.json` (list proxy interface combos), `resource-config.json`, `serialization-config.json`. Location: `META-INF/native-image/<groupId>/<artifactId>/`. Generate automatically with the GraalVM tracing agent: `java -agentlib:native-image-agent=config-output-dir=META-INF/native-image -jar myapp.jar`. Spring Boot 3+ has built-in Native Image support via `spring-aot-maven-plugin`.
What is `MethodHandles` and how does it differ from reflection?	`java.lang.invoke.MethodHandle` is a typed, directly-invocable ref to a method, constructor, or field. Obtained via `MethodHandles.lookup()`. Differences from reflection: (1) **Performance** ‚Äî MethodHandles are JIT-optimizable; after warm-up they are as fast as direct calls. `Method.invoke()` involves repeated array boxing and unpacking. (2) **Type safety** ‚Äî `MethodType` encodes exact signature; type mismatches fail at lookup time, not call time. (3) **Access control** ‚Äî `Lookup` captures access context at creation point; respects encapsulation properly. Use: `MethodHandle mh = MethodHandles.lookup().findVirtual(String.class, "length", MethodType.methodType(int.class)); int len = (int) mh.invokeExact("hello");`. Used by Java internals, Groovy, Kotlin, and `invokedynamic`.
What is `invokedynamic` and how does Java use it?	`invokedynamic` (JEP 292, Java 7+) is a JVM bytecode instruction that defers the method resolution target to a **bootstrap method** defined by the programmer (or JDK). The bootstrap method returns a `CallSite` (the resolved target); subsequent invocations use that `CallSite` directly. Uses in the JDK: (1) **Lambda expressions** ‚Äî compiled to `invokedynamic` with `LambdaMetafactory` as bootstrap; the actual class generation is deferred to runtime, avoiding class file proliferation. (2) **String concatenation** (Java 9+) ‚Äî `StringConcatFactory` emits optimised assembly at runtime instead of `StringBuilder` chains. (3) **Kotlin** and **Groovy** dynamic dispatch. Benefits: the JIT can re-optimize `CallSite` targets as program behaviour is learned.
What is GraalVM Native Image constraints and how do you handle reflection?	GraalVM Native Image performs AOT (ahead-of-time) compilation to a native binary. It uses closed-world analysis ‚Äî it must know all classes, methods, and resources at build time. Dynamic features that break by default: reflection, JDK proxies, serialization, resources (`Class.forName()`, `getDeclaredMethods()`, etc.). Fix with **reachability metadata** JSON files: `reflect-config.json` (list classes/methods/fields used reflectively), `proxy-config.json` (list proxy interface combos), `resource-config.json`, `serialization-config.json`. Location: `META-INF/native-image/<groupId>/<artifactId>/`. Generate automatically with the GraalVM tracing agent: `java -agentlib:native-image-agent=config-output-dir=META-INF/native-image -jar myapp.jar`. Spring Boot 3+ has built-in Native Image support via `spring-aot-maven-plugin`.
What is AOP (Aspect-Oriented Programming) and what are its core concepts?	AOP modularizes cross-cutting concerns (logging, security, transactions, caching) that span many classes, separating them from business logic. Core concepts: Aspect ‚Äî the module of cross-cutting concern. Advice ‚Äî code to execute (Before, After, AfterReturning, AfterThrowing, Around). Pointcut ‚Äî expression matching join points where advice applies. Join point ‚Äî specific point in execution (method call, field access). Weaving ‚Äî applying aspects at compile time (AspectJ), load time (AspectJ LTW), or runtime (Spring AOP via proxies). Spring AOP supports only method-level join points; AspectJ supports field access and constructors.
What is the `VarHandle` API (Java 9+) and why was it introduced?	`VarHandle` (`java.lang.invoke`) provides typed references to variables (instance fields, array elements, static fields) with multiple memory access modes: plain, opaque, acquire/release (one-sided barriers), and volatile (full barrier). Enables atomic operations: `compareAndSet()`, `getAndAdd()`. Advantages over `sun.misc.Unsafe`: type-safe, bounds-checked, part of the public API. Advantages over `AtomicFieldUpdater`: faster (JIT-friendly), more access modes. Used in JDK's own concurrent data structures; replaces most legitimate uses of `Unsafe` in library code.
What is bytecode manipulation and what libraries support it?	Bytecode manipulation directly reads, modifies, or generates `.class` file bytecode, enabling AOP weaving, mocking, code generation, profiling, and hot patching. Libraries: ASM ‚Äî low-level visitor API, minimal overhead, used by almost all other tools. Javassist ‚Äî source-level API using Java source strings. ByteBuddy ‚Äî fluent, type-safe API used by Mockito (mock generation), Hibernate (proxy creation), and many frameworks. CGLIB ‚Äî generates subclasses dynamically, used by Spring/Hibernate for proxies. Manipulation can happen at compile time (via annotation processors), load time, or runtime (via agents).
What is Netty?	An asynchronous, event-driven NIO framework for building high-performance network servers and clients. Abstracts the complexity of raw NIO into a clean pipeline model. Used by Kafka, Cassandra, gRPC, Vert.x, and many others.
What is Netty's `EventLoop`?	A single-threaded executor that drives I/O for a set of channels. Runs the I/O Selector loop, fires events, and executes submitted tasks ‚Äî all on one thread, avoiding synchronisation. Multiple `EventLoop`s form an `EventLoopGroup`.
What is an `EventLoopGroup`?	A pool of `EventLoop`s. Netty servers typically use two groups: **boss group** (accepts connections) and **worker group** (handles I/O on accepted connections). `NioEventLoopGroup` uses NIO; `EpollEventLoopGroup` uses Linux epoll.
What is a `Channel` in Netty?	Represents an open network connection. `NioServerSocketChannel` (TCP server), `NioSocketChannel` (TCP client), `NioDatagramChannel` (UDP). The channel has a `pipeline` of handlers.
What is a `ChannelPipeline`?	A chain of `ChannelHandler`s processing inbound and outbound events on a channel. Inbound events flow from head to tail; outbound events from tail to head. Each handler can transform, filter, or respond to events.
What is a `ChannelHandler`?	A handler that processes I/O events or operations. `ChannelInboundHandlerAdapter` (read data, channel active/inactive). `ChannelOutboundHandlerAdapter` (write, connect, close). `@ChannelHandler.Sharable` if safe to share across channels.
What is `ChannelHandlerContext`?	Passed to handler methods. Use it to interact with the pipeline: `ctx.fireChannelRead(msg)` (forward to next handler), `ctx.writeAndFlush(msg)` (write outbound), `ctx.channel()` (access the channel), `ctx.executor()` (the EventLoop).
What is `ByteBuf`?	Netty's byte buffer, replacing `java.nio.ByteBuffer`. Two separate read and write indexes (no `flip()` required), pooled allocators, composite buffers, reference counting. Obtain via `ctx.alloc().buffer()`.
What is reference counting in Netty `ByteBuf`?	`ByteBuf` is released to its pool when `refCnt` reaches zero. `buf.retain()` increments, `buf.release()` decrements. Passing a `ByteBuf` to `writeAndFlush` automatically releases it after writing.
What is a `ChannelFuture` in Netty?	An asynchronous result for channel I/O operations. `channelFuture.addListener(future -> { if (future.isSuccess()) ... })`. Do not block on `channelFuture.sync()` inside an `EventLoop` thread ‚Äî causes deadlock.
What is a Netty codec?	A `ChannelHandler` that converts between raw bytes and higher-level messages. `ByteToMessageDecoder` (bytes ‚Üí objects), `MessageToByteEncoder` (objects ‚Üí bytes). Combined: `ByteToMessageCodec`. Examples: `HttpServerCodec`, `LengthFieldBasedFrameDecoder`.
What is `LengthFieldBasedFrameDecoder`?	A Netty decoder that reads a length field at the start of each frame to determine message boundaries, solving the TCP framing (stream-based, no message boundaries) problem cleanly.
What is `HttpServerCodec` in Netty?	A combined encoder+decoder for HTTP/1.1: handles request decoding (`HttpRequest`, `HttpContent`, `LastHttpContent`) and response encoding. Add to the pipeline before your business handler.
How do you bootstrap a Netty TCP server?	`new ServerBootstrap().group(bossGroup, workerGroup).channel(NioServerSocketChannel.class).childHandler(new ChannelInitializer<SocketChannel>() { protected void initChannel(SocketChannel ch) { ch.pipeline().addLast(new MyHandler()); } }).bind(8080).sync();`
What does `childHandler` vs `handler` do in `ServerBootstrap`?	`handler`: applied to the server channel itself (e.g. `LoggingHandler` for accept events). `childHandler`: applied to each accepted child channel (most business logic goes here).
What is `ChannelInitializer`?	A special one-use handler that initialises a channel's pipeline when it is registered. `initChannel` is called once; the initialiser removes itself from the pipeline afterwards.
How do you write a Netty echo server handler?	`public class EchoHandler extends ChannelInboundHandlerAdapter { @Override public void channelRead(ChannelHandlerContext ctx, Object msg) { ctx.writeAndFlush(msg); } @Override public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) { cause.printStackTrace(); ctx.close(); } }`
How do you add TLS to a Netty server?	Create an `SslContext` via `SslContextBuilder.forServer(certChainFile, keyFile).build()`, then add `sslContext.newHandler(ch.alloc())` as the first handler in the pipeline.
How do you enable TLS on a plain Java `SSLSocket`?	`SSLContext ctx = SSLContext.getInstance("TLS"); ctx.init(keyManagers, trustManagers, null); SSLSocketFactory factory = ctx.getSocketFactory(); SSLSocket socket = (SSLSocket) factory.createSocket(host, port); socket.startHandshake();`
What is Project Panama and what problems does it solve compared to JNI?	Project Panama (finalized in Java 22, JEP 454) provides the Foreign Function & Memory API as a modern replacement for JNI (Java Native Interface). JNI problems: verbose C headers, error-prone memory management, unsafe `Unsafe` usage, no type safety, requires a C bridge layer even for simple calls, very hard to debug. Panama advantages: (1) Pure Java ‚Äî no separate C stub code. (2) Type-safe method handles with `FunctionDescriptor`. (3) Explicit, bounded `MemorySegment` allocations instead of raw pointers. (4) Scoped lifetimes via `Arena` ‚Äî automatic off-heap memory cleanup. (5) `jextract` tool auto-generates Java bindings from C headers. (6) Supported by JIT ‚Äî downcall handles are as fast as JNI at steady state.
What is `MemorySegment` and how does `Arena` control its lifetime?	`MemorySegment` represents a contiguous region of memory ‚Äî either on-heap or off-heap (native). It carries bounds and an access scope; out-of-bounds and use-after-free are caught at runtime. Allocation is managed by an `Arena`: `Arena.ofConfined()` ‚Äî access restricted to the creating thread; closed manually or with try-with-resources. `Arena.ofShared()` ‚Äî shared across threads; reference-counted, closed explicitly. `Arena.ofAuto()` ‚Äî GC-managed; memory freed when segment becomes unreachable (no manual close). `Arena.global()` ‚Äî never freed; use for static native structures. Example: `try (Arena arena = Arena.ofConfined()) { MemorySegment seg = arena.allocate(1024); seg.set(ValueLayout.JAVA_INT, 0, 42); }` ‚Äî memory is freed on scope exit.
What is `MemoryLayout` and how do you describe a C struct?	`MemoryLayout` describes the shape of off-heap memory ‚Äî size, alignment, and structure ‚Äî without allocating. Building a C struct equivalent: `StructLayout point = MemoryLayout.structLayout( ValueLayout.JAVA_INT.withName("x"), ValueLayout.JAVA_INT.withName("y") ).withName("Point");`. Variants: `ValueLayout` (primitive types: `JAVA_INT`, `JAVA_LONG`, `ADDRESS`), `SequenceLayout` (arrays), `StructLayout` (struct), `UnionLayout` (union), `PaddingLayout` (alignment bytes). Access via `VarHandle`: `VarHandle xHandle = point.varHandle(MemoryLayout.PathElement.groupElement("x"));`. This makes struct field access type-safe and offset-calculated automatically.
What is a downcall handle and how do you call a native function?	A downcall handle is a `MethodHandle` that calls a native (C) function from Java. Steps: (1) Find the symbol: `SymbolLookup.libraryLookup("libm.so.6", Arena.global()).find("cos").get()` or `Linker.nativeLinker().defaultLookup().find("strlen").get()`. (2) Describe the function signature with `FunctionDescriptor`: `FunctionDescriptor.of(ValueLayout.JAVA_DOUBLE, ValueLayout.JAVA_DOUBLE)` = C `double cos(double)`. (3) Create the handle: `MethodHandle cos = Linker.nativeLinker().downcallHandle(symbol, descriptor);`. (4) Invoke: `double result = (double) cos.invokeExact(1.0);`. The linker handles marshaling between Java types and native ABI conventions (registers, stack layout).
What is an upcall stub and when do you need one?	An upcall stub allows native C code to call back into Java ‚Äî the inverse of a downcall. Use case: passing a Java callback to a C function like `qsort` (which takes a comparator function pointer). Steps: (1) Write the Java method to serve as callback. (2) Create a `MethodHandle` to it. (3) Convert to a native function pointer (upcall stub): `MemorySegment callback = Linker.nativeLinker().upcallStub(handle, FunctionDescriptor.of(JAVA_INT, ADDRESS, ADDRESS), arena);`. (4) Pass `callback` as an `ADDRESS` argument to the native function. The stub bridges the native ABI calling convention back to a Java `MethodHandle` invocation. The stub is valid for the lifetime of the owning `Arena`.
What is `jextract` and how does it accelerate Panama adoption?	`jextract` is a command-line tool that reads C header files and generates corresponding Java bindings using the Foreign Function & Memory API. Given a C header, it generates: constant definitions (as `static final`), struct layouts (`StructLayout`, `VarHandle` accessors), function downcall handles. Usage: `jextract --output src/generated -t com.example.binding /usr/include/openssl/ssl.h`. The generated bindings use the Panama API idiomatically. This eliminates the manual, error-prone work of writing `FunctionDescriptor` and `MemoryLayout` definitions. `jextract` is distributed separately from the JDK but tracks JDK releases. Especially valuable for large C APIs (OpenSSL, LAPACK, system APIs).
What is `SegmentAllocator` and how do you allocate native structs?	`SegmentAllocator` is an interface for allocating `MemorySegment` instances within a given `Arena`. Concrete allocators: (1) `arena.allocate(layout)` ‚Äî allocates a struct-sized segment with the given layout. (2) `arena.allocateFrom(ValueLayout.JAVA_INT, 42)` ‚Äî allocates and initialises a scalar value. (3) `arena.allocateArray(ValueLayout.JAVA_INT, 1, 2, 3)` ‚Äî allocates and fills an array. (4) `SegmentAllocator.prefixAllocator(segment)` ‚Äî sub-allocates from an existing segment. All allocations are automatically freed when the `Arena` is closed. This replaces `sun.misc.Unsafe.allocateMemory()` and the fragile manual `ByteBuffer.allocateDirect()` approach with safe, lifecycle-managed native memory.
What is `SymbolLookup` and how do you locate native functions?	`SymbolLookup` resolves function names and global variable names in native libraries to `MemorySegment` addresses. Three sources: `Linker.nativeLinker().defaultLookup()` ‚Äî symbols from the standard C library (`libc` / `CRT`). `SymbolLookup.loaderLookup()` ‚Äî symbols from libraries already loaded by the JVM's native class loader (via `System.loadLibrary()`). `SymbolLookup.libraryLookup(path, arena)` ‚Äî loads a specific native library and provides lookup within it; the library is unloaded when the arena closes. Combine lookups: `defaultLookup.or(SymbolLookup.loaderLookup())`. `find(name)` returns `Optional<MemorySegment>` ‚Äî empty if the symbol is not found.
What is the Panama safety model and how does it prevent memory errors?	The Foreign Function & Memory API enforces safety through **Arenas** and **access modes**. An `Arena` bounds the lifetime of all its `MemorySegment`s ‚Äî for confined arenas only the owning thread may access the segment; for shared arenas, multi-threaded access is allowed (with memory ordering guarantees); the auto arena uses a `Cleaner` for GC-based release; the global arena never closes. Accessing a `MemorySegment` after its arena is closed throws `IllegalStateException` ‚Äî detecting use-after-free at the Java level instead of crashing the JVM. Spatial bounds: every segment carries base + size; any out-of-bounds dereference throws `IndexOutOfBoundsException`. Thread confinement on confined arenas prevents data races. This contrasts with `Unsafe.allocateMemory()` and JNI, where nothing prevents use-after-free or buffer overflows.	JNI (Java Native Interface) is the classic mechanism for calling C/C++ code from Java. Problems: (1) Requires writing C glue code with `JNIEnv*` pointer. (2) Error-prone memory management ‚Äî `GetStringUTFChars` must be paired with `ReleaseStringUTFChars`. (3) No type safety ‚Äî mismatched signatures cause hard crashes. (4) Performance overhead from JNI transitions (GC safepoint checks, thread state transitions). (5) JNI inhibits JIT inlining across the Java/native boundary. Panama (JEP 454, Java 22 final) replaces JNI with: type-safe `MethodHandle`-based downcalls, explicit `MemorySegment` management via finalized `Arena`, `jextract` for binding generation, and cheaper transitions that the JIT can reason about.
What is Project Valhalla and what does it aim to introduce to Java?	Project Valhalla introduces value types (primitive classes / value classes) ‚Äî objects that behave like primitives: no identity (no `==` identity comparison), stored inline by value (not by reference), no null. Benefits: (1) Eliminates boxing overhead for user-defined types (e.g., `Complex`, `Point`). (2) Arrays of value types are flat (no pointer indirection) ‚Äî much better cache locality. (3) "Codes like a class, works like an int." Also introduces Universal Generics ‚Äî generics over primitives (`List<int>`) without boxing. As of Java 23+, value classes are in preview. Will fundamentally change Java's performance and memory characteristics.
What was the Java Security Manager and why was it removed in Java 17/18?	The `SecurityManager` allowed Java code to define fine-grained access control policies (which files to read, which sockets to open, etc.) via a `Policy` and stack-based permission checks. It was deprecated in Java 17 (JEP 411) and removed in Java 24. Reasons: (1) High maintenance burden with major changes to every JVM release. (2) Rarely used in practice ‚Äî most security policies were misconfigured or overly permissive. (3) Modern alternatives (containers, OS capabilities, module system encapsulation) provide better isolation. (4) Performance overhead on every access check. Modern security: use OS-level isolation (containers/namespaces), module system, careful dependency management, and network policies.
How do you defend against insecure Java deserialization?	Java deserialization (`ObjectInputStream.readObject()`) is a critical attack vector ‚Äî crafting a malicious serialized payload can achieve RCE via "gadget chains" in common libraries (Apache Commons Collections, Spring, etc.). Defenses: (1) `ObjectInputFilter` (Java 9+) ‚Äî whitelist allowed classes: `ObjectInputFilter.Config.setSerialFilter(ObjectInputFilter.allowFilter(c -> c == MyClass.class, Status.REJECTED))`. Since Java 17, a configurable JVM-wide filter (`jdk.serialFilter` system property). (2) Avoid Java serialization entirely ‚Äî use JSON, Protobuf, or Avro. (3) Never deserialize data from untrusted sources. (4) Dependency scanning (OWASP Dependency-Check) to exclude known-Bad gadget-library versions. (5) `transient` on sensitive fields to reduce payload surface.
What is XXE (XML External Entity) injection and how do you prevent it in Java?	XXE occurs when an XML parser resolves external entity references (`<!ENTITY foo SYSTEM "file:///etc/passwd">`), allowing file disclosure, SSRF, or DoS. Fix `DocumentBuilderFactory`: `DocumentBuilderFactory dbf = DocumentBuilderFactory.newInstance(); dbf.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true); dbf.setFeature("http://xml.org/sax/features/external-general-entities", false); dbf.setFeature("http://xml.org/sax/features/external-parameter-entities", false); dbf.setXIncludeAware(false); dbf.setExpandEntityReferences(false);`. Same hardening needed for `SAXParserFactory`, `XMLInputFactory` (StAX), XSLT transformers, and Xerces. Modern Jackson XML and JAXB implementations typically disable XXE by default ‚Äî verify your versions.
What is SSRF and what mitigations are available in Java?	SSRF (Server-Side Request Forgery) occurs when attacker-controlled input makes the server perform HTTP/DNS requests to internal resources (cloud metadata at 169.254.169.254, internal services, localhost). Mitigations: (1) Allowlist: only permit requests to explicitly approved hostnames/IPs. (2) Resolve hostname to IP and validate against allowed CIDR ranges before connecting. (3) Use a dedicated egress proxy that enforces allowlisting. (4) Disable URL redirects if following redirected URLs. (5) Don't use user-supplied URLs directly with `HttpClient` or `URL.openConnection()`. (6) In cloud environments, block metadata endpoint at network level. Java doesn't have built-in SSRF protection ‚Äî it must be implemented in application logic.
What is STRIDE threat modeling and how would you apply it to a Java service?	STRIDE is a threat categorization framework for systematic threat identification: **S**poofing (authentication bypass ‚Äî mitigate: JWT validation, mTLS, strong authn), **T**ampering (data integrity ‚Äî mitigate: HMAC, digital signatures, HTTPS, parameterized queries), **R**epudiation (lack of audit trail ‚Äî mitigate: structured audit logging with immutable log store), **I**nformation Disclosure (data leak ‚Äî mitigate: encryption at rest/transit, minimal data exposure, least privilege), **D**enial of Service (resource exhaustion ‚Äî mitigate: rate limiting, circuit breakers, resource quotas), **E**levation of Privilege (access control bypass ‚Äî mitigate: RBAC, ABAC, method-level security). Apply: diagram data flows ‚Üí identify trust boundaries ‚Üí apply STRIDE to each element ‚Üí document threats + mitigations in a threat register.
What are the Java 24 post-quantum cryptography algorithms (JEP 496/497)?	Java 24 standardizes NIST-approved post-quantum cryptographic algorithms: **ML-KEM** (JEP 496, formerly CRYSTALS-Kyber) ‚Äî Module-Lattice Key Encapsulation Mechanism for key agreement; replaces RSA/ECDH for the quantum era. **ML-DSA** (JEP 497, formerly CRYSTALS-Dilithium) ‚Äî Module-Lattice Digital Signature Algorithm; replaces RSA/ECDSA for signing. Both are lattice-based algorithms believed to be secure against quantum computers (Shor's algorithm breaks RSA and ECC). Usage via standard JCA: `KeyPairGenerator.getInstance("ML-KEM-768")`, `KeyPairGenerator.getInstance("ML-DSA-65")`. Hybrid approaches (classical + PQC) recommended during transition. SLH-DSA (SPHINCS+) is provided in JDK 24 preview.
How do you implement structured audit logging for security events in Java?	Security audit logs must be tamper-evident, complete, and queryable. Principles: (1) Log who, what, when, result, source IP: use structured JSON (not string concatenation). (2) Log all authentication events (success, failure, lockout), privilege escalations, sensitive data access, configuration changes. (3) Never log PII, passwords, tokens, or card numbers ‚Äî log user IDs, redacted fields. (4) Use a dedicated logger (MDC with request ID, user ID): `MDC.put("userId", user.getId())`. (5) Ship logs to an immutable store (SIEM, append-only S3, Splunk, ELK) ‚Äî application should not be able to delete them. (6) Include a correlation/trace ID across microservice calls. Java: Logback + Logstash encoder for structured JSON, OpenTelemetry for trace IDs. Consider `@Audit` custom annotation + AOP to intercept sensitive method calls.
How do you scan for known vulnerable dependencies in Java projects?	Dependency scanning detects libraries with published CVEs before they reach production. Tools: (1) OWASP Dependency-Check ‚Äî Maven/Gradle plugin; checks dependencies against NVD (National Vulnerability Database); `mvn dependency-check:check`. (2) Snyk ‚Äî cloud and CLI tool; pays attention to transitive dependencies and license issues. (3) GitHub Dependabot ‚Äî creates pull requests when a dependency has a CVE. (4) Trivy ‚Äî container and filesystem scanner that reads POM/Gradle files. Integrate into CI pipeline as a build-breaking quality gate. Review and update regularly ‚Äî most exploited vulnerabilities have patches available days before attacks occur.
What is supply chain security for Java applications and what is an SBOM?	Supply chain attacks target upstream dependencies (e.g., malicious NPM/Maven package, SolarWinds-style tampering). Mitigations: (1) SBOM (Software Bill of Materials) ‚Äî a formal list of all components in an application: name, version, license, origin. Formats: SPDX (ISO standard), CycloneDX. Tools: `cyclonedx-maven-plugin` generates CycloneDX SBOMs. (2) Artifact signing ‚Äî verify Maven Central artifacts with GPG signatures; use Sigstore for keyless signing. (3) Private artifact repositories (Sonatype Nexus, JFrog Artifactory) ‚Äî proxy and scan before caching. (4) Dependency pinning ‚Äî pin transitive dependency versions. (5) Reproducible builds ‚Äî verify the same source always produces the same binary.
How do you configure mTLS (mutual TLS) in Java?	mTLS requires both server and client to present and verify X.509 certificates ‚Äî stronger than one-way TLS. Server configuration: set `clientAuth` to `need` (require a client cert) or `want`. In Spring Boot: `server.ssl.client-auth=need`, `server.ssl.trust-store=classpath:truststore.p12`, `server.ssl.trust-store-password=...`. Java client: configure `SSLContext` with a `KeyManager` loaded from a keystore containing the client's private key and certificate: `KeyManagerFactory.getInstance("SunX509"); kmf.init(keyStore, password)`. Use with `HttpClient.newBuilder().sslContext(sslContext).build()`. Use case: service-to-service auth in microservices ‚Äî no bearer tokens needed, certificate acts as the credential.
What are Java serialization gadget chains and what CVEs have exploited them?	A gadget chain exploits the fact that `ObjectInputStream.readObject()` invokes methods on deserialized objects before the application code can validate them. Attackers craft a malicious byte stream that, when deserialized, chains together innocent library classes (gadgets) whose method calls trigger arbitrary code execution ‚Äî typically via `Runtime.exec()`. Famous exploits: **Apache Commons Collections** gadget chain (2015, CVE-2015-4852 and many more) ‚Äî `InvokerTransformer` chains; affected WebLogic, JBoss, Jenkins, WebSphere. **Log4Shell (CVE-2021-44228)** ‚Äî used JNDI lookup, not Java serialization directly, but triggered by deserialization of `LogEvent`s in Log4j2. **Spring4Shell (CVE-2022-22965)** ‚Äî data binding vulnerability, not serialization. Mitigation: `ObjectInputFilter` whitelist; `jvm.serialFilter` system property (Java 9+); replace Java serialization entirely (Jackson, Protobuf); use agent-based detection (Serial Killer, NotSoSerial).
What penetration testing tools and concepts are relevant for Java web applications?	DAST (Dynamic Application Security Testing) tools test a running application from outside. **OWASP ZAP (Zed Attack Proxy)** ‚Äî open-source; spider + active scanner; intercept proxy; scriptable (Groovy, Python); good for CI integration with `zap-baseline-scan.sh`. **Burp Suite** ‚Äî industry-standard proxy; Repeater (replay/modify requests), Intruder (fuzzing), Scanner (paid). **Common test categories for Java apps:** SQL injection via query params/JSON bodies; XXE via XML uploads; deserialization via RMI/JMX/custom endpoints; SSRF via URL parameters consumed by `HttpClient`; path traversal in file upload handlers; CSRF on stateful APIs; JWT `alg:none` / weak secret; Spring Boot Actuator endpoints exposed without auth (`/env`, `/heapdump`). **Shift-left approach:** integrate SAST (Semgrep, SpotBugs + FindSecBugs) and SCA (OWASP Dependency-Check) in CI to catch issues before DAST.	FIPS 140-2 and 140-3 are US government standards specifying approved cryptographic algorithms and module requirements. Compliance is required for US government contracts and regulated industries (finserv, healthcare). In Java: use a FIPS-certified JCE provider such as Bouncy Castle FIPS, IBM JSSE FIPS, or the Red Hat OpenSSL FIPS provider (for RHEL/OpenJDK). Forbidden in FIPS mode: MD5 for security purposes, RC4, DES, 3DES, RSA keys < 2048-bit, TLS 1.0/1.1. Using BC FIPS: add `BouncyCastleFipsProvider` via `Security.addProvider()` or configure via `java.security` file. Always validate with a lab-accredited certification ‚Äî `FIPS mode enabled` logging alone is not compliance proof.
What is the fundamental difference between platform threads and virtual threads in Java 21+?	Platform threads are OS threads ‚Äî one JVM thread maps to one OS thread. They are expensive to create (~1 MB stack, OS context-switch overhead), so thread pools limit concurrency. Virtual threads are JVM-managed, extremely lightweight threads (a few hundred bytes when not running). They are multiplexed onto a small pool of OS "carrier" threads by the JVM scheduler. You can create millions of virtual threads. When a virtual thread blocks (I/O, sleep, lock), the carrier thread is released and can run another virtual thread ‚Äî making the CPU never idle waiting on I/O.
How do you create and use virtual threads?	Several ways: (1) `Thread.ofVirtual().start(runnable)` ‚Äî create and immediately start. (2) `Thread.ofVirtual().name("worker").unstarted(runnable)` ‚Äî create without starting. (3) `Executors.newVirtualThreadPerTaskExecutor()` ‚Äî an `ExecutorService` that creates a new virtual thread per submitted task (don't pool them ‚Äî they're cheap). (4) `Thread.startVirtualThread(runnable)` ‚Äî shorthand. Virtual threads are daemon threads by default. Naming: `Thread.ofVirtual().name("req-", 0)` names threads `req-0`, `req-1`, etc.
What is virtual thread pinning and how do you detect it?	Pinning occurs when a virtual thread is prevented from releasing its carrier thread while blocked. This defeats the purpose of virtual threads and reduces concurrency. Causes of pinning: (1) `synchronized` blocks or methods ‚Äî the carrier is pinned for the full duration. (2) `Object.wait()` inside a `synchronized` block. Detect pinning: start JVM with `-Djdk.tracePinnedThreads=full` (logs a stack trace whenever a virtual thread pins) or `-Djdk.tracePinnedThreads=short`. Fix by replacing `synchronized` with `ReentrantLock` (which correctly unmounts the virtual thread on `lock()`). Note: Java 24+ partially addresses the `synchronized` pinning issue.
Why should you NOT pool virtual threads?	Virtual threads are designed to be created per task and discarded ‚Äî they are so cheap (~few hundred bytes when parked) that pooling adds overhead without benefit. Thread pools were invented to amortize the high cost of OS thread creation. A `VirtualThreadPerTaskExecutor` creates a new virtual thread for each task. Pooling virtual threads (e.g., using `newFixedThreadPool` with virtual threads) actively reduces concurrency by artificially limiting the number of tasks that can run and blocking threads unnecessarily.
How does blocking I/O work with virtual threads?	When a virtual thread performs blocking I/O (file read, HTTP call, JDBC query, `Thread.sleep()`), the JVM intercepts the system call, unmounts the virtual thread from its carrier, and parks the virtual thread. The carrier thread is then free to execute other virtual threads. When the I/O completes, the virtual thread is rescheduled onto an available carrier. This happens transparently ‚Äî the code uses ordinary blocking APIs (no async/await, no callbacks, no reactive operators needed). The effective model is synchronous code with asynchronous efficiency.
What is Structured Concurrency (Java 24+) and what problem does it solve?	Structured Concurrency (`java.util.concurrent.StructuredTaskScope`) ensures concurrent subtasks have the same lifetime as the scope that spawned them ‚Äî they cannot outlive the enclosing block. This prevents "orphaned" threads running after the parent exits. Two built-in policies: `ShutdownOnFailure` ‚Äî if any subtask fails, cancel all others and propagate the exception; use for "all must succeed". `ShutdownOnSuccess` ‚Äî as soon as any subtask succeeds, cancel others and return the result; use for "first wins" (e.g., call two services, take fastest). Example: `try (var scope = new StructuredTaskScope.ShutdownOnFailure()) { var a = scope.fork(taskA); var b = scope.fork(taskB); scope.join().throwIfFailed(); return combine(a.get(), b.get()); }`.
What are Scoped Values (Java 24+) and why are they preferred over `ThreadLocal` with virtual threads?	`ScopedValue<T>` provides immutable, thread-local-like values that are bound for the duration of a code block and automatically inherited by child threads within a `StructuredTaskScope`. API: `ScopedValue.where(KEY, value).run(() -> { /* KEY.get() available here and in child tasks */ })`. Advantages over `ThreadLocal`: (1) Immutable ‚Äî no accidental mutation. (2) Automatically scoped ‚Äî no `remove()` needed (no memory leaks in thread pools). (3) GC-friendly ‚Äî values eligible for collection when scope exits. (4) Inherited correctly by structured concurrency child tasks. (5) Much better performance under virtual threads (no per-thread map overhead).
What should you avoid to get the most from virtual threads?	(1) Avoid `synchronized` on hot paths ‚Äî replace with `ReentrantLock`. (2) Avoid CPU-intensive work in virtual threads ‚Äî they are optimised for I/O-bound tasks; use a dedicated platform-thread pool for CPU work. (3) Don't pool virtual threads ‚Äî use `newVirtualThreadPerTaskExecutor()`. (4) Avoid `ThreadLocal` for mutable state ‚Äî use `ScopedValue` instead. (5) Don't use `Thread.currentThread().isVirtual()` to branch logic ‚Äî code should be unaware and thread-agnostic. (6) Existing synchronized JDBC drivers may pin ‚Äî check driver compatibility.
How does Spring MVC process an HTTP request?	(1) Request arrives at `DispatcherServlet` ‚Äî the single front controller registered with the Servlet container. (2) `DispatcherServlet` consults `HandlerMapping` to find the right `@Controller` method based on URL, HTTP method, headers, etc. (3) `HandlerAdapter` invokes the handler method, processing `@RequestParam`, `@PathVariable`, `@RequestBody` etc. (4) Method returns a view name (for MVC) or writes directly to the response (for `@RestController` using `HttpMessageConverter` to serialize to JSON/XML). (5) `DispatcherServlet` renders the view if needed.
What does Spring Boot auto-configuration do?	Spring Boot's auto-configuration (`@EnableAutoConfiguration`, part of `@SpringBootApplication`) scans the classpath and automatically configures Spring beans based on what's present. Example: if Jackson is on the classpath, `MappingJackson2HttpMessageConverter` is configured. If Spring Data JPA + a datasource are present, `JpaRepositories` are set up. Each auto-configuration class is annotated with `@ConditionalOnClass`, `@ConditionalOnMissingBean`, etc. You can override by defining your own beans (presence of your bean disables the auto-configured one) or with `application.properties` / `application.yml`. View what was configured: `--debug` flag shows auto-configuration report.
What is the difference between `@Controller` and `@RestController`?	`@RestController` is a convenience annotation that combines `@Controller` (registers the class as a Spring MVC controller) and `@ResponseBody` (each handler method's return value is written directly to the HTTP response body via a `HttpMessageConverter` ‚Äî typically JSON). With plain `@Controller`, without `@ResponseBody`, the return value is interpreted as a view name for template resolution. Use `@RestController` for REST APIs; use `@Controller` for MVC apps rendering HTML templates (Thymeleaf, Freemarker).
How does Spring Security's filter chain work?	Spring Security inserts a `DelegatingFilterProxy` into the Servlet filter chain before any servlet processing. This delegates to a Spring-managed `FilterChainProxy` which contains an ordered chain of security filters. Key filters: `SecurityContextPersistenceFilter` (loads/saves `SecurityContext` from session), `UsernamePasswordAuthenticationFilter` (handles login), `JwtAuthFilter` (custom ‚Äî validates JWT and sets `SecurityContext`), `ExceptionTranslationFilter` (converts `AccessDeniedException` / `AuthenticationException` to HTTP responses), `FilterSecurityInterceptor` (authorization checks). Configure with `SecurityFilterChain` bean and `HttpSecurity` fluent builder.
How does `@Transactional` work in Spring?	`@Transactional` on a method (or class) wraps it in a database transaction managed by Spring's `PlatformTransactionManager`. When a `@Transactional` method is called through a Spring proxy, Spring begins a transaction before the method executes and commits (or rolls back on exception) after it returns. Default rollback on unchecked exceptions (`RuntimeException`, `Error`); checked exceptions do NOT trigger rollback by default ‚Äî override with `rollbackFor = IOException.class`. Key attributes: `propagation` (REQUIRED, REQUIRES_NEW, NESTED), `isolation` (READ_COMMITTED, SERIALIZABLE), `readOnly = true` (optimization for queries), `timeout`.
What is the difference between eager and lazy loading in JPA?	Eager loading (`FetchType.EAGER`) loads the related entities immediately when the parent entity is loaded. Lazy loading (`FetchType.LAZY`) loads the related entities on first access (via a proxy). Default: `@ManyToOne` and `@OneToOne` are EAGER; `@OneToMany` and `@ManyToMany` are LAZY. Pitfall: LazyInitializationException ‚Äî accessing a lazy collection outside a Hibernate session (e.g., after the transaction closes). Solutions: `@EntityGraph` to fetch eagerly per query, `JOIN FETCH` in JPQL, or `@Transactional` on the calling method to keep the session open. Avoid `FetchType.EAGER` on collections ‚Äî it causes N+1 problems.
What is Project Reactor and what is the difference between `Mono` and `Flux`?	Project Reactor is a reactive programming library for the JVM based on the Reactive Streams specification. `Mono<T>` represents an asynchronous sequence of 0 or 1 element. `Flux<T>` represents an asynchronous sequence of 0 to N elements. Both are lazy ‚Äî nothing happens until you subscribe. Key operators: `map`, `flatMap`, `filter`, `zip`, `merge`, `concatMap` (sequential flatMap), `buffer`, `window`, `onErrorResume`, `retry`. Subscribe: `flux.subscribe(onNext, onError, onComplete)` or block (avoid): `mono.block()`. Spring WebFlux is the reactive web framework built on Reactor.
What is Spring WebFlux and when should you choose it over Spring MVC?	Spring WebFlux is the reactive, non-blocking web framework for Spring. Built on Netty (or other non-blocking servers) and Project Reactor. Benefit: handles many concurrent requests with a small thread pool ‚Äî no thread-per-request; threads don't block on I/O. Use when: (1) High concurrency with I/O-bound work (many parallel service calls). (2) Streaming responses (SSE, WebSocket). (3) You're comfortable with reactive programming. Don't switch for: CPU-bound work, existing blocking code/libraries (e.g., regular JDBC), simpler applications ‚Äî Spring MVC + virtual threads (Java 21+) often achieves similar throughput with less complexity.
What is gRPC and how does it work in Java?	gRPC is a high-performance RPC framework by Google using HTTP/2 and Protocol Buffers (Protobuf). You define services and messages in `.proto` files; the `protoc` compiler with the gRPC plugin generates Java stubs. Communication types: unary (request/response), server streaming, client streaming, bidirectional streaming. Java library: `grpc-java`. Server: extends generated `*Grpc.*ImplBase`. Client: creates a `ManagedChannel` and a blocking or async stub. Advantages over REST: strongly-typed contracts, binary encoding (smaller, faster), HTTP/2 multiplexing, built-in streaming. Disadvantage: not human-readable, harder to test with curl/browser, requires Protobuf tooling.
What is the difference between WebSockets and Server-Sent Events (SSE)?	WebSockets: full-duplex, bidirectional, persistent TCP connection. Either side can send messages at any time. Use for: real-time chat, collaborative editing, games. Higher complexity; uses `ws://` / `wss://`. Server-Sent Events (SSE): unidirectional, server ‚Üí client, over standard HTTP. Client opens a connection; server pushes events when available. Use for: live dashboards, notifications, streaming API responses. Simpler (standard HTTP, `text/event-stream`); automatic reconnect built into the browser; multiplexed over HTTP/2. In Spring: `SseEmitter` for SSE, `@MessageMapping` + STOMP over WebSocket for WebSocket.
What are the common strategies for REST API versioning?	(1) URI path versioning ‚Äî `GET /v1/users`, `GET /v2/users`. Simple, cacheable, visible in logs. Most widely adopted. Drawback: the version is in the URL which is technically part of the resource identity. (2) Accept header versioning ‚Äî `Accept: application/vnd.myapp.v2+json`. Purist REST approach; keeps URLs clean. Harder to test directly in browser. (3) Query parameter ‚Äî `GET /users?version=2`. Easy to implement but adds noise. (4) Custom header ‚Äî `X-API-Version: 2`. Cleanest URLs, but non-standard. Best practice: support at least two versions simultaneously; deprecate with a `Deprecation` or `Sunset` response header; document migration path.
How do you integrate JWT authentication in Spring Security?	Add a custom filter that runs before `UsernamePasswordAuthenticationFilter`: extend `OncePerRequestFilter`, extract the `Authorization: Bearer <token>` header, validate the token (signature, expiry, claims) using a library like `io.jsonwebtoken` (JJWT) or `com.nimbusds`, then construct an `UsernamePasswordAuthenticationToken` with the principal and authorities and set it in `SecurityContextHolder`. Register in your `SecurityFilterChain`: `http.addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)`. Method-level security: enable with `@EnableMethodSecurity` on the config class; use `@PreAuthorize("hasRole('ADMIN')")` or SpEL expressions (`@PreAuthorize("#userId == authentication.principal.id")`) on `@Service` or `@RestController` methods. Common pitfalls: not validating token signature, trusting `alg:none` claims, missing expiry check.
What is JAX-RS (Jakarta REST) and how do you define endpoints?	JAX-RS (now Jakarta REST, spec 3.x) is the standard Java annotation-based API for building RESTful services. Core annotations: `@Path("/users/{id}")` ‚Äî maps URI path (on class and/or method). `@GET`, `@POST`, `@PUT`, `@DELETE`, `@PATCH` ‚Äî HTTP method binding. `@Produces(MediaType.APPLICATION_JSON)` ‚Äî declares response content type. `@Consumes(MediaType.APPLICATION_JSON)` ‚Äî restricts accepted request body type. `@PathParam("id")` ‚Äî binds URI template variable to parameter. `@QueryParam("page")` ‚Äî binds query string parameter. `@RequestBody` / `@BeanParam` ‚Äî bind request body (impl-specific). Implementations: Jersey (reference RI), RESTEasy (WildFly), CXF. In Quarkus/Micronaut, JAX-RS annotations work natively. JAX-RS resource classes are CDI beans ‚Äî can `@Inject` services directly.	Rate limiting protects services from overload and abuse. Strategies: (1) Token bucket / leaky bucket algorithms ‚Äî allow bursting up to a limit, then throttle. (2) Fixed window counter ‚Äî reset a counter each time window expires. (3) Sliding window log ‚Äî tracks individual request timestamps. Libraries: Resilience4j `RateLimiter`, Bucket4j (token-bucket with Hazelcast/Redis backend for distributed limiting), Spring Cloud Gateway built-in rate filter. In a distributed system, the counter must be shared ‚Äî use Redis with atomic INCR/EXPIRY. HTTP 429 Too Many Requests with `Retry-After` header is the standard response. API gateway (NGINX, AWS API Gateway) can enforce limits before traffic hits application code.
