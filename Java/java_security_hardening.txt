# Java Security Hardening & Threat Modeling
# Tier: Expert
# Format: Front[TAB]Back

What was the Java Security Manager and why was it removed in Java 17/18?	The `SecurityManager` allowed Java code to define fine-grained access control policies (which files to read, which sockets to open, etc.) via a `Policy` and stack-based permission checks. It was deprecated in Java 17 (JEP 411) and removed in Java 24. Reasons: (1) High maintenance burden with major changes to every JVM release. (2) Rarely used in practice — most security policies were misconfigured or overly permissive. (3) Modern alternatives (containers, OS capabilities, module system encapsulation) provide better isolation. (4) Performance overhead on every access check. Modern security: use OS-level isolation (containers/namespaces), module system, careful dependency management, and network policies.
How do you defend against insecure Java deserialization?	Java deserialization (`ObjectInputStream.readObject()`) is a critical attack vector — crafting a malicious serialized payload can achieve RCE via "gadget chains" in common libraries (Apache Commons Collections, Spring, etc.). Defenses: (1) `ObjectInputFilter` (Java 9+) — whitelist allowed classes: `ObjectInputFilter.Config.setSerialFilter(ObjectInputFilter.allowFilter(c -> c == MyClass.class, Status.REJECTED))`. Since Java 17, a configurable JVM-wide filter (`jdk.serialFilter` system property). (2) Avoid Java serialization entirely — use JSON, Protobuf, or Avro. (3) Never deserialize data from untrusted sources. (4) Dependency scanning (OWASP Dependency-Check) to exclude known-Bad gadget-library versions. (5) `transient` on sensitive fields to reduce payload surface.
What is XXE (XML External Entity) injection and how do you prevent it in Java?	XXE occurs when an XML parser resolves external entity references (`<!ENTITY foo SYSTEM "file:///etc/passwd">`), allowing file disclosure, SSRF, or DoS. Fix `DocumentBuilderFactory`: `DocumentBuilderFactory dbf = DocumentBuilderFactory.newInstance(); dbf.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true); dbf.setFeature("http://xml.org/sax/features/external-general-entities", false); dbf.setFeature("http://xml.org/sax/features/external-parameter-entities", false); dbf.setXIncludeAware(false); dbf.setExpandEntityReferences(false);`. Same hardening needed for `SAXParserFactory`, `XMLInputFactory` (StAX), XSLT transformers, and Xerces. Modern Jackson XML and JAXB implementations typically disable XXE by default — verify your versions.
What is SSRF and what mitigations are available in Java?	SSRF (Server-Side Request Forgery) occurs when attacker-controlled input makes the server perform HTTP/DNS requests to internal resources (cloud metadata at 169.254.169.254, internal services, localhost). Mitigations: (1) Allowlist: only permit requests to explicitly approved hostnames/IPs. (2) Resolve hostname to IP and validate against allowed CIDR ranges before connecting. (3) Use a dedicated egress proxy that enforces allowlisting. (4) Disable URL redirects if following redirected URLs. (5) Don't use user-supplied URLs directly with `HttpClient` or `URL.openConnection()`. (6) In cloud environments, block metadata endpoint at network level. Java doesn't have built-in SSRF protection — it must be implemented in application logic.
What is STRIDE threat modeling and how would you apply it to a Java service?	STRIDE is a threat categorization framework for systematic threat identification: **S**poofing (authentication bypass — mitigate: JWT validation, mTLS, strong authn), **T**ampering (data integrity — mitigate: HMAC, digital signatures, HTTPS, parameterized queries), **R**epudiation (lack of audit trail — mitigate: structured audit logging with immutable log store), **I**nformation Disclosure (data leak — mitigate: encryption at rest/transit, minimal data exposure, least privilege), **D**enial of Service (resource exhaustion — mitigate: rate limiting, circuit breakers, resource quotas), **E**levation of Privilege (access control bypass — mitigate: RBAC, ABAC, method-level security). Apply: diagram data flows → identify trust boundaries → apply STRIDE to each element → document threats + mitigations in a threat register.
What are the Java 24 post-quantum cryptography algorithms (JEP 496/497)?	Java 24 standardizes NIST-approved post-quantum cryptographic algorithms: **ML-KEM** (JEP 496, formerly CRYSTALS-Kyber) — Module-Lattice Key Encapsulation Mechanism for key agreement; replaces RSA/ECDH for the quantum era. **ML-DSA** (JEP 497, formerly CRYSTALS-Dilithium) — Module-Lattice Digital Signature Algorithm; replaces RSA/ECDSA for signing. Both are lattice-based algorithms believed to be secure against quantum computers (Shor's algorithm breaks RSA and ECC). Usage via standard JCA: `KeyPairGenerator.getInstance("ML-KEM-768")`, `KeyPairGenerator.getInstance("ML-DSA-65")`. Hybrid approaches (classical + PQC) recommended during transition. SLH-DSA (SPHINCS+) is provided in JDK 24 preview.
How do you implement structured audit logging for security events in Java?	Security audit logs must be tamper-evident, complete, and queryable. Principles: (1) Log who, what, when, result, source IP: use structured JSON (not string concatenation). (2) Log all authentication events (success, failure, lockout), privilege escalations, sensitive data access, configuration changes. (3) Never log PII, passwords, tokens, or card numbers — log user IDs, redacted fields. (4) Use a dedicated logger (MDC with request ID, user ID): `MDC.put("userId", user.getId())`. (5) Ship logs to an immutable store (SIEM, append-only S3, Splunk, ELK) — application should not be able to delete them. (6) Include a correlation/trace ID across microservice calls. Java: Logback + Logstash encoder for structured JSON, OpenTelemetry for trace IDs. Consider `@Audit` custom annotation + AOP to intercept sensitive method calls.

# === Dependency & Supply Chain ===
How do you scan for known vulnerable dependencies in Java projects?	Dependency scanning detects libraries with published CVEs before they reach production. Tools: (1) OWASP Dependency-Check — Maven/Gradle plugin; checks dependencies against NVD (National Vulnerability Database); `mvn dependency-check:check`. (2) Snyk — cloud and CLI tool; pays attention to transitive dependencies and license issues. (3) GitHub Dependabot — creates pull requests when a dependency has a CVE. (4) Trivy — container and filesystem scanner that reads POM/Gradle files. Integrate into CI pipeline as a build-breaking quality gate. Review and update regularly — most exploited vulnerabilities have patches available days before attacks occur.
What is supply chain security for Java applications and what is an SBOM?	Supply chain attacks target upstream dependencies (e.g., malicious NPM/Maven package, SolarWinds-style tampering). Mitigations: (1) SBOM (Software Bill of Materials) — a formal list of all components in an application: name, version, license, origin. Formats: SPDX (ISO standard), CycloneDX. Tools: `cyclonedx-maven-plugin` generates CycloneDX SBOMs. (2) Artifact signing — verify Maven Central artifacts with GPG signatures; use Sigstore for keyless signing. (3) Private artifact repositories (Sonatype Nexus, JFrog Artifactory) — proxy and scan before caching. (4) Dependency pinning — pin transitive dependency versions. (5) Reproducible builds — verify the same source always produces the same binary.
How do you configure mTLS (mutual TLS) in Java?	mTLS requires both server and client to present and verify X.509 certificates — stronger than one-way TLS. Server configuration: set `clientAuth` to `need` (require a client cert) or `want`. In Spring Boot: `server.ssl.client-auth=need`, `server.ssl.trust-store=classpath:truststore.p12`, `server.ssl.trust-store-password=...`. Java client: configure `SSLContext` with a `KeyManager` loaded from a keystore containing the client's private key and certificate: `KeyManagerFactory.getInstance("SunX509"); kmf.init(keyStore, password)`. Use with `HttpClient.newBuilder().sslContext(sslContext).build()`. Use case: service-to-service auth in microservices — no bearer tokens needed, certificate acts as the credential.

What are Java serialization gadget chains and what CVEs have exploited them?	A gadget chain exploits the fact that `ObjectInputStream.readObject()` invokes methods on deserialized objects before the application code can validate them. Attackers craft a malicious byte stream that, when deserialized, chains together innocent library classes (gadgets) whose method calls trigger arbitrary code execution — typically via `Runtime.exec()`. Famous exploits: **Apache Commons Collections** gadget chain (2015, CVE-2015-4852 and many more) — `InvokerTransformer` chains; affected WebLogic, JBoss, Jenkins, WebSphere. **Log4Shell (CVE-2021-44228)** — used JNDI lookup, not Java serialization directly, but triggered by deserialization of `LogEvent`s in Log4j2. **Spring4Shell (CVE-2022-22965)** — data binding vulnerability, not serialization. Mitigation: `ObjectInputFilter` whitelist; `jvm.serialFilter` system property (Java 9+); replace Java serialization entirely (Jackson, Protobuf); use agent-based detection (Serial Killer, NotSoSerial).
What penetration testing tools and concepts are relevant for Java web applications?	DAST (Dynamic Application Security Testing) tools test a running application from outside. **OWASP ZAP (Zed Attack Proxy)** — open-source; spider + active scanner; intercept proxy; scriptable (Groovy, Python); good for CI integration with `zap-baseline-scan.sh`. **Burp Suite** — industry-standard proxy; Repeater (replay/modify requests), Intruder (fuzzing), Scanner (paid). **Common test categories for Java apps:** SQL injection via query params/JSON bodies; XXE via XML uploads; deserialization via RMI/JMX/custom endpoints; SSRF via URL parameters consumed by `HttpClient`; path traversal in file upload handlers; CSRF on stateful APIs; JWT `alg:none` / weak secret; Spring Boot Actuator endpoints exposed without auth (`/env`, `/heapdump`). **Shift-left approach:** integrate SAST (Semgrep, SpotBugs + FindSecBugs) and SCA (OWASP Dependency-Check) in CI to catch issues before DAST.	FIPS 140-2 and 140-3 are US government standards specifying approved cryptographic algorithms and module requirements. Compliance is required for US government contracts and regulated industries (finserv, healthcare). In Java: use a FIPS-certified JCE provider such as Bouncy Castle FIPS, IBM JSSE FIPS, or the Red Hat OpenSSL FIPS provider (for RHEL/OpenJDK). Forbidden in FIPS mode: MD5 for security purposes, RC4, DES, 3DES, RSA keys < 2048-bit, TLS 1.0/1.1. Using BC FIPS: add `BouncyCastleFipsProvider` via `Security.addProvider()` or configure via `java.security` file. Always validate with a lab-accredited certification — `FIPS mode enabled` logging alone is not compliance proof.
