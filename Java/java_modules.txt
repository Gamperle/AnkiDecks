# Java Module System (JPMS), Flow API & ServiceLoader
# Tier: Expert
# Format: Front[TAB]Back

What is JPMS and what problems does it solve?	JPMS (Java Platform Module System), introduced in Java 9 via Project Jigsaw, addresses: (1) **Classpath hell** — unclear which JARs provide which types; conflicting versions silently shadow each other. (2) **Encapsulation gaps** — all public classes on the classpath are accessible to all other code, including internal JDK APIs. (3) **Slow startup / large runtime** — the entire JDK was monolithic. JPMS introduces named modules: each has a `module-info.java` declaring what it requires and what it exports. Only exported packages are accessible to dependent modules, even if classes are `public`. Enables `jlink` to create minimal JDK runtimes.
What are the key directives in `module-info.java`?	A `module-info.java` lives at the source root and uses these directives: `requires <module>` — declares a dependency. `requires transitive <module>` — transitive: dependents of this module also see it (e.g., a library that exposes API types from another module). `exports <package>` — makes the package accessible to all modules. `exports <package> to <module>` — qualified export to a specific module only. `opens <package>` — allows full reflective access at runtime (needed for frameworks like Spring, Jackson). `opens <package> to <module>` — qualified open. `uses <service>` — declares this module consumes a service. `provides <service> with <impl>` — registers an implementation for `ServiceLoader`.
What is the difference between named, automatic, and unnamed modules?	(1) **Named module** — has a `module-info.class`; participates fully in the module system with explicit `requires`/`exports`. (2) **Automatic module** — a regular JAR on the module path (not classpath) without `module-info.class`. Gets a module name from its JAR filename or `Automatic-Module-Name` manifest entry. It exports all its packages and reads all other modules — a migration aid. (3) **Unnamed module** — code on the classpath. Reads all named and automatic modules, but named modules cannot `requires` the unnamed module. Allows old classpath code to call module code but not vice versa. Migration path: classpath → automatic module → named module.
How does `--add-opens` work and why is it sometimes needed?	`--add-opens <module>/<package>=<target-module>` opens a package for deep reflective access at runtime, bypassing strong encapsulation. Used when frameworks (Spring, Hibernate, Jackson, test runners) need to reflectively access JDK internals that are no longer `opens` by default. Example: `--add-opens java.base/java.lang=ALL-UNNAMED`. Add to JVM args in `maven-surefire-plugin`, `java` command, or `JAVA_TOOL_OPTIONS`. Also `--add-exports` for compile-time access to non-exported packages. Both are escape hatches; prefer migrating dependencies to module-aware versions. Java 17+ enforces strong encapsulation by default (JEP 396/403).
What is the Flow API (Java 9+) and what does it provide?	`java.util.concurrent.Flow` is the standard Java API for reactive streams (Reactive Streams specification). It defines four nested interfaces: `Flow.Publisher<T>` — produces items; `subscribe(Subscriber)` method. `Flow.Subscriber<T>` — consumes items; `onSubscribe`, `onNext`, `onError`, `onComplete`. `Flow.Subscription` — the link between Publisher and Subscriber; `request(n)` for back-pressure, `cancel()` to stop. `Flow.Processor<T,R>` — both Publisher and Subscriber (pipeline stage). Core concept: **back-pressure** — the subscriber controls the rate by calling `subscription.request(n)`. Java provides only `SubmissionPublisher<T>` as a concrete publisher. Project Reactor (Spring) / RxJava bridge to Flow via `FlowAdapters`.
How does `ServiceLoader` work and how does the module system integrate with it?	`ServiceLoader<S>` dynamically discovers implementations of a service interface at runtime. Classic usage: place a file `META-INF/services/<fully.qualified.Interface>` in a JAR containing the implementation class names (one per line). Module system usage: in `module-info.java`, the consumer declares `uses com.example.Plugin;` and the provider declares `provides com.example.Plugin with com.example.MyPlugin;`. Load: `ServiceLoader<Plugin> loader = ServiceLoader.load(Plugin.class); for (Plugin p : loader) { ... }` or `loader.findFirst()`. Benefits: loose coupling, replaceable implementations without recompilation. Used by the JDK itself (e.g., `java.nio.file.spi.FileSystemProvider`, JDBC drivers, `javax.xml` parsers).
What is a multi-release JAR and how does it work?	A multi-release JAR (JEP 238, Java 9+) allows a single JAR to contain version-specific class files for multiple Java versions. Structure: `META-INF/versions/11/com/example/Foo.class` overrides the root-level `com/example/Foo.class` when running on Java 11+. The `META-INF/MANIFEST.MF` must contain `Multi-Release: true`. Build support: Maven `maven-compiler-plugin` `<release>` config per version, Gradle `multiReleaseJar`. Use case: libraries that want to take advantage of newer API (e.g., `StackWalker`, `VarHandle`) while still supporting older JVMs. The root-level classes are the fallback for any JVM that doesn't understand multi-release (Java 8 and earlier).
