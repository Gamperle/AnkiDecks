What is type erasure in Java and what are its runtime implications?	At compile time, generics provide type-checking. At runtime, all generic type parameters are erased and replaced with their bounds (usually `Object`). `List<String>` becomes `List` at bytecode level; `List<Integer>` also becomes `List`. Implications: (1) Cannot use generics in `instanceof` checks: `x instanceof List<String>` is illegal (use `instanceof List<?>`). (2) Cannot create generic arrays: `new T[]` is illegal. (3) Cannot catch a generic type: `catch (MyException<T> e)` is illegal. (4) Static fields cannot be of type T. (5) Overloads only differing by type parameter won't compile (same erasure). Type information is retained in class metadata (via `getGenericSuperclass()`, reflection) but not in object instances themselves.
What are bridge methods and why does the compiler generate them?	When a class implements or extends a generic type, the compiler generates synthetic "bridge methods" to maintain binary compatibility with the erased base type. Example: `class IntBox implements Comparable<Integer>`. The interface requires `compareTo(Object)` (erased) but the class provides `compareTo(Integer)`. The compiler generates a bridge `compareTo(Object e) { return compareTo((Integer) e); }` that casts and delegates. This allows code compiled against the raw interface to call the typed implementation. Bridge methods are an implementation detail; you'll see them in `getDeclaredMethods()` with `isBridge() == true`.
What is PECS and when should you use wildcards?	PECS = Producer Extends, Consumer Super. It guides which wildcard to use when designing APIs: (1) If a collection is a **Producer** (you read values out of it) → use `? extends T`. The collection produces T values; you can safely read T references. Example: `void copy(List<? extends Number> src, ...)`. (2) If a collection is a **Consumer** (you write values into it) → use `? super T`. The collection can accept T values and any supertype's consumers work. Example: `void addNumbers(List<? super Integer> dest)`. (3) If you both read and write → use exact type `T` (no wildcard). Rule: you **cannot add** to a `? extends T` list (except null), and you can only read as `Object` from a `? super T` list.
What is a recursive (F-bounded) type bound and where is it used?	A recursive type bound constrains a type parameter using itself: `<T extends Comparable<T>>`. This says "T must be comparable to itself." The Java `Comparable` interface uses this: `interface Comparable<T>` combined with `T extends Comparable<T>` ensures type-safe comparison. Used in: (1) Builder patterns: `class Builder<B extends Builder<B>>` allows `this` references to return the concrete subtype. (2) Enum: `enum` inhericitly uses `Enum<E extends Enum<E>>`. (3) Fluent APIs where each method returns `this` typed to the concrete class. Common in APIs like Google Guava, Lombok `@SuperBuilder`.
What is heap pollution and what causes it?	Heap pollution occurs when a variable of a parameterized type refers to an object that is not of that type — violating the type safety guarantee. It typically happens via: (1) Raw types: `List rawList = new ArrayList<>(); rawList.add("hello"); List<Integer> nums = rawList; Integer x = nums.get(0); // ClassCastException`. (2) Unchecked casts: `(List<Integer>) someObject`. (3) Varargs with generics: `void foo(List<T>... lists)` — the varargs creates an array of the generic type, which is inherently unsafe. The `@SafeVarargs` annotation suppresses the unchecked warning when you've manually verified safety. At runtime, heap pollution → `ClassCastException` at an unexpected location.
How can you recover generic type information at runtime using reflection?	You can access retained generic type info via `java.lang.reflect.Type` and its subtypes. Methods: (1) `getClass().getGenericSuperclass()` returns a `ParameterizedType` if the superclass is generic: `((ParameterizedType) getClass().getGenericSuperclass()).getActualTypeArguments()[0]` gives the type argument. Pattern used by Gson's `TypeToken<List<String>> {}` (anonymous subclass) and Jackson's `TypeReference`. (2) `Method.getGenericParameterTypes()` and `getGenericReturnType()` — return type arguments declared in source. (3) `Field.getGenericType()`. Note: This works because class file metadata retains type signatures even after erasure. It does NOT work on instances — only on class structure.
What are the differences between `List<?>`, `List<Object>`, and raw `List`?	They are all different: (1) `List<?>` (unbounded wildcard) — a list of some unknown type. You can read as `Object`, but you cannot add anything except null (unsafe). Useful for code that doesn't care about the type: `printAll(List<?> list)`. (2) `List<Object>` — a list that holds any `Object`. You can add `String`, `Integer`, etc. But `List<String>` is NOT a `List<Object>` (generics are invariant). A method `void foo(List<Object> l)` won't accept a `List<String>`. (3) Raw `List` — no type parameter at all. Bypasses all generic checks; heap pollution risk; generates unchecked warnings. Only for interop with pre-Java 5 APIs. Summary: prefer `<?>` for read-only unknowns, `<T>` with bounds for flexibility, avoid raw types.

What is a bounded type parameter in Java generics?	Bounded type parameters restrict the types that can be used as type arguments. Upper bound: `<T extends Number>` — T must be Number or a subclass. Multiple bounds: `<T extends Comparable<T> & Serializable>`. Lower bound wildcard: `<? super Integer>` — accepts Integer, Number, or Object. Unbounded wildcard: `<?>`. Bounds enable calling methods of the bound type on T: `T extends Number` allows `t.doubleValue()`. Wildcards appear in usage (method signatures); type parameters appear in declaration (class/method declarations).
What are wildcards in Java generics and what are the three forms?	Wildcards (`?`) represent an unknown type in generic type arguments. Unbounded: `List<?>` — any type; can only add null, read as Object. Upper bounded: `List<? extends Number>` — Number or subtype; read as Number, cannot add (type unknown). Lower bounded: `List<? super Integer>` — Integer supertype; can add Integer/subtype, read as Object. Wildcards appear in usage, not declaration. Apply PECS (Producer Extends, Consumer Super) to choose the right form.
What is a generic method in Java?	A generic method introduces its own type parameter(s) independent of the class's type parameters: `public <T extends Comparable<T>> T max(T a, T b) { return a.compareTo(b) > 0 ? a : b; }`. Type parameters are declared before the return type. Type is inferred at the call site; explicit invocation: `Collections.<String>emptyList()`. Generic methods can exist in non-generic classes. Benefits: type-safe without casts, reusable for multiple types. Examples from JDK: `Collections.sort()`, `Collections.swap()`, `Optional.of()`.
