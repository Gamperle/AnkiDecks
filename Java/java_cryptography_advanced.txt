# === TLS / SSL ===
What happens during a TLS handshake?	(1) ClientHello — client sends supported TLS versions, cipher suites, a random nonce. (2) ServerHello — server selects TLS version, cipher suite, sends its certificate, its random nonce. (3) Certificate validation — client verifies the server certificate's chain of trust to a trusted root CA, signature, expiry, and hostname. (4) Key Exchange — e.g., ECDHE: both sides compute the same shared premaster secret without transmitting it. (5) Session keys derived — both sides derive symmetric encryption and MAC keys from the shared secret + nonces. (6) Finished — both sides send a HMAC of the handshake transcript to confirm integrity. All subsequent traffic is encrypted with the session keys.
How do you configure SSL/TLS in Java?	Use `SSLContext`: `SSLContext ctx = SSLContext.getInstance("TLS"); ctx.init(keyManagers, trustManagers, new SecureRandom());`. `KeyManager` holds the server's (or client's) certificate and private key — loaded from a `KeyStore`. `TrustManager` determines which server certificates to trust — loaded from a `TrustStore` (contains trusted CA certificates). Apply to `HttpsURLConnection`: `conn.setSSLSocketFactory(ctx.getSocketFactory())`. For `HttpClient` (Java 11+): `HttpClient.newBuilder().sslContext(ctx).build()`. For Spring Boot: configure via `server.ssl.*` properties.
What is the difference between a Keystore and a Truststore in Java?	A Keystore holds private keys and associated certificate chains — identifying YOU to others. Used when YOUR server needs to present a certificate (server auth) or when using mutual TLS (client cert auth). A Truststore holds trusted CA certificates — determining which certificates YOU trust. Used to validate the certificates presented by peers. In Java: both use `KeyStore` class; distinguished by role. JVM's default truststore is `$JAVA_HOME/lib/security/cacerts` (contains common root CAs). Formats: `JKS` (Java-specific, legacy), `PKCS12` (standard, recommended). Managed with `keytool`.

# === Certificates & PKI ===
What is a certificate chain and how is it validated?	A certificate chain is a sequence: your server cert → intermediate CA cert(s) → root CA cert. Each cert is signed by the private key of the cert above it. Validation: (1) Check each certificate's signature using the public key of the issuing CA. (2) Verify each cert is not expired (`notBefore`/`notAfter`). (3) Check certificate revocation (CRL or OCSP). (4) Verify the leaf cert's hostname matches the `Subject Alternative Name` (SAN) / Common Name. (5) Verify the root CA is in the truststore. If any step fails, the TLS connection is rejected.
How do you create digital signatures in Java?	`java.security.Signature`. Signing (with private key): `Signature sig = Signature.getInstance("SHA256withRSA"); sig.initSign(privateKey); sig.update(dataBytes); byte[] signature = sig.sign();`. Verification (with public key): `sig.initVerify(publicKey); sig.update(dataBytes); boolean valid = sig.verify(signature);`. Common algorithms: `SHA256withRSA`, `SHA256withECDSA` (faster, smaller signatures). Digital signatures provide authenticity (only the private-key holder could sign) and non-repudiation.

# === JCA / JCE Provider Model ===
What is the Java Cryptography Architecture (JCA) and its provider model?	JCA is the framework governing cryptographic operations in Java (`java.security`). It is algorithm-agnostic: applications request services by name (`MessageDigest.getInstance("SHA-256")`) and JCA dispatches to the appropriate provider. Providers (e.g., SunEC, SunJCE, BouncyCastle) register algorithm implementations. Add a provider: `Security.addProvider(new BouncyCastleProvider())` or list it in `java.security` config file. This enables swapping implementations for compliance (e.g., FIPS-certified providers) without changing application code.
What is ECDH key agreement and how is it used in Java?	ECDH (Elliptic Curve Diffie-Hellman) allows two parties to establish a shared secret over an insecure channel without transmitting the secret itself. Both parties generate ephemeral key pairs; they exchange public keys; each computes the same shared secret using the other's public key and their own private key. In Java: `KeyAgreement ka = KeyAgreement.getInstance("ECDH"); ka.init(myPrivateKey); ka.doPhase(theirPublicKey, true); byte[] sharedSecret = ka.generateSecret();`. Derive a symmetric key from the shared secret using HKDF or SHA-256. TLS uses a similar approach (ECDHE — ephemeral ECDH) for forward secrecy.

# === Key Derivation & Secrets ===
How do you derive a key from a password using PBKDF2 in Java?	`SecretKeyFactory skf = SecretKeyFactory.getInstance("PBKDF2WithHmacSHA256"); PBEKeySpec spec = new PBEKeySpec(password.toCharArray(), salt, 310_000, 256); byte[] keyBytes = skf.generateSecret(spec).getEncoded(); SecretKey key = new SecretKeySpec(keyBytes, "AES");`. Key parameters: salt — random 16+ bytes per credential (generated with `SecureRandom`); iterations — high enough that hashing takes ~100ms (tune to your hardware; OWASP recommends 600,000 for PBKDF2-HMAC-SHA256); output length — 256 bits for AES-256. NEVER hardcode salt; store it alongside the derived key.
What does AES-GCM authentication tag protect against?	AES-GCM (Galois/Counter Mode) is authenticated encryption: it produces a ciphertext AND an authentication tag. The tag is a MAC (Message Authentication Code) covering both the ciphertext and optional Additional Authenticated Data (AAD — like a header or context). If an attacker modifies even one bit of the ciphertext, IV, or AAD, tag verification fails and `javax.crypto.AEADBadTagException` is thrown — the data is rejected. This prevents chosen-ciphertext attacks, padding oracle attacks, and undetected tampering. Critical: never reuse an IV with the same key — IV reuse with GCM is catastrophic and breaks both confidentiality and authenticity.
What are the best practices for secrets management in Java applications?	(1) Never hardcode secrets (API keys, DB passwords, crypto keys) in source code or config files checked into version control. (2) Use environment variables or a secrets management service: HashiCorp Vault (`vault-java-driver`), AWS Secrets Manager, Azure Key Vault, GCP Secret Manager. (3) In Spring Boot: `@Value("${DB_PASSWORD}")` reads from environment; Spring Cloud Vault integrates Vault natively. (4) Rotate secrets regularly; automate rotation where possible. (5) Audit secret access. (6) In Kubernetes: use Secrets objects mounted as environment variables or files (not stored in env vars in pod specs in plaintext YAML). Seal with Sealed Secrets or external-secrets-operator.
