What is annotation processing and how does `AbstractProcessor` work?	Annotation processing runs at compile time via the Java Compiler Plugin (APT). Subclass `javax.annotation.processing.AbstractProcessor` and annotate it with `@SupportedAnnotationTypes("com.example.MyAnnotation")` and `@SupportedSourceVersion(SourceVersion.RELEASE_17)`. Override `process(Set<? extends TypeElement> annotations, RoundEnvironment roundEnv)`. Use `roundEnv.getElementsAnnotatedWith(MyAnnotation.class)` to get annotated elements (`TypeElement`, `MethodElement`, etc.). Generate source files via `processingEnv.getFiler().createSourceFile(className).openWriter()`. Register the processor in `META-INF/services/javax.annotation.processing.Processor` or via `@AutoService`. Examples: Lombok (code generation), Dagger (DI graph), MapStruct (mapper generation), Immutables.
How do JDK dynamic proxies work and what are their limitations?	JDK dynamic proxies created via `Proxy.newProxyInstance(classLoader, interfaces, handler)` generate a class at runtime that implements given interfaces. All method calls are dispatched to the `InvocationHandler.invoke(Object proxy, Method method, Object[] args)` method. Uses: AOP, logging, mocking (Mockito), JPA lazy loading. Limitations: (1) Can only proxy **interfaces** — not concrete classes. (2) Method must be declared in an interface to be intercepted. (3) Calls from within the target object to itself (`this.method()`) bypass the proxy. When a class (not interface) must be proxied, CGLIB or ByteBuddy generates a subclass proxy at runtime by subclassing with bytecode generation.
What is the difference between JDK proxy and CGLIB/ByteBuddy proxies?	JDK proxy: generates a class implementing specified interfaces; only intercepts interface methods; uses `InvocationHandler`; part of the JDK (`java.lang.reflect.Proxy`), no extra dependency. CGLIB (Code Generation Library): subclasses the target class; intercepts all non-final, non-private methods; requires CGLIB JAR (bundled in Spring). Used by Spring AOP when no interface exists. ByteBuddy: modern, powerful, type-safe bytecode library; used by Mockito 2+, Hibernate 5+. All have the same internal-call limitation — `this.method()` bypasses the proxy since the original instance doesn't know about the proxy wrapper. Spring recommends prefer-interface design and `@Transactional` on public methods for this reason.
What are GraalVM Native Image constraints and how do you handle reflection?	GraalVM Native Image performs AOT (ahead-of-time) compilation to a native binary. It uses closed-world analysis — it must know all classes, methods, and resources at build time. Dynamic features that break by default: reflection, JDK proxies, serialization, resources (`Class.forName()`, `getDeclaredMethods()`, etc.). Fix with **reachability metadata** JSON files: `reflect-config.json` (list classes/methods/fields used reflectively), `proxy-config.json` (list proxy interface combos), `resource-config.json`, `serialization-config.json`. Location: `META-INF/native-image/<groupId>/<artifactId>/`. Generate automatically with the GraalVM tracing agent: `java -agentlib:native-image-agent=config-output-dir=META-INF/native-image -jar myapp.jar`. Spring Boot 3+ has built-in Native Image support via `spring-aot-maven-plugin`.
What is `MethodHandles` and how does it differ from reflection?	`java.lang.invoke.MethodHandle` is a typed, directly-invocable ref to a method, constructor, or field. Obtained via `MethodHandles.lookup()`. Differences from reflection: (1) **Performance** — MethodHandles are JIT-optimizable; after warm-up they are as fast as direct calls. `Method.invoke()` involves repeated array boxing and unpacking. (2) **Type safety** — `MethodType` encodes exact signature; type mismatches fail at lookup time, not call time. (3) **Access control** — `Lookup` captures access context at creation point; respects encapsulation properly. Use: `MethodHandle mh = MethodHandles.lookup().findVirtual(String.class, "length", MethodType.methodType(int.class)); int len = (int) mh.invokeExact("hello");`. Used by Java internals, Groovy, Kotlin, and `invokedynamic`.
What is `invokedynamic` and how does Java use it?	`invokedynamic` (JEP 292, Java 7+) is a JVM bytecode instruction that defers the method resolution target to a **bootstrap method** defined by the programmer (or JDK). The bootstrap method returns a `CallSite` (the resolved target); subsequent invocations use that `CallSite` directly. Uses in the JDK: (1) **Lambda expressions** — compiled to `invokedynamic` with `LambdaMetafactory` as bootstrap; the actual class generation is deferred to runtime, avoiding class file proliferation. (2) **String concatenation** (Java 9+) — `StringConcatFactory` emits optimised assembly at runtime instead of `StringBuilder` chains. (3) **Kotlin** and **Groovy** dynamic dispatch. Benefits: the JIT can re-optimize `CallSite` targets as program behaviour is learned.

What is AOP (Aspect-Oriented Programming) and what are its core concepts?	AOP modularizes cross-cutting concerns (logging, security, transactions, caching) that span many classes, separating them from business logic. Core concepts: Aspect — the module of cross-cutting concern. Advice — code to execute (Before, After, AfterReturning, AfterThrowing, Around). Pointcut — expression matching join points where advice applies. Join point — specific point in execution (method call, field access). Weaving — applying aspects at compile time (AspectJ), load time (AspectJ LTW), or runtime (Spring AOP via proxies). Spring AOP supports only method-level join points; AspectJ supports field access and constructors.
What is the `VarHandle` API (Java 9+) and why was it introduced?	`VarHandle` (`java.lang.invoke`) provides typed references to variables (instance fields, array elements, static fields) with multiple memory access modes: plain, opaque, acquire/release (one-sided barriers), and volatile (full barrier). Enables atomic operations: `compareAndSet()`, `getAndAdd()`. Advantages over `sun.misc.Unsafe`: type-safe, bounds-checked, part of the public API. Advantages over `AtomicFieldUpdater`: faster (JIT-friendly), more access modes. Used in JDK's own concurrent data structures; replaces most legitimate uses of `Unsafe` in library code.
What is bytecode manipulation and what libraries support it?	Bytecode manipulation directly reads, modifies, or generates `.class` file bytecode, enabling AOP weaving, mocking, code generation, profiling, and hot patching. Libraries: ASM — low-level visitor API, minimal overhead, used by almost all other tools. Javassist — source-level API using Java source strings. ByteBuddy — fluent, type-safe API used by Mockito (mock generation), Hibernate (proxy creation), and many frameworks. CGLIB — generates subclasses dynamically, used by Spring/Hibernate for proxies. Manipulation can happen at compile time (via annotation processors), load time, or runtime (via agents).
