What is the difference between <b>process</b> and <b>thread</b> in Java?	A <b>process</b> is an independent program instance with its own dedicated memory space (heap, stack, code, data segments) managed by the OS. Processes are isolated — one crash doesn't affect others. A <b>thread</b> is a lightweight unit of execution <em>within</em> a process. All threads of a process share the same heap and code, but each has its own stack and program counter. Thread creation is cheaper than process creation. Java threads map to OS threads (platform threads) by default, or to virtual threads (Java 21+) managed by the JVM.

What does the <b>volatile</b> keyword guarantee in Java?	<b>volatile</b> provides two guarantees: <b>1) Visibility</b> — a write to a volatile variable is immediately flushed to main memory, and any subsequent read by another thread reads from main memory, not from a CPU cache. <b>2) Ordering</b> — prevents instruction reordering across the volatile access (happens-before guarantee). It does <b>NOT</b> guarantee atomicity for compound operations like <code>i++</code> (which is read-modify-write). Use <code>AtomicInteger</code> or <code>synchronized</code> for atomicity.

What is a <b>race condition</b>?	A race condition occurs when two or more threads access shared mutable data concurrently, and the final outcome depends on the unpredictable interleaving of their operations. Example: two threads both read <code>counter = 5</code>, both increment it, and both write <code>6</code> instead of the expected <code>7</code>. Race conditions are prevented using <code>synchronized</code>, <code>ReentrantLock</code>, or atomic classes. They are notoriously hard to reproduce because they depend on thread scheduling timing.

What is the difference between <b>synchronized method</b> and <b>synchronized block</b>?	A <b>synchronized method</b> acquires the intrinsic lock (monitor) on <code>this</code> (or the Class object for static methods) for the entire method duration — coarse-grained. A <b>synchronized block</b> acquires a lock on a specified object for only a portion of the code — fine-grained, reducing contention. Example: <code>synchronized(this) { ... }</code> or <code>synchronized(lockObject) { ... }</code>. Prefer synchronized blocks when only part of the method needs protection, improving throughput.

What is a <b>deadlock</b> and how can you prevent it?	A deadlock occurs when thread A holds lock X and waits for lock Y, while thread B holds lock Y and waits for lock X — both block forever. <b>Prevention strategies:</b> (1) <b>Lock ordering</b>: always acquire locks in the same global order. (2) <b>tryLock() with timeout</b>: <code>lock.tryLock(500, TimeUnit.MILLISECONDS)</code> to avoid indefinite waiting. (3) <b>Avoid nested locks</b>: minimize holding multiple locks simultaneously. (4) Use higher-level concurrency utilities (e.g., <code>java.util.concurrent</code>). Deadlocks can be detected with thread dumps using <code>jstack</code>.

What is the difference between <b>ReentrantLock</b> and <b>synchronized</b>?	Both provide mutual exclusion, but <b>ReentrantLock</b> offers additional capabilities: (1) <b>tryLock()</b> — non-blocking acquisition attempt. (2) <b>tryLock(timeout)</b> — timed acquisition. (3) <b>lockInterruptibly()</b> — interruptible waiting. (4) <b>Fairness policy</b> — constructor: <code>new ReentrantLock(true)</code> ensures longest-waiting thread gets the lock. (5) <b>Multiple Condition variables</b> via <code>lock.newCondition()</code>. Both are reentrant (same thread can re-acquire). <code>synchronized</code> is simpler and automatically released on exception; ReentrantLock must be unlocked in a <code>finally</code> block.

What is a <b>ThreadLocal</b> variable?	<b>ThreadLocal</b> provides each thread with its own independent copy of a variable, stored in the thread's own map (<code>Thread.threadLocals</code>). Used for per-thread state like database connections, user sessions, or date formatters without synchronization. Example: <code>ThreadLocal&lt;SimpleDateFormat&gt; sdf = ThreadLocal.withInitial(() -&gt; new SimpleDateFormat("yyyy-MM-dd"));</code>. <b>Warning:</b> always call <code>remove()</code> when done — especially in thread pool threads — to prevent memory leaks and stale data leaking across requests.

What is the <b>happens-before</b> relationship in Java Memory Model?	A <b>happens-before</b> (HB) relationship guarantees that all memory writes visible before action A are visible to action B. Key HB rules: (1) Program order within a thread. (2) Monitor unlock HB monitor re-lock on the same object. (3) <code>volatile</code> write HB subsequent volatile read of the same variable. (4) <code>Thread.start()</code> HB any action in the started thread. (5) All actions in a thread HB <code>Thread.join()</code> return. (6) Object construction HB finalizer. Without HB, the JVM or CPU may reorder or cache writes, making them invisible to other threads.

What is the purpose of <b>CountDownLatch</b>?	<b>CountDownLatch</b> is a synchronization aid initialized with a count. Threads call <code>await()</code> to block until the count reaches zero. Other threads call <code>countDown()</code> to decrement it. <b>Use cases:</b> (1) Wait for multiple services to initialize before starting. (2) Wait for N worker threads to complete before aggregating results. <b>Key limitation:</b> it is <b>not reusable</b> — once the count reaches zero it cannot be reset. Use <code>CyclicBarrier</code> or <code>Phaser</code> for reusable barriers.

What is the difference between <b>CountDownLatch</b> and <b>CyclicBarrier</b>?	<b>CountDownLatch:</b> One or more threads wait for N other threads to complete tasks. Count goes one way to zero. <b>Not reusable.</b> Threads calling <code>countDown()</code> don't wait themselves. <b>CyclicBarrier:</b> A fixed number of threads all wait for each other at a barrier point before proceeding together. <b>Reusable</b> — resets automatically after each cycle. An optional <code>Runnable</code> barrier action runs when the barrier trips. Ideal for iterative parallel algorithms (e.g., matrix computation phases).

What is a <b>Semaphore</b> in Java concurrency?	A <b>Semaphore</b> maintains a set of permits. <code>acquire()</code> blocks until a permit is available (and takes one); <code>release()</code> returns a permit. <b>Use cases:</b> (1) Limiting concurrent access to a resource pool (e.g., max 10 DB connections). (2) Producer-consumer flow control. Can be initialized with <b>fairness</b>: <code>new Semaphore(5, true)</code> to prevent thread starvation. Unlike locks, permits can be acquired and released by <em>different</em> threads.

What is the <b>Fork/Join framework</b>?	The Fork/Join framework (<code>java.util.concurrent.ForkJoinPool</code>) is designed for <b>divide-and-conquer</b> parallel algorithms. Tasks extend <code>RecursiveTask&lt;V&gt;</code> (returns a result) or <code>RecursiveAction</code> (no result). Tasks <b>fork</b> subtasks (submit asynchronously) and <b>join</b> their results. Uses <b>work-stealing</b>: idle threads steal tasks from the tail of other threads' queues, maximizing CPU utilization. Parallel streams use ForkJoinPool.commonPool() internally.

What is <b>CompletableFuture</b>?	<b>CompletableFuture&lt;T&gt;</b> (Java 8+) is an enhanced Future supporting: (1) <b>Explicit completion</b>: <code>complete(value)</code>, <code>completeExceptionally(ex)</code>. (2) <b>Chaining</b>: <code>thenApply()</code>, <code>thenCompose()</code>, <code>thenCombine()</code>, <code>thenAccept()</code>, <code>thenRun()</code>. (3) <b>Exception handling</b>: <code>exceptionally()</code>, <code>handle()</code>. (4) <b>Combining</b>: <code>allOf()</code>, <code>anyOf()</code>. (5) <b>Async variants</b>: <code>thenApplyAsync()</code> runs on a thread pool. By default uses ForkJoinPool.commonPool(). Avoids callback hell with a readable fluent pipeline.

What is the difference between <b>thenApply</b> and <b>thenCompose</b> in CompletableFuture?	<b>thenApply(Function&lt;T,U&gt;)</b>: transforms the result synchronously — analogous to <code>map()</code>. Returns <code>CompletableFuture&lt;U&gt;</code>. If the function itself returns a <code>CompletableFuture</code>, you get a nested <code>CompletableFuture&lt;CompletableFuture&lt;U&gt;&gt;</code>. <b>thenCompose(Function&lt;T, CompletableFuture&lt;U&gt;&gt;)</b>: chains a dependent async computation — analogous to <code>flatMap()</code>. Flattens the result to <code>CompletableFuture&lt;U&gt;</code>. Use <code>thenCompose</code> when the next step is itself async (e.g., another HTTP call), to avoid nesting.

What is a <b>blocking queue</b>? Name implementations.	A <b>BlockingQueue</b> adds blocking semantics to a queue: <code>put()</code> blocks if the queue is full (capacity-bounded), <code>take()</code> blocks if empty. Used in producer-consumer patterns. Implementations: <b>ArrayBlockingQueue</b> (bounded, array-backed, FIFO), <b>LinkedBlockingQueue</b> (optionally bounded, linked nodes), <b>PriorityBlockingQueue</b> (unbounded, priority-ordered), <b>SynchronousQueue</b> (zero capacity — each put must wait for a take), <b>DelayQueue</b> (elements become available after a delay), <b>LinkedTransferQueue</b> (high-performance, transfer semantics).

What is the <b>Executor framework</b>?	The Executor framework (<code>java.util.concurrent</code>) decouples <em>task definition</em> from <em>thread management</em>. Key types: <code>Executor</code> (simple execute(Runnable)), <code>ExecutorService</code> (lifecycle management + submit returning Future), <code>ScheduledExecutorService</code> (delayed/periodic tasks), <code>ThreadPoolExecutor</code> (fully configurable pool: core/max pool size, keep-alive, work queue, rejection policy). Factory class <code>Executors</code> provides convenient factory methods. Benefits: thread reuse, controlled concurrency, graceful shutdown.

What are the thread pool types available via <b>Executors</b> factory?	<b>newFixedThreadPool(n)</b>: fixed number of threads, unbounded queue — good for CPU-bound tasks. <b>newCachedThreadPool()</b>: unbounded threads created on demand, idle threads recycled after 60s — good for many short-lived tasks. <b>newSingleThreadExecutor()</b>: single thread, tasks queued sequentially — guarantees ordering. <b>newScheduledThreadPool(n)</b>: supports scheduling with fixed-rate/fixed-delay. <b>newWorkStealingPool()</b> (Java 8+): ForkJoinPool with parallelism = available processors, work-stealing. <b>Note:</b> for production use, prefer manually configured <code>ThreadPoolExecutor</code> with explicit queue bounds and rejection policies.

What is <b>livelock</b>?	A livelock occurs when threads continuously respond to each other's actions without making progress — unlike deadlock, the threads are <em>not blocked</em> but are busy doing useless work. Classic example: two threads each "politely" back off and retry simultaneously, forever. Prevention: introduce randomized back-off delays (e.g., Ethernet CSMA/CD collision resolution), or designate one thread as the "winner" via priority or ordering.

What does <b>AtomicInteger</b> provide over <code>volatile int</code>?	<code>volatile int</code> only guarantees visibility of a single read or write. It cannot atomically perform compound operations. <b>AtomicInteger</b> provides lock-free atomic compound operations using <b>CAS (Compare-And-Swap)</b> hardware instructions: <code>getAndIncrement()</code>, <code>incrementAndGet()</code>, <code>getAndAdd(delta)</code>, <code>compareAndSet(expected, update)</code>, <code>accumulateAndGet()</code>. These guarantee that read-modify-write sequences are atomic without using <code>synchronized</code>, making them highly performant in concurrent scenarios.

What is <b>CAS (Compare-And-Swap)</b>?	CAS is a hardware-level atomic instruction: given a memory address, an expected value, and a new value — it atomically updates the memory location to the new value <em>only if</em> it currently contains the expected value, returning success/failure. This is the foundation of lock-free programming. All <code>java.util.concurrent.atomic</code> classes use CAS via <code>sun.misc.Unsafe</code> (or <code>VarHandle</code> in Java 9+). <b>ABA problem:</b> CAS can incorrectly succeed if a value changes A→B→A; solved with <code>AtomicStampedReference</code>.

What is a <b>Stream</b> in Java?	A <b>Stream&lt;T&gt;</b> (Java 8+) is a declarative pipeline for processing sequences of elements. Key characteristics: (1) <b>Lazy evaluation</b> — intermediate operations build a pipeline; execution only starts when a terminal operation is called. (2) <b>Not a data structure</b> — doesn't store data; reads from a source (collection, array, I/O). (3) <b>Consumed once</b> — a stream cannot be reused after a terminal operation. (4) <b>Supports parallelism</b> via <code>parallelStream()</code> or <code>stream().parallel()</code>. (5) Primitive specializations: <code>IntStream</code>, <code>LongStream</code>, <code>DoubleStream</code> avoid boxing overhead.

What is the difference between <b>intermediate</b> and <b>terminal</b> operations in Streams?	<b>Intermediate operations</b> are lazy — they return a new Stream and don't execute until a terminal operation triggers the pipeline. Examples: <code>filter(Predicate)</code>, <code>map(Function)</code>, <code>flatMap()</code>, <code>sorted()</code>, <code>distinct()</code>, <code>limit(n)</code>, <code>skip(n)</code>, <code>peek()</code>. <b>Terminal operations</b> are eager — they trigger the pipeline, consume the stream, and produce a result or side effect. Examples: <code>collect()</code>, <code>forEach()</code>, <code>reduce()</code>, <code>count()</code>, <code>findFirst()</code>, <code>anyMatch()</code>, <code>toList()</code> (Java 16+). A stream can have many intermediate ops but only one terminal op.

What is the difference between <b>map</b> and <b>flatMap</b> in Streams?	<b>map(Function&lt;T,R&gt;)</b>: transforms each element to exactly one element. Result: <code>Stream&lt;R&gt;</code>. If R is a list, you get <code>Stream&lt;List&gt;</code>. <b>flatMap(Function&lt;T, Stream&lt;R&gt;&gt;)</b>: transforms each element to a <em>stream</em> of elements and <b>flattens</b> all resulting streams into one. Example: converting a list of sentences to a stream of words: <code>stream.flatMap(sentence -&gt; Arrays.stream(sentence.split(" ")))</code>. Rule of thumb: use <code>flatMap</code> when your mapping function produces a collection/stream per element.

What does <b>Collectors.groupingBy</b> do?	<code>Collectors.groupingBy(classifier)</code> groups stream elements by a classifier function, producing a <code>Map&lt;K, List&lt;V&gt;&gt;</code>. Example: <code>people.stream().collect(groupingBy(Person::getDepartment))</code>. <b>Downstream collectors</b> can be combined for further aggregation: <code>groupingBy(Person::getDept, counting())</code>, <code>groupingBy(Person::getDept, summingInt(Person::getSalary))</code>, <code>groupingBy(Person::getDept, mapping(Person::getName, toList()))</code>. Multi-level grouping: nest <code>groupingBy</code> as a downstream: <code>groupingBy(dept, groupingBy(gender))</code>.

What is the difference between <b>findFirst()</b> and <b>findAny()</b>?	Both return an <code>Optional&lt;T&gt;</code> and short-circuit the stream. <b>findFirst()</b> always returns the <em>first element</em> in encounter order — deterministic. <b>findAny()</b> returns <em>any</em> element — in sequential streams it usually returns the first, but in <b>parallel streams</b> it returns whichever thread finds one first, making it faster at the cost of non-determinism. Prefer <code>findAny()</code> in parallel streams where order doesn't matter.

What are <b>method references</b> in Java? Give examples.	Method references are concise lambda syntax for calling existing methods. Four types: (1) <b>Static method</b>: <code>Integer::parseInt</code> ≡ <code>x -&gt; Integer.parseInt(x)</code>. (2) <b>Bound instance method</b>: <code>str::toUpperCase</code> ≡ <code>() -&gt; str.toUpperCase()</code>. (3) <b>Unbound instance method</b>: <code>String::toUpperCase</code> ≡ <code>s -&gt; s.toUpperCase()</code>. (4) <b>Constructor reference</b>: <code>ArrayList::new</code> ≡ <code>() -&gt; new ArrayList()</code>. Method references improve readability when the lambda just delegates to an existing method. They are compiled using <code>invokedynamic</code>, not anonymous classes.

What is a <b>functional interface</b>?	A functional interface has <b>exactly one abstract method</b> (SAM — Single Abstract Method). It can have multiple <code>default</code> and <code>static</code> methods. The <code>@FunctionalInterface</code> annotation enforces this at compile time. They are the target types for lambda expressions and method references. Core built-in functional interfaces in <code>java.util.function</code>: <b>Predicate</b>, <b>Function</b>, <b>Consumer</b>, <b>Supplier</b>, <b>BiFunction</b>, <b>UnaryOperator</b>, <b>BinaryOperator</b>. Historic examples: <b>Runnable</b>, <b>Callable</b>, <b>Comparator</b>.

What is the difference between <b>Predicate</b>, <b>Function</b>, <b>Consumer</b>, and <b>Supplier</b>?	<b>Predicate&lt;T&gt;</b>: takes T, returns boolean. Used for filtering. Methods: <code>test()</code>, <code>and()</code>, <code>or()</code>, <code>negate()</code>. <b>Function&lt;T,R&gt;</b>: takes T, returns R. Used for transformation. Methods: <code>apply()</code>, <code>andThen()</code>, <code>compose()</code>. <b>Consumer&lt;T&gt;</b>: takes T, returns void. Used for side effects (printing, saving). Methods: <code>accept()</code>, <code>andThen()</code>. <b>Supplier&lt;T&gt;</b>: takes nothing, returns T. Used for lazy value creation or factories. Method: <code>get()</code>. Bi-variants exist: <code>BiPredicate</code>, <code>BiFunction</code>, <code>BiConsumer</code>.

What is <b>Optional</b> and why use it?	<b>Optional&lt;T&gt;</b> (Java 8+) is a container that explicitly models the possible absence of a value, replacing <code>null</code> returns to avoid <code>NullPointerException</code>. Creation: <code>Optional.of(v)</code> (non-null), <code>Optional.ofNullable(v)</code>, <code>Optional.empty()</code>. Key methods: <code>isPresent()</code>, <code>isEmpty()</code> (Java 11+), <code>get()</code> (throws if empty), <code>orElse(default)</code>, <code>orElseGet(Supplier)</code>, <code>orElseThrow()</code>, <code>map()</code>, <code>flatMap()</code>, <code>filter()</code>, <code>ifPresent(Consumer)</code>, <code>ifPresentOrElse()</code> (Java 9+). <b>Best practice:</b> use as return type, not as field or parameter type.

What is the difference between <b>Stream.of()</b> and <b>Arrays.stream()</b>?	<b>Arrays.stream(arr)</b> handles primitive arrays (<code>int[]</code>, <code>long[]</code>, <code>double[]</code>) by returning specialized primitive streams (<code>IntStream</code>, <code>LongStream</code>, <code>DoubleStream</code>) — no boxing. Also supports sub-arrays: <code>Arrays.stream(arr, start, end)</code>. <b>Stream.of(arr)</b> treats a primitive array as a single element, returning <code>Stream&lt;int[]&gt;</code> (boxed wrapper). For object arrays both work the same, but <code>Arrays.stream()</code> is more idiomatic. For primitives, always prefer <code>Arrays.stream()</code> to avoid boxing overhead.

What is a <b>parallel stream</b>? When should you use it?	A parallel stream splits data into chunks and processes them concurrently using <code>ForkJoinPool.commonPool()</code>. Enable with <code>collection.parallelStream()</code> or <code>stream.parallel()</code>. <b>Use when:</b> large data set (hundreds of thousands+ elements), operations are CPU-intensive, stateless, and associative, and encounter order doesn't matter. <b>Avoid when:</b> data set is small (thread overhead dominates), operations have side effects or depend on order, IO-bound tasks (threads are wasted blocking). Always benchmark — parallel streams often perform worse than sequential for small or IO-bound workloads.

What is <b>reduce()</b> in Streams?	<code>reduce()</code> folds all stream elements into a single result using an associative accumulation function. Three forms: (1) <code>reduce(identity, BinaryOperator)</code> — always returns T. (2) <code>reduce(BinaryOperator)</code> — returns <code>Optional&lt;T&gt;</code> (empty stream safe). (3) <code>reduce(identity, BiFunction, BinaryOperator)</code> — for parallel streams combining different types. Example: <code>stream.reduce(0, Integer::sum)</code>. For mutable accumulation (e.g., building a list), prefer <code>collect()</code> — it uses a mutable container and is more efficient.

What is the difference between <b>Comparable</b> and <b>Comparator</b>?	<b>Comparable&lt;T&gt;</b>: implemented <em>by the class itself</em> via <code>compareTo(T other)</code>, defining the <b>natural ordering</b> (e.g., String alphabetical, Integer numeric). Used by <code>Collections.sort()</code> and <code>TreeSet</code> by default. <b>Comparator&lt;T&gt;</b>: external strategy for ordering, passed as a parameter. Allows multiple orderings without modifying the class. In Java 8+: <code>Comparator.comparing(Person::getName).thenComparing(Person::getAge).reversed()</code>. Use Comparator when you don't control the class or need a different ordering.

What is the JVM&#39;s <b>Heap</b> divided into?	The JVM Heap is divided into: <b>Young Generation</b> — where new objects are allocated. Subdivided into <b>Eden</b> (allocation) and two <b>Survivor spaces (S0/S1)</b> (objects that survived minor GC alternate between them). <b>Old Generation (Tenured)</b> — long-lived objects promoted from Young Gen after surviving several GC cycles (threshold controlled by <code>-XX:MaxTenuringThreshold</code>). With <b>G1GC</b>, the heap is split into equal-sized (~1-32MB) <b>regions</b> dynamically assigned as Eden, Survivor, or Old. <b>Metaspace</b> (not in heap) stores class metadata.

What is the difference between <b>Minor GC</b> and <b>Major GC</b>?	<b>Minor GC</b>: collects the <b>Young Generation</b> only. Fast (usually milliseconds) because the Young Gen is small. Uses a copying algorithm — live objects are moved to Survivor/Old spaces, dead ones discarded. Frequent. <b>Major GC (Full GC)</b>: collects <b>Old Generation</b> (and often Young Gen too). Slower (can be seconds) because the Old Gen is larger and uses mark-sweep-compact. Causes longer <b>Stop-The-World</b> pauses. Triggered when Old Gen fills up, or explicitly by <code>System.gc()</code>. Modern GCs (G1, ZGC) try to make most collection concurrent to reduce pause times.

What is <b>Stop-The-World</b> (STW) in GC?	STW is a phase during garbage collection where the JVM pauses <em>all</em> application threads to ensure heap consistency — no object references can change while GC analyzes the heap. Duration varies by GC algorithm: Parallel GC has long STW pauses, G1GC aims for short predictable pauses, ZGC and Shenandoah aim for sub-millisecond STW. STW is necessary for phases like root scanning, object relocation, and reference updates. Minimizing STW is crucial for latency-sensitive applications.

What is a <b>memory leak</b> in Java?	A memory leak occurs when objects are no longer needed but remain <em>strongly reachable</em> (referenced), preventing GC from collecting them. The heap grows until <code>OutOfMemoryError</code>. <b>Common causes:</b> (1) Static collections that grow indefinitely. (2) Listeners/callbacks registered but never removed. (3) Caches without eviction policies. (4) ThreadLocal values not removed in thread pools. (5) Unclosed streams/connections holding resources. (6) Inner class instances holding implicit references to outer class. Detect with heap dumps + <b>Eclipse MAT</b> or <b>YourKit</b>.

What are the main GC algorithms in HotSpot JVM?	<b>Serial GC</b> (<code>-XX:+UseSerialGC</code>): single-threaded, for small heaps/single-core. <b>Parallel GC</b> (<code>-XX:+UseParallelGC</code>): multi-threaded Young Gen collection, throughput-focused, default before Java 9. <b>CMS (Concurrent Mark Sweep)</b>: mostly concurrent, low-pause, deprecated in Java 9, removed in Java 14. <b>G1GC</b> (<code>-XX:+UseG1GC</code>): default since Java 9, region-based, predictable pauses. <b>ZGC</b> (<code>-XX:+UseZGC</code>): ultra-low latency (&lt;1ms), concurrent, scalable to TB heaps, production-ready Java 15+. <b>Shenandoah</b>: similar to ZGC, developed by Red Hat, concurrent evacuation.

What is <b>G1GC</b>?	G1 (Garbage First) GC (default since Java 9) divides the heap into equal-sized regions (~1-32MB, configurable with <code>-XX:G1HeapRegionSize</code>). Regions are dynamically assigned as Eden, Survivor, Old, or Humongous (for large objects). G1 first collects regions with the most garbage ("garbage first"), making collection more efficient. It runs mostly concurrently (concurrent marking) and uses <b>evacuation</b> (copying) to compact live objects. Target pause time set with <code>-XX:MaxGCPauseMillis=200</code>. Balances throughput and latency — suitable for most server applications.

What is the difference between <b>WeakReference</b>, <b>SoftReference</b>, and <b>PhantomReference</b>?	All three are in <code>java.lang.ref</code> and allow GC to reclaim objects that would otherwise be retained. <b>WeakReference</b>: collected at the <em>next</em> GC cycle regardless of memory. Used in <code>WeakHashMap</code> for automatic key removal. <b>SoftReference</b>: collected only when the JVM is <em>running low on memory</em>. Ideal for memory-sensitive caches. <b>PhantomReference</b>: collected <em>after</em> finalization; <code>get()</code> always returns null. Used with <code>ReferenceQueue</code> for post-GC cleanup actions (replacement for <code>finalize()</code>). All can be paired with a <code>ReferenceQueue</code> to be notified when collected.

What is <b>metaspace</b> in Java 8+?	Metaspace replaced <b>PermGen</b> (Permanent Generation) in Java 8. It stores class metadata: class definitions, method bytecode, constant pools, annotations. Unlike PermGen (fixed size in JVM heap), Metaspace resides in <b>native (off-heap) memory</b> and grows dynamically by default. This eliminates the infamous <code>OutOfMemoryError: PermGen space</code>. Configure with: <code>-XX:MetaspaceSize</code> (initial size), <code>-XX:MaxMetaspaceSize</code> (cap it to prevent unbounded growth). Metaspace leaks can occur with excessive class loading (e.g., dynamic proxy generation, scripting engines).

What is <b>escape analysis</b> in JVM?	Escape analysis is a JIT compiler optimization that determines whether an object's reference "escapes" its creation method or thread. If an object <b>doesn't escape</b>: (1) <b>Stack allocation</b>: the object can be allocated on the stack instead of heap, reducing GC pressure. (2) <b>Scalar replacement</b>: the object is decomposed into its primitive fields, allocated in registers. (3) <b>Lock elision</b>: synchronization on a non-escaping object (provably accessed by only one thread) is removed. Enabled by default in HotSpot. Can be disabled with <code>-XX:-DoEscapeAnalysis</code>.

What is the <b>JIT compiler</b>?	The Just-In-Time compiler identifies <b>hot spots</b> (frequently executed bytecode) and compiles them to optimized native machine code at runtime. HotSpot uses <b>tiered compilation</b>: <b>C1</b> (Client compiler): quick compilation with basic optimizations — reduces startup latency. <b>C2</b> (Server compiler): aggressive long-running optimizations (inlining, escape analysis, loop unrolling) — maximizes steady-state throughput. JIT also performs <b>deoptimization</b> — reverting to interpreted mode if an optimization assumption is invalidated (e.g., a new subclass is loaded). GraalVM provides an alternative JIT.

What does <b>-Xms</b> and <b>-Xmx</b> JVM flags control?	<b>-Xms&lt;size&gt;</b>: sets the <b>initial heap size</b> allocated at JVM startup. <b>-Xmx&lt;size&gt;</b>: sets the <b>maximum heap size</b> the JVM can use. If heap exceeds this, <code>OutOfMemoryError</code> is thrown. Example: <code>-Xms512m -Xmx4g</code>. <b>Best practice for production:</b> set <code>-Xms == -Xmx</code> to avoid heap resizing overhead (resizing causes full GC pauses). Related flags: <code>-Xss</code> (thread stack size), <code>-XX:MetaspaceSize</code>, <code>-XX:MaxMetaspaceSize</code>, <code>-XX:NewRatio</code> (Young/Old ratio), <code>-XX:SurvivorRatio</code>.

What is the <b>Singleton</b> design pattern? How do you implement it thread-safely?	Singleton ensures only <b>one instance</b> of a class exists per JVM. Thread-safe implementations: (1) <b>Enum Singleton</b> (preferred by Effective Java): <code>enum MySingleton { INSTANCE; }</code> — JVM guarantees single instance, serialization-safe, reflection-safe. (2) <b>Bill Pugh / static inner class</b>: <code>private static class Holder { static final S INSTANCE = new S(); }</code> — lazy, thread-safe via class loading. (3) <b>Double-checked locking (DCL)</b>: <code>if (instance == null) { synchronized(this) { if (instance == null) instance = new S(); } }</code> — requires <code>volatile</code> to prevent partial construction. Avoid eager initialization in non-lazy scenarios.

What is the <b>Factory Method</b> pattern?	<b>Factory Method</b> defines an abstract method in a superclass/interface for creating an object, letting <b>subclasses decide</b> which concrete class to instantiate. Decouples the client from concrete types. Example: <code>DocumentCreator.createDocument()</code> returns a <code>Document</code> — subclasses <code>PdfCreator</code>, <code>WordCreator</code> return specific types. Follows the <b>Open/Closed Principle</b>: add new product types by adding new subclasses without modifying existing code. Contrast with <b>Simple Factory</b> (not a GoF pattern) which uses a static method.

What is the <b>Abstract Factory</b> pattern?	<b>Abstract Factory</b> provides an interface for creating <em>families of related objects</em> without specifying their concrete classes — a "factory of factories." Example: a <code>UIFactory</code> interface with methods <code>createButton()</code> and <code>createTextField()</code> — <code>WindowsUIFactory</code> and <code>MacUIFactory</code> produce platform-consistent widgets. Ensures that products from the same factory are compatible with each other. More complex than Factory Method — use when multiple product hierarchies must vary together.

What is the <b>Builder</b> pattern?	Builder separates the <em>construction</em> of a complex object from its representation, enabling step-by-step construction with a readable API. Solves the "telescoping constructor" anti-pattern. Example: <code>new Person.Builder("Alice").age(30).email("a@b.com").build()</code>. Benefits: (1) Handles optional parameters cleanly. (2) Immutable object construction. (3) Self-documenting code. In Java: Lombok's <code>@Builder</code>, record builders, Guava's builders. <b>Director</b> variant: a Director class orchestrates the builder steps.

What is the <b>Prototype</b> pattern?	Prototype creates new objects by <b>cloning</b> an existing object (the prototype) instead of instantiating via constructor. Useful when object creation is expensive (DB load, complex initialization). Java implementation: implement <code>Cloneable</code> and override <code>clone()</code> (shallow copy by default) or use a copy constructor. <b>Shallow copy</b>: copies fields by reference — nested objects are shared. <b>Deep copy</b>: recursively copies nested objects. Prefer copy constructors/factories over <code>clone()</code> (Effective Java Item 13). Used in prototype registries for configuration templates.

What is the <b>Adapter</b> pattern?	Adapter converts an incompatible interface into one that the client expects — "makes things work together." Two implementations: (1) <b>Object adapter</b>: wraps an adaptee instance — preferred in Java (uses composition). (2) <b>Class adapter</b>: extends adaptee — requires multiple inheritance (not possible in Java). Example: wrapping a legacy <code>LegacyPrinter</code> to implement a modern <code>Printer</code> interface. Java examples: <code>Arrays.asList()</code> adapts arrays to List, <code>InputStreamReader</code> adapts byte streams to character streams.

What is the <b>Decorator</b> pattern?	Decorator dynamically adds responsibilities to an object by wrapping it in decorator objects, as an alternative to subclassing. Each decorator implements the same interface as the component it wraps, delegating calls and adding behavior before/after. <b>Example:</b> Java I/O: <code>new BufferedInputStream(new FileInputStream(path))</code> — each wrapper adds behavior (buffering, compression, encryption). Benefits: follows Open/Closed Principle, composable at runtime, avoids class explosion. Compare with Proxy (controls access) and Adapter (converts interface).

What is the <b>Proxy</b> pattern?	Proxy provides a <b>surrogate</b> for another object, controlling access to it. Types: (1) <b>Virtual proxy</b>: lazy initialization — creates expensive objects on first use. (2) <b>Protection proxy</b>: access control based on permissions. (3) <b>Remote proxy</b>: represents an object in another JVM/machine (RMI). (4) <b>Caching proxy</b>: caches results. (5) <b>Logging/monitoring proxy</b>. Java provides <b>dynamic proxies</b>: <code>java.lang.reflect.Proxy.newProxyInstance()</code> creates a proxy at runtime for interfaces. Spring AOP uses JDK dynamic proxies (interfaces) and CGLIB (classes).

What is the <b>Observer</b> pattern?	Observer defines a <b>one-to-many</b> dependency: a Subject (Observable) maintains a list of Observers and notifies them automatically on state changes. Decouples subject from observers — subject only knows the Observer interface. Java built-ins: <code>java.util.Observable</code>/<code>Observer</code> (deprecated), <code>PropertyChangeListener</code>. Foundation of: event listeners (Swing, JavaFX), Spring's <code>ApplicationEventPublisher</code>, reactive streams. <b>Push vs Pull</b>: push sends data with notification; pull lets observers query state after notification.

What is the <b>Strategy</b> pattern?	Strategy defines a family of interchangeable algorithms, encapsulates each one in a class, and selects the algorithm at runtime via composition rather than subclassing. Example: <code>Sorter</code> class with a <code>SortStrategy</code> field — swap between <code>BubbleSort</code>, <code>QuickSort</code>, <code>MergeSort</code> without changing <code>Sorter</code>. In Java 8+: strategies are often implemented as lambda expressions or method references since they're functional interfaces. Follows <b>Open/Closed Principle</b>. Used in <code>Comparator</code>, <code>Comparator.comparing()</code>.

What is the <b>Template Method</b> pattern?	Template Method defines the <b>skeleton of an algorithm</b> in a base class method, deferring specific steps to subclasses via abstract or hook methods. The overall structure (order of steps) is fixed; subclasses customize specific steps without changing the algorithm. Example: <code>DataProcessor.process()</code> calls <code>readData()</code>, <code>processData()</code>, <code>writeData()</code> — subclasses override individual steps. Follows <b>Hollywood Principle</b>: "Don't call us, we'll call you." Used extensively in Java frameworks (JUnit lifecycle, Spring's <code>JdbcTemplate</code>).

What is the <b>Command</b> pattern?	Command encapsulates a request as an object with an <code>execute()</code> method, enabling: (1) <b>Parameterization</b>: pass commands as arguments. (2) <b>Queuing</b>: queue commands for deferred execution. (3) <b>Logging</b>: log executed commands. (4) <b>Undo/Redo</b>: store command history and call <code>undo()</code>. (5) <b>Macros</b>: composite commands. Components: Command (interface), ConcreteCommand, Receiver (actual logic), Invoker (triggers command), Client (creates commands). Example: GUI button actions, task queues, transaction scripts.

What is the <b>Chain of Responsibility</b> pattern?	Chain of Responsibility passes a request along a chain of handlers. Each handler decides to: (1) Handle it and stop propagation, or (2) Pass it to the next handler. Decouples senders from receivers and allows dynamic chain construction. Example: HTTP filter chains (Servlet filters, Spring's <code>HandlerInterceptor</code>), logging level hierarchy, event bubbling in UIs, Spring Security filter chain. In Java 8+: handlers can be chained as functions using <code>andThen()</code>. Difference from Decorator: CoR passes the request to one handler; Decorator applies all wrappers.

What is the difference between <b>HashMap</b> and <b>ConcurrentHashMap</b>?	<b>HashMap</b>: not thread-safe — concurrent modification causes data corruption or infinite loops. <b>ConcurrentHashMap</b>: thread-safe without locking the entire map. Java 7: used segment-based locking (16 segments by default). Java 8+: uses per-bin synchronization — CAS for inserts into empty bins, <code>synchronized</code> on the bin head for updates. Reads are entirely lock-free. Supports concurrent bulk operations: <code>forEach()</code>, <code>reduce()</code>, <code>search()</code>. Does <b>not</b> allow null keys/values (unlike HashMap). Significantly higher throughput than <code>Hashtable</code> or <code>Collections.synchronizedMap()</code>.

What happens when two keys have the same <b>hashCode</b> in a HashMap?	This is a <b>hash collision</b> — both keys map to the same bucket. Java HashMap handles it with <b>chaining</b>: (1) Before Java 8: linked list in each bucket — O(n) worst case. (2) Java 8+: list until bin size reaches <b>TREEIFY_THRESHOLD (8)</b>, then converts to a <b>red-black tree</b> for O(log n) lookup. Tree is untreeified back to a list if size drops below <b>UNTREEIFY_THRESHOLD (6)</b>. Note: treeification requires keys to implement <code>Comparable</code> (or falls back to identity hash comparison). Hash DoS attacks with poor <code>hashCode()</code> are mitigated by the tree conversion.

What is the default <b>load factor</b> of HashMap and what does it mean?	Default load factor is <b>0.75</b>, balancing time vs. space. When <code>size &gt; capacity × loadFactor</code>, the map is <b>rehashed</b>: a new backing array of 2× size is allocated, and all entries are redistributed (re-bucketed). Rehashing is O(n) and expensive. <b>Lower load factor</b> (e.g., 0.5): fewer collisions, faster lookups, more memory wasted. <b>Higher load factor</b> (e.g., 0.9): more collisions, slower lookups, less memory. Pre-size with <code>new HashMap&lt;&gt;(expectedSize / 0.75 + 1)</code> to avoid rehashing. Initial capacity defaults to 16 (always a power of 2).

What is the difference between <b>HashSet</b> and <b>TreeSet</b>?	<b>HashSet</b>: backed by a <code>HashMap</code>, stores only keys (value is a dummy object). O(1) average add/remove/contains. No ordering guarantee. Allows one null. <b>TreeSet</b>: backed by a <code>TreeMap</code> (red-black tree). O(log n) for all operations. Elements always <b>sorted</b> by natural ordering or a provided <code>Comparator</code>. Implements <code>NavigableSet</code>: supports <code>floor()</code>, <code>ceiling()</code>, <code>headSet()</code>, <code>tailSet()</code>. Does not allow null. <b>LinkedHashSet</b>: HashSet with insertion-order iteration (backed by LinkedHashMap).

What is <b>LinkedHashMap</b>?	<b>LinkedHashMap</b> extends <code>HashMap</code> with a <b>doubly-linked list</b> connecting all entries, maintaining either <b>insertion order</b> (default) or <b>access order</b> (constructor: <code>new LinkedHashMap&lt;&gt;(capacity, loadFactor, true)</code>). Iteration is O(n) in insertion/access order — unlike HashMap's arbitrary order. Access-order mode combined with <code>removeEldestEntry()</code> override enables an <b>LRU cache</b>. Slightly higher memory overhead than HashMap. Used internally by <code>LinkedHashSet</code>.

What is the difference between <b>ArrayList</b> and <b>LinkedList</b>?	<b>ArrayList</b>: backed by a dynamic array. O(1) amortized add-to-end, O(1) random access by index, O(n) insert/delete in middle (shifts elements). Compact memory, excellent cache locality. <b>LinkedList</b>: doubly-linked list. O(1) insert/delete at known node, O(n) random access (traversal). Implements both <code>List</code> and <code>Deque</code>. Higher memory overhead (node object + two pointers per element), poor cache locality. <b>In practice:</b> ArrayList outperforms LinkedList in most scenarios due to cache effects. Use LinkedList only when frequent head/tail operations and no random access.

What is <b>PriorityQueue</b>?	<b>PriorityQueue</b> is an unbounded priority queue backed by a <b>binary min-heap</b>. Elements are ordered by natural ordering or a provided <code>Comparator</code>. <code>peek()</code> and <code>poll()</code> return the smallest (highest-priority) element in O(1) and O(log n) respectively. <code>offer()/add()</code> is O(log n). Iteration order is <b>not</b> sorted — only head is guaranteed to be minimum. Not thread-safe — use <code>PriorityBlockingQueue</code> for concurrency. Used in Dijkstra's algorithm, task scheduling, event simulation.

What is <b>Collections.unmodifiableList()</b> vs <b>List.of()</b>?	<b>Collections.unmodifiableList(list)</b>: returns a <em>view</em> — throws <code>UnsupportedOperationException</code> on modification attempts, but the <em>underlying list</em> can still be modified (the view reflects changes). Allows null elements. <b>List.of()</b> (Java 9+): truly immutable — structural modifications and <code>set()</code> both throw <code>UnsupportedOperationException</code>. Does <b>not</b> allow null elements (throws NullPointerException). Stored compactly. Also <b>List.copyOf()</b> creates an immutable copy of an existing collection. Use <code>List.of()</code> for constants; <code>Collections.unmodifiableList()</code> when wrapping a mutable list you control.

What are <b>EnumSet</b> and <b>EnumMap</b>?	<b>EnumSet</b>: a specialized Set for enum types backed by a <b>bit vector</b> (long bitmask for ≤64 values, or array for more). Extremely fast O(1) operations with minimal memory. No null allowed. Factory methods: <code>EnumSet.of(A, B)</code>, <code>EnumSet.allOf(E.class)</code>, <code>EnumSet.range(MON, FRI)</code>. <b>EnumMap</b>: a Map with enum keys backed by an <b>array</b> indexed by enum ordinal. O(1) operations, compact, iteration in enum declaration order. Both are not thread-safe — wrap with <code>Collections.synchronized*()</code> if needed. Strongly preferred over <code>HashSet</code>/<code>HashMap</code> with enum keys for performance.

What is <b>Java reflection</b>?	Reflection (<code>java.lang.reflect</code>) allows runtime inspection and manipulation of classes, methods, fields, and constructors, even private ones (with <code>setAccessible(true)</code>). Key classes: <code>Class&lt;?&gt;</code>, <code>Method</code>, <code>Field</code>, <code>Constructor</code>. Capabilities: instantiate classes dynamically, invoke methods by name, inspect annotations, get/set field values. Heavily used by: Spring (dependency injection, AOP), Hibernate (ORM), JUnit, Jackson (JSON serialization). Java 9+ module system restricts reflective access to non-exported packages — <code>--add-opens</code> may be needed.

What is the performance cost of <b>reflection</b>?	Reflection is significantly slower than direct invocation due to: (1) <b>Security checks</b> on every access (skipped after <code>setAccessible(true)</code>). (2) <b>Dynamic dispatch</b> and vtable bypassing. (3) <b>No JIT inlining</b> — JIT cannot inline reflective calls as easily. (4) <b>Boxing/unboxing</b> of primitive parameters/return values. (5) <b>Method lookup overhead</b>. Benchmarks show 10-100× slower than direct calls for individual invocations. <b>Mitigations:</b> cache <code>Method</code>/<code>Field</code>/<code>Constructor</code> objects, use <code>MethodHandle</code> (JIT-friendly), or code generation (ByteBuddy, ASM). Avoid in hot paths.

What are <b>annotations</b> in Java?	Annotations are metadata markers attached to code elements (classes, methods, fields, parameters, packages, local variables). They don't affect runtime behavior directly but are read by: (1) <b>Compiler</b>: <code>@Override</code>, <code>@SuppressWarnings</code>, <code>@FunctionalInterface</code>. (2) <b>Annotation processors (APT)</b>: Lombok, MapStruct generate code at compile time. (3) <b>Runtime reflection</b>: Spring, JUnit, Jackson read annotations at runtime via <code>@Retention(RUNTIME)</code>. Custom annotations: <code>@interface MyAnnotation { String value() default ""; }</code>. Annotations can be repeated with <code>@Repeatable</code>.

What are <b>meta-annotations</b>? Name them.	Meta-annotations annotate annotation type declarations: <b>@Retention</b>: how long the annotation is retained — <code>SOURCE</code> (discarded after compilation), <code>CLASS</code> (in .class but not at runtime, default), <code>RUNTIME</code> (accessible via reflection). <b>@Target</b>: which elements can be annotated — <code>TYPE</code>, <code>METHOD</code>, <code>FIELD</code>, <code>PARAMETER</code>, <code>CONSTRUCTOR</code>, <code>LOCAL_VARIABLE</code>, etc. <b>@Documented</b>: include annotation in Javadoc. <b>@Inherited</b>: annotation on a superclass is inherited by subclasses (only class-level). <b>@Repeatable</b>: the annotation can be applied multiple times to the same element (requires a container annotation).

What is <b>annotation processing</b>?	Annotation Processing (APT) runs custom processors (<code>javax.annotation.processing.Processor</code>) during <code>javac</code> compilation. Processors use the <code>javax.lang.model</code> API to inspect annotated elements and generate new source files, class files, or resources. Multiple rounds: generated files can trigger further processing. <b>Examples:</b> <b>Lombok</b> generates getters/setters/builders, <b>MapStruct</b> generates mapper implementations, <b>Dagger</b> generates DI code, <b>AutoValue</b> generates immutable value classes. No runtime reflection overhead — all code generation happens at compile time. Configure in <code>META-INF/services/javax.annotation.processing.Processor</code>.

What is the difference between <b>checked</b> and <b>unchecked exceptions</b>?	<b>Checked exceptions</b>: extend <code>Exception</code> (but not <code>RuntimeException</code>). The compiler enforces that they are either caught or declared with <code>throws</code>. Represent recoverable conditions: <code>IOException</code>, <code>SQLException</code>, <code>ParseException</code>. <b>Unchecked exceptions</b>: extend <code>RuntimeException</code>. No compiler enforcement — represent programming errors: <code>NullPointerException</code>, <code>IllegalArgumentException</code>, <code>IndexOutOfBoundsException</code>. <b>Error</b>: extend <code>Throwable</code> directly — JVM-level problems (<code>OutOfMemoryError</code>, <code>StackOverflowError</code>) — should not be caught. Modern best practice (Clean Code, Effective Java): prefer unchecked exceptions; use checked only for truly recoverable conditions.

What is <b>exception chaining</b>?	Exception chaining (also called <em>exception wrapping</em>) preserves the original cause when rethrowing at a higher abstraction level. Example: catching <code>SQLException</code> in a DAO and rethrowing as <code>DataAccessException("DB error", sqlEx)</code>. The original exception is attached as the <em>cause</em> via the constructor: <code>new RuntimeException("msg", cause)</code> or <code>initCause(cause)</code>, and retrieved with <code>getCause()</code>. The full chain appears in stack traces. <b>Why it matters:</b> without chaining, the root cause is lost, making debugging much harder. Always preserve the cause when translating exceptions.

What is a <b>try-with-resources</b> statement?	Introduced in Java 7, try-with-resources automatically closes resources implementing <code>AutoCloseable</code> (or <code>Closeable</code>) when the try block exits — whether normally or via exception. Syntax: <code>try (InputStream is = new FileInputStream(f); Connection c = ds.getConnection()) { ... }</code>. Resources are closed in <b>reverse order of declaration</b>. Eliminates boilerplate <code>finally</code> blocks and the risk of forgetting to close. Java 9+: can use effectively-final variables declared outside: <code>try (existing) { ... }</code>. Always prefer over manual <code>finally</code> for resource management.

What is <b>exception masking</b> and how does try-with-resources handle it?	Exception masking occurs when an exception thrown in a <code>finally</code> block <em>replaces</em> (masks) the original exception from the <code>try</code> block, losing the root cause. Classic example: <code>close()</code> throws in <code>finally</code>, hiding the original exception. <b>Try-with-resources solution:</b> if both the body and the <code>close()</code> throw, the body's exception is preserved as the primary exception, and the <code>close()</code> exception is added as a <b>suppressed exception</b>, accessible via <code>primaryException.getSuppressed()</code>. This way both exceptions are preserved and neither is lost.

What is a <b>multi-catch block</b>?	Java 7+ allows catching multiple unrelated exception types in a single <code>catch</code> clause: <code>catch (IOException | SQLException | ParseException e) { ... }</code>. This reduces code duplication when handling multiple exceptions identically. The caught variable is <b>implicitly final</b> — you cannot reassign it. <b>Limitation:</b> cannot catch related types (e.g., <code>catch (Exception | IOException e)</code> is a compile error since IOException is a subtype of Exception). Multi-catch produces a single exception handler in bytecode, not duplicate handlers.

What is the <b>difference between final, finally, and finalize</b>?	<b>final</b>: keyword with three uses — (1) final variable: constant, cannot be reassigned. (2) final method: cannot be overridden. (3) final class: cannot be subclassed (e.g., <code>String</code>, <code>Integer</code>). <b>finally</b>: block in a try-catch-finally construct that always executes (even on return or exception), used for cleanup. Skipped only on <code>System.exit()</code> or JVM crash. <b>finalize()</b>: deprecated instance method called by GC before reclaiming an object. Unreliable (no guarantee when/if called), can resurrect objects, causes GC overhead. Removed in Java 18+. Use try-with-resources or <code>Cleaner</code> API instead.

What is <b>type erasure</b> in Java generics?	Type erasure is the process by which the Java compiler removes all generic type information from compiled bytecode, replacing type parameters with their bounds (upper bound or <code>Object</code> if unbounded). This ensures backward compatibility with pre-generics Java. <b>Consequences:</b> (1) Cannot use <code>new T()</code>, <code>new T[]</code>, or <code>instanceof T</code> at runtime. (2) <code>List&lt;String&gt;</code> and <code>List&lt;Integer&gt;</code> are the same type at runtime (<code>List</code>). (3) Heap pollution with unchecked casts. (4) Cannot overload methods that differ only in generic type parameters. Workaround: pass <code>Class&lt;T&gt;</code> tokens or use <code>TypeToken</code> (Guava) for runtime type information.

What is a <b>bounded type parameter</b>? Give examples.	Bounded type parameters restrict the types that can be used as type arguments. <b>Upper bound</b>: <code>&lt;T extends Number&gt;</code> — T must be Number or a subclass. Multiple bounds: <code>&lt;T extends Comparable&lt;T&gt; &amp; Serializable&gt;</code>. <b>Lower bound wildcard</b>: <code>&lt;? super Integer&gt;</code> — accepts Integer, Number, Object. <b>Unbounded wildcard</b>: <code>&lt;?&gt;</code> — accepts any type. Bounds enable calling methods of the bound type on T: <code>T extends Number</code> allows <code>t.doubleValue()</code>. Wildcards are used in method signatures, type parameters in class/method declarations.

What is the <b>PECS principle</b>?	PECS: <b>Producer Extends, Consumer Super</b> — from Effective Java by Joshua Bloch. <b>Use <code>? extends T</code></b> (upper-bounded wildcard) when a collection <em>produces</em> T values (you only read from it). Example: <code>void copy(List&lt;? extends Number&gt; src, ...)</code>. <b>Use <code>? super T</code></b> (lower-bounded wildcard) when a collection <em>consumes</em> T values (you only write to it). Example: <code>void addNumbers(List&lt;? super Integer&gt; dest, ...)</code>. <b>Use exact type</b> when both reading and writing. Memory trick: <b>GET (extends)</b>, <b>PUT (super)</b>. Applicable to all generic APIs designing for maximum flexibility.

What is a <b>wildcard</b> in generics?	Wildcards (<code>?</code>) represent an unknown type in generic type arguments. <b>Unbounded</b>: <code>List&lt;?&gt;</code> — any type; can only add null, read as Object. <b>Upper bounded</b>: <code>List&lt;? extends Number&gt;</code> — Number or subtype; read as Number, cannot add (type unknown). <b>Lower bounded</b>: <code>List&lt;? super Integer&gt;</code> — Integer supertype; can add Integer/subtype, read as Object. Key constraint: wildcard types are <em>not</em> type parameters — they appear in usage, not declaration. Wildcards enable API flexibility (PECS principle). <b>Capture conversion</b>: the compiler can sometimes infer the wildcard type in specific contexts.

What is a <b>generic method</b>?	A generic method introduces its own type parameter(s) independent of the class's type parameters: <code>public &lt;T extends Comparable&lt;T&gt;&gt; T max(T a, T b) { return a.compareTo(b) &gt; 0 ? a : b; }</code>. Type parameters are declared before the return type. Type is inferred at the call site — explicit invocation: <code>Collections.&lt;String&gt;emptyList()</code>. Generic methods can be in non-generic classes. <b>Benefits:</b> type-safe without casts, reusable for multiple types, enables utility methods (<code>Collections.sort()</code>, <code>Collections.swap()</code>).

What are <b>sealed classes</b> in Java 17?	Sealed classes (JEP 409, finalized Java 17) restrict which classes can extend or implement them using the <code>permits</code> clause: <code>public sealed class Shape permits Circle, Rectangle, Triangle {}</code>. Permitted subclasses must be in the same package/module and must be <code>final</code>, <code>sealed</code>, or <code>non-sealed</code>. <b>Benefits:</b> (1) Models <em>closed</em> type hierarchies exhaustively. (2) Enables <b>exhaustive pattern matching</b> in switch without a default case (Java 21). (3) Better than enums for hierarchies that need polymorphic behavior. Sealed interfaces work the same way.

What are <b>records</b> in Java 16?	Records (JEP 395, finalized Java 16) are immutable data carriers declared as: <code>record Point(int x, int y) {}</code>. The compiler auto-generates: canonical constructor, accessors (<code>x()</code>, <code>y()</code>), <code>equals()</code>, <code>hashCode()</code>, <code>toString()</code>. Records are implicitly <code>final</code>, cannot extend other classes (but can implement interfaces). Support compact constructors for validation: <code>record Range(int min, int max) { Range { if (min &gt; max) throw new IllegalArgumentException(); } }</code>. Can have static fields, static methods, and instance methods. Used with Jackson (2.12+), JPA projections.

What are <b>pattern matching</b> improvements in Java 16+?	<b>instanceof pattern matching</b> (Java 16): <code>if (obj instanceof String s) { s.length(); }</code> — eliminates explicit cast. Works with <code>&amp;&amp;</code>: <code>if (obj instanceof String s &amp;&amp; s.length() &gt; 5)</code>. <b>Switch pattern matching</b> (Java 21): <code>switch (obj) { case Integer i -&gt; i * 2; case String s -&gt; s.length(); case null -&gt; 0; default -&gt; -1; }</code>. <b>Guarded patterns</b>: <code>case String s when s.length() &gt; 5 -&gt; ...</code>. <b>Record patterns</b> (Java 21): <code>if (obj instanceof Point(int x, int y)) { ... }</code> — destructures records directly.

What are <b>text blocks</b> in Java 15?	Text blocks (JEP 378, finalized Java 15) are multi-line string literals delimited by triple quotes <code>"""</code>: <pre>String json = """
    {
        "name": "Alice"
    }
    """;</pre>. Features: (1) <b>Incidental whitespace</b> is stripped based on indentation of the closing <code>"""</code>. (2) Preserves intentional line breaks. (3) No need to escape double quotes or newlines. (4) New escape sequences: <code>\</code> (line continuation), <code>\s</code> (trailing space). Ideal for JSON, HTML, SQL, XML snippets embedded in code. The result is a plain <code>String</code> — no new type.

What is the <b>module system</b> (JPMS) introduced in Java 9?	The Java Platform Module System (Project Jigsaw, JEP 261) introduces explicit module boundaries via <code>module-info.java</code>: <code>module com.example.app { requires com.example.lib; exports com.example.api; opens com.example.impl to framework; }</code>. Benefits: (1) <b>Strong encapsulation</b>: internal packages inaccessible by default. (2) <b>Reliable configuration</b>: missing dependencies detected at startup. (3) <b>Reduced footprint</b>: create custom runtime images with <code>jlink</code>. Challenges: many libraries pre-date JPMS; unnamed module/open modules ease migration. JDK itself is fully modularized (25+ modules in <code>java.*</code> and <code>jdk.*</code>).

What is the difference between <b>String</b>, <b>StringBuilder</b>, and <b>StringBuffer</b>?	<b>String</b>: immutable — every modification creates a new object. Thread-safe by nature. Stored in String Pool if literal. <b>StringBuilder</b>: mutable, not thread-safe, O(1) amortized append. Backed by a char array that grows as needed (initial capacity 16). <b>StringBuffer</b>: mutable, thread-safe via synchronized methods, slightly slower than StringBuilder. <b>Use StringBuilder</b> for single-threaded string building. <b>Use String</b> for immutable values. <b>Avoid StringBuffer</b> unless you genuinely need thread safety (prefer other synchronization strategies). The JDK compiler automatically uses StringBuilder for <code>+</code> concatenation in loops (Java 9+ uses <code>invokedynamic</code> via StringConcatFactory for even better performance).

What is <b>String interning</b>?	String interning stores string objects in a <b>String Pool</b> (part of the heap since Java 8, previously PermGen). String literals are automatically interned — <code>"abc" == "abc"</code> is true. <code>new String("abc")</code> creates a new object outside the pool. <code>str.intern()</code> returns the pool's canonical copy (or adds it). Benefits: memory savings when many strings have the same value. <b>Caveats:</b> <code>intern()</code> is relatively expensive (hashing, synchronization). Only intern strings that will have many duplicates with long lifetimes. In Java 8+, the pool is GC'd like normal heap objects.

What is <b>String.format()</b> vs <b>String.formatted()</b>?	Both produce formatted strings using printf-style specifiers (<code>%s</code>, <code>%d</code>, <code>%f</code>, <code>%n</code>, etc.). <b>String.format(String fmt, Object... args)</b>: static method, available since Java 5. <b>"Hello %s".formatted(name)</b> (Java 15+): instance method — more readable in method chains. Both use <code>java.util.Formatter</code> internally and support locale-aware formatting via overloads. <b>Performance note:</b> both create a new String object; for very high-frequency formatting, consider StringBuilder or templating alternatives. Java 21+ introduces <b>String Templates</b> (preview) for safer embedded interpolation.

What is the contract between <b>equals()</b> and <b>hashCode()</b>?	The contract (from <code>Object</code> specification): if <code>a.equals(b)</code> is true, then <code>a.hashCode() == b.hashCode()</code> must be true. <b>The reverse is NOT required</b> — two unequal objects may have the same hash code (collision). <b>Violations break hash-based collections</b>: HashMap, HashSet, Hashtable store objects by hash — if two equal objects have different hashes, one won't be found after insertion. <b>Best practices:</b> always override <code>hashCode()</code> when overriding <code>equals()</code>. Use IDE generation or <code>Objects.hash(field1, field2)</code>. <code>equals()</code> must be: reflexive, symmetric, transitive, consistent, and <code>x.equals(null) == false</code>.

What is the difference between <b>==</b> and <b>equals()</b> in Java?	<b>==</b>: (1) For <em>primitives</em>: compares values. (2) For <em>objects</em>: compares <b>references</b> (memory addresses) — two different objects with identical content are not <code>==</code>. <b>equals()</b>: defined in <code>Object</code> as reference equality by default, but typically overridden to compare <b>content/value</b>. Examples: <code>new String("abc") == new String("abc")</code> is false; <code>new String("abc").equals(new String("abc"))</code> is true. <b>Common trap:</b> comparing boxed types (<code>Integer</code>) with <code>==</code> works for cached range (-128..127) but fails outside. Always use <code>equals()</code> for object comparisons; use <code>Objects.equals(a, b)</code> for null-safe comparison.

What is <b>autoboxing</b> and <b>unboxing</b>?	<b>Autoboxing</b>: automatic conversion from primitive to wrapper type (e.g., <code>int</code> → <code>Integer</code>, <code>double</code> → <code>Double</code>) by the compiler. <b>Unboxing</b>: reverse conversion (wrapper → primitive). Occurs transparently in assignments, method calls, and collections. <b>Pitfalls:</b> (1) <b>NullPointerException</b> on unboxing a null wrapper. (2) <b>Performance</b>: creates/destroys wrapper objects in tight loops — use primitive streams (<code>IntStream</code>) or arrays. (3) <b>Unexpected == comparisons</b>: autoboxed Integer outside the -128..127 cache range produces different objects. (4) Confusing overloaded methods: <code>list.remove(1)</code> (index) vs <code>list.remove(Integer.valueOf(1))</code> (element).

What is the <b>Integer cache</b> in Java?	Java caches <code>Integer</code> objects for values <b>-128 to 127</b> in a static array (initialized at class loading). <code>Integer.valueOf(100)</code> always returns the same cached object — <code>== </code> comparison holds. <code>Integer.valueOf(200)</code> creates a new object — <code>==</code> fails. The upper bound can be increased with <code>-XX:AutoBoxCacheMax=&lt;max&gt;</code> JVM flag. Similar caches exist for <code>Boolean</code> (TRUE/FALSE), <code>Byte</code> (all), <code>Short</code> (-128..127), <code>Long</code> (-128..127), <code>Character</code> (0..127). <b>Lesson:</b> always use <code>equals()</code> for Integer comparisons; never rely on <code>==</code>.

What is <b>var</b> (local variable type inference) in Java 10?	<code>var</code> (JEP 286) instructs the compiler to infer the type of a local variable from the initializer. <code>var list = new ArrayList&lt;String&gt;()</code> infers <code>ArrayList&lt;String&gt;</code>. <b>Still statically typed</b> — not dynamic like JavaScript. Restrictions: only for local variables with initializers, not for fields, method parameters, return types, or catch variables (Java 10). Java 11 allows <code>var</code> in lambda parameters for annotations: <code>(@NonNull var x) -&gt; x</code>. <b>When to use:</b> reduces verbosity for obvious types (<code>var map = new HashMap&lt;String, List&lt;Integer&gt;&gt;()</code>). <b>When to avoid:</b> when the type is not obvious from the right-hand side — hurts readability.

What is <b>instanceof</b> pattern matching?	Java 16 finalized pattern matching for <code>instanceof</code> (JEP 394). Combines a type check with a <b>binding variable</b>: <code>if (obj instanceof String s) { ... use s directly ... }</code> — no explicit cast needed. The binding variable <code>s</code> is in scope and definitely typed within the true branch. Works in compound conditions: <code>if (obj instanceof String s &amp;&amp; s.length() &gt; 5)</code>. Also negated: <code>if (!(obj instanceof String s)) return; // s in scope after</code>. Java 21 extends this to switch expressions with guarded patterns and record deconstruction.

What are <b>virtual threads</b> (Project Loom) in Java 21?	Virtual threads (JEP 444, finalized Java 21) are JVM-managed lightweight threads, not mapped 1:1 to OS threads. Millions can exist simultaneously (each using ~few KB of heap). When a virtual thread blocks on I/O, the JVM <b>unmounts</b> it from its carrier OS thread (from ForkJoinPool), freeing the carrier for other virtual threads. Creation: <code>Thread.ofVirtual().start(runnable)</code>, <code>Thread.startVirtualThread(runnable)</code>, or <code>Executors.newVirtualThreadPerTaskExecutor()</code>. Key benefit: write blocking-style code (simple, readable) with async scalability (no callback hell). Ideal for I/O-bound, request-per-thread models (REST servers, database calls).

What is the difference between <b>virtual threads</b> and platform threads?	<b>Platform threads</b>: 1:1 mapping to OS threads. Each costs ~1MB of stack memory. Creating thousands is expensive. Blocking a platform thread blocks the OS thread. Managed by the OS scheduler. <b>Virtual threads</b>: M:N mapping — many virtual threads share few OS carrier threads. Each costs only a few KB of heap. Blocking is cheap — the JVM parks the virtual thread and reuses the carrier. Managed by the JVM scheduler (ForkJoinPool). <b>Don't use virtual threads for CPU-intensive tasks</b> — they still run on carrier threads and don't add parallelism; use platform threads or ForkJoinPool for those. Avoid thread-local variables in virtual threads at scale (different semantics with many short-lived threads).

What are <b>structured concurrency</b> improvements in Java 21?	Structured concurrency (JEP 453, preview Java 21) treats a group of related concurrent tasks as a single unit. <code>StructuredTaskScope</code> ensures all spawned subtasks complete (or are cancelled) before the scope closes: <pre>try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    var f1 = scope.fork(task1);
    var f2 = scope.fork(task2);
    scope.join().throwIfFailed();
    return f1.get() + f2.get();
}</pre>. <code>ShutdownOnFailure</code>: cancels all on first failure. <code>ShutdownOnSuccess</code>: cancels rest on first success. Benefits: predictable lifetime, better error handling, no orphaned threads, simpler reasoning than CompletableFuture chains.

What is <b>sequenced collections</b> (Java 21)?	JEP 431 introduces three new interfaces into the Collections framework: <b>SequencedCollection&lt;E&gt;</b> (extends Collection): <code>getFirst()</code>, <code>getLast()</code>, <code>addFirst()</code>, <code>addLast()</code>, <code>removeFirst()</code>, <code>removeLast()</code>, <code>reversed()</code>. <b>SequencedSet&lt;E&gt;</b> (extends Set + SequencedCollection): like LinkedHashSet. <b>SequencedMap&lt;K,V&gt;</b> (extends Map): <code>firstEntry()</code>, <code>lastEntry()</code>, <code>putFirst()</code>, <code>putLast()</code>, <code>sequencedKeySet()</code>, <code>reversed()</code>. Solves the inconsistency where different collections had different ways to access first/last elements (e.g., <code>Deque.peekFirst()</code> vs <code>LinkedHashMap</code> having no direct API).

What is <b>ClassLoader</b> hierarchy in Java?	Java has three built-in ClassLoaders in a parent-child hierarchy: (1) <b>Bootstrap ClassLoader</b>: written in C, loads core JDK classes from <code>rt.jar</code>/modules (<code>java.lang</code>, <code>java.util</code>, etc.). Has no parent — the root. (2) <b>Platform ClassLoader</b> (Extension ClassLoader before Java 9): loads JDK extension/platform modules (<code>java.se</code>, etc.). (3) <b>Application (System) ClassLoader</b>: loads classes from the application classpath (<code>-classpath</code> or <code>CLASSPATH</code> env). Application ClassLoaders (servlet containers, OSGi) may add more layers. Each ClassLoader has a reference to its parent.

What is the <b>parent delegation model</b> of ClassLoaders?	When a ClassLoader receives a class loading request, it <em>first</em> delegates to its <b>parent</b> ClassLoader. Only if the parent (and its ancestors) cannot find the class does the child attempt to load it. This ensures: (1) Core JDK classes are always loaded by Bootstrap ClassLoader — cannot be replaced by user code. (2) No duplicate class definitions across loaders in the same hierarchy. (3) Security: malicious classes masquerading as <code>java.lang.String</code> are rejected. <b>Exception</b>: OSGi and application servers intentionally break delegation for class isolation (child-first loading).

What is a <b>custom ClassLoader</b> used for?	Custom ClassLoaders (extending <code>ClassLoader</code>, overriding <code>findClass()</code>) enable: (1) <b>Non-standard sources</b>: loading classes from network, database, encrypted jars. (2) <b>Hot reloading</b>: discard old ClassLoader instance to "reload" classes at runtime (used in development servers like JRebel, Spring DevTools). (3) <b>Class isolation</b>: plugin systems (OSGi, Java EE containers) give each plugin its own ClassLoader namespace to avoid version conflicts. (4) <b>Bytecode transformation</b>: decrypt or instrument classes before defining them. Important: classes from different ClassLoaders are incompatible even if same binary name.

What is <b>bytecode</b> in Java?	Bytecode is the <b>platform-independent intermediate representation</b> of compiled Java programs — instruction set for the JVM, not the CPU. Stored in <code>.class</code> files. The JVM interprets bytecode (or JIT-compiles to native code). Bytecode is more compact and higher-level than native code. Tools: <code>javap -c MyClass</code> disassembles bytecode. Bytecode instructions include: <code>iload</code>, <code>istore</code>, <code>invokevirtual</code>, <code>invokespecial</code>, <code>invokedynamic</code>, <code>new</code>, <code>checkcast</code>, etc. Java isn't unique — Kotlin, Scala, Groovy, Clojure all compile to JVM bytecode, enabling cross-language interoperability.

What is <b>bytecode manipulation</b>? Name libraries.	Bytecode manipulation directly reads, modifies, or generates <code>.class</code> file bytecode, enabling capabilities beyond what Java source allows: AOP weaving, mocking, code generation, profiling, hot patching. <b>Libraries:</b> (1) <b>ASM</b>: low-level, visitor API, minimal overhead, used by almost all other tools. (2) <b>Javassist</b>: source-level API — manipulate classes using Java source strings. (3) <b>ByteBuddy</b>: fluent API, type-safe, used by Mockito (mock generation), Hibernate (proxy creation), and many frameworks. (4) <b>CGLIB</b>: extends classes dynamically, used by Spring/Hibernate for proxies. Manipulation can happen at compile time (via annotation processors) or at load/runtime (via agents).

What is a <b>Java agent</b>?	A Java agent instruments classes via <code>java.lang.instrument.Instrumentation</code> API. <b>Load-time agent</b>: JVM flag <code>-javaagent:agent.jar</code> invokes <code>premain(String args, Instrumentation inst)</code> before <code>main()</code>. <b>Dynamic agent</b>: attached at runtime via <code>VirtualMachine.attach()</code>, invokes <code>agentmain()</code>. Agents register a <code>ClassFileTransformer</code> that intercepts and transforms bytecode for each class as it is loaded. <b>Use cases:</b> APM tools (New Relic, Datadog), profilers (YourKit, JProfiler), code coverage (JaCoCo), AOP frameworks, mock libraries, hot patching. Agents can also retransform already-loaded classes.

What is <b>AOP (Aspect-Oriented Programming)</b>?	AOP modularizes <b>cross-cutting concerns</b> (logging, security, transactions, caching, metrics) that span many classes, separating them from business logic. Core concepts: <b>Aspect</b>: the module of cross-cutting concern. <b>Advice</b>: code to execute (Before, After, AfterReturning, AfterThrowing, Around). <b>Pointcut</b>: expression matching join points where advice applies. <b>Join point</b>: specific point in execution (method call, field access). <b>Weaving</b>: applying aspects — at compile time (AspectJ), load time (AspectJ LTW), or runtime (Spring AOP via proxies). Spring AOP supports only method-level join points; full AspectJ supports field access, constructors, etc.

What is the difference between <b>JDK dynamic proxy</b> and <b>CGLIB proxy</b>?	<b>JDK Dynamic Proxy</b>: built into the JDK (<code>java.lang.reflect.Proxy</code>). Works <em>only</em> with interfaces — creates a proxy class implementing the target's interfaces. Invocation handler intercepts calls. Slightly faster to create. <b>CGLIB (Code Generation Library)</b>: generates a <em>subclass</em> of the target class at runtime, overriding non-final methods to add advice. Works without interfaces. Cannot proxy <code>final</code> classes or methods. Slightly slower to create (subclass generation), but similar runtime performance. <b>Spring AOP:</b> uses JDK proxy when target implements an interface (default), CGLIB otherwise. Can force CGLIB with <code>@EnableAspectJAutoProxy(proxyTargetClass=true)</code>. Spring Boot defaults to CGLIB since Spring 5.

What is <b>serialization</b> in Java?	Java serialization converts an object graph to a byte stream for storage or network transfer and back. Requirements: class must implement <code>java.io.Serializable</code> (marker interface). <code>serialVersionUID</code> (static final long): used for version compatibility — if not declared, computed from class structure; mismatch on deserialization throws <code>InvalidClassException</code>. <code>transient</code> fields are not serialized. <b>Process:</b> <code>ObjectOutputStream.writeObject(obj)</code> / <code>ObjectInputStream.readObject()</code>. Custom hooks: <code>writeObject()</code>/<code>readObject()</code>, <code>writeReplace()</code>/<code>readResolve()</code>. Entire reachable object graph is serialized recursively — all referenced objects must also be Serializable.

What are the risks of Java <b>serialization</b>?	(1) <b>Security</b>: deserialization of untrusted data can execute arbitrary code via "gadget chains" — critical CVEs in Apache Commons Collections, Spring, etc. Use <code>ObjectInputFilter</code> (Java 9+) to whitelist allowed classes. (2) <b>Performance</b>: slow, verbose binary format. (3) <b>Brittleness</b>: tight coupling to class structure — adding a field can break deserialization without careful <code>serialVersionUID</code> management. (4) <b>No schema</b>: no contract enforcement between producer and consumer. <b>Alternatives:</b> JSON (Jackson, Gson), Protocol Buffers (Protobuf), Avro, Thrift, Kryo — faster, language-agnostic, schema-controlled.

What is <b>externalizable</b> vs <b>serializable</b>?	<b>Serializable</b>: marker interface — Java automatically serializes all non-transient fields using reflection. Simple but slow and inflexible. <b>Externalizable</b>: extends Serializable and requires implementing <code>writeExternal(ObjectOutput)</code> and <code>readExternal(ObjectInput)</code>. You control exactly what is serialized — can be significantly faster and produce a smaller payload. <b>Requirement:</b> Externalizable classes must have a public no-arg constructor (used by deserialization before calling <code>readExternal()</code>). Use Externalizable when performance is critical and you need precise control; otherwise modern alternatives (Protobuf, Kryo) are better choices.

What is <b>NIO (New I/O)</b> in Java?	<code>java.nio</code> (Java 4+) provides a more efficient I/O model: (1) <b>Channels</b> (<code>FileChannel</code>, <code>SocketChannel</code>, <code>ServerSocketChannel</code>): bidirectional I/O to/from Buffers. (2) <b>Buffers</b> (<code>ByteBuffer</code>, <code>IntBuffer</code>, etc.): fixed-size containers holding data; key positions: <code>position</code>, <code>limit</code>, <code>capacity</code>; <code>flip()</code> switches read/write mode. (3) <b>Selectors</b>: multiplexes multiple non-blocking channels on a single thread — event-driven I/O (like epoll/select). <b>NIO.2</b> (Java 7, <code>java.nio.file</code>): <code>Path</code>, <code>Files</code>, <code>WatchService</code>, async file I/O (<code>AsynchronousFileChannel</code>).

What is the difference between <b>blocking</b> and <b>non-blocking I/O</b>?	<b>Blocking I/O</b> (traditional <code>java.io</code>): the calling thread blocks (sleeps) until data is available or written. Simple programming model but each concurrent connection requires a dedicated thread — poor scalability. <b>Non-blocking I/O</b> (java.nio): <code>channel.configureBlocking(false)</code> makes operations return immediately if data isn't ready (returning 0 or -1). Used with a <b>Selector</b>: one thread monitors many channels, processing whichever are ready. Enables high connection concurrency with few threads. <b>Asynchronous I/O</b> (NIO.2): callbacks or Futures notified upon completion — no polling needed.

What is <b>memory-mapped file I/O</b>?	<code>FileChannel.map(MapMode, position, size)</code> maps a region of a file directly into JVM memory space as a <code>MappedByteBuffer</code>. The OS handles paging data between disk and memory on demand (<b>demand paging</b>). <b>Benefits:</b> eliminates explicit read/write system calls for subsequent access — treat file content like an in-memory array. Extremely fast for large files (logs, databases). Changes to the buffer are automatically persisted (for READ_WRITE mode). <b>Caveats:</b> the mapping remains until the buffer is GC'd (no explicit unmap in Java); can cause <code>OutOfMemoryError</code> if too many mappings exist.

What is the <b>ServiceLoader</b> in Java?	<code>ServiceLoader&lt;S&gt;</code> (Java 6+) implements the <b>Service Provider Interface (SPI)</b> pattern for runtime plugin discovery without hard-coded dependencies. Providers declare themselves in: (1) <code>META-INF/services/fully.qualified.ServiceInterface</code> (text file listing provider class names). (2) <code>module-info.java</code>: <code>provides com.example.Service with com.example.impl.ServiceImpl;</code> (Java 9+). Usage: <code>ServiceLoader.load(Service.class).forEach(...)</code>. Used by: JDBC drivers (auto-registration), logging frameworks (SLF4J), <code>java.nio.file.FileSystemProvider</code>, <code>javax.xml</code> parsers, Spring factories.

What is <b>method inlining</b> in JIT?	Method inlining is the JIT's most impactful optimization: it replaces a method call site with the method body directly, eliminating call overhead (stack frame setup/teardown, argument passing) and enabling further optimizations on the combined code (constant folding, escape analysis, loop unrolling). <b>Triggers:</b> method is called frequently ("hot") and is small enough (≤35 bytecodes for hot methods, ≤325 for trivial). <code>final</code>, <code>private</code>, and <code>static</code> methods are easier to inline (no virtual dispatch). Too-large methods won't inline — keep hot-path methods small. HotSpot inlines up to 9 levels deep. Use <code>-XX:+PrintInlining</code> to observe.

What is <b>on-stack replacement (OSR)</b>?	OSR is a JIT technique that replaces an <em>actively executing</em> method's interpreter stack frame with an optimized JIT-compiled version, while the method is still running — typically in the middle of a long loop. Without OSR, the JVM would have to wait until the method returns to switch to compiled code. OSR enables long-running loops to benefit from JIT optimization without restarting. The transition involves mapping interpreter state (local variables, stack) to the compiled frame's representation. Triggered when a method's loop back-edge counter exceeds a compilation threshold.

What is <b>HotSpot&#39;s tiered compilation</b>?	HotSpot uses five compilation tiers: <b>Level 0</b>: pure interpretation — initial execution. <b>Level 1</b>: C1 compiled, no profiling — trivial methods. <b>Level 2</b>: C1 compiled, limited profiling. <b>Level 3</b>: C1 compiled, full profiling (method/branch/type profiles collected). <b>Level 4</b>: C2 fully optimized — uses profiles from Level 3 for aggressive optimizations (inlining, devirtualization, loop transformations). Methods advance through tiers based on invocation count and back-edge count. Hot methods reach Level 4; less hot stay at Level 1-3. Tiered compilation is default since Java 8. Controlled with <code>-XX:+TieredCompilation</code>/<code>-XX:-TieredCompilation</code>.

What is <b>deoptimization</b> in JVM?	Deoptimization rolls back a JIT-compiled method to interpreted mode when a speculative optimization assumption is violated. Examples of invalidated assumptions: (1) A new subclass is loaded, invalidating a devirtualized call. (2) A type profile changes (previously monomorphic call site becomes polymorphic). (3) An uncommon trap is triggered (e.g., null check that was optimized away). The JVM transfers execution state from compiled to interpreter frame. Deoptimization is normal and expected — HotSpot may recompile with less aggressive optimizations afterward. Frequent deoptimization ("deopt storms") can hurt performance; observable with <code>-XX:+PrintDeoptimization</code>.

What is the difference between <b>static binding</b> and <b>dynamic binding</b>?	<b>Static binding (early binding)</b>: the method to be called is determined at <b>compile time</b>. Applies to: <code>private</code> methods (not overridable), <code>static</code> methods (belong to class), <code>final</code> methods (not overridable), and constructors. <b>Dynamic binding (late binding)</b>: the method is resolved at <b>runtime</b> based on the actual type of the object, not the reference type. This is the default for <code>public</code> and <code>protected</code> instance methods — implements polymorphism via the <b>vtable (virtual method table)</b>. JIT can optimize dynamic dispatch via devirtualization if the actual type is provably unique (monomorphic call site).

What is <b>covariant return type</b>?	Java 5+ allows an overriding method to return a <b>more specific (subtype)</b> return type than the parent method's declared return type. Example: <code>Object.clone()</code> returns <code>Object</code>, but <code>ArrayList.clone()</code> can return <code>ArrayList</code>. This avoids unchecked casts at the call site. The JVM implements this via a <b>bridge method</b> (generated by the compiler) with the parent's return type that delegates to the actual method. Covariant returns improve API design — builder patterns use them extensively (<code>return this</code> in subclass methods returning the subclass type).

What is the difference between <b>overloading</b> and <b>overriding</b>?	<b>Overloading</b>: multiple methods in the same class (or hierarchy) with the same name but <em>different parameter lists</em>. Resolved at <b>compile time</b> (static polymorphism / ad-hoc polymorphism). Return type and exceptions can differ. <b>Overriding</b>: a subclass redefines a method from the superclass with the <em>same signature</em>. Resolved at <b>runtime</b> (dynamic polymorphism). Rules: same name, same parameters, return type must be same or covariant, cannot reduce visibility, cannot throw broader checked exceptions. Use <code>@Override</code> annotation to get compile-time verification. Static methods are hidden (not overridden) — no polymorphism for static methods.

What is <b>covariance</b> and <b>contravariance</b> in Java?	<b>Covariance</b> (<code>? extends T</code>): a subtype can be used where a supertype is expected. Java arrays are covariant (<code>String[]</code> is a <code>Object[]</code>) — but this is <em>unsafe</em> (ArrayStoreException at runtime). Generic wildcards: <code>List&lt;? extends Number&gt;</code> is covariant — you can read Numbers but not write (type unknown). Covariant return types in overriding. <b>Contravariance</b> (<code>? super T</code>): a supertype is used where a subtype is expected. <code>List&lt;? super Integer&gt;</code> — you can write Integers but reading yields Object. <b>Invariance</b>: <code>List&lt;String&gt;</code> is NOT a <code>List&lt;Object&gt;</code> — neither covariant nor contravariant.

What is the difference between <b>interface default methods</b> and <b>abstract class methods</b>?	Both provide concrete implementations that subclasses can inherit or override. Key differences: (1) <b>State</b>: abstract classes can have instance fields; interfaces cannot (only public static final constants). (2) <b>Inheritance</b>: a class can <code>implement</code> multiple interfaces but <code>extend</code> only one abstract class. (3) <b>Constructor</b>: abstract classes have constructors; interfaces don't. (4) <b>Access modifiers</b>: abstract class methods can be any visibility; interface default methods are implicitly public. (5) <b>Purpose</b>: default methods enable interface evolution (add methods without breaking existing implementations); abstract classes model shared state + behavior.

What is <b>diamond problem</b> in Java interfaces?	The diamond problem occurs when a class <code>C</code> implements two interfaces <code>A</code> and <code>B</code> that both have a <code>default</code> method with the same signature. Java requires <code>C</code> to explicitly override the method to resolve the ambiguity. To call a specific interface's default: <code>A.super.method()</code>. <b>Resolution rules:</b> (1) Class/superclass wins over interface default. (2) More specific interface wins over less specific (if one interface extends the other). (3) Otherwise, explicit override required. This issue doesn't apply to abstract methods (no implementation to conflict).

What is <b>static initializer block</b>?	A <code>static { ... }</code> block in a class executes <em>exactly once</em> when the class is first loaded by the ClassLoader, before any constructor or static method call. Multiple static blocks execute in <b>declaration order</b>. Used for: complex static field initialization, loading native libraries (<code>System.loadLibrary()</code>), initializing static collections, validation. <b>Instance initializer blocks</b> (<code>{ ... }</code> without static) run before every constructor call (copied into each constructor by the compiler). Static initializer exceptions wrap in <code>ExceptionInInitializerError</code>, permanently disabling the class.

What is <b>double-checked locking</b> (DCL) and when is it needed?	DCL is a pattern for <b>lazy initialization</b> of a shared resource with minimal synchronization overhead after initialization: <pre>private volatile Instance instance;
public Instance getInstance() {
    if (instance == null) {           // 1st check (no lock)
        synchronized (this) {
            if (instance == null) {    // 2nd check (with lock)
                instance = new Instance();
            }
        }
    }
    return instance;
}</pre>. <b>Why volatile is essential:</b> without it, the JVM or CPU may reorder the write to <code>instance</code> before the object's constructor completes — another thread sees a non-null but partially constructed object. <code>volatile</code> prevents this reordering (happens-before guarantee). Prefer <b>Bill Pugh (static inner class)</b> or <b>enum singleton</b> as simpler alternatives.

What is the <b>Flyweight</b> pattern?	Flyweight minimizes memory by sharing as much data as possible between many similar objects. Divides state into: <b>intrinsic</b> (shared, stored in flyweight) and <b>extrinsic</b> (context-dependent, passed by client). Example: a text editor stores one Character object per glyph type, not per occurrence — position (extrinsic) is passed separately. Java examples: <b>String pool</b> (interned strings), <b>Integer cache</b> (-128..127), <b>Boolean.TRUE/FALSE</b>, <b>Character cache</b>. Works well with a factory that returns cached instances: <code>Character.valueOf(c)</code>. Trade-off: reduced memory at the cost of more complex code.

What is the <b>State</b> pattern?	State allows an object to change its behavior when its internal state changes, appearing to change its class. Instead of large <code>if/switch</code> on a state field, each state is a separate class implementing a common interface. The context delegates behavior to the current state object and can change state by swapping the reference. Example: <code>TrafficLight</code> context with <code>RedState</code>, <code>GreenState</code>, <code>YellowState</code> — each handles <code>next()</code> differently. Benefits: eliminates state-based conditionals, easy to add new states, single responsibility per state. Common in parsers, workflow engines, game entities.

What is the <b>Composite</b> pattern?	Composite organizes objects into <b>tree structures</b> to represent part-whole hierarchies. Both leaf and composite nodes implement the same <b>Component</b> interface, so clients treat them uniformly. Example: a file system — <code>File</code> (leaf) and <code>Directory</code> (composite, contains Files and Directories) both implement <code>FileSystemItem</code> with <code>getSize()</code>. Calling <code>getSize()</code> on a directory recursively sums its children. Java AWT/Swing uses Composite: <code>Component</code> (leaf) and <code>Container</code> (composite). Useful when client code should ignore the difference between individual and composed objects.

What is the <b>Facade</b> pattern?	Facade provides a <b>simplified, high-level interface</b> to a complex subsystem, reducing coupling between clients and the subsystem's internals. The facade class delegates to the appropriate subsystem classes. Example: <code>HomeTheaterFacade</code> with a single <code>watchMovie()</code> method that internally coordinates projector, amplifier, DVD player, and lights. Java examples: <code>java.net.URL.openStream()</code> (hides socket/HTTP/stream setup), Spring's <code>JdbcTemplate</code> (hides JDBC boilerplate), <code>SLF4J</code> (hides logging framework). Difference from Adapter: Facade simplifies; Adapter converts.

What is <b>StackOverflowError</b> and what causes it?	<code>StackOverflowError</code> is thrown when the JVM's thread stack exceeds its allocated size. Each method call pushes a new frame (local variables, operand stack, return address) onto the stack. <b>Causes:</b> (1) Infinite or excessively deep recursion (most common). (2) Very deep call chains with large stack frames. (3) Mutual recursion. <b>Solutions:</b> add a base case to recursion, convert to iterative with an explicit stack, increase thread stack size with <code>-Xss</code> (e.g., <code>-Xss4m</code>) — only for legitimate deep stacks. Default stack size is ~512KB–1MB depending on OS and JVM version.

What is <b>OutOfMemoryError</b>? Name common causes.	<code>OutOfMemoryError</code> is thrown when the JVM cannot allocate an object due to insufficient memory. <b>Common causes:</b> (1) <b>Java heap space</b>: heap is full — memory leak or insufficient <code>-Xmx</code>. (2) <b>GC overhead limit exceeded</b>: JVM spending >98% of time in GC with <2% heap recovered. (3) <b>Metaspace</b>: class metadata area full — too many classes loaded (e.g., excessive dynamic proxies). (4) <b>Unable to create native thread</b>: OS thread limit reached — too many threads. (5) <b>Direct buffer memory</b>: NIO direct buffers exhausted. (6) <b>Requested array size exceeds VM limit</b>: array allocation too large. Enable <code>-XX:+HeapDumpOnOutOfMemoryError</code> for post-mortem analysis.

What is a <b>heap dump</b> and how do you analyze it?	A heap dump is a point-in-time snapshot of all objects in the JVM heap, including their types, sizes, references, and field values. <b>Capture methods:</b> <code>jmap -dump:format=b,file=heap.hprof &lt;pid&gt;</code>, JVM flag <code>-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/</code>, VisualVM, JConsole, jcmd. <b>Analysis tools:</b> <b>Eclipse MAT (Memory Analyzer Tool)</b>: finds memory leaks via Leak Suspects, analyzes retained heap, finds GC roots. <b>VisualVM</b>: basic heap analysis. <b>YourKit / JProfiler</b>: commercial profilers. Key concepts: <b>retained heap</b> (memory freed if object GC'd), <b>dominator tree</b>, <b>GC roots</b> analysis.

What is <b>JMX (Java Management Extensions)</b>?	JMX (JSR 3) provides a standardized API for <b>monitoring and managing Java applications at runtime</b>. Core concept: <b>MBeans (Managed Beans)</b> — objects exposing attributes and operations via the MBeanServer. Types: Standard MBean (interface-based), Dynamic MBean, Open MBean, MXBean (uses open types). The JDK exposes many built-in MXBeans: <code>MemoryMXBean</code>, <code>ThreadMXBean</code>, <code>RuntimeMXBean</code>, <code>GarbageCollectorMXBean</code> — accessible via <code>ManagementFactory</code>. Clients: <b>JConsole</b>, <b>VisualVM</b>, <b>Java Mission Control</b>. Remote management via JMX connector (RMI or JMXMP). Used by APM agents and application servers for runtime metrics.

What is the difference between <b>fail-fast</b> and <b>fail-safe</b> iterators?	<b>Fail-fast</b> iterators detect structural modification during iteration by checking a <code>modCount</code> counter. If the collection is modified (add/remove) by any means other than the iterator's own <code>remove()</code>, they throw <code>ConcurrentModificationException</code> immediately. Not guaranteed in all cases — best-effort. Examples: <code>ArrayList</code>, <code>HashMap</code>, <code>HashSet</code> iterators. <b>Fail-safe</b> (weakly consistent) iterators work on a snapshot or are designed to tolerate concurrent modification. They never throw CME but may not reflect modifications made after iterator creation. Examples: <code>CopyOnWriteArrayList</code>, <code>ConcurrentHashMap</code> iterators.

What is <b>CopyOnWriteArrayList</b>?	<code>CopyOnWriteArrayList</code> (and <code>CopyOnWriteArraySet</code>) is a thread-safe variant where every mutating operation (<code>add</code>, <code>set</code>, <code>remove</code>) creates a <b>fresh copy</b> of the underlying array. Reads operate on the existing array without any locking — completely lock-free reads. Iterators snapshot the array at creation time — never throw <code>ConcurrentModificationException</code>, but don't reflect subsequent changes. <b>Best for:</b> collections with very infrequent writes but frequent reads (e.g., event listener lists, routing tables). <b>Avoid for:</b> write-heavy workloads — O(n) write cost plus GC pressure from array copies.

What is a <b>Phaser</b> in Java concurrency?	<code>Phaser</code> (Java 7+) is the most flexible synchronization barrier, subsuming both <code>CountDownLatch</code> and <code>CyclicBarrier</code>. Key features: (1) <b>Dynamic party registration</b>: parties can register/deregister at any time with <code>register()</code>/<code>arriveAndDeregister()</code>. (2) <b>Multiple phases</b>: automatically advances to the next phase when all parties arrive — reusable like CyclicBarrier. (3) <b>Tiered Phasers</b>: parent-child hierarchy to reduce contention at large party counts. (4) Override <code>onAdvance(phase, parties)</code> to run code at each phase transition or to terminate. (5) Supports both blocking (<code>arriveAndAwaitAdvance()</code>) and non-blocking (<code>arrive()</code>) participation.

What is <b>StampedLock</b>?	<code>StampedLock</code> (Java 8+) provides three locking modes via stamp (long token) return values: (1) <b>Write lock</b>: exclusive, blocks all readers and writers: <code>long stamp = lock.writeLock()</code>. (2) <b>Read lock</b>: shared, blocks writers: <code>long stamp = lock.readLock()</code>. (3) <b>Optimistic read</b>: non-blocking, no actual lock acquired: <code>long stamp = lock.tryOptimisticRead()</code> — validate afterward with <code>lock.validate(stamp)</code> and retry with a real read lock if invalidated. <b>Tradeoffs:</b> higher throughput than <code>ReentrantReadWriteLock</code> for read-heavy workloads, but StampedLock is NOT reentrant and does not support condition variables.

What is <b>LockSupport</b>?	<code>LockSupport</code> provides the foundational primitives for building custom synchronizers. <b><code>LockSupport.park()</code></b>: suspends the current thread (like <code>Object.wait()</code> but without a monitor lock). <b><code>LockSupport.unpark(thread)</code></b>: gives a permit to the specified thread — if it's parked, it unblocks; if it's not yet parked, the next <code>park()</code> returns immediately (permit is consumed). <b>Advantages over <code>wait()/notify()</code>:</b> no monitor required, unpark can be called before park (no missed signal), targeted unpark (by thread reference). <b>Used by:</b> <code>AbstractQueuedSynchronizer</code> (the backbone of ReentrantLock, Semaphore, CountDownLatch).

What is <b>reactive programming</b> in Java?	Reactive programming models asynchronous data flows as observable streams with back-pressure support. <b>Core concepts:</b> Publisher emits items, Subscriber consumes them, Processor transforms. <code>java.util.concurrent.Flow</code> (Java 9) standardizes the Reactive Streams interfaces: <code>Flow.Publisher</code>, <code>Flow.Subscriber</code>, <code>Flow.Subscription</code>, <code>Flow.Processor</code>. Implementations: <b>Project Reactor</b> (Spring WebFlux): <code>Mono&lt;T&gt;</code> (0-1 items), <code>Flux&lt;T&gt;</code> (0-N items). <b>RxJava</b>: <code>Observable</code>, <code>Single</code>, <code>Maybe</code>, <code>Flowable</code>. Benefits: non-blocking I/O, efficient resource use for I/O-bound apps. Drawback: steep learning curve, harder to debug stack traces.

What is <b>backpressure</b> in reactive streams?	Backpressure is the mechanism allowing a <b>Subscriber to signal</b> to the Publisher how many items it is ready to receive — preventing a fast producer from overwhelming a slow consumer. In the Reactive Streams spec: <code>Subscription.request(n)</code> tells the publisher to emit at most <code>n</code> items. The publisher must not emit more. Strategies when consumer can't keep up: (1) <b>Buffer</b>: queue excess items (risk: OOM). (2) <b>Drop</b>: discard excess items. (3) <b>Error</b>: signal overflow. (4) <b>Throttle/sample</b>: emit only latest. Without backpressure (e.g., RxJava Observable), overflow must be handled manually. Reactive Streams spec (and Java Flow) mandates backpressure support.

What is the difference between <b>submit()</b> and <b>execute()</b> in ExecutorService?	<b><code>execute(Runnable)</code></b>: defined in the <code>Executor</code> interface. Fire-and-forget — no return value. Exceptions from the task are propagated to the thread's <code>UncaughtExceptionHandler</code>. <b><code>submit(Callable&lt;T&gt;)</code></b> / <b><code>submit(Runnable)</code></b>: defined in <code>ExecutorService</code>. Returns a <code>Future&lt;T&gt;</code> — allows: (1) Retrieving the result via <code>future.get()</code>. (2) Exception handling — exceptions are captured in the Future and rethrown wrapped in <code>ExecutionException</code> on <code>get()</code>. (3) Cancellation via <code>future.cancel()</code>. Always prefer <code>submit()</code> when you need the result or proper exception handling.

What is <b>ScheduledExecutorService</b>?	<code>ScheduledExecutorService</code> extends <code>ExecutorService</code> for time-based task scheduling: <b><code>schedule(callable, delay, unit)</code></b>: run once after a delay. <b><code>scheduleAtFixedRate(runnable, initialDelay, period, unit)</code></b>: run periodically — period starts at task start. <b><code>scheduleWithFixedDelay(runnable, initialDelay, delay, unit)</code></b>: run with fixed delay between end of one and start of next. Returns <code>ScheduledFuture</code> for cancellation. Replacement for error-prone <code>Timer</code>/<code>TimerTask</code> — supports multiple threads, proper exception handling, and doesn't stop on task exception. Create with <code>Executors.newScheduledThreadPool(n)</code>.

What is the difference between <b>scheduleAtFixedRate</b> and <b>scheduleWithFixedDelay</b>?	<b><code>scheduleAtFixedRate(task, initialDelay, period, unit)</code></b>: attempts to run task every <code>period</code>. The period is measured from the <em>start</em> of each execution. If a task takes longer than the period, the next execution starts immediately after completion (no overlap — tasks run sequentially in a single-threaded executor). Useful when you want consistent tick rates (e.g., every 10s regardless of task duration). <b><code>scheduleWithFixedDelay(task, initialDelay, delay, unit)</code></b>: waits <code>delay</code> time after each task <em>completes</em> before starting the next. Useful when you want breathing room between executions regardless of task duration.

What is <b>invokeAll()</b> vs <b>invokeAny()</b> in ExecutorService?	<b><code>invokeAll(Collection&lt;Callable&gt;)</code></b>: submits all tasks, blocks until ALL complete (or timeout), returns a <code>List&lt;Future&gt;</code> where all futures are done. Exceptions are available per-future via <code>get()</code>. Use when you need results from all tasks. <b><code>invokeAny(Collection&lt;Callable&gt;)</code></b>: submits all tasks, blocks until the FIRST one completes successfully, returns its result, and cancels all remaining tasks. Throws if all tasks fail. Use for redundant/speculative execution where you only need one result (e.g., querying multiple servers). Both accept an optional timeout variant.

What is <b>try-finally vs try-with-resources</b> for performance?	<b>try-with-resources</b> generates slightly more bytecode than manual <code>finally</code> because it calls <code>Throwable.addSuppressed()</code> to handle suppressed exceptions. However, the performance difference is <b>negligible in practice</b> — measured in nanoseconds. The JIT compiler optimizes both paths similarly. <b>try-with-resources is always preferred</b> for <code>AutoCloseable</code> resources because: (1) Guarantees resource closure even on all exit paths. (2) Properly handles suppressed exceptions (no masking). (3) More readable and less error-prone. (4) Supports multiple resources in declaration order. Never use <code>finally</code> manually for closeable resources.

What are <b>switch expressions</b> in Java 14+?	Switch expressions (JEP 361, finalized Java 14) produce a value. Two syntaxes: (1) <b>Arrow (<code>-&gt;</code>) labels</b>: <code>int result = switch (day) { case MON, TUE -&gt; 1; case WED -&gt; 2; default -&gt; 0; };</code> — no fall-through, single expression per arm. (2) <b>Traditional colon with <code>yield</code></b>: <code>case WED: yield compute(); break;</code> — for multi-statement arms. <b>Exhaustiveness</b>: switch expressions must cover all cases (default or all enum values). Eliminates accidental fall-through bugs. Works with int, String, enum, and sealed class hierarchies. Java 21 adds pattern matching in switch arms.

What is <b>instanceof</b> with switch patterns (Java 21)?	Java 21 finalized switch pattern matching (JEP 441): <pre>switch (obj) {
    case Integer i    -&gt; System.out.println("int: " + i);
    case String s when s.length() &gt; 5 -&gt; System.out.println("long string");
    case String s     -&gt; System.out.println("short string");
    case null         -&gt; System.out.println("null");
    default           -&gt; System.out.println("other");
}</pre>. Features: (1) <b>Type patterns</b>: bind and test type in one step. (2) <b>Guarded patterns (<code>when</code>)</b>: add conditions to patterns. (3) <b>Null handling</b>: <code>case null</code> instead of pre-check. (4) <b>Exhaustiveness checking</b>: compiler verifies all sealed subtypes covered.

What is a <b>MethodHandle</b>?	<code>MethodHandle</code> (<code>java.lang.invoke</code>, Java 7+) is a typed, directly executable reference to a method, constructor, field getter/setter, or other operation. Obtained via <code>MethodHandles.lookup().findVirtual(cls, "name", type)</code>. <b>Advantages over reflection:</b> (1) JIT-friendly — can be inlined like direct calls after warm-up. (2) Type-safe invocation at runtime. (3) Supports method transformations: <code>asType()</code> (type adaptation), <code>bindTo()</code> (partial application / currying), <code>filterArguments()</code>, <code>foldArguments()</code>. (4) Access checking at creation time, not invocation time. Used internally by lambda implementation (<code>LambdaMetafactory</code>), <code>invokedynamic</code> bootstrap methods.

What is <b>invokedynamic</b>?	<code>invokedynamic</code> (JEP 292, Java 7) is a JVM bytecode instruction that allows the method to be invoked to be determined <em>at runtime</em> via a <b>bootstrap method</b>. The first invocation calls the bootstrap method, which returns a <code>CallSite</code> (containing a <code>MethodHandle</code>). Subsequent invocations reuse the linked <code>CallSite</code>. <b>Java uses invokedynamic for:</b> (1) <b>Lambda expressions</b>: <code>LambdaMetafactory</code> generates functional interface implementations at runtime — more efficient than anonymous classes (no new .class file per lambda). (2) <b>String concatenation</b> (Java 9+): <code>StringConcatFactory</code> for optimized <code>+</code>. (3) Dynamic languages on JVM (Groovy, JRuby, Nashorn).

What is the <b>VarHandle</b> API?	<code>VarHandle</code> (<code>java.lang.invoke</code>, Java 9) provides typed references to variables (instance fields, array elements, static fields) with support for multiple <b>memory access modes</b>: plain, opaque, acquire/release (one-sided barriers), and volatile (full barrier). Enables atomic operations: <code>compareAndSet()</code>, <code>getAndAdd()</code>, etc. <b>Advantages over <code>sun.misc.Unsafe</code>:</b> type-safe, bounds-checked, part of the public API. <b>Advantages over <code>AtomicFieldUpdater</code>:</b> faster (JIT-friendly), more access modes. Used in JDK's own concurrent data structures. Replaces most legitimate uses of Unsafe in library code.

What is <b>Project Panama</b>?	Project Panama improves Java-native code interoperability. <b>Foreign Function &amp; Memory (FFM) API</b> (finalized Java 22, JEP 454) provides: (1) <b><code>MemorySegment</code></b>: safe, bounded access to native/heap/mapped memory with lifetime control via <code>Arena</code>. (2) <b><code>Linker</code></b>: call native C functions directly without JNI: <code>linker.downcallHandle(addr, descriptor)</code>. (3) <b><code>SymbolLookup</code></b>: find native symbols in loaded libraries. <b>Advantages over JNI:</b> pure Java, no native code required, safe (bounds/lifetime checked), more ergonomic, JIT-optimizable. Also includes <b>Vector API</b> (SIMD intrinsics via <code>jdk.incubator.vector</code>).

What is <b>Project Valhalla</b>?	Project Valhalla aims to introduce <b>value types</b> (primitive classes / value classes) — objects that behave like primitives: no identity (no <code>==</code> identity comparison), stored inline by value (not by reference), no null. Benefits: (1) Eliminates boxing overhead for user-defined types (e.g., <code>Complex</code>, <code>Point</code>, <code>OptionalInt</code>). (2) Arrays of value types are flat (no pointer indirection) — much better cache locality. (3) "Codes like a class, works like an int." Also introduces <b>Universal Generics</b> — generics over primitives (<code>List&lt;int&gt;</code>) without boxing. As of Java 23+, value classes are in preview. Will fundamentally change Java's performance characteristics.

What is <b>JNI (Java Native Interface)</b>?	JNI is the standard API for Java code to call and be called by native code (C/C++). The Java method is declared <code>native</code>; the implementation is in a shared library (<code>.so</code>/<code>.dll</code>). JNI uses <code>System.loadLibrary("name")</code> to load the native library. <b>Challenges:</b> (1) Complex, error-prone boilerplate (JNIEnv pointer, type mangling). (2) Memory management must be manual in native code. (3) Bypasses JVM safety guarantees — native crashes kill the JVM. (4) JIT cannot optimize across the JNI boundary. (5) Difficult to debug. <b>Being superseded</b> by Project Panama's Foreign Function &amp; Memory API for most use cases.

What is <b>-XX:+UseG1GC</b> vs <b>-XX:+UseZGC</b>?	<b>G1GC</b> (<code>-XX:+UseG1GC</code>): default since Java 9. Region-based, concurrent marking + evacuation, targets predictable pause times (default 200ms via <code>-XX:MaxGCPauseMillis</code>). Handles heaps from GBs to ~TB. Good balance of throughput and latency for most workloads. <b>ZGC</b> (<code>-XX:+UseZGC</code>): ultra-low latency, production-ready Java 15+, generational ZGC Java 21+ (JEP 439). Concurrent relocation — pauses &lt;1ms regardless of heap size. Scalable to multi-TB heaps. Slightly lower throughput than G1. Use ZGC for latency-sensitive apps (trading, gaming, real-time). Use G1 for general-purpose. <b>Shenandoah</b>: similar goals to ZGC, Red Hat developed.

What is <b>GraalVM</b> and <b>Native Image</b>?	<b>GraalVM</b> is a high-performance polyglot VM that replaces HotSpot's JIT with the Graal JIT compiler (written in Java), claiming better optimizations for some workloads. Also runs JavaScript, Python, Ruby, R, WASM via Truffle framework. <b>Native Image</b> (<code>native-image</code> tool): ahead-of-time (AOT) compiles Java to a standalone native binary. <b>Benefits:</b> instant startup (milliseconds), very low memory footprint — ideal
