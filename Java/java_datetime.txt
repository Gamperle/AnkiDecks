# Java Date & Time API (java.time)
# Tier: Intermediate
# Format: Front[TAB]Back

# === Why java.time ===
Why was `java.time` (JSR-310) introduced in Java 8 to replace `java.util.Date` and `Calendar`?	`java.util.Date` and `Calendar` had severe design problems: (1) Mutability — `Date` is mutable, making it unsafe to share. (2) Poor API — `Calendar.get(MONTH)` is 0-based (January = 0). (3) No timezone clarity — `Date` doesn't carry a timezone but its `toString()` formats in the local zone. (4) Not thread-safe. `java.time` (based on Joda-Time) is immutable, fluent, clearly separates date/time concepts, and has an excellent API. All `java.time` classes are immutable and thread-safe.

# === Core Types ===
What is `LocalDate` in `java.time`?	`LocalDate` represents a date (year, month, day) without time or timezone. Example: `2026-03-09`. Created with `LocalDate.of(2026, 3, 9)`, `LocalDate.now()`, or `LocalDate.parse("2026-03-09")`. Common methods: `getYear()`, `getMonth()`, `getDayOfWeek()`, `plusDays(n)`, `minusMonths(n)`, `isAfter(other)`, `isBefore(other)`, `withDayOfMonth(1)`. Immutable — all modification methods return a new instance.
What is `LocalTime` in `java.time`?	`LocalTime` represents a time of day (hours, minutes, seconds, nanoseconds) without date or timezone. Example: `14:30:00`. Created with `LocalTime.of(14, 30)`, `LocalTime.now()`, `LocalTime.parse("14:30:00")`. Useful for business hours, scheduling logic where the calendar date is irrelevant. Methods: `getHour()`, `plusMinutes(n)`, `isBefore()`, `truncatedTo(ChronoUnit.MINUTES)`.
What is `LocalDateTime`?	`LocalDateTime` represents a date AND time without timezone information. It is essentially a `LocalDate` + `LocalTime`. Use when you need both date and time but timezone is handled separately (e.g., storing in UTC, displaying locally). Created with `LocalDateTime.of(date, time)`, `LocalDateTime.now()`. Convert to `ZonedDateTime` by adding a `ZoneId`: `localDT.atZone(ZoneId.of("Europe/Berlin"))`.
What is `ZonedDateTime` and when should you use it?	`ZonedDateTime` represents a date and time with a full timezone (`ZoneId`), including DST (Daylight Saving Time) transitions. Use it when you need to display a time in a specific user's timezone or perform timezone-aware arithmetic. Example: `ZonedDateTime.now(ZoneId.of("America/New_York"))`. When you add one hour across a DST change, `ZonedDateTime` correctly adjusts. Store dates in UTC (`Instant`) and convert to `ZonedDateTime` for display.
What is `Instant` and when should you use it?	`Instant` represents a single point on the UTC timeline — epoch seconds plus nanoseconds. It has no timezone, no DST, no calendar concepts. Use it for: logging timestamps, event ordering, persistence (store as UTC). Created with `Instant.now()`, `Instant.ofEpochSecond(seconds)`. Convert to `ZonedDateTime` for display: `instant.atZone(ZoneId.of("Europe/Berlin"))`. `Instant.now()` is nearly equivalent to `System.currentTimeMillis()` but more precise.
What is the difference between `Duration` and `Period`?	`Duration` measures a machine-based time span in seconds and nanoseconds: `Duration.ofHours(2)`, `Duration.between(start, end)`. Use for measuring elapsed time (benchmarks, timeouts). `Period` measures a human-based date span in years, months, and days: `Period.ofMonths(3)`, `Period.between(date1, date2)`. Use for calendar arithmetic: "3 months later" with `Period` handles different month lengths correctly. `Duration.ofDays(30)` ≠ `Period.ofMonths(1)`.
What is `ZoneId` vs `ZoneOffset`?	`ZoneId` (e.g., `ZoneId.of("Europe/Berlin")`) represents a full timezone with DST rules — the offset might be +01:00 in winter and +02:00 in summer. Always prefer `ZoneId` for user-facing code. `ZoneOffset` (e.g., `ZoneOffset.of("+02:00")`) represents a fixed offset from UTC with no DST rules — constant. Use `ZoneOffset.UTC` for UTC. `OffsetDateTime` uses `ZoneOffset`; `ZonedDateTime` uses `ZoneId`.

# === Formatting & Parsing ===
How does `DateTimeFormatter` work in `java.time`?	`DateTimeFormatter` formats and parses `java.time` objects. It is immutable and thread-safe (unlike `SimpleDateFormat`). Use built-in constants: `DateTimeFormatter.ISO_LOCAL_DATE`, `DateTimeFormatter.RFC_1123_DATE_TIME`. Or create custom: `DateTimeFormatter.ofPattern("dd.MM.yyyy HH:mm")`. Format: `localDate.format(formatter)`. Parse: `LocalDate.parse("09.03.2026", formatter)`. For localized formats: `DateTimeFormatter.ofLocalizedDate(FormatStyle.LONG).withLocale(Locale.GERMAN)`.

# === Arithmetic & Utilities ===
How do you perform date/time arithmetic in `java.time`?	All `java.time` types have `plus*` and `minus*` methods: `date.plusDays(7)`, `dateTime.minusHours(3)`. For more complex adjustments use `with(TemporalAdjuster)`: `date.with(TemporalAdjusters.lastDayOfMonth())`, `date.with(TemporalAdjusters.nextOrSame(DayOfWeek.MONDAY))`. `TemporalAdjusters` provides factory methods for common adjustments. All operations return new instances — the originals are unchanged.
How do you calculate the difference between two dates/times in `java.time`?	Use `ChronoUnit.X.between(start, end)` for simple differences in a single unit: `ChronoUnit.DAYS.between(date1, date2)`, `ChronoUnit.HOURS.between(dt1, dt2)`. Use `Duration.between(temporal1, temporal2)` for time-based spans with precision. Use `Period.between(localDate1, localDate2)` for date-based spans in years/months/days. Note: `ChronoUnit.MONTHS.between()` calculates whole months, which handles varying month lengths correctly.
How do you convert between legacy `java.util.Date` / `Calendar` and `java.time`?	`Date` → `Instant`: `date.toInstant()`. `Instant` → `Date`: `Date.from(instant)`. `Date` → `LocalDate` (via `ZoneId`): `date.toInstant().atZone(ZoneId.systemDefault()).toLocalDate()`. `Calendar` → `Instant`: `calendar.toInstant()`. `LocalDate` → `Date`: `Date.from(localDate.atStartOfDay(ZoneId.systemDefault()).toInstant())`. When working with JDBC: use `PreparedStatement.setObject(1, localDate)` for JDBC 4.2+ drivers which support `java.time` natively.
What is `Clock` and why is it useful?	`Clock` provides access to the current instant, date, and time with a timezone. `Clock.systemUTC()` and `Clock.systemDefaultZone()` are standard clocks. `Clock.fixed(instant, zone)` returns a fixed-time clock — invaluable for testing: inject a `Clock` into services and use it instead of `Instant.now()`/`LocalDate.now()`, making tests deterministic and fast. Example: `LocalDate.now(clock)` instead of `LocalDate.now()`. This is the clean, testable design for time-dependent code.
