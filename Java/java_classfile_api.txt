# Java Class-File API (JEP 484, Java 24)
# Tier: Expert
# Format: Front[TAB]Back

[Pro] What is the Class-File API and why was it introduced?	The Class-File API (JEP 484, finalized in Java 24) provides a standard JDK API for reading, writing, and transforming Java `.class` files. Before it, the ecosystem relied on third-party libraries like ASM, Javassist, and BCEL. Problems with that approach: (1) Each JDK release could change the class-file format, requiring ASM updates before frameworks could work on new JDKs. (2) The JDK itself internally used ASM (which caused versioning conflicts). (3) No standard, stable API. The Class-File API is maintained in sync with JVM spec changes, eliminating the "ASM lag" problem. It is already used internally by the JDK (`javac`, proxy generation, `MethodHandles` internals).
[Pro] How do you read and inspect a class file with the Class-File API?	Use `ClassFile.of()` to get the entry point, then `parse()` to read bytes: `ClassFile cf = ClassFile.of(); ClassModel model = cf.parse(Path.of("MyClass.class"));`. Navigate the model: `model.thisClass().asInternalName()` → class name. `model.methods()` → list of `MethodModel`. `model.fields()` → list of `FieldModel`. Per method: `method.methodName().stringValue()`, `method.flags()`, and `method.code().ifPresent(code -> code.forEach(element -> { if (element instanceof Instruction i) { ... } }))`. The model is a rich tree of `ClassElement`, `MethodElement`, `CodeElement` subtypes — visitor-like iteration without requiring explicit visitors.
[Pro] How do you generate a new class from scratch with the Class-File API?	`ClassFile.of().build(ClassDesc.of("com.example.Generated"), classBuilder -> { classBuilder .withFlags(ClassFile.ACC_PUBLIC) .withSuperclass(ClassDesc.of("java.lang.Object")) .withMethod("greet", MethodTypeDesc.of(CD_String), ClassFile.ACC_PUBLIC, methodBuilder -> { methodBuilder.withCode(codeBuilder -> { codeBuilder .ldc("Hello from generated class") .areturn(); }); }); });`. Returns a `byte[]` which can be loaded with a `ClassLoader`. `ClassDesc` encodes class descriptors type-safely; `MethodTypeDesc` encodes method signatures; `CodeBuilder` exposes typed instruction-emitting methods (`invokevirtual`, `aload`, `ireturn`, etc.).
[Pro] How does `ClassFile.transformClass()` work for bytecode transformation?	`transformClass` takes a `ClassModel` and a `ClassTransform` and produces a new `byte[]` with the transformation applied. `ClassTransform` is a functional interface: `(ClassBuilder builder, ClassElement element) -> { /* inspect element, emit to builder */ }`. To selectively modify methods: `ClassTransform transform = ClassTransform.transformingMethods( method -> method.methodName().stringValue().startsWith("log"), (methodBuilder, methodElement) -> { // intercept or modify method elements methodBuilder.accept(methodElement); });`. Chain transforms: `transform1.andThen(transform2)`. This is the pattern used for AOP-like instrumentation, profiling agents, coverage tools, and framework code generation — without a separate ASM dependency.
[Pro] How does the Class-File API compare to ASM?	Both read/write class files at the bytecode level. Differences: (1) **Stability** — Class-File API is a JDK standard, versioned with the JVM spec; ASM requires explicit updates per JDK version. (2) **API style** — Class-File uses immutable models + functional builders; ASM uses a mutable visitor pattern (harder to compose). (3) **Type safety** — Class-File uses `ClassDesc`, `MethodTypeDesc`, `DirectMethodHandleDesc` to avoid string-based errors; ASM uses raw internal name strings (`Ljava/lang/String;`). (4) **Performance** — comparable; both zero-copy parse where possible. (5) **Ecosystem** — ASM has a massive existing ecosystem and is mature; Class-File API is new (Java 24) but will grow. For new projects targeting Java 24+, Class-File API is the forward-looking choice; for maximum compatibility, ASM remains practical.
