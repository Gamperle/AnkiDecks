# Java Web Security

# === OWASP & Injection ===
[Beginner] What is the OWASP Top 10 and what are the most critical categories?	The OWASP Top 10 is a standard document listing the most critical web application security risks. Key categories: (1) Broken Access Control — IDOR, missing authorization. (2) Cryptographic Failures — weak encryption, plaintext secrets. (3) Injection — SQL, LDAP, OS command. (4) Insecure Design — missing threat modeling. (5) Security Misconfiguration — default creds, verbose errors. (6) Vulnerable Components — outdated libraries with known CVEs. (7) Identification/Authentication Failures — weak passwords, session issues. (8) Software/Data Integrity Failures — insecure CI/CD, deserialization. (9) Logging/Monitoring Failures. (10) SSRF.
[Intermediate] What is SQL injection, how does it work, and how is it prevented in Java?	SQL injection occurs when user input is concatenated directly into a SQL query string, letting attackers modify the query structure. Example: `"SELECT * FROM users WHERE email='" + email + "'"` — input of `' OR '1'='1' --` returns all rows. Prevention: always use `PreparedStatement` with parameterized queries: `PreparedStatement stmt = conn.prepareStatement("SELECT * FROM users WHERE email = ?"); stmt.setString(1, email);`. The driver safely escapes parameters. Also applies to JPA/JPQL: use `@Query` with named parameters (`?1` or `:name`), not string concatenation.
[Intermediate] What is XSS and how is it prevented?	XSS (Cross-Site Scripting) is an injection attack where malicious JavaScript is inserted into web pages served to other users. Types: Stored (persisted in DB), Reflected (in URL/response), DOM-based (JavaScript manipulates DOM). Impact: session cookie theft, keylogging, page defacement. Prevention: (1) HTML-encode all user-supplied output: `&` → `&amp;`, `<` → `&lt;`, `"` → `&quot;`, etc. — use a library like OWASP Java HTML Sanitizer. (2) Use Content-Security-Policy header. (3) `HttpOnly` and `SameSite` flags on cookies. (4) Do NOT use `innerHTML` with user data; use `textContent`.
[Intermediate] What is CSRF and how is it prevented?	CSRF (Cross-Site Request Forgery) tricks an authenticated user's browser into sending unintended requests to a site where they're logged in. Example: a malicious page contains `<img src="https://bank.com/transfer?to=attacker&amount=1000">` — the browser automatically sends the user's session cookie with the request. Prevention: (1) CSRF tokens — a unique random token per session embedded in forms; server validates it (token not sent automatically by browsers). (2) `SameSite=Strict` or `SameSite=Lax` cookie attribute — prevents browser from sending cookies on cross-origin requests. (3) `Origin`/`Referer` header validation (secondary). Spring Security provides built-in CSRF protection.

# === Transport & Headers ===
[Intermediate] What are the most important security response headers?	`Strict-Transport-Security: max-age=31536000; includeSubDomains` — forces HTTPS for future visits (HSTS). `Content-Security-Policy: default-src 'self'` — restricts sources of scripts, styles, etc.; mitigates XSS. `X-Content-Type-Options: nosniff` — prevents MIME-type sniffing. `X-Frame-Options: DENY` — prevents clickjacking by disallowing iframes. `Referrer-Policy: strict-origin-when-cross-origin` — controls referrer header. `Permissions-Policy` — restricts browser feature API access. In Spring Security, `HttpSecurity.headers()` configures these automatically.
[Intermediate] What is CORS and how does it work?	CORS (Cross-Origin Resource Sharing) is a browser security mechanism that restricts cross-origin HTTP requests made from JavaScript. A "cross-origin" request is one with a different scheme, host, or port. For "simple" requests the browser sends the request with an `Origin` header; the server must respond with `Access-Control-Allow-Origin: <origin>` or `*`. For "non-simple" requests (e.g., PUT, DELETE, or custom headers) the browser first sends a preflight `OPTIONS` request. The server's response headers (`Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`, `Access-Control-Allow-Credentials`) determine if the actual request is permitted. CORS is enforced by the BROWSER only — server-side APIs must still authorize.
[Intermediate] What is clickjacking and how is it prevented?	Clickjacking tricks users into clicking elements on a site loaded invisibly in a transparent iframe on a malicious page. The user thinks they're clicking innocuous content but is actually interacting with the hidden site. Prevention: (1) `X-Frame-Options: DENY` or `SAMEORIGIN` response header — instructs browsers not to render the page in a frame. (2) `Content-Security-Policy: frame-ancestors 'none'` (modern equivalent, more flexible). Both should be set; CSP takes precedence in modern browsers.

# === Authentication & Tokens ===
[Intermediate] What is the structure of a JWT (JSON Web Token) and what are its security pitfalls?	A JWT has three Base64Url-encoded parts separated by `.`: `header.payload.signature`. Header: `{"alg":"HS256","typ":"JWT"}`. Payload: claims like `{"sub":"user123","exp":1741478400}`. Signature: HMAC or RSA sign of header+payload. Common pitfalls: (1) `alg: none` attack — reject tokens with `alg: none`; always validate the algorithm. (2) Not validating expiration (`exp` claim). (3) Trusting the payload without verifying the signature. (4) Storing JWTs in `localStorage` (XSS risk) — prefer `HttpOnly` cookies. (5) Not revoking JWTs on logout — use short expiry + refresh token or a revocation list.
[Intermediate] What are the main OAuth2 flows and when is each used?	Authorization Code (+ PKCE): for user-facing apps (web, mobile). User is redirected to auth server, grants consent, server returns authorization code to redirect URI, code exchanged for tokens. PKCE (Proof Key for Code Exchange) is required for public clients. Client Credentials: for machine-to-machine (server-to-server) — no user interaction; client authenticates directly with auth server using clientId + clientSecret. Device Code: for devices without browsers (smart TVs, CLI tools). Implicit flow: DEPRECATED — don't use.
[Intermediate] What is OpenID Connect and how does it relate to OAuth2?	OpenID Connect (OIDC) is an identity layer built on top of OAuth2. OAuth2 is for AUTHORIZATION (granting access to resources). OIDC adds AUTHENTICATION (verifying user identity). OIDC adds the `openid` scope and returns an ID Token (a JWT with user identity claims: `sub`, `email`, `name`, etc.) in addition to the OAuth2 access token. The `/userinfo` endpoint returns the full user profile. Use OIDC when you need to know who the user is (login), not just what they're allowed to do.

# === Session Security ===
[Intermediate] What is session fixation and how is it prevented?	Session fixation is an attack where an attacker sets a user's session ID to a known value before the user logs in, then after login the user's authenticated session is accessible to the attacker. Prevention: always regenerate the session ID upon successful authentication. In Servlet 3.1+: `request.changeSessionId()`. In Spring Security: `SessionManagementFilter` does this by default via `sessionFixation().migrateSession()` (regenerates session ID while keeping attributes) or `.newSession()` (creates fresh session).
[Intermediate] What is timing-safe comparison and why is it needed for hashes?	Standard string equality (`equals()`) short-circuits — it returns false as soon as it finds the first differing character. This is vulnerable to timing attacks: by measuring how long comparison takes, an attacker can determine how many bytes of a token/hash match and brute-force the rest. Timing-safe comparison (constant-time) always checks all bytes regardless of where a difference is. In Java: `MessageDigest.isEqual(calculatedHmac, expectedHmac)` is constant-time. In Spring Security: `DigestUtils` provides constant-time comparison. NEVER use `.equals()` to validate passwords, HMAC signatures, or tokens.
