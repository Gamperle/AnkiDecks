# Java Enterprise & Architectural Patterns
# Tier: Advanced
# Format: Front[TAB]Back

# === Foundational ===
What is Dependency Injection (DI) and Inversion of Control (IoC)?	IoC means the framework creates and wires objects instead of application code doing `new Dependency()`. DI is the most common form of IoC: dependencies are "injected" into a class rather than the class creating them itself. Three types: (1) Constructor injection — dependencies passed in constructor (preferred: objects are always valid, fields can be final, easy to test). (2) Setter injection — dependencies set via setters (optional dependencies). (3) Field injection (`@Autowired` on field) — convenient but hides dependencies and makes testing harder. Benefits: loose coupling, testability (inject mocks), configurability.
What is the Repository pattern?	Repository abstracts data access behind a collection-like interface for a domain aggregate, decoupling business logic from persistence details. The service layer works with the repository interface; different implementations can use JPA, JDBC, an in-memory store, or a remote API. Spring Data JPA's `JpaRepository<T, ID>` provides standard CRUD, pagination, and sorting. Add custom queries with method name derivation (`findByEmailAndActive(String, boolean)`) or `@Query("SELECT u FROM User u WHERE ...")`. Benefits: business logic doesn't know about SQL; easy to mock in tests; swap implementations without changing services.
What is the Service Layer pattern?	The Service Layer defines application operations (use cases) and coordinates domain objects and repositories to fulfil them. It is the transaction boundary: `@Transactional` annotates service methods, ensuring the whole operation succeeds or rolls back atomically. Services should not contain persistence logic (that's repositories) or presentation logic (that's controllers). Keep services focused on business workflows. In Spring: annotate with `@Service`; inject repositories and other services via constructor injection.
What is a DTO (Data Transfer Object)?	A DTO is a plain object whose sole purpose is carrying data between layers or across process boundaries — it has no business behaviour. Use DTOs to: decouple the API contract from the domain model (API can evolve independently), avoid exposing sensitive fields, reduce over-fetching (only include needed fields). Example: a `UserResponse` DTO omits the password field from the `User` entity. Mapping libraries: MapStruct (compile-time, type-safe, fast) or ModelMapper (runtime reflection). In Java 16+ a `record` is a concise DTO: `public record UserResponse(String name, String email) {}`.
What is a Value Object?	A Value Object is defined entirely by its value, not by identity — two Value Objects with the same values are considered equal. Value Objects should be immutable. Examples: `Money(amount, currency)`, `EmailAddress(value)`, `DateRange(start, end)`. Java `record` is the ideal Value Object: `public record Money(BigDecimal amount, String currency) {}` — provides immutability, `equals()`, `hashCode()`, `toString()` automatically. Contrast with Entity, which has identity (ID) independent of its attribute values.

# === Resilience Patterns ===
What is the Circuit Breaker pattern?	Circuit Breaker prevents cascading failures by monitoring calls to a remote service. States: CLOSED (normal operation — calls pass through; failures counted). OPEN (failure threshold exceeded — calls immediately fail without reaching the remote service, allowing it time to recover). HALF-OPEN (after a timeout, limited calls are allowed through to test recovery). If successful, returns to CLOSED; if failing, returns to OPEN. Java implementations: Resilience4j `@CircuitBreaker`, Spring Cloud Circuit Breaker. Configurable: failure rate threshold, slow call threshold, wait duration in OPEN state.
What is the Retry pattern with exponential backoff and jitter?	Retry automatically re-executes a failed operation, useful for transient failures (network blips, temporary service unavailability). Naive retry (fixed interval, unlimited) can cause thundering herd problems. Best practice: exponential backoff — double the wait time after each failure: 1s, 2s, 4s, 8s... Plus jitter (random variation) — avoids all clients retrying at the same moment after a shared outage. Limit maximum retries (e.g., 3) and maximum delay (e.g., 30s). Resilience4j: `@Retry(name="myService")` with `waitDuration`, `maxAttempts`, `enableExponentialBackoff`, and `randomizedWaitFactor`.
What is the Bulkhead pattern?	Bulkhead isolates components so a failure in one doesn't cascade to others, named after watertight compartments in ships. In Java: assign separate, bounded thread pools to different downstream services — if one service is slow and fills its pool, other services are unaffected. Resilience4j supports two bulkhead implementations: `ThreadPoolBulkhead` (separate thread pool per service) and `SemaphoreBulkhead` (max concurrent calls). Key config: `maxConcurrentCalls`, `maxWaitDuration`. Essential for services that call multiple downstream dependencies.
What is the Outbox Pattern?	The Outbox Pattern reliably publishes events to a message broker as part of a database transaction, solving the dual-write problem (atomically updating DB AND publishing an event is impossible without distributed transactions). Implementation: write the event to an `outbox` table in the SAME database transaction as the data change. A separate "relay" service reads the outbox and publishes events to the broker, removing them after confirmation. This guarantees at-least-once delivery. CDC (Change Data Capture) tools like Debezium can act as the relay by reading the database's WAL.
What is CQRS (Command Query Responsibility Segregation)?	CQRS separates the model used to write data (Commands that change state) from the model used to read data (Queries that return data). Benefits: read models can be optimized independently (denormalized projections, caches), write models can focus on enforcing business rules, scalability (scale reads and writes independently). In practice: Commands go to a service that updates the domain model and emits events; Queries read from a separate, optimized read model updated by event handlers. Often combined with Event Sourcing. Overhead: eventual consistency between write and read models; increased complexity — only adopt when the benefits justify it.

# === Domain-Driven Design Concepts ===
What is a Domain Event?	A Domain Event is a record of something significant that happened in the domain, expressed in domain language: `OrderPlaced`, `PaymentFailed`, `UserRegistered`. They are immutable value objects (use `record`). Benefits: decoupled components react to events without tight coupling; natural fit for audit logging; enables Event Sourcing. In Spring: publish with `ApplicationEventPublisher.publishEvent(event)`; handle with `@EventListener` (sync) or `@TransactionalEventListener(AFTER_COMMIT)` for reliably publishing after the transaction commits.
