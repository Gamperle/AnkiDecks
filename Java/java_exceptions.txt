# Java Exceptions
# Tier: Beginner → Intermediate
# Format: Front[TAB]Back

# === Basics ===
What is the difference between a checked and an unchecked exception in Java?	Checked exceptions are subclasses of `Exception` (but not `RuntimeException`) and must be declared with `throws` or caught at compile time. Examples: `IOException`, `SQLException`. Unchecked exceptions are subclasses of `RuntimeException` (or `Error`) and do NOT need to be declared or caught. Examples: `NullPointerException`, `IllegalArgumentException`, `ArrayIndexOutOfBoundsException`. The rationale: checked exceptions represent recoverable conditions the caller should handle; unchecked represent programming mistakes.
What is the exception class hierarchy in Java?	`Throwable` is the root. Two direct subclasses: `Error` (JVM-level problems like `OutOfMemoryError`, `StackOverflowError` — normally not caught) and `Exception` (application-level problems). `Exception` splits into checked exceptions (direct subclasses like `IOException`) and unchecked exceptions (subclasses of `RuntimeException` like `NullPointerException`).
What does `try / catch / finally` guarantee?	The `finally` block ALWAYS executes after the `try` block completes — whether normally, via a caught exception, or an uncaught exception. The only exceptions are `System.exit()` calls or the JVM crashing. Use `finally` for cleanup (closing resources). Note: if `finally` contains a `return` statement, it suppresses any exception that was propagating.
What is the difference between `throw` and `throws`?	`throw` is a statement that actually throws an exception object: `throw new IllegalArgumentException("bad input")`. `throws` is a method declaration keyword that advertises which checked exceptions a method may propagate: `void readFile() throws IOException`. A method can `throws` multiple exceptions: `throws IOException, SQLException`.
What are the most common built-in exceptions to know?	`NullPointerException` — dereferencing null. `IllegalArgumentException` — invalid argument value. `IllegalStateException` — method called at wrong time. `IndexOutOfBoundsException` / `ArrayIndexOutOfBoundsException` — invalid index. `ClassCastException` — invalid cast. `UnsupportedOperationException` — operation not supported (e.g., immutable collection mutation). `StackOverflowError` — recursion too deep. `OutOfMemoryError` — heap exhausted.
What is `Error` vs `Exception` in Java?	`Error` represents serious JVM-level problems that are almost always unrecoverable: `OutOfMemoryError`, `StackOverflowError`, `AssertionError`. Applications should generally NOT catch `Error`. `Exception` represents conditions an application might reasonably handle. Catching `Throwable` catches both, which is almost never appropriate.

# === Intermediate ===
How do you create a custom exception in Java?	Extend `RuntimeException` for unchecked or `Exception` for checked. Convention: provide at least two constructors — one with a message string and one with a message + cause. Example: `public class OrderNotFoundException extends RuntimeException { public OrderNotFoundException(String message) { super(message); } public OrderNotFoundException(String message, Throwable cause) { super(message, cause); } }`. Prefer unchecked for application-specific exceptions; checked only for genuinely recoverable conditions.
What is exception chaining and how do you use it?	Exception chaining preserves the original cause when wrapping exceptions to add context. Pass the original as the `cause` parameter: `throw new ServiceException("Failed to load user", originalIOException)`. Retrieve with `ex.getCause()`. This keeps the full stack trace chain. `initCause(Throwable)` is an older alternative. Always chain when translating low-level exceptions into higher-level ones to avoid losing diagnostic info.
What is multi-catch syntax and when is it useful?	Java 7+ allows catching multiple exception types in one block: `catch (IOException | SQLException e)`. The caught variable is implicitly `final`. This avoids duplicating identical handler code for different exception types. Restriction: the types in a multi-catch cannot be related by inheritance (compiler error).
What is try-with-resources and how does it work?	Introduced in Java 7. Resources implementing `AutoCloseable` (or `Closeable`) are declared in the `try` clause and automatically closed at the end, even if an exception is thrown. Syntax: `try (var in = new FileInputStream(path)) { ... }`. Multiple resources can be declared (closed in reverse order). If both the body and `close()` throw, the `close()` exception is suppressed (accessible via `Throwable.getSuppressed()`). `Closeable.close()` may throw `IOException`; `AutoCloseable.close()` may throw `Exception`.
What are suppressed exceptions?	When try-with-resources closes a resource and `close()` throws while an exception from the body is already propagating, the `close()` exception is "suppressed" — attached to the primary exception but not the one that propagates. Retrieve with `primaryException.getSuppressed()`. You can add suppressed exceptions manually with `addSuppressed(Throwable)`.
What are Helpful NullPointerExceptions introduced in Java 14?	Before Java 14 an NPE message was just `NullPointerException` with no details. From Java 14+ (enabled by default from Java 15+) the JVM generates a detailed message identifying exactly what was null. Example: `Cannot read field "address" because "customer" is null`. Also identifies null array elements, null method receiver, etc. Enable explicitly in Java 14 with `-XX:+ShowCodeDetailsInExceptionMessages`.
What is the best practice for exception handling?	(1) Catch the most specific exception type, not `Exception` or `Throwable`. (2) Never swallow exceptions silently with an empty catch block. (3) Log with full stack trace at the boundary where you handle it, not at every rethrow. (4) Wrap low-level exceptions with context and chain the original cause. (5) Avoid using exceptions for flow control — they are expensive. (6) Declare only the exceptions a method actually throws; don't over-declare. (7) Always use try-with-resources for `AutoCloseable` resources.
What is the difference between `Exception.getMessage()` and `Exception.toString()`?	`getMessage()` returns just the detail message string passed to the constructor (or null). `toString()` returns the class name followed by the message: `java.io.IOException: File not found`. `printStackTrace()` prints `toString()` plus the full stack trace to stderr. For logging, prefer passing the exception object to the logger, not just the message string, so the stack trace is included.

What is exception masking, and how does try-with-resources handle it?	Exception masking occurs when a `finally` block throws an exception that replaces the original exception from the `try` block, losing the root cause. Try-with-resources solution: if both the body and `close()` throw, the body's exception is preserved as the primary exception, and the `close()` exception is added as a suppressed exception, accessible via `primaryException.getSuppressed()`. This way both exceptions are preserved and neither is lost.
What is the difference between `final`, `finally`, and `finalize` in Java?	`final`: keyword with three uses — (1) final variable: cannot be reassigned. (2) final method: cannot be overridden. (3) final class: cannot be subclassed (e.g., `String`, `Integer`). `finally`: block in try-catch-finally that always executes (even on `return` or exception), used for cleanup. Skipped only on `System.exit()` or JVM crash. `finalize()`: deprecated instance method called by GC before reclaiming an object — unreliable, removed in Java 18+. Use try-with-resources or the `Cleaner` API instead.
Is there a performance difference between try-with-resources and manual try-finally?	try-with-resources generates slightly more bytecode because it calls `Throwable.addSuppressed()` to handle suppressed exceptions. However, the performance difference is negligible in practice — measured in nanoseconds and eliminated by JIT. Always prefer try-with-resources for `AutoCloseable` resources: it guarantees closure on all exit paths, correctly handles suppressed exceptions (no masking), supports multiple resources in declaration order, and is more readable.
