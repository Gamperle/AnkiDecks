# Java Lock-Free & Non-Blocking Concurrency
# Tier: Expert
# Format: Front[TAB]Back

What is Compare-And-Swap (CAS) and how does it underpin lock-free algorithms?	CAS is a CPU-level atomic instruction: `compareAndSwap(address, expectedValue, newValue)` — atomically: if the current value at address equals expected, replace it with new and return true; otherwise return false without modifying memory. This is the foundation of all Java lock-free code. CAS eliminates mutex locks: instead of "lock → read → modify → write → unlock", lock-free code does "read → compute new value → CAS(old, new); retry on failure." The key property: exactly one thread wins a CAS; others retry without blocking. All `java.util.concurrent.atomic` classes use CAS internally (`AtomicInteger.compareAndSet(expect, update)`). On x86 this is the `LOCK CMPXCHG` instruction.
What is the ABA problem and how does `AtomicStampedReference` solve it?	ABA: thread reads value A, another thread changes A→B→A. The first thread's CAS succeeds (sees A as expected) but doesn't detect that an intermediate change occurred — the data structure may be in an inconsistent state (e.g., node was removed and re-added to a linked list). `AtomicStampedReference<V>` pairs the value with a monotonically incrementing integer stamp (version number). The CAS checks both value AND stamp: `ref.compareAndSet(expectedRef, newRef, expectedStamp, expectedStamp + 1)`. Since the stamp is always incremented, A→B→A results in stamps 0→1→2; the original thread checks stamp=0 which no longer matches stamp=2 → CAS fails correctly. `AtomicMarkableReference` is similar but uses a boolean mark instead of a counter.
When should you use `LongAdder`/`LongAccumulator` over `AtomicLong`?	`AtomicLong` is a single memory location updated with CAS. Under high contention, many threads compete for the same CAS, causing repeated retries (spinning) — throughput degrades significantly. `LongAdder` maintains an array of cells (per-thread stripes) + a base cell. Increment: each thread updates its own cell with low contention. `sum()`: adds all cells — not atomic, so see a recent-but-not-instant value. Use when you need: (1) High-throughput parallel increments (counters, accumulators). (2) The final sum, not a per-operation precise value. Use `AtomicLong` when you need `get()` to reflect every prior update instantly (e.g., sequence number generation, flag, compare-and-act). `LongAccumulator` generalizes `LongAdder` to arbitrary accumulation functions: `new LongAccumulator(Long::max, Long.MIN_VALUE)`.
What are `VarHandle` memory ordering modes and when does each apply?	`VarHandle` (Java 9+) gives controlled access to fields/array elements with explicit memory ordering. Modes from weakest to strongest: **Plain** — no ordering guarantees; JIT may reorder freely. Use only for provably single-threaded access. **Opaque** — reads/writes are atomic (no tearing) and not reordered with other opaque accesses on the same variable. **Acquire (read) / Release (write)** — acquire read sees all writes before the corresponding release write on another thread. Classic "publish a new object" pattern: write fields, then release-set flag; reader acquire-reads flag, then safely reads fields. **Volatile** — full sequential consistency (total order); all threads see the same order of volatile ops. Costs increase with strength. Use the weakest mode that meets your correctness requirement.
How do you implement a correct non-blocking stack using CAS?	A non-blocking (Treiber) stack stores a `AtomicReference<Node<T>> top`. Push: `Node<T> newNode = new Node<>(item); do { newNode.next = top.get(); } while (!top.compareAndSet(newNode.next, newNode));`. Pop: `Node<T> node; Node<T> next; do { node = top.get(); if (node == null) return empty; next = node.next; } while (!top.compareAndSet(node, next)); return node.item;`. Progress guarantee: **lock-free** — at least one thread makes progress per finite number of steps (no thread can be blocked by another). Not wait-free (individual threads may spin). ABA risk: if nodes are recycled and re-pushed, the same pointer may reappear. Mitigate with `AtomicStampedReference` or a memory allocator that never reuses pointers.
What distinguishes lock-free, wait-free, and obstruction-free algorithms?	These are formally defined progress guarantees for non-blocking algorithms: (1) **Obstruction-Free** — a thread makes progress if it runs alone (no contention). Weakest; rarely used. (2) **Lock-Free** — at least one thread in the system makes progress in finite steps regardless of what other threads do. Failed CAS retries don't count as progress for the retrying thread but another thread succeeds. Most `java.util.concurrent` algorithms (e.g., `AtomicInteger`, `ConcurrentLinkedQueue`) are lock-free. (3) **Wait-Free** — every thread makes progress in a bounded number of steps regardless of contention. Strongest; very difficult to implement; few practical examples. Lock-free algorithms can cause starvation of individual threads; wait-free eliminates that. For most real-world Java code, lock-free is the practical target.
