# Java JVM Internals
# Format: Front[TAB]Back

[Intermediate] What are the main memory areas of the JVM?	Heap, Method Area (Metaspace in Java 8+), JVM Stack (per thread), Native Method Stack, Program Counter Register.
[Intermediate] What is the Heap in the JVM?	The runtime data area shared among all threads where objects and arrays are allocated. It is the primary target of garbage collection.
[Intermediate] What is the Stack in the JVM?	A per-thread memory area that stores stack frames for each method call, holding local variables, operand stack, and return values.
[Intermediate] What is the Metaspace (Java 8+)?	The replacement for the old PermGen; stores class metadata in native memory rather than the heap, growing automatically.
[Intermediate] What is Just-In-Time (JIT) compilation?	The JVM compiles frequently executed bytecode (hot spots) into native machine code at runtime to improve performance.
[Intermediate] What is garbage collection?	The automatic process by which the JVM reclaims heap memory occupied by objects that are no longer reachable.
[Intermediate] What is the difference between Minor GC and Major GC?	Minor GC collects the Young Generation (Eden + Survivor spaces); Major (or Full) GC collects the Old Generation (Tenured space), usually more expensive.
[Intermediate] What is the Young Generation?	The heap region where newly created objects are allocated. It consists of Eden space and two Survivor spaces.
[Intermediate] What is the Old Generation?	The heap region where long-lived objects (that survived multiple Minor GCs) are promoted. Collected less frequently but with greater cost.
[Intermediate] What is the G1 Garbage Collector?	A region-based, server-style GC (default since Java 9) that divides the heap into equal-sized regions and aims for predictable pause times.
[Intermediate] What is a ClassLoader?	A component that loads class files into the JVM at runtime. The three built-in loaders are Bootstrap, Extension (Platform), and Application ClassLoader.
[Intermediate] What is bytecode?	The intermediate, platform-independent instruction set that Java source code compiles into; it is executed or JIT-compiled by the JVM.
[Intermediate] What is the difference between `-Xms` and `-Xmx` JVM flags?	`-Xms` sets the initial heap size; `-Xmx` sets the maximum heap size.
[Intermediate] What is a memory leak in Java?	A situation where objects are no longer needed but are still referenced (e.g., in a static collection), preventing GC from reclaiming the memory.
[Intermediate] What is `finalize()`?	A method called by the GC before reclaiming an object's memory. Deprecated since Java 9; use `Cleaner` or try-with-resources instead.

# === Expert JVM ===
[Pro] What is escape analysis and how does the JVM use it?	Escape analysis is a JIT optimization that determines whether an object's reference can escape its creation method or thread. If an object does not escape: (1) Stack allocation — the object can be allocated on the stack and collected when the frame exits, reducing GC pressure. (2) Scalar replacement — the object is broken into its primitive fields and allocated in CPU registers. (3) Lock elision — `synchronized` on a provably thread-local object is eliminated entirely. Enabled by default: `-XX:+DoEscapeAnalysis`. Disable with `-XX:-DoEscapeAnalysis` to diagnose allocation issues.
[Pro] What is false sharing and how do you avoid it with `@Contended`?	False sharing occurs when two threads write to different variables that happen to reside in the same CPU cache line (typically 64 bytes). Although the variables are logically independent, the CPU must invalidate and reload the entire cache line on every write by the other CPU, causing a significant performance hit in high-throughput concurrent code. Detection: use Java Flight Recorder or JMH profiling; look for high L1/L2 cache miss rates. Solution: pad the variable to fill a full cache line. Java 8+: annotate the field or class with `@jdk.internal.vm.annotation.Contended` (requires `-XX:-RestrictContended`). Alternatively pad manually with 7 unused `long` fields around the hot field.
[Pro] What is JVMTI and how are Java agents used for instrumentation?	JVMTI (JVM Tool Interface) is the native API for inspecting and controlling the JVM — used by profilers, debuggers, and agents. A Java agent is a JAR with a `premain(String args, Instrumentation inst)` method. Attach at startup with `-javaagent:agent.jar`. A dynamic agent uses `VirtualMachine.attach(pid)` to attach to a running JVM and invokes `agentmain()`. Agents register a `ClassFileTransformer` that intercepts every class as it is loaded; the transformer can modify bytecode before the class is defined. Uses: APM (New Relic, Datadog, Elastic APM), code coverage (JaCoCo), test mocking, AOP weaving. Tools built on agents: ByteBuddy, ASM.
[Pro] What is AOT Class Loading & Linking (Java 24, JEP 483)?	JEP 483 adds a class-data cache that persists the result of class loading and linking across JVM runs. Training run: `java -XX:AOTCache=app.aot -cp ... MyApp` — warms the JVM, records loaded and linked class data into the cache file. Production run: `java -XX:AOTCache=app.aot -cp ... MyApp` — restores pre-loaded and pre-linked class data from the cache, skipping parsing and verification for cached classes. Startup time improvement: 10–40% reduction for large applications. Difference from CDS (Class Data Sharing): CDS only caches JDK bootstrap class data; JEP 483 extends this to application classes. Complements JEP 490 (AOT Method Profiling) which persists JIT profile data.
[Pro] What are `StackWalker` API (Java 9+) and `ProcessHandle` API (Java 9+)?	`StackWalker` replaces `Thread.getStackTrace()` for efficient, lazy stack frame traversal. `StackWalker.getInstance(Option.RETAIN_CLASS_REFERENCE).walk(stream -> stream.limit(5).collect(toList()))` — only walks as many frames as requested; much faster than `getStackTrace()` which allocates all frames eagerly. `ProcessHandle` represents an operating system process. `ProcessHandle.current()` — current JVM process. `.pid()`, `.info()` (command, arguments, start time, CPU time). `.children()` / `.descendants()` — stream of child/descendent processes. `.onExit()` — returns a `CompletableFuture<ProcessHandle>` that completes when the process exits. Used to manage subprocesses started with `ProcessBuilder`.
[Pro] What is the difference between `WeakReference`, `SoftReference`, and `PhantomReference`?	All three allow an object to be GC'd while still holding a reference. Strengths: `SoftReference` — cleared before OOM; good for memory-sensitive caches. `WeakReference` — cleared at the next GC cycle when no strong/soft refs remain; good for canonical maps (`WeakHashMap`), listener registries. `PhantomReference` — never returns the referent (`get()` always returns null); enqueued into a `ReferenceQueue` *after* the object is finalized but *before* its memory is freed; used for post-mortem cleanup (replacing `finalize()`). All can be used with a `ReferenceQueue` to track when objects are collected.
[Pro] What causes `StackOverflowError` and how can you prevent or diagnose it?	`StackOverflowError` is thrown when the JVM stack runs out of space — typically caused by infinite or very deep recursion (each method call pushes a new stack frame). Common causes: missing base case in recursion, mutual recursion, extremely deep call chains. Diagnosis: the error includes a partial stack trace. Prevention: (1) Add or fix the base case. (2) Convert recursion to iteration using an explicit `Deque` stack. (3) Use trampolining for tail-recursive algorithms. (4) Increase stack size with `-Xss` (e.g., `-Xss4m`) — temporary fix only. Virtual threads (Java 21+) have growable stacks and are immune to `StackOverflowError` for typical use.
[Pro] What are the variants of `OutOfMemoryError`?	`OutOfMemoryError` has distinct messages identifying the memory space that exhausted: (1) `Java heap space` — heap full; tune `-Xmx` or fix leaks. (2) `GC overhead limit exceeded` — GC spending >98% of time reclaiming <2% of heap; app effectively stalled. (3) `Metaspace` (or PermGen pre-Java 8) — too many classes loaded; common with classloader leaks or code generation libraries. (4) `Unable to create native thread` — OS process limit on threads; reduce thread count or increase OS limits. (5) `Direct buffer memory` — `ByteBuffer.allocateDirect()` exceeded `-XX:MaxDirectMemorySize`. (6) `Native memory` — off-heap memory exhausted. Use heap dumps and `jmap -histo:live` to diagnose.
[Pro] How do you trigger and analyse a heap dump?	Trigger: `jmap -dump:format=b,file=heap.hprof <pid>`, or add JVM flag `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/` for automatic dumps on OOM. Analysis tools: **Eclipse MAT (Memory Analyser Tool)** — best-in-class. Use the **Leak Suspects Report** to identify the likely root. Key concepts: **retained heap** — total heap freed if the object were collected (includes its entire reachability subgraph). **Dominator tree** — shows which objects hold the most retained heap. **GC roots** — static fields, thread locals, JNI globals; objects reachable from roots cannot be collected. Shallow heap vs retained heap: a 16-byte object holding a 100MB array has tiny shallow heap but huge retained heap.
[Pro] What is JMX and how do you use it for JVM monitoring?	JMX (Java Management Extensions) provides a standardized API for monitoring and managing JVM resources. JVM exposes built-in **MBeans** (Managed Beans) via `ManagementFactory`: `MemoryMXBean` — heap/non-heap usage. `ThreadMXBean` — thread counts, deadlock detection (`findDeadlockedThreads()`). `GarbageCollectorMXBean` — GC counts and cumulative times. `RuntimeMXBean` — uptime, JVM input arguments. Connect with `jconsole`, `jvisualvm`, or programmatically: `ManagementFactory.getMemoryMXBean().getHeapMemoryUsage()`. Enable remote JMX: `-Dcom.sun.management.jmxremote.port=9010 -Dcom.sun.management.jmxremote.authenticate=false`. Prometheus exporters (e.g., JMX Exporter agent) scrape MBeans and expose metrics for Grafana dashboards.
[Pro] What is deoptimization in the JVM?	Deoptimization discards a compiled (JIT-optimized) method and falls back to interpreted mode when a speculative assumption is invalidated. Example: HotSpot optimistically inlines a virtual call assuming only one subtype exists (monomorphic call site); if a second subtype is loaded later, the inlined code is no longer valid and is deoptimized. Other triggers: class loading (`instanceof` assumptions), uncommon traps (rarely-executed branches that were never compiled), OSR (on-stack replacement) reversals. A high deoptimization rate (`-XX:+PrintCompilation` output showing `made not entrant`) indicates polymorphic code or unstable type hierarchies. Fix: reduce polymorphism, avoid reflective class loading after warm-up.

[Pro] What are the three phases of class loading in the JVM?	Class loading has three phases: **(1) Loading** — the ClassLoader reads the `.class` file bytes from the source (filesystem, JAR, network) and creates a `Class<?>` object in the method area. **(2) Linking** — three sub-steps: **Verification** checks bytecode correctness and safety (type checks, branch targets); **Preparation** allocates memory for static fields and sets them to default values (zero, null, false); **Resolution** replaces symbolic references (class names, method names) in the constant pool with direct references (pointers) — may be eagerly or lazily resolved. **(3) Initialization** — runs the class's `<clinit>` static initializer block and initialises static fields to their declared values. Initialization is triggered lazily on first active use (object creation, static method call, static field read/write).
[Pro] What is Java Flight Recorder (JFR) and how do you use it?	JFR (open-sourced in Java 11, JEP 328) is a low-overhead, always-on profiling framework built into the JVM. Overhead is typically < 1%. It records predefined and custom events (GC, JIT, thread activity, I/O, exceptions, allocations). Start programmatically: `jcmd <pid> JFR.start duration=60s filename=app.jfr` or via `-XX:StartFlightRecording=duration=60s,filename=app.jfr` JVM flag. Stop: `jcmd <pid> JFR.stop` / `JFR.dump`. Analyse recordings in JDK Mission Control (JMC). Events appear in categories: GC (pause times, reasons), JIT (compilations, deoptimisations), threads (locks, sleep, blocking), heap (TLAB sizes, allocation rates). Custom events: extend `jdk.jfr.Event` with `@Label`, `@Description` annotations. Ideal for production profiling when `jstack`/`jmap` are too invasive.
[Pro] What is the `RandomGenerator` interface (Java 17+)?	`java.util.random.RandomGenerator` (Java 17+, JEP 356) is a common interface over all random number generators. Allows code to accept any RNG implementation without knowing the concrete type. Algorithm families available: **LXM family** (`L64X128MixRandom`, `L64X256MixRandom`, etc.) — excellent statistical quality, very fast, not secure. **Xoshiro/Xoroshiro** (`Xoshiro256PlusPlus`, `Xoroshiro128PlusPlus`) — fast, small state. `SplittableRandom` — supports `split()` for independent streams in parallel code. Obtain by name: `RandomGenerator rng = RandomGeneratorFactory.of("L64X128MixRandom").create()`. `SecureRandom` also implements `RandomGenerator`. `RandomGeneratorFactory.all()` returns a stream of all available factories.
[Pro] What is `HexFormat` (Java 17+)?	`java.util.HexFormat` (Java 17+, JEP 378) provides utility methods for formatting and parsing hex strings. Create: `HexFormat hex = HexFormat.of()` (lowercase, no separator); `HexFormat.ofDelimiter(":")` (colon-separated bytes). Formatting: `hex.formatHex(byte[])` — byte array to hex string; `hex.toHexDigits(int)` — integer to 8-digit hex; `hex.toHexDigits(byte)` — byte to 2-digit hex. Parsing: `hex.parseHex(String)` — hex string to `byte[]`; throws `IllegalArgumentException` on invalid input. Replaces manual `String.format("%02x", b)` loops, `DatatypeConverter.printHexBinary()`, and similar boilerplate. Useful for logging binary data, cryptographic keys, and protocol debugging.
[Pro] What is Generational ZGC (Java 21+, JEP 439)?	ZGC (Z Garbage Collector) was non-generational until Java 21: it treated the entire heap as one space, which is safe but means short-lived objects are swept at the same cost as long-lived ones. Generational ZGC (JEP 439, production-ready Java 21) adds a young/old generation split, exploiting the **generational hypothesis** (most objects die young). Young generation is collected much more frequently than old generation. Benefits: higher throughput (less work per collection), lower allocation stall frequency, similar pause times (sub-millisecond). Enable: `-XX:+UseZGC -XX:+ZGenerational` (Java 21), or just `-XX:+UseZGC` in Java 23+ where generational is the default. Compare: with non-generational ZGC every minor allocation storm triggers a full-heap scan.
[Pro] How do you analyse GC logs and tune G1GC pause times?	Enable GC logging: `-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m`. Key metrics: **pause time** (`GC(x) Pause Young/Old`), **allocation rate** (promotion rate + Eden size flush frequency), **GC frequency** (how often Minor/Full GC fires). GC viewers: GCEasy.io, GCViewer, JDK Mission Control. G1 tuning: `-XX:MaxGCPauseMillis=200` (default) — G1 tries to stay under this target; lowering it forces smaller regions and more frequent collections. `-XX:G1HeapRegionSize=n[m]` — fixed region size (1–32 MB, must be power of 2); larger regions reduce fragmentation at the cost of reclamation granularity. `-XX:G1NewSizePercent` / `G1MaxNewSizePercent` — tune young gen size range. Symptom: long Mixed GC pauses → increase `-XX:G1MixedGCCountTarget`; humongous object allocations → choose region size > half the big object size.
[Pro] What is Java object layout and how do you inspect it with JOL?	Every Java object on the heap has a **header** (12–16 bytes on 64-bit JVM): **mark word** (8 bytes) — stores GC age, identity hash code, lock state (biased/thin/fat), monitor pointer. **Klass pointer** (4 bytes with compressed oops, 8 bytes without) — points to the class metadata. After the header come the instance fields (aligned/padded to 8-byte boundaries). **Object alignment**: each object starts on an 8-byte boundary; padding bytes fill the space after fields to the next alignment boundary. JOL (Java Object Layout) tool: `ClassLayout.parseClass(MyClass.class).toPrintable()` — prints field offsets, sizes, padding. Why it matters: understanding layout helps minimise memory usage and avoid false sharing. Compact records and `@Contended` both rely on layout knowledge.
[Pro] What is AOT Method Profiling (Java 24+, JEP 490)?	JEP 490 (Java 24) complements AOT Class Loading (JEP 483) by persisting JIT **profile data** across JVM runs. Training run: the JVM collects profiling data (which methods are hot, which branches are taken, which call sites are monomorphic/bimorphic) and dumps it to an AOT cache file alongside the class-data cache. Production run: the JIT reads the persisted profile at startup, allowing C2 to apply aggressive optimisations (inlining decisions, speculative optimisations) earlier — without waiting for the warm-up period. Result: applications reach peak performance faster, reducing the latency spike during the first minutes of operation that is characteristic of JIT-compiled Java. Combined with JEP 483, startup time and time-to-peak-performance can both be improved significantly.	OSR allows the JVM to replace an interpreter-executing method with its compiled version *while the method is still on the stack* (e.g., mid-loop). Without OSR, a hot loop would run interpreted until the method returns and is re-entered. OSR inserts a transition point at the loop back-edge: if the loop body becomes hot, the interpreter hands off to the compiled version. Visible in `-XX:+PrintCompilation` as `%` suffix on the method. OSR compilations are slightly less optimized than normal compilations because the live local variables constrain register allocation. Long-running loop-heavy code is the primary beneficiary.

[Intermediate] What is the JVM Heap divided into?	The JVM heap is divided into: Young Generation — where new objects are allocated, subdivided into Eden (allocation) and two Survivor spaces (S0/S1, objects alternate between them after each minor GC). Old Generation (Tenured) — long-lived objects promoted from Young Gen after surviving several GC cycles (controlled by `-XX:MaxTenuringThreshold`). With G1GC, the heap is split into equal-sized regions (~1–32 MB) dynamically assigned as Eden, Survivor, Old, or Humongous (large objects). Metaspace (not in heap) stores class metadata in native memory.
[Intermediate] What is a Stop-The-World (STW) pause in GC?	STW is a GC phase where the JVM pauses all application threads to ensure heap consistency — no object references can change while GC analyzes the heap. Duration varies by GC algorithm: Parallel GC has long STW pauses; G1GC aims for short predictable pauses; ZGC and Shenandoah aim for sub-millisecond STW. STW is necessary for root scanning, object relocation, and reference updates. Minimizing STW is crucial for latency-sensitive applications.
[Intermediate] What are the main GC algorithms in HotSpot JVM?	Serial GC (`-XX:+UseSerialGC`): single-threaded, for small heaps/single-core. Parallel GC (`-XX:+UseParallelGC`): multi-threaded, throughput-focused, default before Java 9. G1GC (`-XX:+UseG1GC`): region-based, predictable pauses, default since Java 9. ZGC (`-XX:+UseZGC`): ultra-low latency (<1ms), concurrent, scalable to TB heaps, production-ready Java 15+; generational ZGC is default in Java 23+. Shenandoah: concurrent evacuation, developed by Red Hat. CMS was deprecated in Java 9 and removed in Java 14.
[Intermediate] What is the ClassLoader hierarchy in Java?	Java has three built-in ClassLoaders in a parent-child hierarchy: (1) Bootstrap ClassLoader: written in C, loads core JDK classes (`java.lang`, `java.util`, etc.); has no parent — the root. (2) Platform ClassLoader (Extension ClassLoader before Java 9): loads JDK extension/platform modules. (3) Application (System) ClassLoader: loads classes from the application classpath. Application servers and OSGi may add more layers. Each ClassLoader holds a reference to its parent.
[Intermediate] What is the parent delegation model of ClassLoaders?	When a ClassLoader receives a load request, it first delegates to its parent ClassLoader. Only if the parent cannot find the class does the child attempt to load it. This ensures: (1) Core JDK classes are always loaded by Bootstrap ClassLoader — cannot be replaced by user code. (2) No duplicate class definitions in the same hierarchy. (3) Security: malicious classes masquerading as `java.lang.String` are rejected. Exception: OSGi and application servers intentionally use child-first loading for class isolation.
[Pro] What is a custom ClassLoader used for?	Custom ClassLoaders (extending `ClassLoader`, overriding `findClass()`) enable: (1) Non-standard sources — loading classes from a network, database, or encrypted JARs. (2) Hot reloading — discard old ClassLoader instance to "reload" classes at runtime (Spring DevTools, JRebel). (3) Class isolation — plugin systems (OSGi, Java EE containers) give each plugin its own ClassLoader namespace to prevent version conflicts. (4) Bytecode transformation — decrypt or instrument classes before defining them. Note: classes from different ClassLoaders are incompatible even with the same binary name.
[Pro] What is method inlining in the JIT compiler?	Method inlining replaces a method call site with the method body directly, eliminating call overhead (stack frame setup/teardown, argument passing) and enabling further optimizations (constant folding, escape analysis, loop unrolling). Triggers: method is hot and small enough (≤35 bytecodes for hot methods). `final`, `private`, and `static` methods are easier to inline (no virtual dispatch). Too-large methods won't inline — keep hot-path methods small. HotSpot inlines up to 9 levels deep. Use `-XX:+PrintInlining` to observe inlining decisions.
[Pro] What is HotSpot's tiered compilation?	HotSpot uses five tiers: Level 0 (interpreted), Level 1 (C1 compiled, no profiling), Level 2 (C1, limited profiling), Level 3 (C1, full profiling — method/branch/type data collected), Level 4 (C2 fully optimized — uses Level 3 profiles for inlining, devirtualization, loop transforms). Methods advance through tiers based on invocation and back-edge counts. Hot methods reach Level 4; less hot stay at Level 1–3. Default since Java 8. Controlled with `-XX:+TieredCompilation`.
[Pro] What is the difference between G1GC (`-XX:+UseG1GC`) and ZGC (`-XX:+UseZGC`)?	G1GC (default since Java 9): region-based, concurrent marking + evacuation, targets predictable pause times (default 200ms via `-XX:MaxGCPauseMillis`). Good balance of throughput and latency for most server workloads. ZGC (production-ready Java 15+): ultra-low latency, sub-millisecond pauses regardless of heap size, concurrent relocation, scalable to multi-TB heaps. Generational ZGC is the default in Java 23+. Slightly lower throughput than G1. Use ZGC for latency-sensitive apps (trading, real-time); G1 for general-purpose applications.
