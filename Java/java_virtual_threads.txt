# Java Virtual Threads (Project Loom)
# Tier: Expert
# Format: Front[TAB]Back

What is the fundamental difference between platform threads and virtual threads in Java 21+?	Platform threads are OS threads — one JVM thread maps to one OS thread. They are expensive to create (~1 MB stack, OS context-switch overhead), so thread pools limit concurrency. Virtual threads are JVM-managed, extremely lightweight threads (a few hundred bytes when not running). They are multiplexed onto a small pool of OS "carrier" threads by the JVM scheduler. You can create millions of virtual threads. When a virtual thread blocks (I/O, sleep, lock), the carrier thread is released and can run another virtual thread — making the CPU never idle waiting on I/O.
How do you create and use virtual threads?	Several ways: (1) `Thread.ofVirtual().start(runnable)` — create and immediately start. (2) `Thread.ofVirtual().name("worker").unstarted(runnable)` — create without starting. (3) `Executors.newVirtualThreadPerTaskExecutor()` — an `ExecutorService` that creates a new virtual thread per submitted task (don't pool them — they're cheap). (4) `Thread.startVirtualThread(runnable)` — shorthand. Virtual threads are daemon threads by default. Naming: `Thread.ofVirtual().name("req-", 0)` names threads `req-0`, `req-1`, etc.
What is virtual thread pinning and how do you detect it?	Pinning occurs when a virtual thread is prevented from releasing its carrier thread while blocked. This defeats the purpose of virtual threads and reduces concurrency. Causes of pinning: (1) `synchronized` blocks or methods — the carrier is pinned for the full duration. (2) `Object.wait()` inside a `synchronized` block. Detect pinning: start JVM with `-Djdk.tracePinnedThreads=full` (logs a stack trace whenever a virtual thread pins) or `-Djdk.tracePinnedThreads=short`. Fix by replacing `synchronized` with `ReentrantLock` (which correctly unmounts the virtual thread on `lock()`). Note: Java 24+ partially addresses the `synchronized` pinning issue.
Why should you NOT pool virtual threads?	Virtual threads are designed to be created per task and discarded — they are so cheap (~few hundred bytes when parked) that pooling adds overhead without benefit. Thread pools were invented to amortize the high cost of OS thread creation. A `VirtualThreadPerTaskExecutor` creates a new virtual thread for each task. Pooling virtual threads (e.g., using `newFixedThreadPool` with virtual threads) actively reduces concurrency by artificially limiting the number of tasks that can run and blocking threads unnecessarily.
How does blocking I/O work with virtual threads?	When a virtual thread performs blocking I/O (file read, HTTP call, JDBC query, `Thread.sleep()`), the JVM intercepts the system call, unmounts the virtual thread from its carrier, and parks the virtual thread. The carrier thread is then free to execute other virtual threads. When the I/O completes, the virtual thread is rescheduled onto an available carrier. This happens transparently — the code uses ordinary blocking APIs (no async/await, no callbacks, no reactive operators needed). The effective model is synchronous code with asynchronous efficiency.
What is Structured Concurrency (Java 24+) and what problem does it solve?	Structured Concurrency (`java.util.concurrent.StructuredTaskScope`) ensures concurrent subtasks have the same lifetime as the scope that spawned them — they cannot outlive the enclosing block. This prevents "orphaned" threads running after the parent exits. Two built-in policies: `ShutdownOnFailure` — if any subtask fails, cancel all others and propagate the exception; use for "all must succeed". `ShutdownOnSuccess` — as soon as any subtask succeeds, cancel others and return the result; use for "first wins" (e.g., call two services, take fastest). Example: `try (var scope = new StructuredTaskScope.ShutdownOnFailure()) { var a = scope.fork(taskA); var b = scope.fork(taskB); scope.join().throwIfFailed(); return combine(a.get(), b.get()); }`.
What are Scoped Values (Java 24+) and why are they preferred over `ThreadLocal` with virtual threads?	`ScopedValue<T>` provides immutable, thread-local-like values that are bound for the duration of a code block and automatically inherited by child threads within a `StructuredTaskScope`. API: `ScopedValue.where(KEY, value).run(() -> { /* KEY.get() available here and in child tasks */ })`. Advantages over `ThreadLocal`: (1) Immutable — no accidental mutation. (2) Automatically scoped — no `remove()` needed (no memory leaks in thread pools). (3) GC-friendly — values eligible for collection when scope exits. (4) Inherited correctly by structured concurrency child tasks. (5) Much better performance under virtual threads (no per-thread map overhead).
What should you avoid to get the most from virtual threads?	(1) Avoid `synchronized` on hot paths — replace with `ReentrantLock`. (2) Avoid CPU-intensive work in virtual threads — they are optimised for I/O-bound tasks; use a dedicated platform-thread pool for CPU work. (3) Don't pool virtual threads — use `newVirtualThreadPerTaskExecutor()`. (4) Avoid `ThreadLocal` for mutable state — use `ScopedValue` instead. (5) Don't use `Thread.currentThread().isVirtual()` to branch logic — code should be unaware and thread-agnostic. (6) Existing synchronized JDBC drivers may pin — check driver compatibility.
