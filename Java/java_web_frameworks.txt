# === Spring Core ===
[Pro] How does Spring MVC process an HTTP request?	(1) Request arrives at `DispatcherServlet` — the single front controller registered with the Servlet container. (2) `DispatcherServlet` consults `HandlerMapping` to find the right `@Controller` method based on URL, HTTP method, headers, etc. (3) `HandlerAdapter` invokes the handler method, processing `@RequestParam`, `@PathVariable`, `@RequestBody` etc. (4) Method returns a view name (for MVC) or writes directly to the response (for `@RestController` using `HttpMessageConverter` to serialize to JSON/XML). (5) `DispatcherServlet` renders the view if needed.
[Pro] What does Spring Boot auto-configuration do?	Spring Boot's auto-configuration (`@EnableAutoConfiguration`, part of `@SpringBootApplication`) scans the classpath and automatically configures Spring beans based on what's present. Example: if Jackson is on the classpath, `MappingJackson2HttpMessageConverter` is configured. If Spring Data JPA + a datasource are present, `JpaRepositories` are set up. Each auto-configuration class is annotated with `@ConditionalOnClass`, `@ConditionalOnMissingBean`, etc. You can override by defining your own beans (presence of your bean disables the auto-configured one) or with `application.properties` / `application.yml`. View what was configured: `--debug` flag shows auto-configuration report.
[Pro] What is the difference between `@Controller` and `@RestController`?	`@RestController` is a convenience annotation that combines `@Controller` (registers the class as a Spring MVC controller) and `@ResponseBody` (each handler method's return value is written directly to the HTTP response body via a `HttpMessageConverter` — typically JSON). With plain `@Controller`, without `@ResponseBody`, the return value is interpreted as a view name for template resolution. Use `@RestController` for REST APIs; use `@Controller` for MVC apps rendering HTML templates (Thymeleaf, Freemarker).
[Pro] How does Spring Security's filter chain work?	Spring Security inserts a `DelegatingFilterProxy` into the Servlet filter chain before any servlet processing. This delegates to a Spring-managed `FilterChainProxy` which contains an ordered chain of security filters. Key filters: `SecurityContextPersistenceFilter` (loads/saves `SecurityContext` from session), `UsernamePasswordAuthenticationFilter` (handles login), `JwtAuthFilter` (custom — validates JWT and sets `SecurityContext`), `ExceptionTranslationFilter` (converts `AccessDeniedException` / `AuthenticationException` to HTTP responses), `FilterSecurityInterceptor` (authorization checks). Configure with `SecurityFilterChain` bean and `HttpSecurity` fluent builder.

# === Spring Data & Transactions ===
[Pro] How does `@Transactional` work in Spring?	`@Transactional` on a method (or class) wraps it in a database transaction managed by Spring's `PlatformTransactionManager`. When a `@Transactional` method is called through a Spring proxy, Spring begins a transaction before the method executes and commits (or rolls back on exception) after it returns. Default rollback on unchecked exceptions (`RuntimeException`, `Error`); checked exceptions do NOT trigger rollback by default — override with `rollbackFor = IOException.class`. Key attributes: `propagation` (REQUIRED, REQUIRES_NEW, NESTED), `isolation` (READ_COMMITTED, SERIALIZABLE), `readOnly = true` (optimization for queries), `timeout`.
[Pro] What is the difference between eager and lazy loading in JPA?	Eager loading (`FetchType.EAGER`) loads the related entities immediately when the parent entity is loaded. Lazy loading (`FetchType.LAZY`) loads the related entities on first access (via a proxy). Default: `@ManyToOne` and `@OneToOne` are EAGER; `@OneToMany` and `@ManyToMany` are LAZY. Pitfall: LazyInitializationException — accessing a lazy collection outside a Hibernate session (e.g., after the transaction closes). Solutions: `@EntityGraph` to fetch eagerly per query, `JOIN FETCH` in JPQL, or `@Transactional` on the calling method to keep the session open. Avoid `FetchType.EAGER` on collections — it causes N+1 problems.

# === Reactive & Modern ===
[Pro] What is Project Reactor and what is the difference between `Mono` and `Flux`?	Project Reactor is a reactive programming library for the JVM based on the Reactive Streams specification. `Mono<T>` represents an asynchronous sequence of 0 or 1 element. `Flux<T>` represents an asynchronous sequence of 0 to N elements. Both are lazy — nothing happens until you subscribe. Key operators: `map`, `flatMap`, `filter`, `zip`, `merge`, `concatMap` (sequential flatMap), `buffer`, `window`, `onErrorResume`, `retry`. Subscribe: `flux.subscribe(onNext, onError, onComplete)` or block (avoid): `mono.block()`. Spring WebFlux is the reactive web framework built on Reactor.
[Pro] What is Spring WebFlux and when should you choose it over Spring MVC?	Spring WebFlux is the reactive, non-blocking web framework for Spring. Built on Netty (or other non-blocking servers) and Project Reactor. Benefit: handles many concurrent requests with a small thread pool — no thread-per-request; threads don't block on I/O. Use when: (1) High concurrency with I/O-bound work (many parallel service calls). (2) Streaming responses (SSE, WebSocket). (3) You're comfortable with reactive programming. Don't switch for: CPU-bound work, existing blocking code/libraries (e.g., regular JDBC), simpler applications — Spring MVC + virtual threads (Java 21+) often achieves similar throughput with less complexity.
[Pro] What is gRPC and how does it work in Java?	gRPC is a high-performance RPC framework by Google using HTTP/2 and Protocol Buffers (Protobuf). You define services and messages in `.proto` files; the `protoc` compiler with the gRPC plugin generates Java stubs. Communication types: unary (request/response), server streaming, client streaming, bidirectional streaming. Java library: `grpc-java`. Server: extends generated `*Grpc.*ImplBase`. Client: creates a `ManagedChannel` and a blocking or async stub. Advantages over REST: strongly-typed contracts, binary encoding (smaller, faster), HTTP/2 multiplexing, built-in streaming. Disadvantage: not human-readable, harder to test with curl/browser, requires Protobuf tooling.

# === WebSockets & SSE ===
[Pro] What is the difference between WebSockets and Server-Sent Events (SSE)?	WebSockets: full-duplex, bidirectional, persistent TCP connection. Either side can send messages at any time. Use for: real-time chat, collaborative editing, games. Higher complexity; uses `ws://` / `wss://`. Server-Sent Events (SSE): unidirectional, server → client, over standard HTTP. Client opens a connection; server pushes events when available. Use for: live dashboards, notifications, streaming API responses. Simpler (standard HTTP, `text/event-stream`); automatic reconnect built into the browser; multiplexed over HTTP/2. In Spring: `SseEmitter` for SSE, `@MessageMapping` + STOMP over WebSocket for WebSocket.

# === API Design ===
[Pro] What are the common strategies for REST API versioning?	(1) URI path versioning — `GET /v1/users`, `GET /v2/users`. Simple, cacheable, visible in logs. Most widely adopted. Drawback: the version is in the URL which is technically part of the resource identity. (2) Accept header versioning — `Accept: application/vnd.myapp.v2+json`. Purist REST approach; keeps URLs clean. Harder to test directly in browser. (3) Query parameter — `GET /users?version=2`. Easy to implement but adds noise. (4) Custom header — `X-API-Version: 2`. Cleanest URLs, but non-standard. Best practice: support at least two versions simultaneously; deprecate with a `Deprecation` or `Sunset` response header; document migration path.

[Pro] How do you integrate JWT authentication in Spring Security?	Add a custom filter that runs before `UsernamePasswordAuthenticationFilter`: extend `OncePerRequestFilter`, extract the `Authorization: Bearer <token>` header, validate the token (signature, expiry, claims) using a library like `io.jsonwebtoken` (JJWT) or `com.nimbusds`, then construct an `UsernamePasswordAuthenticationToken` with the principal and authorities and set it in `SecurityContextHolder`. Register in your `SecurityFilterChain`: `http.addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)`. Method-level security: enable with `@EnableMethodSecurity` on the config class; use `@PreAuthorize("hasRole('ADMIN')")` or SpEL expressions (`@PreAuthorize("#userId == authentication.principal.id")`) on `@Service` or `@RestController` methods. Common pitfalls: not validating token signature, trusting `alg:none` claims, missing expiry check.
[Pro] What is JAX-RS (Jakarta REST) and how do you define endpoints?	JAX-RS (now Jakarta REST, spec 3.x) is the standard Java annotation-based API for building RESTful services. Core annotations: `@Path("/users/{id}")` — maps URI path (on class and/or method). `@GET`, `@POST`, `@PUT`, `@DELETE`, `@PATCH` — HTTP method binding. `@Produces(MediaType.APPLICATION_JSON)` — declares response content type. `@Consumes(MediaType.APPLICATION_JSON)` — restricts accepted request body type. `@PathParam("id")` — binds URI template variable to parameter. `@QueryParam("page")` — binds query string parameter. `@RequestBody` / `@BeanParam` — bind request body (impl-specific). Implementations: Jersey (reference RI), RESTEasy (WildFly), CXF. In Quarkus/Micronaut, JAX-RS annotations work natively. JAX-RS resource classes are CDI beans — can `@Inject` services directly.	Rate limiting protects services from overload and abuse. Strategies: (1) Token bucket / leaky bucket algorithms — allow bursting up to a limit, then throttle. (2) Fixed window counter — reset a counter each time window expires. (3) Sliding window log — tracks individual request timestamps. Libraries: Resilience4j `RateLimiter`, Bucket4j (token-bucket with Hazelcast/Redis backend for distributed limiting), Spring Cloud Gateway built-in rate filter. In a distributed system, the counter must be shared — use Redis with atomic INCR/EXPIRY. HTTP 429 Too Many Requests with `Retry-After` header is the standard response. API gateway (NGINX, AWS API Gateway) can enforce limits before traffic hits application code.
