# Java Reflection & Annotations
# Tier: Advanced
# Format: Front[TAB]Back

# === Reflection ===
What is Java reflection and what can you do with it?	Reflection (`java.lang.reflect`) lets you inspect and manipulate classes, fields, methods, and constructors at runtime — even private ones (with `setAccessible(true)`). Key classes: `Class<?>`, `Method`, `Field`, `Constructor`. Capabilities: instantiate classes by name (`Class.forName("...")`), invoke methods dynamically (`Method.invoke(target, args)`), read/write field values, inspect annotations, discover parameter types. Used heavily by Spring (DI, AOP), Hibernate (ORM mapping), JUnit (test discovery), Jackson (JSON serialization). Java 9+ module system restricts reflective access to non-exported packages — `--add-opens` or `opens` in `module-info.java` may be needed.
What is the performance cost of reflection and how do you mitigate it?	Reflection is 10–100× slower than direct calls due to: (1) Security checks on every `invoke()` call (reduced after `setAccessible(true)`). (2) No JIT inlining — reflective call sites are megamorphic. (3) Boxing/unboxing of primitive arguments and return values. (4) Dynamic method lookup overhead. Mitigations: (1) Cache `Method`, `Field`, `Constructor` objects — lookup is expensive, re-invocation is cheaper. (2) Use `MethodHandle` instead — JIT-friendly, can reach near-direct-call speed after warmup. (3) Use code generation at compile time (Lombok, MapStruct, annotation processors) to avoid reflection entirely at runtime. (4) Use ByteBuddy to generate adapter classes at startup rather than reflecting on every call.
How do you use `Class<?>` to inspect a class at runtime?	`Class<?>` is the entry point for reflection. Obtain it via: `MyClass.class` (literal), `obj.getClass()` (instance), `Class.forName("com.example.MyClass")` (by name). Key methods: `getDeclaredFields()` — all fields declared in this class (incl. private). `getFields()` — public fields incl. inherited. `getDeclaredMethods()` — all methods declared here. `getMethods()` — public methods incl. inherited. `getDeclaredConstructors()`, `getAnnotation(Class)`, `isAssignableFrom(Class)`, `getGenericSuperclass()`. Note: `getDeclared*()` vs `get*()` — declared includes private but not inherited; get* includes inherited but only public. Call `setAccessible(true)` on a `Field` or `Method` to bypass access controls.

# === Annotations ===
What are Java annotations and how are they used?	Annotations are metadata markers attached to code elements (classes, methods, fields, parameters, packages). They don't affect runtime behavior directly but are processed by: (1) Compiler: `@Override`, `@SuppressWarnings`, `@FunctionalInterface`, `@Deprecated`. (2) Annotation processors (APT) at compile time: Lombok, MapStruct, Dagger generate code. (3) Frameworks at runtime via reflection: Spring (`@Component`, `@Autowired`), JUnit (`@Test`), Jackson (`@JsonProperty`). Custom annotation syntax: `@interface MyAnnotation { String value() default ""; int priority() default 0; }`. Can be `@Repeatable` for multiple uses on the same element.
What are the five meta-annotations and what does each control?	Meta-annotations annotate annotation type declarations: (1) `@Retention(RetentionPolicy)` — how long the annotation is retained: `SOURCE` (discarded after compilation, e.g., `@Override`), `CLASS` (in .class file but not at runtime, the default), `RUNTIME` (accessible via reflection at runtime, required for Spring/JUnit annotations). (2) `@Target(ElementType...)` — which elements may be annotated: `TYPE`, `METHOD`, `FIELD`, `PARAMETER`, `CONSTRUCTOR`, `LOCAL_VARIABLE`, `ANNOTATION_TYPE`, `PACKAGE`, `TYPE_USE`. (3) `@Documented` — include annotation in Javadoc. (4) `@Inherited` — annotation on a superclass is inherited by subclasses (class-level only). (5) `@Repeatable(Container.class)` — the annotation can be applied multiple times to the same element; requires a container annotation type.
What is compile-time annotation processing and how does it differ from runtime reflection?	Annotation processing runs custom `AbstractProcessor` subclasses during `javac` compilation. Processors use `javax.lang.model` API to inspect annotated source elements and generate new `.java` files or resources. Multiple rounds: generated files can trigger further processing. The processor is run by the compiler — no JVM overhead at runtime. Examples: Lombok (generates getters/setters/builders at compile time — no reflection at runtime), MapStruct (generates mapper implementations), Dagger (generates DI graph), AutoValue (generates value classes). Runtime reflection reads annotations at runtime via `element.getAnnotation(MyAnnotation.class)` — slower and requires `@Retention(RUNTIME)`. Annotation processing produces zero-overhead abstractions; runtime reflection has per-call cost.
What is `Method.invoke()` and what are its pitfalls?	`Method.invoke(Object target, Object... args)` calls the method reflectively. Steps: get `Method` via `Class.getDeclaredMethod("name", paramTypes)`, call `setAccessible(true)` to bypass visibility, then `invoke(target, args)`. Pitfalls: (1) Checked exceptions are wrapped in `InvocationTargetException` — unwrap with `getCause()`. (2) Wrong argument types throw `IllegalArgumentException`. (3) Private method requires `setAccessible(true)` — may fail with `InaccessibleObjectException` in a module. (4) Null target for instance method throws `NullPointerException`. (5) Performance — call overhead on every invocation. For repeated calls, cache the `Method` object or convert to a `MethodHandle` for JIT-friendly invocation.
