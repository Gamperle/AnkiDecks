# Java Foreign Function & Memory API (Project Panama)
# Tier: Expert
# Format: Front[TAB]Back

[Pro] What is Project Panama and what problems does it solve compared to JNI?	Project Panama (finalized in Java 22, JEP 454) provides the Foreign Function & Memory API as a modern replacement for JNI (Java Native Interface). JNI problems: verbose C headers, error-prone memory management, unsafe `Unsafe` usage, no type safety, requires a C bridge layer even for simple calls, very hard to debug. Panama advantages: (1) Pure Java — no separate C stub code. (2) Type-safe method handles with `FunctionDescriptor`. (3) Explicit, bounded `MemorySegment` allocations instead of raw pointers. (4) Scoped lifetimes via `Arena` — automatic off-heap memory cleanup. (5) `jextract` tool auto-generates Java bindings from C headers. (6) Supported by JIT — downcall handles are as fast as JNI at steady state.
[Pro] What is `MemorySegment` and how does `Arena` control its lifetime?	`MemorySegment` represents a contiguous region of memory — either on-heap or off-heap (native). It carries bounds and an access scope; out-of-bounds and use-after-free are caught at runtime. Allocation is managed by an `Arena`: `Arena.ofConfined()` — access restricted to the creating thread; closed manually or with try-with-resources. `Arena.ofShared()` — shared across threads; reference-counted, closed explicitly. `Arena.ofAuto()` — GC-managed; memory freed when segment becomes unreachable (no manual close). `Arena.global()` — never freed; use for static native structures. Example: `try (Arena arena = Arena.ofConfined()) { MemorySegment seg = arena.allocate(1024); seg.set(ValueLayout.JAVA_INT, 0, 42); }` — memory is freed on scope exit.
[Pro] What is `MemoryLayout` and how do you describe a C struct?	`MemoryLayout` describes the shape of off-heap memory — size, alignment, and structure — without allocating. Building a C struct equivalent: `StructLayout point = MemoryLayout.structLayout( ValueLayout.JAVA_INT.withName("x"), ValueLayout.JAVA_INT.withName("y") ).withName("Point");`. Variants: `ValueLayout` (primitive types: `JAVA_INT`, `JAVA_LONG`, `ADDRESS`), `SequenceLayout` (arrays), `StructLayout` (struct), `UnionLayout` (union), `PaddingLayout` (alignment bytes). Access via `VarHandle`: `VarHandle xHandle = point.varHandle(MemoryLayout.PathElement.groupElement("x"));`. This makes struct field access type-safe and offset-calculated automatically.
[Pro] What is a downcall handle and how do you call a native function?	A downcall handle is a `MethodHandle` that calls a native (C) function from Java. Steps: (1) Find the symbol: `SymbolLookup.libraryLookup("libm.so.6", Arena.global()).find("cos").get()` or `Linker.nativeLinker().defaultLookup().find("strlen").get()`. (2) Describe the function signature with `FunctionDescriptor`: `FunctionDescriptor.of(ValueLayout.JAVA_DOUBLE, ValueLayout.JAVA_DOUBLE)` = C `double cos(double)`. (3) Create the handle: `MethodHandle cos = Linker.nativeLinker().downcallHandle(symbol, descriptor);`. (4) Invoke: `double result = (double) cos.invokeExact(1.0);`. The linker handles marshaling between Java types and native ABI conventions (registers, stack layout).
[Pro] What is an upcall stub and when do you need one?	An upcall stub allows native C code to call back into Java — the inverse of a downcall. Use case: passing a Java callback to a C function like `qsort` (which takes a comparator function pointer). Steps: (1) Write the Java method to serve as callback. (2) Create a `MethodHandle` to it. (3) Convert to a native function pointer (upcall stub): `MemorySegment callback = Linker.nativeLinker().upcallStub(handle, FunctionDescriptor.of(JAVA_INT, ADDRESS, ADDRESS), arena);`. (4) Pass `callback` as an `ADDRESS` argument to the native function. The stub bridges the native ABI calling convention back to a Java `MethodHandle` invocation. The stub is valid for the lifetime of the owning `Arena`.
[Pro] What is `jextract` and how does it accelerate Panama adoption?	`jextract` is a command-line tool that reads C header files and generates corresponding Java bindings using the Foreign Function & Memory API. Given a C header, it generates: constant definitions (as `static final`), struct layouts (`StructLayout`, `VarHandle` accessors), function downcall handles. Usage: `jextract --output src/generated -t com.example.binding /usr/include/openssl/ssl.h`. The generated bindings use the Panama API idiomatically. This eliminates the manual, error-prone work of writing `FunctionDescriptor` and `MemoryLayout` definitions. `jextract` is distributed separately from the JDK but tracks JDK releases. Especially valuable for large C APIs (OpenSSL, LAPACK, system APIs).
What is JNI and why does Project Panama replace it?
[Pro] What is `SegmentAllocator` and how do you allocate native structs?	`SegmentAllocator` is an interface for allocating `MemorySegment` instances within a given `Arena`. Concrete allocators: (1) `arena.allocate(layout)` — allocates a struct-sized segment with the given layout. (2) `arena.allocateFrom(ValueLayout.JAVA_INT, 42)` — allocates and initialises a scalar value. (3) `arena.allocateArray(ValueLayout.JAVA_INT, 1, 2, 3)` — allocates and fills an array. (4) `SegmentAllocator.prefixAllocator(segment)` — sub-allocates from an existing segment. All allocations are automatically freed when the `Arena` is closed. This replaces `sun.misc.Unsafe.allocateMemory()` and the fragile manual `ByteBuffer.allocateDirect()` approach with safe, lifecycle-managed native memory.
[Pro] What is `SymbolLookup` and how do you locate native functions?	`SymbolLookup` resolves function names and global variable names in native libraries to `MemorySegment` addresses. Three sources: `Linker.nativeLinker().defaultLookup()` — symbols from the standard C library (`libc` / `CRT`). `SymbolLookup.loaderLookup()` — symbols from libraries already loaded by the JVM's native class loader (via `System.loadLibrary()`). `SymbolLookup.libraryLookup(path, arena)` — loads a specific native library and provides lookup within it; the library is unloaded when the arena closes. Combine lookups: `defaultLookup.or(SymbolLookup.loaderLookup())`. `find(name)` returns `Optional<MemorySegment>` — empty if the symbol is not found.
[Pro] What is the Panama safety model and how does it prevent memory errors?	The Foreign Function & Memory API enforces safety through **Arenas** and **access modes**. An `Arena` bounds the lifetime of all its `MemorySegment`s — for confined arenas only the owning thread may access the segment; for shared arenas, multi-threaded access is allowed (with memory ordering guarantees); the auto arena uses a `Cleaner` for GC-based release; the global arena never closes. Accessing a `MemorySegment` after its arena is closed throws `IllegalStateException` — detecting use-after-free at the Java level instead of crashing the JVM. Spatial bounds: every segment carries base + size; any out-of-bounds dereference throws `IndexOutOfBoundsException`. Thread confinement on confined arenas prevents data races. This contrasts with `Unsafe.allocateMemory()` and JNI, where nothing prevents use-after-free or buffer overflows.	JNI (Java Native Interface) is the classic mechanism for calling C/C++ code from Java. Problems: (1) Requires writing C glue code with `JNIEnv*` pointer. (2) Error-prone memory management — `GetStringUTFChars` must be paired with `ReleaseStringUTFChars`. (3) No type safety — mismatched signatures cause hard crashes. (4) Performance overhead from JNI transitions (GC safepoint checks, thread state transitions). (5) JNI inhibits JIT inlining across the Java/native boundary. Panama (JEP 454, Java 22 final) replaces JNI with: type-safe `MethodHandle`-based downcalls, explicit `MemorySegment` management via finalized `Arena`, `jextract` for binding generation, and cheaper transitions that the JIT can reason about.

[Pro] What is Project Valhalla and what does it aim to introduce to Java?	Project Valhalla introduces value types (primitive classes / value classes) — objects that behave like primitives: no identity (no `==` identity comparison), stored inline by value (not by reference), no null. Benefits: (1) Eliminates boxing overhead for user-defined types (e.g., `Complex`, `Point`). (2) Arrays of value types are flat (no pointer indirection) — much better cache locality. (3) "Codes like a class, works like an int." Also introduces Universal Generics — generics over primitives (`List<int>`) without boxing. As of Java 23+, value classes are in preview. Will fundamentally change Java's performance and memory characteristics.
