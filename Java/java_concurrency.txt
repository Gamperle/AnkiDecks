# Java Concurrency
# Format: Front[TAB]Back

[Beginner] What is a thread in Java?	The smallest unit of execution within a process. Created by extending `Thread` or implementing `Runnable`.
[Beginner] What is the difference between `Runnable` and `Callable`?	`Runnable.run()` returns void and cannot throw checked exceptions; `Callable.call()` returns a value and can throw checked exceptions.
[Beginner] What does `synchronized` do?	Acquires an intrinsic lock (monitor) on an object, ensuring that only one thread at a time can execute the synchronized block or method.
[Beginner] What is a `volatile` variable?	A variable whose reads and writes are always done directly to main memory, preventing CPU cache inconsistency across threads.
[Beginner] What is a race condition?	A bug that occurs when the outcome depends on the timing/interleaving of multiple threads accessing shared data without proper synchronization.
[Beginner] What is a deadlock?	A situation where two or more threads are each waiting for a lock held by the other, causing all of them to block indefinitely.
[Beginner] What is `ExecutorService`?	A higher-level replacement for managing threads directly; it maintains a pool of threads and executes submitted tasks asynchronously.
[Beginner] What is `Future`?	Represents the result of an asynchronous computation. `get()` blocks until the result is available.
[Beginner] What is `CompletableFuture`?	An extension of `Future` that supports composable asynchronous pipelines with callbacks like `thenApply`, `thenCompose`, and `exceptionally`.
[Beginner] What is `AtomicInteger`?	A class in `java.util.concurrent.atomic` that provides atomic (lock-free, thread-safe) operations on an integer value.
[Beginner] What is a `CountDownLatch`?	A synchronization aid that allows one or more threads to wait until a set of operations in other threads completes (countdown reaches zero).
[Beginner] What is a `Semaphore`?	A synchronization construct that controls access to a shared resource by maintaining a count of available permits.
[Beginner] What is the difference between `wait()` and `sleep()`?	`wait()` releases the intrinsic lock and suspends until `notify()` or `notifyAll()`; `sleep()` pauses the thread but does NOT release any locks.
[Beginner] What is `ThreadLocal`?	A class that provides thread-local variables; each thread holds its own independent copy of the variable.
[Beginner] What is the happens-before relationship?	A guarantee in the Java Memory Model that all actions before a write are visible to a thread that subsequently reads that value.

# === Advanced Concurrency ===
[Pro] What is the difference between `ReentrantLock` and `synchronized`?	Both provide mutual exclusion and are reentrant (same thread can re-acquire). `ReentrantLock` extras: `tryLock()` (non-blocking attempt), `tryLock(timeout, unit)` (timed), `lockInterruptibly()` (can be interrupted while waiting), fairness constructor `new ReentrantLock(true)` (longest-waiting thread wins), multiple `Condition` variables via `lock.newCondition()`. `synchronized` is simpler, automatically released on exit/exception. `ReentrantLock` must be unlocked in a `finally` block. Use `synchronized` for simple cases; `ReentrantLock` when you need `tryLock`, fairness, or multiple conditions.
[Pro] What is `StampedLock` and what is optimistic reading?	`StampedLock` (Java 8+) offers three modes via a stamp (long token): (1) Write lock ‚Äî exclusive: `long stamp = lock.writeLock()`. (2) Read lock ‚Äî shared: `long stamp = lock.readLock()`. (3) Optimistic read ‚Äî non-blocking, no actual lock: `long stamp = lock.tryOptimisticRead()` then validate with `lock.validate(stamp)` afterward; if invalidated, fall back to a real read lock. Higher throughput than `ReentrantReadWriteLock` for read-heavy workloads. Caution: NOT reentrant; does not support `Condition` variables.
[Pro] What is `CyclicBarrier` and how does it differ from `CountDownLatch`?	`CyclicBarrier` makes a fixed number of threads all wait for each other at a barrier point before proceeding together. It is **reusable** ‚Äî resets after each cycle. An optional `Runnable` barrier action runs when the barrier trips. Ideal for iterative parallel algorithms (e.g., simulation phases). `CountDownLatch`: one or more threads wait for N other threads to complete a task. Count is one-way to zero. **Not reusable**. Threads calling `countDown()` do not wait themselves.
[Pro] What is `Phaser` and when is it more flexible than `CyclicBarrier`?	`Phaser` (Java 7+) is the most flexible barrier: (1) Dynamic parties ‚Äî register/deregister with `register()` / `arriveAndDeregister()` at any time. (2) Multi-phase ‚Äî automatically advances to next phase when all parties arrive; reusable like `CyclicBarrier`. (3) Tiered phasers ‚Äî parent-child hierarchy reduces contention at large party counts. (4) Override `onAdvance(phase, parties)` for per-phase logic or early termination. Supports both blocking (`arriveAndAwaitAdvance()`) and non-blocking (`arrive()`) participation.
[Pro] What are `BlockingQueue` implementations and when to use each?	`ArrayBlockingQueue(capacity)` ‚Äî bounded, array-backed, FIFO, optional fairness. `LinkedBlockingQueue(optional capacity)` ‚Äî optionally bounded, linked nodes, higher throughput than `ArrayBlockingQueue`. `SynchronousQueue` ‚Äî zero capacity: each `put()` must wait for a matching `take()` (rendezvous); used in cached thread pools. `PriorityBlockingQueue` ‚Äî unbounded, priority-ordered. `DelayQueue` ‚Äî elements become available only after a per-element delay. `LinkedTransferQueue` ‚Äî unbounded, supports `transfer()` (blocks until a consumer takes the element). Use in producer-consumer patterns as the handoff mechanism.
[Pro] What are the `CompletableFuture` additions in Java 9+?	`orTimeout(duration, unit)` ‚Äî completes exceptionally with `TimeoutException` if not done in time. `completeOnTimeout(value, duration, unit)` ‚Äî completes with a default value on timeout instead of an exception. `completeAsync(Supplier, executor)` ‚Äî asynchronously completes using a supplier. `failedFuture(Throwable)` ‚Äî creates an already-failed future. `newIncompleteFuture()` ‚Äî creates a new incomplete future of the same type (useful for subclassing). Combined with the Java 8 combinators (`thenApply`, `thenCompose`, `allOf`, etc.), these cover most async timeout and fallback patterns without external libraries.
[Pro] What is the Fork/Join framework and when should you use it?	`ForkJoinPool` (Java 7+) is designed for divide-and-conquer parallel algorithms. Tasks extend `RecursiveTask<V>` (returns a value) or `RecursiveAction` (no return). Tasks `fork()` subtasks asynchronously and `join()` their results. Uses **work-stealing**: idle threads steal tasks from the tails of other threads' deques, maximizing CPU use. `ForkJoinPool.commonPool()` is used by parallel streams and `CompletableFuture` by default. Use when: tasks naturally split into independent subtasks, work is CPU-bound, and subtasks are roughly equal in cost. Avoid for I/O-bound work (use virtual threads instead).
[Pro] What is `LockSupport` and how does it relate to `AQS`?	`LockSupport.park()` suspends the current thread (like `Object.wait()` but without requiring a lock). `LockSupport.unpark(thread)` permits the given thread to resume. Key differences from `synchronized`/`Object.wait()`: (1) No monitor needed ‚Äî can park/unpark without holding any lock. (2) Unpark can be called before park ‚Äî the permit is stored, so the subsequent `park()` returns immediately (no spurious block). (3) Non-reentrant permit ‚Äî only one stored permit. `AbstractQueuedSynchronizer` (AQS) uses `LockSupport` internally as the primitive to block/unblock threads in its wait queue. `ReentrantLock`, `Semaphore`, `CountDownLatch`, and `CyclicBarrier` are all built on AQS.
[Pro] What is `submit()` vs `execute()` in `ExecutorService`?	`execute(Runnable)` is defined on `Executor`; it runs the task but discards any result or exception ‚Äî if the task throws, the exception is swallowed by the uncaught exception handler of that thread. `submit(Runnable)` and `submit(Callable<V>)` return a `Future<V>`; exceptions are captured inside the `Future` and thrown when you call `future.get()`. Always prefer `submit()` over `execute()` so you can detect and handle exceptions. Common pitfall: calling `executor.submit(task)` and ignoring the returned `Future` ‚Äî exceptions are silently dropped.
[Pro] What is `scheduleAtFixedRate` vs `scheduleWithFixedDelay`?	Both are on `ScheduledExecutorService`. `scheduleAtFixedRate(command, initialDelay, period, unit)` ‚Äî fires every `period` regardless of task duration. If the task takes longer than `period`, the next execution starts immediately after it finishes (no overlap, but rate may slow). `scheduleWithFixedDelay(command, initialDelay, delay, unit)` ‚Äî waits `delay` after each task completes before scheduling the next. Use `AtFixedRate` for heartbeats / metrics collection where wall-clock frequency matters. Use `WithFixedDelay` when you need a guaranteed rest between executions (e.g., polling with cooldown).
[Pro] What is `invokeAll()` vs `invokeAny()` in `ExecutorService`?	`invokeAll(Collection<Callable<T>>)` submits all tasks and blocks until ALL complete (or timeout). Returns a `List<Future<T>>` ‚Äî some may have exceptions. `invokeAny(Collection<Callable<T>>)` submits all tasks and returns the result of the FIRST one to succeed; cancels remaining tasks. Use `invokeAll` for parallel batch processing where you need every result. Use `invokeAny` for redundant requests where the first response wins (e.g., querying multiple servers, speculative execution). Both accept an optional timeout overload.
What is reactive programming and back-pressure?
[Pro] What is a `Condition` and how do you use `await()`, `signal()`, `signalAll()`?	A `Condition` is the `java.util.concurrent.locks` equivalent of `Object.wait/notify`, but tied to a `Lock` rather than a monitor. Obtain one from a `Lock`: `Condition notFull = lock.newCondition()`. A single `ReentrantLock` can have multiple Conditions (e.g., `notFull` and `notEmpty` for a bounded buffer), whereas `Object` only has one wait-set per monitor. `await()` ‚Äî atomically releases the lock and suspends the thread until signalled or interrupted; re-acquires the lock before returning. `signal()` ‚Äî wakes one waiting thread on this Condition. `signalAll()` ‚Äî wakes all waiting threads. Always call `await()`/`signal()` while holding the associated lock; always re-check the condition in a `while` loop to guard against spurious wake-ups.
[Pro] What are the `Executors` factory types and when should you use each?	`Executors.newFixedThreadPool(n)` ‚Äî `n` threads, unbounded `LinkedBlockingQueue`; good for CPU-bound tasks with known parallelism. `newCachedThreadPool()` ‚Äî creates new threads on demand, reuses idle ones; unbounded ‚Äî can exhaust resources under load; good for many short-lived tasks. `newSingleThreadExecutor()` ‚Äî single thread, tasks serialised; useful for ordered background processing. `newScheduledThreadPool(n)` ‚Äî supports `schedule()`, `scheduleAtFixedRate()`, `scheduleWithFixedDelay()`. `newWorkStealingPool()` (Java 8+) ‚Äî backed by `ForkJoinPool` with parallelism = available processors; work-stealing; good for parallel divide-and-conquer. `newVirtualThreadPerTaskExecutor()` (Java 21+) ‚Äî creates one virtual thread per task; ideal for high-concurrency I/O-bound workloads. Prefer explicit `ThreadPoolExecutor` construction for production code to control queue size and rejection policy.	Reactive programming models data flows as asynchronous streams of events. Producers push data; operators transform/filter; consumers process asynchronously. **Back-pressure** is the mechanism by which a slow consumer signals a fast producer to slow down ‚Äî preventing unbounded buffering and OOM. In Java: `java.util.concurrent.Flow` (Reactive Streams spec): `Subscriber.onSubscribe()` receives a `Subscription`; the subscriber calls `subscription.request(n)` to signal it can handle `n` more items. The publisher must not emit more than requested. Libraries: Project Reactor (`Flux`, `Mono`), RxJava (`Observable`, `Flowable` ‚Äî Flowable has back-pressure, Observable does not).

[Intermediate] What is the difference between a process and a thread in Java?	A process is an independent program instance with its own dedicated memory space (heap, stack, code, data segments) managed by the OS. Processes are isolated ‚Äî one crash doesn't affect others. A thread is a lightweight unit of execution within a process. All threads of a process share the same heap and code, but each has its own stack and program counter. Thread creation is cheaper than process creation. Java threads map to OS threads (platform threads) by default, or to virtual threads (Java 21+) managed by the JVM.
[Intermediate] What are the two guarantees provided by the `volatile` keyword in Java?	`volatile` provides two guarantees: (1) Visibility ‚Äî a write to a `volatile` variable is immediately flushed to main memory, and any subsequent read by another thread reads from main memory, not a CPU cache. (2) Ordering ‚Äî prevents instruction reordering across the volatile access (happens-before guarantee). It does NOT guarantee atomicity for compound operations like `i++` (read-modify-write). Use `AtomicInteger` or `synchronized` for atomicity.
[Intermediate] What is the difference between a synchronized method and a synchronized block?	A synchronized method acquires the intrinsic lock on `this` (or the Class object for static methods) for the entire method ‚Äî coarse-grained. A synchronized block acquires a lock on a specified object for only a portion of the code ‚Äî fine-grained, reducing contention. Example: `synchronized(lockObject) { ... }`. Prefer synchronized blocks when only part of the method needs protection to improve throughput.
[Intermediate] What is a deadlock and how can you prevent it in Java?	A deadlock occurs when thread A holds lock X and waits for lock Y, while thread B holds lock Y and waits for lock X ‚Äî both block forever. Prevention strategies: (1) Lock ordering: always acquire locks in the same global order. (2) `tryLock()` with timeout: `lock.tryLock(500, TimeUnit.MILLISECONDS)` to avoid indefinite waiting. (3) Avoid nested locks: minimize holding multiple locks simultaneously. (4) Use higher-level concurrency utilities (`java.util.concurrent`). Deadlocks can be detected with thread dumps using `jstack`.
[Intermediate] What is a livelock in Java?	A livelock occurs when threads continuously respond to each other's actions without making progress ‚Äî unlike deadlock, the threads are not blocked but are busy doing useless work. Classic example: two threads each "politely" back off and retry simultaneously, forever. Prevention: introduce randomized back-off delays, or designate one thread as the "winner" via priority or ordering.
[Intermediate] What does `AtomicInteger` provide over a `volatile int`?	`volatile int` only guarantees visibility of a single read or write ‚Äî it cannot atomically perform compound operations. `AtomicInteger` provides lock-free atomic compound operations using CAS (Compare-And-Swap) hardware instructions: `getAndIncrement()`, `incrementAndGet()`, `getAndAdd(delta)`, `compareAndSet(expected, update)`, `accumulateAndGet()`. These guarantee that read-modify-write sequences are atomic without `synchronized`, making them highly performant in concurrent scenarios.
[Pro] What is CAS (Compare-And-Swap) and what is the ABA problem?	CAS is a hardware-level atomic instruction: given a memory address, an expected value, and a new value ‚Äî it atomically updates the memory only if it currently contains the expected value, returning success/failure. The foundation of lock-free programming. All `java.util.concurrent.atomic` classes use CAS via `VarHandle`. ABA problem: CAS can incorrectly succeed if a value changes A‚ÜíB‚ÜíA; solved with `AtomicStampedReference`.
[Pro] What is the difference between `thenApply()` and `thenCompose()` in `CompletableFuture`?	`thenApply(Function<T,U>)` transforms the result synchronously ‚Äî analogous to `map()`. If the function itself returns a `CompletableFuture<U>`, you get nested `CompletableFuture<CompletableFuture<U>>`. `thenCompose(Function<T, CompletableFuture<U>>)` chains a dependent async computation ‚Äî analogous to `flatMap()`. Flattens to `CompletableFuture<U>`. Use `thenCompose` when the next step is itself async (e.g., another HTTP call), to avoid nesting.
[Pro] What is double-checked locking (DCL) and when is `volatile` essential?	DCL is a lazy initialization pattern that avoids synchronization overhead after initialization: check the field without a lock; if null, lock and check again, then initialize. Without `volatile`, the JVM may reorder the write to the object's fields with the assignment of the reference ‚Äî another thread sees a non-null but partially constructed object. `volatile` on the field inserts a happens-before edge ensuring all writes to the object are visible before the reference is published. Prefer the initialization-on-demand holder (Bill Pugh) pattern or enum singleton as simpler alternatives.
[Intermediate] What is `ScheduledExecutorService` and what does it add over `ExecutorService`?	`ScheduledExecutorService` extends `ExecutorService` for time-based task scheduling: `schedule(callable, delay, unit)` ‚Äî run once after a delay; `scheduleAtFixedRate(runnable, initialDelay, period, unit)` ‚Äî run periodically from task start; `scheduleWithFixedDelay(runnable, initialDelay, delay, unit)` ‚Äî run with fixed delay between task end and next start. Returns `ScheduledFuture` for cancellation. Replacement for error-prone `Timer`/`TimerTask` ‚Äî proper exception handling, multi-thread support. Create with `Executors.newScheduledThreadPool(n)`.
