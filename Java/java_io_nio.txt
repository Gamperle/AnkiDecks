# Java I/O and NIO
# Tier: Intermediate
# Format: Front[TAB]Back

# === java.io ===
[Intermediate] What is the difference between `InputStream`/`OutputStream` and `Reader`/`Writer`?	`InputStream` and `OutputStream` deal with raw bytes (8-bit). Use for binary data: images, audio, arbitrary bytes. `Reader` and `Writer` deal with characters (16-bit Unicode). Use for text data. Key implementations: byte streams — `FileInputStream`, `FileOutputStream`, `ByteArrayInputStream`, `BufferedInputStream`. Char streams — `FileReader`, `FileWriter`, `InputStreamReader` (byte→char bridge using a charset), `BufferedReader`, `PrintWriter`. Always specify a charset explicitly; never rely on the platform default.
[Intermediate] Why should you always wrap I/O streams in a Buffered variant?	Unbuffered I/O issues one system call per byte/character read or written, causing massive performance overhead. `BufferedInputStream` / `BufferedReader` read chunks into an in-memory buffer and serve reads from it. `BufferedOutputStream` / `BufferedWriter` accumulate writes and flush in chunks. Example: reading a file byte-by-byte without buffering may be 100x slower than with buffering. Always: `new BufferedReader(new FileReader(path))`. With try-with-resources the `BufferedReader` is closed automatically, which also closes the underlying `FileReader`.
[Intermediate] What is `InputStreamReader` and why is it important?	`InputStreamReader` is a bridge that converts a byte `InputStream` into a `Reader` by decoding bytes to characters using a specified charset. Always specify the charset explicitly: `new InputStreamReader(stream, StandardCharsets.UTF_8)`. Never use the no-charset constructor — it uses the system default charset, which varies between platforms and causes encoding bugs when deploying across environments.
[Intermediate] What is Java object serialization and what are its risks?	Java serialization converts an object graph to a byte stream (`ObjectOutputStream.writeObject()`) for persistence or transmission; deserialization (`ObjectInputStream.readObject()`) reconstructs it. Classes must implement `Serializable`. `transient` fields are skipped. `serialVersionUID` is used for version compatibility checking. Risks: (1) Security — deserializing untrusted data can execute arbitrary code via gadget chains. (2) Fragility — class changes break compatibility. (3) Performance. Alternatives: JSON (Jackson), Protocol Buffers, CBOR. If you must deserialize, use `ObjectInputFilter` to whitelist allowed classes.

# === java.nio.file ===
[Intermediate] What is the `Path` API in `java.nio.file`?	`Path` (Java 7+) represents a file system path and replaces `java.io.File`. Created with `Path.of("dir/file.txt")` (Java 11+) or `Paths.get("dir/file.txt")`. Key methods: `getFileName()`, `getParent()`, `toAbsolutePath()`, `normalize()` (resolves `.` and `..`), `resolve(other)` (appends a relative path), `relativize(other)` (computes relative path between two paths), `startsWith(prefix)`. `Path` is immutable. Convert to legacy `File` with `.toFile()`.
[Intermediate] What utilities does `java.nio.file.Files` provide?	`Files` is a utility class with static methods for file operations: Reading — `Files.readAllBytes(path)`, `Files.readAllLines(path)`, `Files.readString(path)` (Java 11+), `Files.lines(path)` (lazy stream). Writing — `Files.write(path, bytes)`, `Files.writeString(path, text)` (Java 11+). Copying/moving — `Files.copy(src, dst, CopyOption...)`, `Files.move(src, dst)`. Directory ops — `Files.createDirectories(path)`, `Files.list(dir)` (lazy stream), `Files.walk(dir)` (recursive lazy stream), `Files.find(dir, depth, matcher)`. Metadata — `Files.exists(path)`, `Files.size(path)`, `Files.getLastModifiedTime(path)`, `Files.isRegularFile(path)`.
[Intermediate] How does `Files.walk()` work and what should you watch out for?	`Files.walk(Path, int maxDepth, FileVisitOption...)` returns a lazy `Stream<Path>` of all files/directories under the given root, recursively. The stream must be closed after use (use try-with-resources) to release the underlying directory iterator. Be aware: it does not follow symbolic links by default (pass `FileVisitOption.FOLLOW_LINKS` to enable). For very large trees, filter early in the pipeline to avoid loading all paths into memory. Example: `try (var stream = Files.walk(dir)) { stream.filter(Files::isRegularFile).forEach(System.out::println); }`.
[Intermediate] What is the difference between `Files.readString()` (Java 11+) and `Files.readAllLines()`?	`Files.readString(Path)` reads the entire file content as a single `String` in one call. Optional second argument specifies charset (default UTF-8). Best for small files where you want the entire content. `Files.readAllLines(Path)` reads all lines into a `List<String>`, stripping line terminators. Also reads the whole file into memory. `Files.lines(Path)` returns a lazy `Stream<String>` — better for large files as it reads on demand. Remember to close the stream: `try (var lines = Files.lines(path)) { ... }`.
[Intermediate] What is `Files.mismatch()` (Java 12+)?	`Files.mismatch(Path path1, Path path2)` compares the content of two files byte-by-byte and returns the position (0-based) of the first differing byte, or `-1L` if the files are identical. Useful for file comparison utilities, test assertions, or cache validation without reading entire files into memory. If one file is shorter, the returned position is the length of the shorter file.

# === NIO Channels (Advanced I/O) ===
[Pro] What is a `FileChannel` and when would you use it over `Files`?	`FileChannel` is an NIO channel for reading, writing, mapping, and locking files. Advantages over `Files`: (1) Memory-mapped files — `map(MapMode, position, size)` maps a file region directly into memory for very fast bulk reads/writes. (2) File locking — `lock()` / `tryLock()` for inter-process coordination. (3) Positioned reads/writes — read/write at specific positions without seeking. (4) Efficient transfers between channels — `transferTo()` / `transferFrom()`. Open with `FileChannel.open(path, StandardOpenOption.READ)`. Essential for high-performance or low-level file work.
[Pro] What is the difference between blocking and non-blocking I/O in Java NIO?	Traditional I/O (java.io) is **blocking** — `read()` and `write()` block the calling thread until data is available or written. NIO (java.nio) supports **non-blocking** mode: `channel.configureBlocking(false)`. A non-blocking `read()` returns 0 immediately if no data is ready, without blocking. A `Selector` multiplexes many non-blocking channels onto a single thread: register channels with `channel.register(selector, SelectionKey.OP_READ)`; call `selector.select()` to block until at least one channel is ready; iterate `selector.selectedKeys()` to handle ready channels. This enables high-concurrency servers with few threads. Java 21+ virtual threads largely replace NIO selector-based servers by making blocking I/O cheap.
[Pro] What are memory-mapped files and when should you use them?	A memory-mapped file maps a file (or region) directly into the process address space via `FileChannel.map(MapMode.READ_ONLY, 0, size)`, returning a `MappedByteBuffer`. Reads/writes to the buffer become reads/writes to the file's pages in the OS page cache. Benefits: (1) Zero-copy — no explicit `read()`/`write()` calls; OS demand-pages file content. (2) Shared memory between processes — multiple processes map the same file. (3) Very fast for large sequential or random-access reads. Pitfalls: (1) The mapping occupies virtual address space (32-bit process limit). (2) `MappedByteBuffer` cannot be unmapped explicitly in Java — reference must go out of scope + GC. (3) File changes from another process are immediately visible. Use for large read-only datasets, log scanning, inter-process communication.
[Pro] What is the difference between `Serializable` and `Externalizable`?	`Serializable` is a marker interface — Java handles all serialization automatically using reflection to read/write fields. Add `serialVersionUID` for version control; use `transient` to exclude fields. `Externalizable` extends `Serializable` and requires implementing `writeExternal(ObjectOutput)` and `readExternal(ObjectInput)` — you control exactly what is serialized/deserialized. Externalizable is faster (no reflection) and more compact, but more code. It also requires a public no-arg constructor (called during deserialization before `readExternal`). Modern alternatives (Jackson, Protobuf) are preferred over both for new code — Java serialization has security (deserialization gadget chains) and fragility concerns.

# === From java_advanced_anki_expanded ===
[Intermediate] What is Java NIO (New I/O) and what are its three core abstractions?	`java.nio` (Java 4+) provides a more efficient I/O model with three core abstractions: (1) Channels (`FileChannel`, `SocketChannel`, `ServerSocketChannel`) — bidirectional I/O to/from Buffers; can be non-blocking. (2) Buffers (`ByteBuffer`, `IntBuffer`, etc.) — fixed-size containers with `position`, `limit`, and `capacity`; `flip()` switches between write and read mode. (3) Selectors — multiplex multiple non-blocking channels on a single thread (event-driven I/O, like epoll). NIO.2 (Java 7, `java.nio.file`) adds `Path`, `Files`, `WatchService`, and asynchronous file I/O (`AsynchronousFileChannel`).
