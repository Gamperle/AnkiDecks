# Java Design Patterns — GoF & JDK
# Tier: Advanced
# Format: Front[TAB]Back

# === Creational ===
What is the Singleton pattern and what are the thread-safe implementation options in Java?	Singleton ensures only one instance of a class exists per JVM. Thread-safe options: (1) Eager initialization — `private static final Singleton INSTANCE = new Singleton();` — simple, always thread-safe. (2) Synchronized factory method — `public static synchronized Singleton getInstance()` — correct but slow (lock on every call). (3) Double-checked locking — requires `private static volatile Singleton instance;` — fast after initialization; `volatile` prevents the JVM reordering the partially-constructed object assignment. (4) Enum singleton — `public enum Singleton { INSTANCE; }` — JVM guarantees uniqueness; also serialization-safe. (5) Initialization-on-demand holder (Bill Pugh) — `private static class Holder { static final Singleton INSTANCE = new Singleton(); }` — lazy, thread-safe via class loading guarantee; recommended.
What is the Builder pattern and when should you use it?	Builder separates the construction of a complex object from its representation, using a fluent step-by-step API. Use when: an object has many optional parameters (avoids telescoping constructors), or construction involves validation rules that should be enforced before creating the object. JDK examples: `StringBuilder`, `HttpRequest.newBuilder()`, `ProcessBuilder`, `Stream.builder()`. Common Java implementation: a static nested `Builder` class with setter-style methods returning `this`, and a `build()` method that constructs and validates the final object. Lombok `@Builder` generates this automatically.
What is the Factory Method pattern?	Factory Method defines an interface for creating an object but lets subclasses decide which concrete class to instantiate. The factory method is typically an abstract or overridable method in a base class. Decouples the object creation from its use. JDK examples: `NumberFormat.getInstance()`, `Calendar.getInstance()`, `Charset.forName()`, `Path.of()`. Contrast with Abstract Factory (creates families of related objects) and Static Factory Method (a plain static method that returns instances — not strictly GoF).
What is the Prototype pattern and what are its pitfalls in Java?	Prototype creates new objects by copying (cloning) an existing prototype rather than using `new`. In Java: implement `Cloneable` and override `clone()`. Pitfalls: (1) `Object.clone()` performs a SHALLOW copy by default — fields pointing to mutable objects still share the same reference with the original. (2) `Cloneable` is a marker interface — not calling `super.clone()` breaks the contract. (3) `clone()` bypasses constructors, which can break class invariants. Safer alternatives: copy constructors (`new Person(other)`) or copy factory methods, which give full control over what is copied.

# === Structural ===
What is the Decorator pattern and how does it appear in the JDK?	Decorator attaches additional behaviour to an object dynamically by wrapping it in a decorator that shares the same interface, without using subclassing. Enables constructing complex behaviour through composition of simple objects. JDK examples: `BufferedInputStream wrapping FileInputStream` — adds buffering; `DataOutputStream wrapping FileOutputStream` — adds typed writes; `Collections.unmodifiableList(list)` — adds immutability checks; `Collections.synchronizedList(list)` — adds synchronization. Each wrapper adds one responsibility — the classic "onion" composition.
What is the Proxy pattern and what are the three types in Java?	Proxy provides a surrogate or placeholder to control access to another object. Three main types in Java: (1) Static Proxy — hand-written class implementing the same interface, delegating to the real object. (2) JDK Dynamic Proxy — `java.lang.reflect.Proxy.newProxyInstance(classLoader, interfaces, InvocationHandler)` — works only for interfaces; handler intercepts all calls. (3) CGLIB Proxy — generates a subclass at runtime (used by Spring for concrete classes without interfaces). Uses: lazy initialization, access control, logging, transaction management (Spring AOP), caching.
What is the Adapter pattern?	Adapter converts the interface of a class into another interface that a client expects, making incompatible interfaces work together. Two forms: Class Adapter (uses inheritance — Java's single inheritance limits this), Object Adapter (uses composition — preferred). JDK examples: `Arrays.asList()` — wraps an array in a `List` interface; `InputStreamReader` — adapts `InputStream` (bytes) to `Reader` (chars); `Collections.enumeration(list)` — adapts `List` to legacy `Enumeration`. Distinguish from Bridge: Adapter makes incompatible things work together retroactively; Bridge separates abstraction from implementation by design.
What is the Flyweight pattern?	Flyweight uses shared objects to efficiently support a large number of fine-grained objects, where many share identical intrinsic state. Intrinsic state is shared and immutable; extrinsic state is passed in per-use. JDK examples: `Integer.valueOf(n)` caches `Integer` objects for −128 to 127 — if you call `valueOf(100)` twice you get the SAME object; `Boolean.valueOf()` returns one of two cached instances; `String.intern()` stores only one copy of each distinct string literal in the pool. Common mistake: using `new Integer(n)` (deprecated) bypasses the cache.

# === Behavioral ===
What is the Strategy pattern?	Strategy defines a family of algorithms, encapsulates each one, and makes them interchangeable. The algorithm can vary independently from the clients that use it. JDK examples: `Comparator` passed to `List.sort()` or `TreeMap`; `java.util.function.*` — passing in different functions; `ThreadPoolExecutor.RejectedExecutionHandler` — pluggable rejection policy. In Spring: `PasswordEncoder` strategy; `ResourceLoader`. Enables Open/Closed Principle — add new algorithms without modifying existing code.
What is the Observer pattern?	Observer defines a one-to-many dependency: when the subject (observable) changes state, all its registered observers are notified automatically. JDK examples: `java.util.EventListener` (Swing/AWT events); `PropertyChangeSupport` / `PropertyChangeListener`; `Flow.Publisher` / `Flow.Subscriber` (Java 9+ reactive streams). `java.util.Observable` was deprecated in Java 9 due to design flaws. Spring: `ApplicationEventPublisher` + `@EventListener`. Reactive variants (Reactor, RxJava) are the modern push-based implementation.
What is the Template Method pattern?	Template Method defines the skeleton of an algorithm in a base class, deferring the implementation of specific steps to subclasses. The base class calls the abstract or overridable "hook" methods in a fixed order. JDK examples: `AbstractList` — provides `iterator()`, `contains()`, `indexOf()` implementations based on abstract `get(index)` and `size()`; `HttpServlet.service()` — dispatches to `doGet()` / `doPost()` etc.; `TimerTask.run()`. Prefer composition over inheritance — the Strategy pattern achieves a similar goal via injection rather than subclassing.
What is the Chain of Responsibility pattern?	Chain of Responsibility passes a request along a chain of handlers, where each handler decides to process the request or pass it to the next handler. JDK/Java examples: `ClassLoader` delegation model — child loader asks parent first; `javax.servlet.FilterChain` — each filter calls `chain.doFilter()` to pass to the next; Spring Security's security filter chain; `java.util.logging` Logger parent delegation. The chain can be terminated at any point by not calling the next handler.
What is the Command pattern?	Command encapsulates a request as an object, allowing it to be parameterized, queued, logged, or undone. JDK examples: `Runnable` — encapsulates a unit of work; `Callable` — encapsulates a computation with a return value; `java.awt.event.ActionListener` — encapsulates a UI action. Benefits: decouples invoker from executor; enables undo/redo (by storing inverse commands); enables macro commands (composite of commands); supports queuing and scheduling.
What is the Visitor pattern and where does it appear in the JDK?	Visitor lets you add operations to an object structure (like a composite tree) without modifying its classes. Define a `Visitor` interface with a `visit(ConcreteElement)` method per element type; each element implements `accept(Visitor)` which calls `visitor.visit(this)`. JDK examples: `java.nio.file.FileVisitor` — `Files.walkFileTree(path, visitor)` calls `visitFile`, `preVisitDirectory`, `postVisitDirectory`, etc.; `javax.lang.model.element.ElementVisitor` used in annotation processors. New in Java 24: `ClassFile` API uses visitors for inspecting bytecode elements.

# === JDK Pattern Spotting ===
Where can you see the Facade pattern in the JDK?	`java.net.URL` hides the complexity of socket creation, protocol handling, and connection management behind a simple `openStream()` API. `javax.xml.parsers.DocumentBuilderFactory` provides a simple facade over complex XML parsing machinery. `SLF4J` facades multiple logging backends (Logback, Log4j2, java.util.logging) behind a single API. `javax.faces.context.FacesContext` in JSF. In Spring: `JdbcTemplate` facades raw JDBC (exception translation, resource management). Use Facade when a subsystem has many moving parts — expose only what clients need.
What is the Bridge pattern and how does it differ from Adapter?	Bridge decouples an abstraction from its implementation so that the two can vary independently. You create two separate inheritance hierarchies — one for the abstraction and one for the implementation — and the abstraction holds a reference (the bridge) to an implementation. Example: `Shape` (abstraction) with `DrawingAPI` (implementation) — you can add new shapes and new drawing APIs independently. JDK example: JDBC — `DriverManager` + `Connection` bridge allows different database drivers without changing the client code. Key difference from Adapter: Adapter reconciles incompatible EXISTING interfaces; Bridge is designed up-front to allow abstraction and implementation to evolve independently.
What is the Composite pattern?	Composite lets clients treat individual objects (leaves) and compositions of objects (composites) uniformly through a common interface. Forms a tree structure where both leaves and composites implement the same `Component` interface. JDK examples: `java.awt.Component` (leaf) and `java.awt.Container` (composite — holds other `Component`s); `java.nio.file.Path` represents files or directories uniformly; `javax.swing.JComponent` hierarchy. Use when you have a part-whole hierarchy and want clients to ignore the difference between leaf and composite operations. Common methods: `add(Component)`, `remove(Component)`, `getChildren()`, `operation()`.
What is the Interpreter pattern?	Interpreter defines a grammar for a language and provides an interpreter to deal with that grammar. Each grammar rule becomes a class; an abstract syntax tree (AST) is evaluated by calling `interpret()` on the tree. JDK examples: `java.util.regex.Pattern` — compiles a regex grammar into a matcher; `javax.el.ExpressionLanguage` — evaluates EL expressions in JSP/JSF; Spring Expression Language (SpEL). Limitations: the class explosion for complex grammars makes it unwieldy — most real parsers use generated code (ANTLR) or hand-rolled recursive descent instead.
What is the Mediator pattern?	Mediator defines an object that encapsulates how a set of objects interact, reducing direct dependencies between them (lowering coupling). Objects communicate through the mediator rather than directly. JDK/Java examples: `java.util.concurrent.ExecutorService` mediates between task submitters and worker threads; Spring's `ApplicationEventPublisher` — components publish events and subscribe without knowing each other; ATC (Air Traffic Control) — classic textbook example. In UI: `JFrame` as mediator between Swing components. Benefits: simplifies maintenance — interaction rules are in one place. Risk: mediator itself becomes a "god object" if overloaded.
What is the Memento pattern?	Memento captures and externalises an object's internal state without violating encapsulation, so the object can be restored to that state later (undo/redo). Roles: Originator (object whose state is saved), Memento (snapshot), Caretaker (stores mementos without inspecting them). Java implementation options: serialisation to store/restore complete object state; copy constructors; records (immutable snapshot). JDK examples: `Serializable` objects used as snapshots; `javax.swing.undo.UndoManager` + `UndoableEdit`. Pitfall: deep object graphs require deep copying — shallow memento breaks on mutable nested objects.
What is the State pattern?
What is the Abstract Factory pattern?	Abstract Factory provides an interface for creating **families of related objects** without specifying their concrete classes. Clients use the factory interface; the concrete factory determines which product family is instantiated. Classic example: UI toolkit factory — `GUIFactory` with `createButton()` and `createCheckbox()`; `WindowsFactory` and `MacFactory` implement it and return platform-specific widgets. Difference from Factory Method: Factory Method creates one product; Abstract Factory creates a family of related products (Button + Checkbox + Dialog all consistent for one platform). JDK examples: `DocumentBuilderFactory.newInstance()` — returns a concrete XML parser implementation; `java.sql.Connection` + `Statement` + `ResultSet` are a family created by `DriverManager`-obtained connections.	State allows an object to alter its behaviour when its internal state changes — the object appears to change its class. Extract each state into a separate class implementing a `State` interface; the context delegates behaviour to its current state object. JDK/Java examples: `java.lang.Thread` — lifecycle states (NEW, RUNNABLE, BLOCKED, WAITING, TERMINATED) each with distinct behaviour; TCP connection states (LISTEN, ESTABLISHED, CLOSE_WAIT); `javax.faces.lifecycle.Lifecycle` phases. Difference from Strategy: Strategy is typically set once and focuses on interchangeable algorithms; State transitions happen automatically in response to events and the state itself can trigger transitions.
