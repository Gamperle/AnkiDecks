# ==========================================
# 13. Git Internals
# ==========================================

What are the four object types in Git?	(1) blob — stores file content (no filename). (2) tree — stores directory listings (filenames + pointers to blobs/trees). (3) commit — points to a tree, parent commit(s), author, committer, and message. (4) tag — annotated tag object pointing to another object (usually a commit) with tagger info and message.
How does Git's content-addressable storage work?	Every object is stored under a key computed as the SHA-1 (or SHA-256) hash of its type, size, and content. Identical content always produces the same hash, enabling deduplication — Git stores each unique version of a file exactly once.
What is inside the .git directory?	Key contents: `objects/` (blob, tree, commit, tag objects), `refs/` (branches, tags, remote refs), `HEAD` (pointer to current branch), `index` (staging area binary file), `config` (repo-level settings), `hooks/` (hook scripts), `packed-refs` (compressed refs), `logs/` (reflog).
What are loose objects vs packed objects?	Loose objects are individual files in .git/objects/ (one file per blob/tree/commit). Packed objects are compressed into .pack files with .idx index files in objects/pack/ — Git deltifies similar objects for efficient storage.
How does Git deltify objects in packfiles?	Git finds similar objects and stores the newer one as a delta (binary diff) against the older base object. This dramatically reduces storage since successive file versions share most content. `git pack-objects` and `git gc` perform packing.
What does `git verify-pack -v <pack.idx>` show?	The contents of a packfile: each object's SHA, type (blob/tree/commit), size, packed size, offset, and delta chain depth. Useful for analyzing repository storage efficiency.
What is the index (staging area) at the binary level?	A binary file (.git/index) containing a sorted list of file entries: path, mode (permissions), blob SHA, stat data (ctime, mtime, size, inode), and flags. Git compares working tree stat data against the index to detect modifications efficiently.
What are loose refs vs packed-refs?	Loose refs are individual files in .git/refs/ (e.g., refs/heads/main containing a commit SHA). packed-refs is a single file listing many refs — Git packs refs for efficiency when there are many branches/tags. Loose refs take precedence over packed-refs.
How does Git detect renames?	Git does not track renames explicitly. It detects renames at diff/log time by comparing deleted and added files — if content similarity exceeds a threshold (default 50%), Git reports it as a rename. Use `-M` (or `--find-renames`) to tune the threshold.
What is `git cat-file` used for?	Inspecting raw Git objects: `git cat-file -t <sha>` shows the object type, `git cat-file -s <sha>` shows the size, `git cat-file -p <sha>` pretty-prints the content (blob content, tree listing, commit details).
How does Git store a commit object internally?	A commit object contains: `tree <sha>` (root tree of the snapshot), `parent <sha>` (one per parent, zero for root commit, two for merge), `author <name> <email> <timestamp>`, `committer <name> <email> <timestamp>`, blank line, and the commit message.

# ==========================================
# 14. Advanced History Manipulation
# ==========================================

What is `git filter-repo` and why does it replace `git filter-branch`?	git-filter-repo is a third-party tool (Python) for rewriting repository history — removing files, renaming paths, modifying commit messages. It's orders of magnitude faster than git filter-branch, which is officially deprecated due to performance issues and subtle correctness problems.
How do you remove a file containing secrets from the entire Git history?	Use `git filter-repo --path secret.env --invert-paths` to rewrite all commits removing that file. Or use BFG Repo-Cleaner: `bfg --delete-files secret.env`. Then force-push all branches and have all collaborators re-clone.
What is BFG Repo-Cleaner?	A faster, simpler alternative to filter-branch for common tasks: removing large files (`--strip-blobs-bigger-than 10M`), removing specific files, or replacing text in files across history. Only operates on commits, not the current HEAD.
What is `git replace`?	Creates a replacement mapping — Git pretends one object is another. `git replace <original> <replacement>` makes Git use the replacement commit whenever the original SHA is referenced. Used for grafting history without rewriting SHAs. Replace refs are in refs/replace/.
What is `git rerere` (Reuse Recorded Resolution)?	When enabled (`git config rerere.enabled true`), Git records how you resolve merge conflicts and automatically applies the same resolution if the same conflict occurs again. Invaluable when repeatedly rebasing or re-merging.
What is a custom merge driver?	A program Git invokes to merge files that require special handling. Defined in .gitattributes (e.g., `*.lock merge=ours`) and configured in .git/config with the driver command. Used for lock files, generated content, or binary formats.
What are the built-in merge strategies?	`ort` (default since Git 2.34, replaces recursive): handles renames well. `recursive`: previous default, recursive-merge. `resolve`: simple three-way. `ours`: keeps our version entirely. `octopus`: merges multiple branches at once (no conflicts allowed).
What is an octopus merge?	A merge with more than two parents — merging multiple branches simultaneously. Git uses the octopus strategy, which only works if there are no conflicts. Produces a single merge commit with N parents. Used when integrating multiple independent feature branches.

# ==========================================
# 15. Git at Scale
# ==========================================

What is sparse-checkout?	A feature that lets you only check out a subset of files in the repository. `git sparse-checkout set <dir1> <dir2>` limits the working tree to those directories while still having the full history. Essential for large monorepos where you only need specific projects.
What is cone mode in sparse-checkout?	An optimized sparse-checkout mode (default since Git 2.37) that only supports directory-based patterns (not arbitrary gitignore patterns). Cone mode is significantly faster for large repos because it can short-circuit directory traversal.
What is a partial clone?	A clone that defers downloading some objects from the server. `--filter=blob:none` (blobless clone): downloads commits/trees but fetches blobs on-demand. `--filter=tree:0` (treeless clone): downloads only commits initially. Dramatically reduces initial clone time for large repos.
What is the difference between blobless and treeless partial clones?	Blobless (`blob:none`): has all commits and trees locally, fetches file content on demand. Good for development: checkout and log are fast; file content triggers fetches. Treeless (`tree:0`): has only commits, fetches trees and blobs on demand. Good for CI and build systems that only need specific paths.
What is Scalar?	A tool (now part of Git) that configures large repository optimizations: background maintenance (prefetch, commit-graph, loose-objects, incremental-repack), filesystem monitor (FSMonitor), sparse-checkout, and partial clone. Originally developed by Microsoft for the Windows OS repository.
What is a commit-graph file?	A pre-computed binary file (.git/objects/info/commit-graph) storing commit DAG structure, generation numbers, and timestamps. Enables O(1) ancestor queries and dramatically speeds up `git log`, merge-base computation, and reachability checks on large repos.
What are reachability bitmaps?	Pre-computed bitsets stored alongside packfiles that mark which objects are reachable from specific commits. They make fetch/clone negotiations much faster by avoiding full graph traversal — the server can quickly determine which objects the client needs.
What is the multi-pack-index (MIDX)?	A single index file that spans multiple packfiles, enabling Git to locate objects across packs with a single binary search instead of checking each pack index. Speeds up object lookups in repos with many packfiles.
What is Git LFS (Large File Storage)?	An extension that replaces large files (binaries, media, datasets) with lightweight pointer files in Git, storing the actual content on a separate LFS server. `git lfs track "*.psd"` configures tracking. Keeps the Git repo small while supporting large binary assets.
What is the difference between Git LFS and git-annex?	Git LFS uses pointer files and a central LFS server (tight GitHub/GitLab integration). git-annex is more flexible: supports multiple storage backends (local, S3, rsync), offers fine-grained file availability control, and works without a specific server — better for distributed/scientific workflows.
What is FSMonitor and how does it help performance?	FSMonitor (filesystem monitor) uses OS-level file change notifications (Watchman, or built-in fsmonitor--daemon since Git 2.37) to track which files changed since the last Git operation. This eliminates the need to stat every file in the working tree, speeding up `git status`, `git add`, and `git diff` on repos with millions of files.

# ==========================================
# 16. Git Security
# ==========================================

How does commit signing with GPG work?	`git commit -S` signs the commit with your GPG private key. The signature is embedded in the commit object. Others verify with `git log --show-signature` using your public key. Proves that the commit was created by the holder of the private key.
How does commit signing with SSH keys work?	Since Git 2.34: `git config gpg.format ssh` and `git config user.signingkey <key-path>`. Git uses your SSH private key to sign commits/tags. Verification uses an allowed-signers file listing trusted public keys.
What is vigilant mode on GitHub?	A setting where GitHub marks all unsigned commits as "Unverified" (not just leaving them unmarked). This lets reviewers easily distinguish between verified (signed) and potentially suspicious (unsigned) commits.
What is a git bundle?	A single file containing Git objects and refs, created with `git bundle create`. Used for offline repository transfer — you can carry commits on a USB drive when network access is unavailable. The receiver uses `git bundle verify` and `git fetch <bundle>`.
What are the security considerations for Git transfer protocols?	SSH: strong authentication and encryption. HTTPS: certificate-based encryption, supports credential helpers. Git protocol (git://): no authentication or encryption — read-only and deprecated for anything sensitive. Smart HTTP is preferred for public repositories.
What are branch protection rules?	Server-side settings (GitHub, GitLab, etc.) that enforce policies on branches: require pull request reviews, require status checks (CI), require signed commits, block force-push, require linear history, restrict who can push.
Why is force-push dangerous and how do you prevent it?	`git push --force` overwrites remote history, potentially destroying other developers' work. Prevention: branch protection rules (block force-push on main), use `--force-with-lease` instead (only force-pushes if your local copy of the remote ref matches the actual remote — rejects if someone else pushed).
What is `git push --force-with-lease`?	A safer alternative to --force that only overwrites the remote branch if your local tracking ref matches the remote's current state. If someone else pushed new commits, the force-with-lease fails — preventing accidental overwrite of their work.
What is the SHA-1 collision concern in Git?	SHA-1 has a known collision attack (SHAttered, 2017). Theoretically, an attacker could create two different objects with the same SHA-1 hash. Git added collision detection hardening (sha1dc library) and is migrating to SHA-256 as an optional object format (experimental since Git 2.29).
How does secret scanning work with Git?	Tools (GitHub secret scanning, GitLeaks, TruffleHog, detect-secrets) scan repository history for patterns matching API keys, tokens, passwords, and certificates. Can run as pre-commit hooks (prevent commits) or in CI (detect existing secrets). GitHub also scans public repos automatically and revokes leaked tokens with partner programs.

# ==========================================
# 17. Git Server Administration
# ==========================================

What is a bare repository?	A repository without a working tree — it only contains the .git directory contents (objects, refs, HEAD). Created with `git init --bare`. Used as remote repositories on servers because no one directly edits files in them.
Why should server repositories be bare?	A non-bare repository's working tree would get out of sync when receiving pushes (pushing to a checked-out branch is rejected by default). Bare repos avoid this problem since there's no working tree to desynchronize.
What are receive-pack and upload-pack?	Server-side Git programs invoked during push and fetch. `git-receive-pack` handles incoming pushes (receives objects, updates refs). `git-upload-pack` handles fetches/clones (negotiates objects, sends packfiles). They communicate via the Git wire protocol.
How does repository mirroring work?	`git clone --mirror <url>` creates a bare clone that includes all refs (branches, tags, remote refs, notes). `git push --mirror <target>` pushes all refs to another remote. Used for backup, migration, or maintaining read-only mirrors.
What is `git maintenance run`?	A command (Git 2.29+) that runs optimization tasks: `gc` (garbage collection), `commit-graph` (update commit-graph file), `prefetch` (fetch from remotes in background), `loose-objects` (pack loose objects), `incremental-repack` (optimize packfiles). Can be scheduled with `git maintenance start`.
What is `git maintenance start`?	Registers the repository for automatic background maintenance using the OS scheduler (cron on Linux/macOS, Task Scheduler on Windows). Runs hourly lightweight tasks and daily/weekly heavier tasks. Configurable via maintenance.* settings.
What is an alternates file in Git?	The file .git/objects/info/alternates lists paths to other repositories' object directories. Git checks these locations when looking for objects, enabling shared object stores — multiple clones of the same project can share objects on disk, saving storage.
What is a shared object store?	A single object directory referenced by multiple repositories via alternates. When cloning with `--reference <local-repo>`, Git uses the local repo's objects instead of downloading duplicates. Useful for CI systems that maintain many checkouts of the same project.
How do server-side hooks differ from branch protection rules?	Server-side hooks (pre-receive, update) are shell scripts running on the Git server — flexible, custom logic, but require server access to install. Branch protection rules are a web UI feature of hosting platforms (GitHub, GitLab) — easier to configure but limited to the platform's supported policy options.
What is the Git wire protocol v2?	An updated protocol (Git 2.18+) for client-server communication that is more efficient: the client specifies exactly what it wants (ref advertisements are filtered), reducing network overhead. Enabled by default in modern Git. Key improvement: `ls-refs` command that returns only matching refs instead of all refs.

# 18. Broader Concepts Behind Git
# ==========================================

# --- Data Structures & Graph Theory ---

What is a Directed Acyclic Graph (DAG) and how does Git use it?	A graph where edges have direction and no cycles exist — you can never follow edges back to the same node. Git's commit history is a DAG: each commit points to its parent(s), and you can never create a circular chain of commits. This structure makes ancestor queries and topological sorting efficient.
Why does Git's commit graph being a DAG matter in practice?	It guarantees a well-defined history order: every commit has a clear set of ancestors. This enables efficient merge-base computation, reachability queries ("is commit A an ancestor of commit B?"), and topological sorting for `git log --topo-order`.
What is topological sorting and where does Git use it?	An ordering of DAG nodes such that every parent appears before its children. Git uses topological sorting in `git log` to display commits in dependency order, in garbage collection to find unreachable objects, and in commit-graph generation numbers for fast ancestor queries.
What is the Lowest Common Ancestor (LCA) problem and how does Git solve it?	Finding the most recent commit reachable from two branches — the merge base. `git merge-base A B` computes this. Git walks both branches' history backward until paths converge. The commit-graph file stores generation numbers that accelerate this by pruning large portions of the DAG without traversal.
What is graph reachability and why is it fundamental to Git?	Determining whether one commit is an ancestor of another by traversing the DAG. Git uses reachability for: garbage collection (unreachable objects are candidates for deletion), ref negotiation during fetch (what objects does the client already have?), and branch protection (ensuring required commits are present).

# --- Hashing & Integrity ---

What is a cryptographic hash function and what properties does Git rely on?	A function mapping arbitrary data to a fixed-size digest. Git relies on: (1) determinism — same input always produces the same hash, (2) collision resistance — practically impossible to find two inputs with the same hash, (3) preimage resistance — given a hash, you cannot reconstruct the input. These properties enable content-addressable storage and data integrity verification.
What is content-addressable storage (CAS)?	A storage system where data is retrieved by the hash of its content rather than by a name or location. Git stores every blob, tree, commit, and tag under a key computed from its content. Identical content always yields the same key, ensuring automatic deduplication.
How does Git use Merkle tree properties for integrity?	Each tree object's hash incorporates the hashes of all its children (blobs and subtrees). A commit's hash incorporates its tree hash. Therefore, changing any file deep in the hierarchy changes every hash up the chain to the commit. This makes history tamper-evident: altering any past content produces different commit hashes for all descendants.
What is the difference between SHA-1 and SHA-256 in Git?	SHA-1 (160-bit) is Git's original hash. SHA-256 (256-bit) is available experimentally since Git 2.29 as a more collision-resistant alternative. SHA-1 has a theoretical collision attack (SHAttered, 2017); Git added hardened SHA-1 (sha1dc) to detect known collision patterns while the SHA-256 transition progresses.
What is a hash collision and why does it matter for version control?	Two different inputs producing the same hash digest. In Git, a collision would mean two different objects sharing the same identifier — the repository would silently use one while discarding the other. This is why collision resistance of the hash function is a critical security property for SCM systems.

# --- Storage & Compression ---

How does Git's snapshot model differ from delta-based version control?	Delta-based VCS (e.g., SVN, CVS) stores each revision as a diff from the previous version. Git stores complete snapshots: every commit points to a full tree of blobs. This makes checkout and diff between arbitrary commits equally fast. Git applies delta compression only in packfiles for storage efficiency — a separate layer from the logical model.
What is delta encoding and how do Git packfiles use it?	Delta encoding stores an object as a set of instructions to reconstruct it from a base object (copy region X, insert bytes Y). Git's pack format finds similar objects (often successive versions of the same file) and stores the newer one as a delta against the older base. This dramatically reduces storage while keeping the full-snapshot logical model intact.
What is zlib/DEFLATE compression and where does Git use it?	DEFLATE is a lossless compression algorithm combining LZ77 and Huffman coding. Git compresses every loose object and packfile entry with zlib (a DEFLATE wrapper). This typically shrinks source code objects to 30–50% of their original size and is transparent to the user.
What is the difference between logical storage and physical storage in Git?	Logically, every commit stores a full snapshot of all files via blob and tree objects. Physically, Git optimizes storage via: (1) deduplication — identical blobs are stored once, (2) delta compression — similar blobs are stored as deltas in packfiles, (3) zlib compression — all objects are compressed. This gives the speed of snapshots with the space efficiency of deltas.

# --- Diff & Merge Algorithms ---

What is the Myers diff algorithm and why does Git use it by default?	Eugene Myers' O(ND) algorithm finds the shortest edit script (minimum insertions + deletions) between two sequences. Git uses it as the default diff algorithm because it produces minimal, intuitive diffs. It works by finding the longest common subsequence and treating everything else as additions or deletions.
What is the patience diff algorithm?	An alternative diff algorithm (--diff-algorithm=patience) that first matches unique lines common to both files, then recursively diffs between matched anchors. It produces more readable diffs for code because it avoids misaligning functions or matching closing braces across unrelated blocks.
What is the histogram diff algorithm?	A variant of patience diff (--diff-algorithm=histogram) that handles low-occurrence lines better by counting line frequencies. It tends to produce cleaner diffs than Myers for source code and is slightly faster than patience. Used by default in JGit (Java implementation of Git).
How does a three-way merge algorithm work conceptually?	Given three versions — base (common ancestor), ours (local), and theirs (remote) — the algorithm compares each side's diff against the base. Regions changed on only one side are accepted automatically. Regions changed on both sides identically are also accepted. Regions changed differently on both sides are conflicts requiring manual resolution.
What is the recursive merge strategy?	When two branches have multiple common ancestors (criss-cross merges), the recursive strategy first merges the common ancestors into a virtual base, then performs a three-way merge using that virtual base. This handles complex branch topologies better than a simple three-way merge.
What is the ORT merge strategy?	"Ostensibly Recursive's Twin" — Git's default merge strategy since Git 2.34, replacing recursive. It reimplements the same algorithm with better performance, improved rename detection, and cleaner code. Handles the same criss-cross and rename scenarios but faster.

# --- Distributed Systems Concepts ---

What is optimistic concurrency control and how does `git push` implement it?	A strategy that allows concurrent work without locking, validating at commit time. `git push` implements this: multiple developers work independently; push is rejected if the remote branch has advanced (fast-forward check). No central lock is ever held — conflicts are resolved locally and retried. `--force-with-lease` adds a CAS (compare-and-swap) check for safe force pushes.
How does Git implement eventual consistency in a distributed system?	Each clone is an independent repository with potentially different state. There is no real-time synchronization — changes propagate via explicit fetch/push operations. Given enough time and communication, all clones converge to the same state. Unlike databases, Git has no automatic conflict resolution — humans resolve merge conflicts during pull/merge.
What is the CAP theorem and how does Git relate to it?	CAP states a distributed system can guarantee at most two of: Consistency, Availability, Partition tolerance. Git prioritizes Availability and Partition tolerance: every clone is fully functional offline (available despite network partitions). Consistency is sacrificed — clones can diverge and require manual merge to synchronize.
What is the difference between a centralized locking model and Git's distributed model?	Centralized VCS (SVN, Perforce) uses pessimistic locking: a developer locks a file preventing others from editing it. Git uses optimistic concurrency: everyone edits freely, conflicts are detected and resolved at merge time. This trades simplicity of conflict avoidance for higher parallelism and offline capability.
What is a compare-and-swap (CAS) operation and where does Git use it?	An atomic operation that updates a value only if it matches an expected old value. `git push --force-with-lease` is a CAS: it updates the remote ref only if it still points to the expected commit. If someone else pushed in the meantime, the operation fails rather than overwriting their work.

# --- Immutability & Garbage Collection ---

What does immutability mean in Git's object model?	Once a Git object (blob, tree, commit, tag) is created, its content never changes — its identity is its content hash. Operations like `commit --amend` or `rebase` don't modify objects; they create new objects and update refs to point to them. Old objects remain in the database until garbage-collected.
Why is immutability important for correctness and collaboration?	Immutable objects guarantee that a commit hash uniquely identifies a specific state of the repository forever. If you have commit abc123, you know its tree, parent, and content are exactly what the author created. This enables trust, caching, deduplication, and reliable distributed synchronization.
How does Git's garbage collection relate to mark-and-sweep?	Like mark-and-sweep GC in programming languages: (1) Mark phase — Git walks all refs (branches, tags, reflog, stash) to find reachable objects. (2) Sweep phase — unreachable objects older than the retention period are deleted. `git gc` triggers this. The reflog acts as a temporary root set, keeping otherwise unreachable commits alive for ~30 days.
What is reference counting vs tracing GC, and which does Git use?	Reference counting tracks how many references point to each object — Git does NOT use this (objects don't store reference counts). Git uses tracing GC: starting from known roots (refs, reflog), it traces all reachable objects through parent/tree/blob pointers and collects unreachable objects. This correctly handles cycles (though Git's DAG has none by definition).

# --- Binary Search & Algorithmic Concepts ---

What is binary search and how does `git bisect` use it?	Binary search finds a target in a sorted collection by repeatedly halving the search space: O(log n). `git bisect` applies this to the commit DAG: given a known "good" and "bad" commit, it checks out the midpoint, you test it, and it halves the remaining range. A bug-introducing commit among 1024 commits is found in ~10 steps.
How does Git use binary search in packfile index lookups?	Packfile .idx files contain a sorted list of object SHAs. To find an object, Git performs a fanout-accelerated binary search: first uses a 256-entry fanout table (indexed by the first byte of the SHA) to narrow the range, then binary-searches within that range. This makes object lookup O(log n) even in packfiles containing millions of objects.
What is a generation number and how does it accelerate Git operations?	A topological level number assigned to each commit: a root commit has generation 1, and every other commit's generation is max(parent generations) + 1. Stored in the commit-graph file, generation numbers let Git answer "is A an ancestor of B?" without full graph traversal — if A's generation ≥ B's generation, A cannot be an ancestor of B. This prunes huge portions of the graph.
What is amortized analysis and how does it apply to Git pack operations?	A technique that averages cost over a series of operations. Individual Git commits are O(1) (writing loose objects). Periodically, `git gc` repacks all loose objects into deltified packfiles — an expensive operation. Amortized over many commits, the average cost per commit remains low. Git's auto-gc triggers packing when the number of loose objects exceeds a threshold (~6700), balancing write speed and storage efficiency.

# --- Software Engineering Principles ---

What is idempotency and which Git operations are idempotent?	An operation is idempotent if running it multiple times produces the same result as running it once. Idempotent Git operations: `git fetch` (fetching the same refs again changes nothing), `git checkout <branch>` (switching to the branch you're on is a no-op), `git gc` (running gc twice is safe). Non-idempotent: `git commit` (each run creates a new commit), `git merge` (merging the same branch twice creates a second merge commit).
What is the principle of least surprise and how does Git apply it?	Design principle: a system should behave the way users expect. Git examples: `git status` never modifies state (read-only query), `git fetch` downloads but never alters your working tree, `git stash pop` restores exactly what was stashed. Counterexample: `git checkout` was surprising because it combined branch-switching and file-restoration — Git 2.23 split it into `git switch` and `git restore`.
What is referential transparency in Git's context?	A concept from functional programming where an expression can be replaced by its value. In Git: a commit hash is referentially transparent — wherever you use hash abc123, you get exactly the same tree, parents, and content. This is why commit hashes are used in CI configs, Dockerfiles, and dependency pinning. Branch names are NOT referentially transparent (they change over time).
What is the single-writer principle and how does Git enforce it at the ref level?	Only one process should write to a shared resource at a time. Git enforces this for refs using lock files: updating a ref (e.g., refs/heads/main) creates a .lock file; the update is atomic via rename. On the server side, receive-pack serializes concurrent pushes to the same ref, and --force-with-lease adds CAS semantics for client-side safety.
