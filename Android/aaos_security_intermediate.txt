# 5.2 AAOS Platform Security

# --- VHAL/CarService Security Model ---

How does the CarService security model enforce access control?	CarService checks the calling app's UID, package name, and granted permissions for every CarManager API call. Each car property and service method has an associated permission annotation. Calls without the required permission throw SecurityException. Additionally, SELinux policies restrict which processes can bind to CarService's Binder interfaces.
What is car property authorization and how does it work?	Each VehicleProperty in the AIDL definition is annotated with a required permission (e.g., @RequiresPermission(Car.PERMISSION_SPEED)). When an app calls CarPropertyManager.getProperty() or setProperty(), CarService verifies the caller holds that permission. The permission-to-property mapping is defined in CarPropertyManager and enforced at the framework level.
What is the security difference between CarPropertyManager read vs write access?	Many properties have split access — reading speed requires CAR_SPEED permission (still restricted but informational), while writing a control property like HVAC_TEMPERATURE_SET requires CAR_HVAC permission. Write permissions are more tightly controlled because they command physical vehicle changes. Some properties use separate read and write permissions.
How does CarService validate that a caller is a system or privileged app?	CarService uses the PackageManager to check if the caller's package is installed in /system/priv-app or /product/priv-app, or if the app is signed with the platform signing key. For certain operations, CarService also checks android:sharedUserId="android.uid.system" or specific signature permissions.
What is the intent behind separating car permissions into fine-grained categories?	Fine-grained permissions (CAR_SPEED vs CAR_ENERGY vs CAR_HVAC vs CAR_EXTERIOR_LIGHTS) follow the principle of least privilege — a media app requesting audio focus doesn't need access to vehicle speed. This limits the damage if any single app is compromised.
How are vendor-specific VHAL properties protected by CarService?	Vendor properties in the 0x0100–0x0FFF range require the CAR_VENDOR_EXTENSION permission, which is typically restricted to signature-level (platform-signed) apps. OEMs can further restrict access with custom SELinux policies and additional OEM-defined permissions annotated on each vendor property.

# --- AAOS-Specific SELinux Policies ---

What SELinux domains are unique to AAOS?	vehicle_hal (VHAL daemon process), carservice_app (CarService process), car_watchdog (CarWatchdog daemon), oem_car_service (OEM CarService extensions), evs_app/evs_driver (Exterior View System), and various vendor-specific HAL domains. Each has tailored policies restricting their resource access.
How do SELinux policies protect the VHAL on AAOS?	Only the vehicle_hal domain can access the VHAL's hardware device nodes (e.g., CAN bus interfaces, vehicle bus sockets). The carservice_app domain can communicate with vehicle_hal via Binder/AIDL, but untrusted_app and platform_app domains cannot directly reach vehicle_hal — they must go through CarService.
What is the role of service_contexts in AAOS SELinux?	service_contexts maps registered Binder service names to SELinux types. For AAOS, it labels CarService's Binder interfaces (car_service, car_property_service, etc.) so that SELinux policies can control which domains can discover and call each car service.
How should OEMs write SELinux policies for custom car services?	OEMs should: (1) Define a new domain for the custom service process. (2) Apply least-privilege — only allow access to the specific files, sockets, and Binder services the service needs. (3) Add neverallow rules to prevent untrusted apps from accessing the service. (4) Use audit2allow during development to identify needed permissions, then tighten policies before production.
What are neverallow rules and why are they critical for AAOS?	neverallow rules declare access that must never be granted, even by future allow rules. CTS/VTS tests verify neverallow compliance. In AAOS, neverallow rules prevent untrusted_app from accessing vehicle_hal sockets, car-specific device nodes, and safety-critical system properties.
How do SELinux policies handle multi-user isolation on AAOS?	SELinux labels are user-aware — each Android user's app processes run in per-user SELinux categories. Combined with Linux UID isolation, this prevents user A's apps from accessing user B's files or IPC endpoints, even if both users run the same app.

# --- UX Restriction Security ---

How are driver distraction UX restrictions enforced at the system level?	CarUxRestrictionsService reads the driving state from CarDrivingStateService (which reads VHAL properties like speed and gear). When the state changes to MOVING, the service broadcasts UX restrictions. Car App Library templates enforce restrictions in their rendering code. For native apps, ActivityBlockingActivity can overlay non-compliant activities.
What is ActivityBlockingActivity?	A system activity that overlays the screen when a non-distraction-optimized app is in the foreground while the vehicle is moving. It blocks the user from interacting with the unsafe app and prompts them to switch to a compliant app or wait until parked.
How could an attacker attempt to bypass UX restrictions?	(1) Spoofing the VHAL driving state to PARKED while actually moving. (2) Modifying CarUxRestrictionsService configuration. (3) Disabling ActivityBlockingActivity. (4) Using system overlays to cover the blocking activity. All require system/root access and are prevented by SELinux, verified boot, and signed system images on production devices.
Why is spoofing the VHAL driving state a security concern?	If an attacker could write to VHAL's PERF_VEHICLE_SPEED or CURRENT_GEAR properties (setting speed=0 or gear=PARK), the system would think the vehicle is parked and remove all UX restrictions, allowing video, keyboards, and complex interactions while actually driving — creating a dangerous distraction.
How does AAOS prevent unauthorized VHAL state manipulation?	Only the vehicle_hal domain can write to actual VHAL properties (these come from real vehicle ECUs). The inject-vhal debug command is only available on userdebug/eng builds and is removed from production builds. SELinux prevents apps from communicating with the VHAL daemon directly.

# --- Secure Diagnostics and Maintenance ---

What are UDS (Unified Diagnostic Services) security access levels?	UDS defines security access (service 0x27) with seed-key exchange — the diagnostic tool requests a seed from the ECU, computes a key using a secret algorithm, and sends it back. Levels include: engineering access, manufacturing access, and dealer access, each with different capabilities and key algorithms.
What is an authenticated UDS session?	A diagnostic session where the tester must pass a challenge-response authentication (seed-key in service 0x27, or certificate-based in ISO 14229-1:2020 service 0x29) before performing privileged operations like firmware flashing, DTC clearing, or ECU configuration changes.
How should OBD-II diagnostic port access be protected on AAOS vehicles?	(1) Restrict OBD-II to standardized emissions diagnostics per regulations. (2) Require authentication for extended diagnostics (UDS security access). (3) Implement gateway ECU filtering to prevent OBD-II access from reaching safety-critical ECUs. (4) Log diagnostic access attempts. (5) Consider physical port protection for fleet vehicles.
What is the risk of unrestricted diagnostic access to AAOS?	Diagnostic interfaces can read sensitive vehicle data, clear fault codes masking safety issues, reprogram ECU firmware, and send actuator commands. Unrestricted access through OBD-II or a compromised IVI could allow vehicle theft, safety system manipulation, or odometer fraud.
How should debug-only access be controlled on production AAOS vehicles?	(1) ADB disabled in user builds. (2) Developer options hidden. (3) Engineering diagnostic modes gated behind authenticated UDS sessions. (4) Debug properties (ro.debuggable=0) set. (5) SELinux running in enforcing mode. (6) Test certificates removed from production builds.

# --- OTA Security for AAOS ---

What cryptographic verification does AAOS OTA perform?	The update_engine verifies: (1) RSA/EC signature over the payload metadata (hash manifest). (2) SHA-256 hash of each partition block matches the manifest. (3) Rollback index is >= current device rollback index. (4) New vbmeta is properly signed. All four checks must pass before any blocks are written.
What is a rollback prevention mechanism in OTA?	The bootloader maintains a monotonically increasing rollback index stored in tamper-resistant memory (RPMB or hardware fuses). Each OTA increment includes a minimum rollback index. If an attacker replays an older OTA with known vulnerabilities, the rollback index check fails and the update is rejected.
What is dual-bank integrity verification?	After writing an OTA to the inactive partition slot (Bank B), the bootloader verifies the entire image before switching. If verification fails, the bootloader remains on Bank A. After switching to Bank B and successfully booting, Bank A can be considered for future updates. This ensures a known-good fallback always exists.
How can a compromised OTA delivery channel be mitigated?	(1) Payload signing with offline OEM keys (private key never on internet-connected servers). (2) TLS mutual authentication between device and OTA server. (3) Certificate pinning in the OTA client. (4) Payload hash verification independent of transport encryption. (5) Code signing certificate rotation support in case of key compromise.
What is the delta update security risk?	Delta OTA patches are applied to the existing partition to produce the new image. If an attacker can modify the old partition before delta application, the result could be attacker-controlled. AVB/dm-verity prevents this by verifying the source partition before delta application.

# --- Secure Communication on Vehicle Networks ---

How is TLS used on the in-vehicle Ethernet network?	TLS secures communication between the IVI and connected ECUs over automotive Ethernet — encrypting data in transit (preventing eavesdropping), authenticating endpoints (preventing spoofing), and ensuring integrity (preventing tampering). Used for telematics, OTA downloads, and inter-ECU service calls over SOME/IP.
What is IPsec in the context of automotive networking?	IPsec provides network-layer encryption and authentication for IP traffic between vehicle ECUs. It can protect all traffic between two endpoints transparently, without application-layer changes. Used when blanket protection of an Ethernet segment is needed.
What is MACSec (IEEE 802.1AE) and how is it used in vehicles?	MACSec provides hop-by-hop Layer 2 encryption and integrity for Ethernet frames. Each switch port encrypts/decrypts frames, preventing physical wire-tapping and frame injection attacks. Unlike TLS (application layer) or IPsec (network layer), MACSec protects at the data-link layer.
What is a secure HMI channel?	A protected communication path between the IVI display system and critical vehicle information (e.g., ADAS warnings, safety alerts). It ensures that malware on the IVI cannot suppress, modify, or spoof safety-critical visual or audio notifications to the driver.
How do you choose between TLS, IPsec, and MACSec for vehicle networks?	MACSec: protects individual Ethernet links (Layer 2), lightweight, hardware-accelerated. IPsec: protects IP communication end-to-end (Layer 3), works across switches/routers. TLS: protects specific application connections (Layer 4+), flexible, per-service. In practice, vehicles often layer these — MACSec on critical links, TLS for application services, IPsec for management traffic.

# --- Firewalling and Gateway ECU ---

What is a gateway ECU and why is it a critical security component?	A central ECU that routes and filters messages between different vehicle network domains (powertrain CAN, chassis CAN, body CAN, infotainment Ethernet). It acts as a firewall — preventing messages from the infotainment domain from reaching safety-critical ECUs (brakes, steering, engine).
How does a gateway ECU filter CAN messages?	The gateway maintains an allowlist of CAN message IDs that can pass between domains. Messages from the IVI domain are checked against this allowlist — only authorized passes (e.g., navigation audio control) are forwarded to the body domain. All other messages, especially those targeting powertrain or chassis domains, are dropped.
What filtering rules should a gateway apply between IVI and safety ECUs?	(1) Block all CAN messages from IVI to powertrain/chassis by default (deny-all policy). (2) Allow only specific, well-defined messages needed for legitimate functions (e.g., displaying speed on IVI). (3) Rate-limit allowed messages to prevent flooding. (4) Log and alert on blocked attempts. (5) Implement stateful filtering where possible.
What is network segmentation within an AAOS vehicle?	Dividing the vehicle's network into isolated domains — infotainment (AAOS IVI), body control, powertrain, ADAS, diagnostics — connected through gateway ECUs with strict filtering. This limits the blast radius of a compromise: a hacked IVI cannot directly reach braking ECUs.
How can the gateway ECU itself be secured?	(1) Secure boot to prevent firmware tampering. (2) Hardware-based key storage (HSM) for authentication. (3) Minimal attack surface — no unnecessary interfaces. (4) Verified firmware updates with rollback protection. (5) Watchdog monitoring for availability. (6) Rate limiting and anomaly detection on all ports.

# --- Secure Boot Chain Continuity ---

What does secure boot chain continuity mean in AAOS?	The trust chain extends from hardware (SoC ROM) through bootloader, kernel, init, system partition, vendor partition, all the way to CarService and VHAL processes. Each stage verifies the next. Breaking any link compromises all subsequent stages.
How does dm-verity protect the system partition at runtime?	dm-verity creates a hash tree over the entire read-only system partition. On every block read, the kernel verifies the block's hash against the tree. If any block has been modified (rootkit, malware injection), the read fails and the system detects tampering.
What is fs-verity and how does it complement dm-verity on AAOS?	fs-verity protects individual files (APKs, config files) on read-write partitions. While dm-verity protects entire read-only partitions, fs-verity can verify the integrity of specific installed app files even on the data partition, detecting if an APK has been modified after installation.
How does the boot chain protect CarService specifically?	CarService runs from the system partition (verified by dm-verity). Its APK is signed with the platform key. At runtime, PackageManager verifies the APK signature. SELinux confines CarService to the carservice_app domain. Together, these ensure only the OEM's authentic CarService code executes with car service privileges.

# --- Android Keystore in Automotive ---

How does Android Keystore work on automotive SoCs?	Automotive SoCs include a TEE (Trusted Execution Environment, e.g., ARM TrustZone) or StrongBox (dedicated secure element) that stores cryptographic keys in hardware-isolated memory. Apps generate and use keys via the KeyStore API, but keys never leave the secure hardware — all crypto operations happen inside the TEE/StrongBox.
What is TEE vs StrongBox on AAOS?	TEE (e.g., TrustZone) runs in a secure world on the main CPU — fast but shares silicon. StrongBox is a separate, dedicated security chip with its own CPU, storage, and tamper resistance — slower but provides stronger physical security. StrongBox is preferred for high-value keys (vehicle access, payment).
What is key attestation and why is it useful for automotive backends?	Key attestation provides a certificate chain proving that a key was generated inside a genuine Android device's secure hardware (TEE/StrongBox), on a specific device, with specific security properties. Vehicle backends can verify that a remote command came from the legitimate vehicle IVI, not a cloned or emulated device.
How is the Keystore used for vehicle-specific operations?	(1) TLS client certificates for backend authentication. (2) Signing vehicle telemetry data to prove authenticity. (3) Digital key credential storage. (4) Encrypting user data at rest. (5) Authenticating OTA update requests. (6) Securing inter-service communication credentials.
What key properties can be enforced by hardware-backed Keystore on AAOS?	(1) Key cannot be exported (hardware-bound). (2) Key requires user authentication before use (biometric/PIN-gated). (3) Key is bound to specific device boot state (invalidated if bootloader unlocked). (4) Key has usage expiry. (5) Key is limited to specific crypto operations (sign-only, encrypt-only).

# --- Secure Multi-User and Guest Isolation ---

How does per-user encryption work on AAOS?	Each Android user has a unique encryption key derived from their credentials (PIN/password) combined with a hardware-bound key from the Keystore. User data is stored in per-user directories encrypted with these keys. When a user is not active, their CE storage is locked and their data is cryptographically inaccessible.
What is the difference between credential-encrypted and device-encrypted storage in a multi-user automotive context?	DE storage (alarm, phone connectivity) is accessible after boot without user authentication — needed for incoming calls, vehicle monitoring. CE storage (personal data, navigation history, messages) requires the specific user to authenticate first. Each user has separate DE and CE storage areas.
How should profile wiping work when a vehicle is resold?	(1) Factory reset triggered from CarSettings or infotainment. (2) All CE encryption keys destroyed (data becomes unrecoverable). (3) DE data wiped. (4) Paired Bluetooth devices cleared. (5) Wi-Fi credentials removed. (6) Google/OEM accounts de-registered. (7) Vehicle should prompt setup wizard on next boot.
What security risks exist if multi-user isolation is poorly implemented?	(1) One driver reading another's contacts/messages. (2) Navigation history leaking between profiles. (3) Saved Wi-Fi passwords shared unintentionally. (4) Previous owner's accounts remaining active after resale. (5) Guest user accessing admin settings. (6) Background processes from inactive users continuing to access shared resources.
How does AAOS handle user data when a user is running but not in the foreground?	Background users' processes are stopped or restricted (depending on OEM configuration). Their CE storage remains unlocked only if the system needs ongoing access (e.g., background music). When the system determines background user data is no longer needed, it can lock their CE storage even while the vehicle is powered on.

# --- Network Segmentation ---

What is network segmentation at the SoC level on AAOS?	The SoC itself may have multiple network interfaces — one connected to the infotainment Ethernet domain and another to the vehicle control domain. On-SoC firewalls, IOMMU, or ARM TrustZone partitioning can isolate these interfaces so that a compromised Android instance cannot directly access the vehicle control network.
How do VLANs help segment the in-vehicle network?	VLANs (Virtual LANs) on automotive Ethernet switches create logical network segments — IVI traffic on VLAN 10, ADAS on VLAN 20, body control on VLAN 30. Traffic between VLANs must pass through the gateway ECU's firewall rules, preventing direct cross-domain communication.
What is the role of an IOMMU in AAOS network segmentation?	An IOMMU (Input/Output Memory Management Unit) restricts which memory regions a hardware peripheral (e.g., Ethernet controller, CAN controller) can access. Even if Android is compromised, the IOMMU prevents the IVI's Ethernet controller from DMA-ing into memory regions used by safety-critical software or hardware interfaces.
How does ARM TrustZone contribute to network isolation?	TrustZone separates the SoC into Secure World and Normal World. Safety-critical network handling (CAN bus management, gateway firewall) can run in the Secure World while Android runs in the Normal World. Even a fully compromised Android cannot access Secure World resources.

# --- TCU Security ---

What is a Telematics Control Unit (TCU)?	A vehicle module that provides cellular connectivity (LTE/5G), enabling remote commands (lock/unlock, pre-conditioning), OTA updates, emergency calls (eCall), data telemetry, and cloud connectivity. It bridges the vehicle to the OEM's backend servers.
How should remote commands from the backend be authenticated?	(1) TLS mutual authentication between TCU and backend (both sides present certificates). (2) Command messages signed with backend private key, verified by TCU. (3) Reply-attack prevention using nonces or timestamps. (4) Per-command authorization tokens with scope and expiry. (5) User authentication at the app level (PIN, biometric) before sending sensitive commands.
What is command allow-listing on a TCU?	The TCU maintains a strict list of allowed remote commands (e.g., lock, unlock, climate pre-condition, honk) and rejects any command not on the list. This prevents an attacker who compromises the backend from sending arbitrary vehicle commands like steering or braking through the TCU.
What security risks does a compromised TCU pose?	A compromised TCU could: (1) Execute unauthorized remote commands (unlock doors, disable alarm). (2) Intercept and exfiltrate private data (location, driving patterns). (3) Act as an entry point to the vehicle's internal network. (4) Disable OTA updates. (5) Send false eCall/emergency data. (6) Track the vehicle's location.
How should TLS mutual authentication work between vehicle and backend?	The TCU stores a device-specific TLS client certificate (provisioned at manufacturing, backed by HSM). The backend presents its server certificate. Both sides verify the other's certificate chain against trusted CAs. This ensures the backend knows it's talking to a genuine vehicle and the vehicle knows it's talking to the real backend.

# --- Secure Logging and Audit Trails ---

What is tamper-evident logging on AAOS?	A logging mechanism where each log entry is cryptographically chained (sequential hashing or MAC) so that deletion or modification of any entry is detectable. Used for security-sensitive events like diagnostic access, VHAL writes, user authentication, and OTA updates.
What events should be logged for security auditing in AAOS?	(1) User authentication success/failure. (2) VHAL property write operations (especially control properties). (3) Diagnostic port access and UDS sessions. (4) OTA update attempts (success/failure). (5) App installation/removal. (6) Permission grant/revoke. (7) USB device connections. (8) SELinux denials. (9) ADB connections. (10) Remote command execution via TCU.
What is forensic readiness for AAOS vehicles?	The ability to collect, preserve, and analyze digital evidence from the IVI after a security incident. Requires: tamper-evident logs, sufficient log storage (survive resets), timestamps synchronized to reliable sources, log backup to cloud before local storage fills, and documented log formats for analysis tools.
Why are VHAL write audit logs important?	VHAL writes command physical vehicle actions (HVAC, lights, door locks). Audit logs create an unalterable record of what commands were issued, by which process, at what time. This enables post-incident analysis (e.g., was the door unlock initiated by the owner's app or by malware?).
How should audit logs be protected on AAOS?	(1) Store logs in a dedicated partition or encrypted storage. (2) Only the logging daemon can write to log files (SELinux enforced). (3) Cryptographic chaining for tamper detection. (4) Periodic log upload to cloud backend over authenticated channel. (5) Log rotation with minimum retention period. (6) Prevent factory reset from erasing security logs (if regulation requires).

# --- Certificate Lifecycle Management ---

What certificates does an AAOS vehicle need to manage?	(1) Device identity certificate (for backend TLS mutual auth). (2) OTA signing certificate (for verifying update payloads). (3) Code-signing certificates (for APK and firmware verification). (4) V2X certificates (for vehicle-to-everything communication). (5) Wi-Fi/802.1X certificates (for enterprise network auth). (6) Digital key certificates (CCC Digital Key). (7) EV charging certificates (ISO 15118).
What is certificate provisioning for AAOS vehicles?	The initial installation of device-specific certificates during manufacturing. The vehicle's SoC generates a key pair in its HSM/TEE, creates a CSR (Certificate Signing Request), and an enrollment CA issues a device certificate. This certificate is stored in secure hardware and used for TLS, attestation, and identity throughout the vehicle's lifecycle.
How does certificate rotation work on a vehicle?	Before a certificate expires, the vehicle contacts a certificate management backend, authenticates using its current valid certificate, generates a new key pair in HSM, submits a CSR, receives a new certificate, and atomically transitions to the new credential. The old certificate is retained briefly for rollback if the new one fails.
What is certificate revocation and why is it important for AAOS?	If a vehicle's private key is compromised (stolen HSM, cloned device), the corresponding certificate must be revoked to prevent the attacker from impersonating the vehicle. The backend checks revocation via CRL (Certificate Revocation List) or OCSP (Online Certificate Status Protocol) before accepting connections.
What challenges does certificate management face in automotive?	(1) Vehicles have 10–15 year lifecycles, far exceeding typical certificate validity. (2) Connectivity may be intermittent (no internet in garages). (3) Multiple certificate types with different lifecycles. (4) Manufacturing-time provisioning across global supply chains. (5) Certificate agency root key rotation over vehicle lifetime.

# --- Supply Chain Security ---

What is an SBOM (Software Bill of Materials) for automotive?	A comprehensive inventory of all software components in the vehicle — AOSP version, kernel version, HAL implementations, third-party libraries (OpenSSL, libcurl), OEM apps, supplier-provided binaries — with exact versions, licenses, and known vulnerabilities. Required by UNECE R155 for cybersecurity management.
Why is supply chain security critical for AAOS vehicles?	AAOS IVIs incorporate software from multiple suppliers (SoC vendor BSP, HAL implementations, codec libraries, navigation SDKs). A vulnerability or backdoor in any supplier component can compromise the entire vehicle. Supply chain attacks (like SolarWinds) targeting automotive suppliers could affect millions of vehicles.
How should OEMs verify vendor-supplied components?	(1) Require SBOMs from all suppliers. (2) Conduct vulnerability scanning against CVE databases. (3) Perform binary analysis of closed-source components. (4) Verify code signing matches known supplier keys. (5) Conduct penetration testing of supplier interfaces. (6) Contractual security requirements and audit rights. (7) Regular update and patching SLAs.
What is third-party library auditing for AAOS?	Regular review of open-source and commercial libraries used in the IVI system: checking for known CVEs (e.g., OpenSSL heartbleed equivalents), verifying library versions are current, analyzing dynamic library dependencies, scanning for deprecated/abandoned libraries, and ensuring license compliance.
How can AAOS OEMs protect against firmware supply chain attacks?	(1) Verify BSP integrity from SoC vendor using signed hashes. (2) Build from source when possible (AOSP, kernel). (3) Reproduce builds to verify binary equivalence. (4) Sign all firmware components with OEM keys. (5) Monitor for unauthorized modifications in CI/CD pipeline. (6) Implement hardware-rooted secure boot to reject tampered firmware.
