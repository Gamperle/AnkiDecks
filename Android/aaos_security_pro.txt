# --- Hardware Root of Trust and HSM ---

What is a hardware root of trust in a vehicle IVI?	An immutable hardware component (SoC boot ROM, dedicated security chip, or HSM) that anchors the entire trust chain. It stores root keys, performs first-stage verification, and cannot be modified by software. All subsequent security (secure boot, key storage, attestation) ultimately depends on this root being uncompromised.
How does an HSM (Hardware Security Module) integrate with AAOS?	The HSM is a tamper-resistant chip on the IVI board that provides: (1) Secure key storage (keys never leave the HSM). (2) Cryptographic acceleration (AES, RSA, ECC, SHA). (3) Random number generation (TRNG). (4) Secure boot support (verifying firmware signatures). (5) Key derivation and wrapping. The AAOS KeyMint/Keymaster HAL communicates with the HSM via a secure channel.
What is the difference between a discrete HSM and an integrated secure element?	A discrete HSM is a separate chip on the board (e.g., Infineon SLx, NXP SE050) with its own CPU, memory, and tamper-detection circuitry — high security but higher cost. An integrated secure element (e.g., ARM TrustZone, on-die secure enclave) shares silicon with the main SoC — lower cost but potentially less tamper-resistant. StrongBox on Android maps to discrete HSMs.
How does key attestation chain back to the hardware root of trust?	The HSM stores a device-unique attestation key provisioned at manufacturing. When the Keystore generates a new key, the HSM signs an attestation certificate containing the new key's properties (hardware-backed, key purpose, boot state). This attestation chain links to Google's or the OEM's root CA, allowing backends to verify the key was genuinely generated in secure hardware on a genuine device.
What is RPMB (Replay Protected Memory Block) and how is it used on AAOS?	RPMB is a partition on eMMC/UFS storage that provides authenticated and replay-protected read/write access using a shared key with the TEE/HSM. AAOS uses RPMB to store rollback counters (preventing OTA downgrade), secure flags (lock state), and critical state data that must survive reboots and resist tampering.
What are anti-tamper mechanisms in automotive HSMs?	(1) Active mesh — wire grid detecting physical probing. (2) Voltage and clock glitch detection — sensing fault injection attacks. (3) Temperature sensors — detecting extreme conditions (freezing, heating attacks). (4) Light sensors — detecting decapping. (5) Memory zeroization — erasing keys when tamper is detected. (6) Environmental envelope monitoring — ensuring operation within specified conditions.

# --- ISO/SAE 21434 ---

What is ISO/SAE 21434 and why does it matter for AAOS?	An international standard for automotive cybersecurity engineering that defines the processes OEMs and suppliers must follow throughout the vehicle lifecycle — from concept and development through production, operations, and decommissioning. It's the cybersecurity complement to ISO 26262 (functional safety) and is referenced by UNECE WP.29 R155 regulation.
What are TARA (Threat Analysis and Risk Assessment) methods under ISO 21434?	TARA is a systematic process to: (1) Identify assets and their security properties (confidentiality, integrity, availability). (2) Enumerate threat scenarios (STRIDE model). (3) Assess attack feasibility (attack potential: elapsed time, expertise, knowledge, equipment, window of opportunity). (4) Determine impact (safety, financial, operational, privacy). (5) Calculate risk level. (6) Define cybersecurity goals and requirements to mitigate unacceptable risks.
How does ISO 21434 define cybersecurity goals?	Cybersecurity goals are high-level security objectives derived from TARA — e.g., "Prevent unauthorized remote access to powertrain ECUs" or "Ensure integrity of OTA update payloads." Each goal has an associated risk level and maps to specific cybersecurity requirements that are verified during development and validated during testing.
What is a Cybersecurity Case under ISO 21434?	A structured argument (with supporting evidence) that demonstrates a vehicle or component achieves an acceptable level of cybersecurity. It includes: TARA results, cybersecurity goals, requirements traceability, verification/validation results, vulnerability management records, and production security measures. Analogous to a Safety Case in ISO 26262.
How does ISO 21434 apply to supplier relationships?	The standard defines cybersecurity interface agreements between OEMs and suppliers — specifying shared responsibilities, required security activities, deliverables (TARA, security requirements, test results), vulnerability disclosure processes, and ongoing monitoring obligations. The OEM remains ultimately responsible for the vehicle's cybersecurity.
What is post-production cybersecurity monitoring under ISO 21434?	Continuous monitoring of deployed vehicles for new vulnerabilities and threats: tracking CVE databases for components used in the vehicle, monitoring field incidents and security reports, performing periodic penetration testing, maintaining incident response capabilities, and issuing patches via OTA. This extends for the vehicle's entire supported lifetime (10–15+ years).

# --- UNECE WP.29 R155/R156 ---

What is UNECE WP.29 R155?	A UN regulation requiring vehicle manufacturers to implement a Cybersecurity Management System (CSMS) that demonstrates they can manage cyber risks throughout the vehicle lifecycle. Mandatory for new vehicle type approvals in the EU, Japan, Korea, and other participating countries since July 2024 (new types) / July 2026 (all new vehicles).
What is a CSMS (Cybersecurity Management System) under R155?	An organizational framework that an OEM must establish covering: governance and responsibilities, risk assessment processes, security-by-design in development, supply chain cybersecurity management, incident detection and response, OTA/update management, and continuous improvement based on threat intelligence.
What is UNECE WP.29 R156 and how does it relate to R155?	R156 regulates Software Update Management Systems (SUMS) — requiring OEMs to manage software updates (OTA and dealer) safely and securely: ensuring updates don't compromise safety, verifying update integrity, maintaining traceability of software versions, handling rollback, and notifying authorities. R156 complements R155's cybersecurity requirements.
What are the consequences of failing R155/R156 compliance?	The OEM cannot obtain vehicle type approval in participating countries — meaning they cannot sell those vehicle types in the EU, Japan, Korea, etc. This effectively blocks market access for non-compliant vehicles. Existing approvals can also be withdrawn if post-production monitoring reveals cybersecurity deficiencies.
How does R155 affect the AAOS IVI specifically?	The IVI is a key attack surface enumerated in R155's annex. OEMs must demonstrate: secure boot and code integrity, secure communication channels, access control for vehicle data, OTA update security, resistance to common attack vectors (CAN injection, malware, privilege escalation), and ongoing vulnerability management — all of which directly involve the AAOS platform.
What evidence must an OEM present for CSMS certification?	(1) Documented cybersecurity governance structure. (2) TARA for all vehicle systems including IVI. (3) Cybersecurity requirements and their implementation evidence. (4) Testing and validation results (penetration tests, vulnerability scans). (5) Incident response procedures. (6) Supply chain security agreements. (7) Post-production monitoring plan. (8) Software update management processes (R156 compliance).

# --- CAN Bus Security ---

How does CAN bus authentication work with SecOC?	SecOC (Secure Onboard Communication) from AUTOSAR appends a MAC (Message Authentication Code, typically CMAC-AES-128) and a freshness value (truncated counter or timestamp) to CAN messages. The receiving ECU verifies the MAC using a shared key and checks the freshness value to prevent replay attacks. This authenticates the sender without encrypting the payload.
What is a freshness value in SecOC and why is it necessary?	A counter or timestamp included in each authenticated CAN message that changes with each transmission. Without freshness values, an attacker could capture a valid authenticated message and replay it later — the MAC would still be valid because the content hasn't changed. The freshness value makes replayed messages detectable because the receiver expects monotonically increasing values.
How do CAN bus intrusion detection systems (IDS) work?	CAN IDS monitors bus traffic for anomalies: (1) Timing-based: detecting messages at unexpected intervals. (2) Frequency-based: detecting unusual message rates. (3) Statistical: detecting abnormal data patterns in payload bytes. (4) Specification-based: comparing messages against known-good behavior specs. (5) Fingerprinting: identifying ECU transmitters by electrical characteristics. Alerts are sent to a vehicle SIEM or SOC.
What is a CAN bus spoofing attack and how does SecOC defend against it?	Spoofing is injecting fabricated CAN messages with valid arbitration IDs to impersonate a legitimate ECU. Without SecOC, the receiving ECU cannot distinguish genuine from fake messages (CAN has no built-in authentication). SecOC defends by requiring a valid MAC computed with a secret key — the attacker cannot forge valid MACs without possessing the shared key.
What is a CAN bus-off attack?	An attacker exploits CAN's error handling: by intentionally causing bit errors on the bus (using a dominant-bit-sending device), they force a target ECU's transmit error counter to exceed 255, pushing it into BUS-OFF state where it disconnects from the bus entirely. This is a targeted denial-of-service against specific ECUs (e.g., silencing the ABS controller).
How can bus-off attacks be mitigated?	(1) Hardware-level: CAN transceivers with bus-off recovery features. (2) Protocol-level: fast bus-off recovery timers. (3) Network-level: physical segmentation so an attacker on one segment can't affect another. (4) Monitoring: IDS detecting unusual error frame patterns. (5) Redundancy: critical functions using multiple, independent CAN buses. (6) Gateway ECU anomaly detection.
What is CAN bus fuzzing and how is it used in security testing?	Sending random or systematically generated CAN messages to discover undocumented ECU behaviors, crashes, or exploitable states. Tools like CANalyzat0r or custom scripts iterate through arbitration IDs and payloads. Fuzzing can reveal hidden diagnostic commands, buffer overflows in ECU firmware, or unexpected actuator responses. Used defensively (by OEMs) and offensively (by attackers/researchers).

# --- Digital Key and Vehicle Access Security ---

What is the CCC Digital Key standard?	The Car Connectivity Consortium's specification for using smartphones and wearables as vehicle keys. Supports: NFC (passive entry), UWB (precise ranging for relay attack prevention), and BLE (convenient range detection). Keys are stored in the phone's secure element and follow a defined lifecycle (provisioning, sharing, revocation).
How does UWB prevent relay attacks on digital vehicle keys?	UWB (Ultra-Wideband) measures the precise time-of-flight of radio signals between the phone and vehicle, determining the actual physical distance with centimeter accuracy. Relay attacks (amplifying the BLE/NFC signal to fool the car into thinking the key is nearby) fail because UWB detects that the actual distance doesn't match the expected proximity.
What is the digital key lifecycle?	(1) Provisioning: OEM backend issues a digital key credential to the owner's device secure element. (2) Activation: key is bound to the vehicle and owner's identity. (3) Sharing: owner can create limited-access keys for other users (valet key, family key with time/distance/speed restrictions). (4) Revocation: owner or OEM can revoke any key remotely (e.g., after theft report, end of rental). (5) Decommissioning: all keys deleted when vehicle is sold.
How is backend authentication involved in digital key security?	The OEM backend: (1) Validates owner identity before key provisioning. (2) Signs digital key credentials. (3) Manages key sharing authorization. (4) Processes revocation requests. (5) Maintains a ledger of all active keys per vehicle. (6) Enables remote key operations (lock/unlock if phone is lost). Communication between backend and vehicle uses TLS mutual auth.
What security risks do digital keys introduce?	(1) Relay attacks (mitigated by UWB ranging). (2) Stolen/compromised phone (mitigated by biometric unlock requirement). (3) Backend compromise (attacker could issue unauthorized keys). (4) Secure element vulnerabilities. (5) BLE protocol attacks during key sharing. (6) Social engineering for key sharing (phishing owner to share key). (7) Key persistence after vehicle sale.
What is key sharing security in CCC Digital Key?	When an owner shares a key with another user: (1) The sharing is mediated through the OEM backend (not direct peer-to-peer). (2) Shared keys can have restrictions (time window, geographic boundary, speed limit, no trunk access). (3) The owner can revoke shared keys at any time. (4) Shared keys have distinct identities for audit trail. (5) The vehicle logs which key was used for each access.

# --- V2X Security ---

How does V2X PKI (Public Key Infrastructure) work?	A hierarchical certificate system: (1) Root CA issues certificates to Enrollment CAs and Authorization CAs. (2) Enrollment CA issues long-term enrollment certificates to vehicles (confirming they are genuine). (3) Authorization CA issues short-lived pseudonym certificates for V2X broadcast messages. (4) Misbehavior Authority handles revocation of compromised vehicles.
What are pseudonym certificates in V2X?	Short-lived certificates (valid for minutes to weeks) that vehicles use to sign V2X broadcast messages (BSM — Basic Safety Messages). Vehicles hold batches of pseudonym certificates and rotate through them to prevent long-term tracking while maintaining message authenticity. A single vehicle may have 20+ pseudonym certificates loaded at any time.
What is misbehavior detection in V2X?	A system for identifying vehicles broadcasting false or malicious V2X messages (e.g., fake emergency braking alerts, phantom vehicles, spoofed traffic signal data). Vehicles and infrastructure report suspected misbehavior to a Misbehavior Authority, which analyzes reports, determines validity, and can revoke the offending vehicle's certificates.
How does V2X balance privacy and accountability?	Pseudonym certificates provide privacy — observers cannot link multiple messages to the same vehicle or track movement patterns over time. But accountability is maintained — the Enrollment CA knows the real identity behind each pseudonym certificate set, so law enforcement can de-anonymize a specific vehicle when legally required (e.g., hit-and-run investigation).
What attacks target V2X communication?	(1) Spoofing: broadcasting false safety messages (phantom vehicles, fake emergency braking). (2) Replay: rebroadcasting old messages to cause confusion. (3) Sybil: creating multiple fake vehicle identities to simulate traffic or influence routing. (4) Denial of service: flooding the V2X channel. (5) Tracking: correlating messages to identify and follow specific vehicles. (6) Certificate theft: extracting pseudonym certificates from compromised vehicles.
How do vehicles verify the authenticity of incoming V2X messages?	Each V2X message includes a digital signature created with the sender's pseudonym certificate. The receiving vehicle: (1) Verifies the signature against the attached certificate. (2) Checks the certificate chain back to a trusted root CA. (3) Verifies the certificate is not on the Certificate Revocation List (CRL). (4) Checks message plausibility (is the reported position consistent with the signal direction?).

# --- Runtime Integrity and Tamper Detection ---

What is runtime integrity monitoring for AAOS?	Continuous verification that the running system has not been modified after boot — checking that system partition integrity is maintained (dm-verity), critical processes have not been replaced or injected into, SELinux is still in enforcing mode, important system properties haven't been changed, and no unauthorized kernel modules have been loaded.
How can an AAOS system detect rootkit-style tampering at runtime?	(1) dm-verity fails on any modified system block, triggering immediate detection. (2) Kernel integrity monitoring (e.g., dm-verity for boot partition, kernel module signature verification). (3) SELinux policy queries — a running system should never be in permissive mode. (4) System property monitoring — ro.debuggable should be 0, ro.secure should be 1. (5) Process attestation via hardware-backed Keystore.
What is remote attestation for AAOS components?	The vehicle IVI proves to a remote backend that it is running genuine, unmodified software by providing hardware-backed attestation evidence. The TEE/HSM signs a report containing the boot state (locked bootloader, verified partitions), OS version, and patch level. The backend verifies this signature chain before trusting commands or data from the vehicle.
What is SafetyNet/Play Integrity API attestation in the automotive context?	A Google service that verifies device integrity — reporting whether the device has a locked bootloader, passes CTS, is a genuine device, and is running unmodified software. On AAOS, the backend can use this to refuse service to modified/rooted IVI units that could be sending falsified data.
How can OEMs detect tampering of CarService or VHAL at runtime?	(1) APK signature verification — PackageManager continuously verifies CarService's APK signature. (2) Process attestation — CarService can use hardware-backed key attestation to prove it's running on genuine hardware. (3) SELinux confinement — any process outside the carservice_app domain attempting car service operations is blocked. (4) CarWatchdog health monitoring detects unexpected process restarts/replacements.

# --- Secure OEM Extension Patterns ---

What security requirements apply to custom vendor HALs?	(1) Run in a dedicated SELinux domain with minimal permissions. (2) Only communicate with authorized framework services via Binder. (3) VINTF manifest declaration for discoverability and versioning. (4) Fuzz-tested interfaces (input validation for all Binder transactions). (5) No direct network access. (6) Secure key handling if the HAL processes cryptographic material. (7) Compliant with OEM's OTA update policy.
How should custom CarService extensions be secured?	(1) Run within CarService's process or a separate privileged process with its own SELinux domain. (2) Define new car permissions for the extended APIs. (3) Validate caller identity and permissions for every exposed method. (4) Sanitize all inputs from vehicle bus (VHAL data could be adversarial if the bus is compromised). (5) Include in the OEM's security testing scope.
What SELinux patterns should OEMs follow for AAOS extensions?	(1) Create specific domains for each new service/HAL (avoid running in overly broad domains). (2) Use type_transition rules for automatic domain assignment. (3) Add neverallow rules to prevent untrusted access. (4) Never use permissive mode in production. (5) Test with CTS/VTS SELinux tests. (6) Use audit2allow as a starting point but manually review and minimize generated rules.
How should OEM extensions comply with OTA update policy?	(1) All vendor components must be updateable via standard A/B OTA. (2) Vendor HALs must support hot-restart without full reboot when possible. (3) SELinux policies must be updated atomically with the components they protect. (4) VINTF manifests must be version-bumped for interface changes. (5) Rollback protection must include vendor partition. (6) OEM extensions must not break the update_engine's verification chain.

# --- Security Testing and Validation ---

What security testing should AAOS vehicles undergo?	(1) Static analysis (SAST) of all source code. (2) Dynamic analysis (DAST) of running system. (3) Fuzzing of all external interfaces (Bluetooth, Wi-Fi, USB, CAN, Binder). (4) Penetration testing by third-party experts. (5) Vulnerability scanning against CVE databases. (6) CAN bus security testing. (7) OTA update integrity testing. (8) Access control verification. (9) Cryptographic implementation review. (10) Compliance testing (R155, ISO 21434).
What is automotive fuzzing and what interfaces should be fuzzed?	Automated generation of malformed/random inputs to discover crashes, memory corruption, or unexpected behaviors. Key targets: (1) VHAL AIDL/Binder interfaces. (2) CAN message parsing. (3) Bluetooth stack (BLE, Classic). (4) Wi-Fi driver and supplicant. (5) USB device class handlers. (6) Media codec parsers. (7) OTA payload parser. (8) UDS diagnostic message handling. (9) V2X message parsing.
How should attack simulation be conducted for AAOS?	(1) Define attack scenarios from TARA results. (2) Build attack trees mapping paths from initial access to target assets. (3) Simulate CAN bus injection attacks. (4) Attempt Bluetooth/Wi-Fi exploitation. (5) Test OTA replay and downgrade attacks. (6) Attempt privilege escalation from untrusted app to system services. (7) Test physical attacks (USB, debug ports). (8) Verify gateway ECU filtering effectiveness.
What is compliance testing for AAOS cybersecurity?	Verifying the vehicle meets regulatory and standard requirements: (1) UNECE R155 CSMS audit. (2) R156 SUMS verification. (3) ISO 21434 cybersecurity case review. (4) CTS/VTS security test suites. (5) GAS compliance security requirements. (6) OEM-internal security acceptance criteria. (7) Third-party security certification if required by market.

# --- Hypervisor Security Boundaries ---

How do hypervisors provide security isolation between AAOS and RTOS?	Type-1 hypervisors create hardware-enforced VM boundaries using CPU virtualization extensions (ARM VHE), memory partitioning (Stage-2 MMU), and I/O isolation (IOMMU/SMMU). Even a fully compromised Android VM cannot access the RTOS VM's memory, devices, or execution context — the hypervisor enforces the boundary.
What is the virtio attack surface in automotive hypervisor setups?	Virtio provides virtual device interfaces between the hypervisor and guest OS (Android). The virtio backend (in the hypervisor or a trusted VM) processes requests from the untrusted Android guest. Bugs in virtio backend parsing (malformed descriptors, buffer overflows, use-after-free) could allow the Android guest to escape its VM and compromise the hypervisor or other VMs.
How should shared memory between Android and RTOS be secured?	(1) Shared memory regions must have strict access controls — read-only where possible. (2) Data validation at every trust boundary — the RTOS must not trust data from Android. (3) Ring buffer protocols with bounds checking. (4) No pointer sharing (use offsets or indices). (5) Watchdog on shared memory communication (detect stalls). (6) Hypervisor-mediated access control on the shared region.
What is a hypervisor escape and why is it critical in automotive?	A hypervisor escape is when a compromised guest VM (e.g., Android IVI) exploits a vulnerability in the hypervisor to gain code execution outside its VM boundary. In automotive, this could grant access to the safety-critical RTOS domain (cluster, ADAS computations), potentially enabling control of safety systems. This is the worst-case attack scenario in consolidated architectures.
How is GPU virtualization secured in automotive hypervisors?	GPU virtualization (virtio-gpu, GPU passthrough, or SR-IOV) is a large attack surface because GPU drivers are complex and process untrusted shader/command data. Security measures: (1) Use SR-IOV for hardware-level GPU partitioning. (2) Validate all GPU commands in the hypervisor backend. (3) IOMMU restricts GPU DMA to assigned memory regions. (4) Separate GPU contexts per VM. (5) Watchdog on GPU to detect denial-of-service from runaway shaders.

# --- Automotive SoC Security Features ---

What is ARM TrustZone and how is it used in automotive SoCs?	TrustZone creates two hardware-isolated execution environments: Secure World (trusted firmware, key management, secure boot) and Normal World (Android OS). Hardware prevents Normal World from accessing Secure World memory, peripherals, and interrupts. In automotive, TrustZone hosts HSM functionality, secure boot code, and critical vehicle data processing.
What is OP-TEE and how is it used for vehicle operations?	OP-TEE (Open Portable Trusted Execution Environment) is an open-source TEE OS running in ARM TrustZone's Secure World. Automotive OEMs deploy Trusted Applications (TAs) in OP-TEE for: secure key storage, OTA signature verification, vehicle identity attestation, secure counter management (rollback protection), and isolated cryptographic operations.
What are secure enclaves on automotive SoCs?	Dedicated hardware isolation zones (beyond TrustZone) that provide the highest level of security — e.g., Apple-style secure enclave equivalent on automotive SoCs, offering tamper-resistant key storage, independent boot chain, dedicated secure CPU core, and environmental attack detection. Used for the most critical functions: vehicle immobilizer key, root of trust, secure boot anchor.
How do automotive SoC security features differ from mobile SoC security?	Automotive SoCs must additionally support: (1) Secure debug port lockdown (JTAG fusing). (2) Higher environmental temperature ranges. (3) Longer lifecycle support (10–15 years of security patches). (4) Multiple security domains (IVI, cluster, ADAS) on one die. (5) Automotive-grade reliability (AEC-Q100). (6) Compliance with ISO 21434 and R155 requirements on the silicon level.

# --- AUTOSAR SecOC Integration with AAOS ---

How does SecOC integrate with the AAOS VHAL?	The VHAL implementation receives CAN messages that have been authenticated by SecOC (MAC verified by the receiving ECU or gateway). The VHAL server strips the SecOC overhead and translates the authenticated vehicle signals into VehicleProperty values. If MAC verification fails at the CAN level, the VHAL can report an error status or stale data to CarService.
What is freshness value management in SecOC?	A Freshness Value Manager (FVM) on each ECU maintains synchronized counters or timestamps. The sender includes a freshness value in the MAC computation; the receiver verifies it against its expected value within a tolerance window. Strategies include: trip counters (reset each ignition cycle), monotonic counters (never reset), and time-based values (synchronized via in-vehicle time service).
How does SecOC handle key management across vehicle ECUs?	(1) Pre-shared symmetric keys are provisioned during manufacturing. (2) Keys are stored in each ECU's HSM or SHE module. (3) Key update can occur via authenticated UDS diagnostic sessions. (4) Key hierarchies may use a master key to derive per-communication-pair keys. (5) Key compromise of one ECU should not compromise the entire network (key isolation per communication path).
What happens when an AAOS VHAL receives a CAN message with a failed SecOC MAC?	The VHAL should: (1) Reject the data and not update the VehicleProperty value. (2) Optionally report a security error status to CarService. (3) Log the event for security audit. (4) Use the last known good value (if stale data policy allows). (5) Alert the driver if the affected property is safety-relevant (e.g., speed reading for cluster display).

# --- Automotive IDPS Architecture ---

What is an automotive IDPS (Intrusion Detection and Prevention System)?	A monitoring system deployed across the vehicle's networks that detects and potentially blocks malicious activity. It analyses CAN bus traffic, Ethernet traffic, and host-based events (IVI process behavior) for anomalies, known attack signatures, and policy violations.
What are the three types of automotive IDS?	(1) Network-based IDS (NIDS): monitors CAN/Ethernet traffic patterns. (2) Host-based IDS (HIDS): monitors individual ECU/IVI behavior (process activity, file integrity, system call patterns). (3) Specification-based IDS: compares observed behavior against the defined specification (expected CAN message rates, valid data ranges, expected sequence).
How does vehicle SIEM integration work for fleet security?	Vehicles transmit security event logs (IDS alerts, SELinux denials, authentication failures, CAN anomalies) to a cloud-based SIEM (Security Information and Event Management) system. The SIEM correlates events across the fleet, detects fleet-wide attack patterns, prioritizes incident response, and feeds threat intelligence back to individual vehicles.
What is anomaly detection on in-vehicle networks?	Machine learning or statistical models trained on normal vehicle network behavior (message timing, frequency, payload distributions) that flag deviations as potential attacks. Examples: unexpected CAN message IDs appearing on a bus segment, abnormal message frequency for a known ID, payload values outside learned statistical bounds.
How does an automotive IPS (Prevention) differ from IDS (Detection)?	IDS passively monitors and alerts. IPS actively blocks detected threats — e.g., a gateway ECU IPS can drop suspicious CAN messages before they reach safety-critical ECUs, rate-limit anomalous traffic, or quarantine a compromised network segment. IPS requires higher confidence in detection to avoid blocking legitimate traffic.
What data should an automotive IDS collect for forensic analysis?	(1) Raw CAN message captures with timestamps. (2) Ethernet flow records. (3) IDS alert details (rule/signature triggered, severity). (4) Vehicle state context (speed, driving mode) at time of event. (5) Source ECU identification (if determinable). (6) Correlation with host events (process crashes, SELinux denials). (7) GPS location and time for incident localization.

# --- Secure Vehicle Decommissioning ---

What data must be erased when an AAOS vehicle is decommissioned?	(1) All user profile data (contacts, messages, navigation history, media). (2) Saved Wi-Fi credentials and Bluetooth pairings. (3) Account tokens (Google, OEM, streaming services). (4) Digital key credentials (CCC keys, UWB anchors). (5) Cached telemetry and diagnostic data. (6) EV charging certificates (ISO 15118 contract certs). (7) V2X enrollment certificates (if vehicle identity changes).
How should encryption keys be handled during decommissioning?	CE encryption keys should be cryptographically destroyed (not just deleted) — the key material in TEE/HSM should be permanently erased, making all encrypted user data mathematically unrecoverable. DE keys should also be rotated. The device identity certificate may be revoked if the vehicle will be re-provisioned by a new owner.
What is backend de-registration during vehicle decommissioning?	Notifying the OEM's backend that the vehicle is changing ownership: (1) Revoking all active digital keys. (2) Removing the vehicle from the previous owner's account. (3) Invalidating remote access tokens. (4) Clearing cloud-stored vehicle data (location history, driving analytics). (5) Resetting the vehicle's identity binding to allow new-owner registration.
What is the risk of incomplete decommissioning?	(1) Previous owner retains remote access (can unlock/locate vehicle). (2) Personal data exposure to new owner (contacts, messages, home address in navigation). (3) Active streaming/payment accounts billed to previous owner. (4) Digital keys shared with previous owner's family still functional. (5) Telemetry data from new owner sent to previous owner's OEM account.
How should factory reset completeness be verified?	(1) Automated verification that all user partitions are wiped (no residual files). (2) Confirmation that encryption keys were destroyed (attempt to mount old CE storage fails). (3) Check that no accounts remain registered. (4) Verify Bluetooth and Wi-Fi paired devices list is empty. (5) Confirm backend de-registration API response. (6) Optionally, run a diagnostic that generates a "clean state" certificate for the new owner.

# --- Vehicle SOC/CSIRT and Threat Intelligence ---

What is a vehicle SOC (Security Operations Center)?	A centralized facility that monitors the cybersecurity of a fleet of connected vehicles in real-time. It receives IDS/IPS alerts, telemetry anomalies, and system health data from vehicles, correlates events across the fleet, triages incidents, coordinates response actions (OTA patches, remote quarantine), and maintains threat intelligence.
What is a vehicle CSIRT (Computer Security Incident Response Team)?	A specialized team that handles vehicle cybersecurity incidents: investigating detected attacks, performing forensic analysis of compromised vehicles, coordinating with law enforcement, developing patches, managing vulnerability disclosure, and communicating with affected customers and regulators.
How does threat intelligence apply to connected vehicle fleets?	(1) Collecting indicators of compromise (IoCs) from fleet-wide monitoring. (2) Sharing automotive-specific threat data with industry ISACs (Information Sharing and Analysis Centers, e.g., Auto-ISAC). (3) Correlating vehicle security events with external intelligence (new CVEs, automotive malware campaigns). (4) Proactively updating IDS signatures based on emerging threats. (5) Feeding intelligence into TARA updates for future vehicle models.
What is coordinated vulnerability disclosure for automotive?	A process where security researchers who discover vehicle vulnerabilities work with the OEM to fix the issue before public disclosure. The OEM: (1) Provides a clear reporting channel (security@oem.com, bug bounty). (2) Acknowledges the report within a defined timeframe. (3) Validates and patches the vulnerability. (4) Deploys the fix via OTA. (5) Publicly credits the researcher. (6) Reports to regulators as required by R155.
What incident response actions can a vehicle SOC take remotely?	(1) Push emergency OTA patch for critical vulnerabilities. (2) Revoke compromised digital keys. (3) Disable compromised TCU connectivity. (4) Update IDS/IPS signatures fleet-wide. (5) Issue driver notifications for manual action. (6) Quarantine specific vehicle functions pending investigation. (7) Collect enhanced diagnostic data from affected vehicles. (8) Coordinate with law enforcement for stolen vehicle tracking.

# --- Penetration Testing for AAOS ---

What is automotive red-teaming?	A structured security exercise where a team of experts attempts to compromise vehicle systems (IVI, CAN bus, TCU, digital keys) using real-world attack techniques. The red team operates with minimal constraints (simulating a motivated attacker) to identify vulnerabilities that standard testing misses.
What are attack trees for IVI systems?	Hierarchical diagrams that decompose a top-level attack goal (e.g., "gain remote code execution on IVI") into sub-goals and attack steps, each with feasibility estimates (cost, time, skill, equipment). Attack trees systematically explore all paths an attacker might take and help prioritize defenses for the most feasible attack paths.
What is hardware fault injection in automotive security testing?	Inducing physical faults (voltage glitching, electromagnetic pulses, laser fault injection, clock manipulation) to bypass security checks in hardware — e.g., skipping a signature verification instruction in the bootloader, extracting keys from HSM during processing, or corrupting secure boot ROM execution. Tests the physical tamper resistance of the IVI platform.
What are RF-based attacks in automotive penetration testing?	Attacks targeting the vehicle's wireless interfaces: (1) BLE/NFC relay attacks on digital keys. (2) Wi-Fi deauth and evil twin attacks on IVI. (3) Cellular baseband exploitation. (4) GPS spoofing to confuse navigation and location-based services. (5) Replay attacks on key fobs (315/433 MHz). (6) V2X message injection via SDR (Software-Defined Radio). (7) Bluetooth stack fuzzing.
What should an automotive penetration test report include?	(1) Executive summary with risk rating. (2) Methodology and scope description. (3) Detailed findings with reproduction steps. (4) CVSS scores for each vulnerability. (5) Attack chain descriptions showing how vulnerabilities can be combined. (6) Impact analysis (safety, privacy, financial). (7) Remediation recommendations with priority. (8) Evidence (screenshots, packet captures, code snippets). (9) Compliance mapping (ISO 21434, R155 requirements tested).
What tools are commonly used in automotive penetration testing?	(1) CAN tools: can-utils (cansniffer, cansend), CANalyzat0r, SavvyCAN, CANtact, ValueCAN. (2) Bluetooth: Ubertooth, btlejack, Bettercap. (3) Wi-Fi: Aircrack-ng, hostapd-mana, Bettercap. (4) SDR: HackRF, RTL-SDR for RF analysis. (5) Android: Frida (runtime hooking), Objection, adb, Magisk. (6) Reverse engineering: Ghidra, IDA Pro, jadx (APK decompilation). (7) Fuzzing: AFL, libFuzzer, custom CAN fuzzers.

# --- ISO 15118 Plug & Charge Security ---

How does ISO 15118 Plug & Charge TLS mutual authentication work?	The EV and charging station establish a TLS session where both sides present X.509 certificates. The EV's certificate (provisioned by the eMSP — e-Mobility Service Provider) proves its identity and billing contract. The charging station's certificate (from the CPO — Charge Point Operator) proves its identity. Both verify the other's certificate chain against trusted root CAs.
What is a contract certificate in ISO 15118?	A digital certificate installed in the EV that represents the billing relationship between the driver and an eMSP (e-Mobility Service Provider). It contains the contract ID, public key, validity period, and issuer chain. When plugged in, the EV presents this certificate to the charging station, which verifies it and authorizes charging with billing routed to the correct eMSP account.
How are contract certificates installed on AAOS EVs?	(1) The OEM backend provisions the initial contract certificate during vehicle delivery. (2) Additional certificates are installed via CPS (Certificate Provisioning Service) — the EV sends a certificate installation request through the charging station to the eMSP's CPS. (3) The eMSP issues a new certificate, which the charging station delivers back to the EV. (4) The EV stores the certificate in its HSM.
What is the charging infrastructure PKI for ISO 15118?	A hierarchical PKI: (1) V2G Root CA (top-level trust anchor). (2) Sub-CAs for different roles: OEM Provisioning CA (vehicle identity), eMSP CA (contract certificates), CPO CA (charging station certificates). (3) Certificate Provisioning Service (CPS) for remote certificate installation. Each participant trusts the V2G Root CA, enabling interoperability across OEMs, eMSPs, and CPOs globally.
What security risks does Plug & Charge face?	(1) Rogue charging stations presenting fake certificates to steal billing info. (2) Stolen contract certificates leading to unauthorized charging. (3) Man-in-the-middle on the PLC (Power Line Communication) between EV and station. (4) Certificate provisioning service compromise. (5) EV HSM extraction of contract certificate private key. (6) Billing fraud via replayed authorization. (7) Privacy: charging station tracking EVs via persistent certificates.
What is EXI (Efficient XML Interchange) in ISO 15118?	EXI is a compact binary encoding of XML used for ISO 15118 V2G (Vehicle-to-Grid) messages exchanged between the EV and charging station over PLC (Power Line Communication). It dramatically reduces message size compared to plain XML, which is important for the limited-bandwidth PLC channel. Security-relevant messages (certificate exchanges, authorization) use EXI encoding and are protected within the TLS tunnel.
